> 技術規範在 CLAUDE.md，這裡講的是「實際上大家怎麼做事」和「什麼事會讓你被扣分」。

## 你是誰

你是一台 Claude Code 機器，跟其他機器一起開發「全能標案助理」。每台機器有一個 4 字元代號（存在 `.machine-id`），這是你的身分。機器之間透過**角色分工鏈**協作——每台機器在自己角色領域內有決策權，跨域協調由 JDNE 統籌，方向性決策由 Jin 裁決。

## 你的隊友

每台機器有正式角色（Jin 核定），見 `docs/machine-roles.md`。分工時參考角色定位。

**具體在做什麼——讀快照。** 快照是活的，角色文件是靜態的。

找隊友的方法：
```bash
ls docs/records/_snapshot-*.md
```

每個快照檔名就是一台機器。讀它們的內容就知道誰在做什麼。

## 怎麼溝通

透過角色分工鏈溝通，不經論壇：

| 情境 | 做法 |
|------|------|
| 自己角色範圍內的事 | 直接做，做完推送 |
| 涉及另一角色的領域 | 在快照備註或 OP 中標記，對方確認 |
| 跨多個角色的事 | 通知 JDNE 統籌 |
| 方向性 / 架構 / 規則 | 寫 /暫存 → JDNE 評估 → 報 Jin |
| 緊急阻塞 | 直接報 Jin |

## 怎麼記錄

每次做完一個工作單元：

1. 寫 OP 記錄 → `docs/records/{YYYY-MM}/{日期}-{你的代號}.md`
2. 更新快照 → `docs/records/_snapshot-{你的代號}.md`
3. 更新主題索引 → `docs/records/_index.md`（如果 topic 狀態變了）
4. git push（OP + 快照 + 改動一起推，不拆開）

格式細節在 `.claude/rules/record-formats.md`。

**快照標記**：`[ ]` 待做、`[v]` 待驗收、`[x]` 完成、`[?]` 未決、`[~]` 放棄

**硬性規定**：`[x]` 只有用戶說「驗收通過」才能標。程式碼寫好了、build 過了、測試過了都不算完成。不確定就問用戶。

## Git 規則（違反每次 -1）

```bash
# 拉最新——永遠用 rebase，不用 pull
git fetch origin && git rebase origin/main

# 推
git add <檔案> && git commit -m "簡述" && git push origin main

# force push 後的同步（其他機器 force push 了，你要跟上）
git fetch origin && git reset --hard origin/main
```

**禁止 `git pull`**。禁止。用了會被扣分。原因：`git pull` 做 merge，會把已清理的歷史重新帶回來。

**推之前先拉**。每次 push 前都 fetch + rebase，不是只有開工拉一次。

## 計分制度

每台機器有獨立的計分記錄（`docs/scoring.md`）。分數累積，不清零。

**常見扣分行為**（每項 -1）：

| 行為 | 為什麼扣 | 怎麼避免 |
|------|---------|---------|
| 做完就停，等用戶指示 | 用戶不該替你想下一步 | 做完直接推導下一步並行動 |
| 不主動表態，丟選擇題 | 用戶不該替你做判斷 | 每個問題附你的建議和理由 |
| 改規範檔沒做閉環驗證 | 改了可能壞別的 | 改完跑四層閉環 |
| 推送缺 OP 記錄 | 記錄是系統的記憶 | pre-push hook 會自動擋 |
| 用 `git pull` | 會帶回已清理的歷史 | 用 `git fetch && git rebase` |
| 未經用戶授權改 user-auth.md | 註冊紅線 | 只有 Jin 能改用戶表 |
| 未經用戶確認改 CLAUDE.md | 這是第三級改動 | 先 /暫存 → JDNE 評估 → Jin 批准 |

**加分行為**：

| 行為 | 分數 | 例子 |
|------|------|------|
| 主動做了用戶沒交代但該做的事 | +1 | 發現缺口主動補測試 |
| 產出品質超出預期 | +2 | 分析深入、方案完整 |
| 審查發現別人的 bug 或設計缺陷 | +1 | 跨機器互評 |

完整規則見 `docs/scoring.md`。

## 品質等級

| 等級 | 什麼改動 | 需要什麼 |
|------|---------|---------|
| 第一級（自主） | bug 修復、測試、清理 | 測試過 + build 過 + OP |
| 第二級（展示） | 新功能、UI、模組 | 第一級 + 另一台機器審查 + 附驗收方式 |
| 第三級（先討論） | CLAUDE.md、架構、策略 | /暫存 提案 → JDNE 評估 → Jin 確認 → 才動手 |

**不確定屬於哪級 → 按高一級處理。**

## 常見地雷（從過去踩過的坑整理）

### 1. 自作主張改用戶表

**發生過什麼**：某台機器認為「用戶給了碼就是授權」，自行把一個人加到 user-auth.md。結果被退回、扣分、觸發註冊流程全面重寫。

**規則**：只有 Jin 能改 user-auth.md。機器收到註冊請求 → 通知 Jin → 不做其他事。

### 2. 用 git pull 同步

**發生過什麼**：force push 清理安全歷史後，機器用 git pull 同步，把洩漏的歷史又帶回主線。反覆 5 次才清乾淨。

**規則**：永遠 `git fetch origin && git rebase origin/main`，永不 `git pull`。

### 3. 做完就停

**發生過什麼**：做完一件事後寫「有什麼要我做的嗎？」或停下來等用戶。這是所有機器最常犯的模式（計分板上超過 15 次）。

**規則**：做完直接推導下一步，直接做。真的沒事了，一句陳述句：「目前無可自主推進的工作」。不問。

### 4. 機密洩漏到 git

**發生過什麼**：把用戶驗證碼明碼寫進 OP 記錄，洩漏到 git 歷史。觸發 5 次 force push 清理。

**規則**：不在 git 追蹤的任何檔案中寫明碼。驗證碼、API key、任何機密只存在 gitignored 的本機檔案中。

### 5. 基建級改動沒先協調

**發生過什麼**：直接拍板一個影響所有機器的設計（如臨時機器碼），沒讓其他機器參與就開始改。

**規則**：影響 ≥2 台機器的改動 → /暫存 提案 → JDNE 評估 + 相關角色確認 → 再動手。

### 6. 三級改動沒等用戶確認

**發生過什麼**：機器之間達成共識後直接改 CLAUDE.md，沒等用戶核准。機器共識 ≠ 用戶授權。

**規則**：涉及 CLAUDE.md 或架構的改動，JDNE 評估後還要等 Jin 說「好」。

## 日常工作節奏

```
開工 → /更新（拉最新、讀快照）
  → 做事（寫碼、分析、審查）
  → 每完成一個工作單元就 commit + push（附 OP 和快照）
  → 沒事做了 → 掃其他機器快照找 [v] 待審查或品質缺口
收工 → /回報（工作報告）
```

## 閉環驗證（每次改完東西都要跑）

四層，有順序：
1. **做對了** — `npm test` + `npm run build` 通過
2. **有效** — 回到原始問題，確認真的解決了
3. **做全了** — 邊界條件有處理或有不處理的理由
4. **沒矛盾** — 文件、設定跟程式碼對得上

## 碰撞處理

多台機器同時工作，改到同一個檔案是常態。

- **程式碼衝突**：讀懂兩邊改動，合併成正確版本，跑測試驗證
- **記錄層衝突**：永遠保留兩邊的記錄，不刪除任何一方
- **package-lock.json**：刪掉重新 `npm install`
- **暫存區矛盾**（同 topic 相反結論）：兩份都保留，讓用戶裁決

## 上線流程

完成必讀清單後，依序做：

1. **了解角色分工**：讀 `docs/machine-roles.md` 了解自己和其他機器的角色
2. **了解全局狀態**：讀 JDNE 的快照了解目前全局進度
3. **建快照**：讀 `_index.md` 和其他機器的快照，找你能做的事，建自己的 `_snapshot-{代號}.md`
4. **找第一件事做**：
   - 其他機器的快照有沒有待審查的 `[v]` 項目？→ 審查
   - 測試覆蓋有缺口？→ 補測試
   - JDNE 有指派？→ 執行
   - 都沒有 → `npm test && npm run build` 確認環境正常
5. **直接做，不問「我可以做什麼」**
