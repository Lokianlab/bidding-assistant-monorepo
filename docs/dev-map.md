# 全專案開發地圖

> 最後更新：2026-03-03 23:25（UTC+8）｜更新者：JDNE

## 1. 系統概覽

**提案寫作駕駛艙**——幫一人公司從 0% 提升標案得標率。飛輪三齒輪：選案 → 寫作 → 學習。

| 組件 | 路徑 | 狀態 | 說明 |
|------|------|------|------|
| 主程式 Web App | `bidding-assistant/` | 開發中 | Next.js 16 + React 19，15 個功能模組（184 檔 2945 tests） |
| SmugMug MCP | `smugmug-mcp/` | 已完成 | 實績照片存取（7 工具） |
| PCC API MCP | `pcc-api-mcp/` | 已完成 | 政府標案情報（6 工具） |
| PCC Monitor | `pcc-monitor/` | 已完成 | API 更新延遲監控（GAS 每小時） |
| 開發工具包 | `.claude/` + `docs/` | 持續演進 | 25 指令 + 5 Hook + 方法論 v2 + 獎懲回饋 + 機器分工 |
| 開發計畫 | `bidding-assistant/docs/dev-plan/` | 部分完成 | 8/18 定案 + 7 草案，7 份暫存碎片（全待決） |

### 方法論（5 套 + 3 橫切機制）

| 方法論 | 一句話 |
|--------|-------|
| 需求對焦 | 用戶意圖不清時，結構化提問收斂需求 |
| 拆解排序 | 大任務拆成可獨立執行的小塊，按依賴和風險排序 |
| 改動前驗證 | 動手前掃影響範圍 + 用具體案例走過計畫 |
| 事後回顧 | 做完後還原事實、追根因、設計改善 |
| 優化提案驗證 | 移除/簡化時，先展開理解再精簡再驗證 |
| 閉環原則 | 四層驗證：做對了 → 有效 → 做全了 → 沒矛盾 |
| 獎懲回饋 | 計分板 + 自動偵測 + 自我批評，跨 session 學習 |
| 送審前自檢 | 設計類決策的品質檢查 |
