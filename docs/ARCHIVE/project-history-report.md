## 一、為什麼做這個專案

### 背景

大員洛川顧問有限公司，15 人團隊，16 年政府標案經驗，核心業務涵蓋課程、導覽、傳統服飾、綠色照護等政府標案。年營收目標 3,000 萬。

**去年成績：投了 46 案，中了 0 案。得標率 0%。**

這些案子大多是「最有利標」——不比價格，比的是建議書品質。問題不是報價太高，而是寫出來的建議書不夠好。

### 觸發點

兩件事同時發生：

1. **0% 得標率的壓力**——46 案全輸不是運氣問題，是系統性問題
2. **AI 工具成熟度**——Claude Code Agent 能力已足以組建全自動開發團隊

Jin（公司創辦人 / 唯一技術決策者）決定：用 AI 開發團隊打造標案輔助工具，從根本解決建議書品質問題。

### 核心公式

```
營收 = 投標數 × 得標率 × 平均合約金額 × 利潤率
     = (找到的機會 × 篩選通過率) × 得標率 × 平均金額 × 利潤率
```

**最大槓桿點是得標率**——從 0% 提升到 20% 就是從零到有的質變。

## 二、問題出在哪裡

### 現行流程（壞掉的）

```
招標文件 → Perplexity 搜尋背景 → Gemini 發想 → Claude 寫初稿
→ Nanobanana 做視覺 → 手動拼貼 → 套用舊版本 → 排版 → 送出 → 落標 → 不檢討 → 重來
```

六個工具互不相通，每個 AI 都從零開始，沒有知識累積。

### 失敗根因分析（基於 184 案歷史資料）

團隊分析了 112-114 年（2023-2025）三年的標案數據，發現失敗原因逐年演化：

| 年度 | 失敗案件數 | 主要敗因 | 市場訊號 |
|------|-----------|---------|---------|
| 112（2023） | 59 | 資格文件不合規（程序問題） | 「文件備齊就能過」 |
| 113（2024） | 65 | 內容品質不足 | 「文件是底線，內容決勝負」 |
| 114（2025） | 60 | 競爭激烈（減價失利） | 「文件+內容都好，成本還是輸」 |

市場門檻逐年提高：程序合規 → 內容品質 → 成本競爭力，三層疊加。

### 五大痛點（按影響排序）

1. **知識庫是空的**——AI 只能寫泛泛的內容，因為它不知道公司做過什麼
2. **選案靠直覺**——46 案全投但品質分散，不如精選 10 案全力以赴
3. **產出仍像初稿**——沒有系統性品質檢查，缺陷在落標後才發現
4. **在 5 個 AI 之間跳來跳去**——沒有工具串接，資料要手動搬運
5. **不累積經驗**——每次投標都從零開始，成功與失敗的經驗都沒留下來

## 三、解決方案是什麼

### 產品定位

**提案寫作駕駛艙**（Proposal Writing Cockpit）——不是儀表板、不是看板、不是 RAG 聊天機器人。

像飛行員的駕駛艙一樣，把所有資訊集中在一個介面上：情報搜尋、戰略分析、內容組裝、品質檢查、文件輸出，一站完成。

### 六大核心能力

| # | 能力 | 對應痛點 | 對應模組 |
|---|------|---------|---------|
| 1 | 精準選案 | 選案靠直覺 | M01 情報模組 + M03 戰略分析 |
| 2 | 知識驅動寫作 | 知識庫是空的 | M02 知識庫模組 |
| 3 | 結構化引導 | 在 AI 之間跳來跳去 | 八層結構寫作引導 |
| 4 | 即時品質把關 | 產出像初稿 | M04 品質閘門 |
| 5 | 經驗累積 | 不累積經驗 | 得標/落標回饋迴路 |
| 6 | 一鍵輸出 | 手動拼貼 | M06 版面輸出 |

### 八層黃金範本

從 30+ 成功案例中萃取出建議書的標準結構：

1. 封面資訊（案號、公司、聯絡人）
2. 目錄（自動生成）
3. 計畫摘要（量化目標、預算摘要、精準時程）
4. 核心活動方案（場次、人數、日期、場地、講師、產出物）
5. 行銷策略（預熱期、執行期、收尾期三階段）
6. 時程 / 甘特圖（月里程碑、期中報告、請款節點）
7. 團隊組織（公司簡介、成員履歷、**履約實績**——最關鍵的一項）
8. 經費表（逐項成本、單價、說明）

### 四種失敗模式

| 模式 | 特徵 | 典型缺陷 |
|------|------|---------|
| A：好點子、準備差 | 概念有趣但沒數據支撐 | 活動空泛 + 無實績 + 無行銷 |
| B：散彈槍 | 塞 20+ 不相關活動 | 時程爆滿 + 預算零散 |
| C：口號多、證據少 | 新公司承諾多 | 無經驗 + 空話 + 行銷空泛 |
| D：結構不完整 | 基本資料缺漏 | 評審讀到一半放棄 |

## 四、怎麼做的——開發歷程

### 時間軸

```
2026-02-18  Day 1  ─── 專案啟動
2026-02-19  Day 2  ─── 基礎建設
2026-02-20  Day 3  ─── 開發規劃
2026-02-21  Day 4  ─── 治理+品質+重構
2026-02-22  Day 5  ─── 產品功能衝刺
```

### Phase 0：基礎建設（Day 1-2）

**做了什麼**：

- 建立 monorepo（`bidding-assistant/` + `pcc-api-mcp/` + `smugmug-mcp/` + `pcc-monitor/`）
- 發現 PCC API（政府電子採購網）的能力遠超預期：評審名單、決標金額、敗因都拿得到
- 建立記錄層（三層檢索架構：主題索引 → 回報 → OP 記錄）
- 建立論壇系統（跨機器非同步通訊）
- 建立方法論 v1（需求對齊、分解排序、變更前驗證、事後覆盤、優化方案驗證）
- 完成 PCC API MCP Server（6 個工具）+ 監控（GAS 每小時檢查）
- 完成 SmugMug MCP Server（7 個工具，作品集照片存取）

**關鍵決策**：
- Discord Bot 架構放棄，改走 SaaS 網頁
- 記錄層取代簡單操作日誌（從「什麼時候做了什麼」升級為「為什麼做、學到什麼」）

### Phase 1：治理與品質（Day 3-4）

**做了什麼**：

- 建立治理機制：共識規則（24h 沒反對即通過）、機器身份（正式/見習）、分工機制（宣告制+快照互讀）
- 建立三級品質制度：自主（bug 修復）→ 展示（新功能）→ 先議（架構變更）
- 建立 CLAUDE.md 修改規則：錯字直接改、調整走共識、刪除要全員同意 + Jin 批准
- 方法論升級到 v2：5 個方法 + 3 個橫切機制，覆蓋 26 個場景
- 大規模模組重構：品質、報價、組裝模組全部抽到 `src/lib/`，建立純函式 + 完整測試

**關鍵決策**：
- 論壇共識：「用戶沒反對 ≠ 用戶同意」，第三級事項必須明確批准
- 品管重構：「自我 QA」方法被降級（同一個 AI 共享盲點），改為可選的預提交自查
- 開始建立商業脈絡文件（23 個痛點、營收公式、產品階段定義）

### Phase 2：產品功能（Day 4-5）

**做了什麼**：

- **M03 戰略分析引擎**（A44T）：五維適配度評分（需求匹配、公司實力、競爭態勢、資源負荷、策略契合）+ 雷達圖 + 風險評估。82 個測試全過。
- **M04 品質閘門**（Z1FV）：四道檢查（文字品質、事實查核、需求對照、實務檢驗）+ 驗收報告 + UI。114 個測試全過。
- **PCC 情報整合**（ITEJ）：搜尋 + 詳情 + 機關情報 + 競爭分析 + 市場趨勢 + P 偵察 + 公司設定 + 全模組快取。從 MCP Server 整合進 Web App。1,132 個測試全過。
- **文件生成**（多台合作）：章節 → DOCX 下載 + markdown 表格支援 + 封面 + 目錄。
- **跨模組導航**（A44T）：情報 → 戰略 → 組裝 → 品質 → 匯出，一鍵串流，資料自動帶入。
- **儀表板快速開始**（A44T）：知識庫未填時顯示引導卡片。
- **提案分析**（AINL）：分析 30+ 成功案例、184 失敗案例，萃取八層黃金範本和四種失敗模式，對 6 個進行中案件做出成功率預測（30%-70%）。

**關鍵決策**：
- 最小展示版確定為三件：M03 + M04 + PCC 情報搜尋
- 「治理階段結束，轉向產品優先」宣告
- 論壇系統廢除（改為角色鏈直接溝通）

## 八、下一步

### 最大的瓶頸

**知識庫裡的真實資料。** 系統已經蓋好了，但知識庫是空的。AI 不知道公司做過什麼案子、團隊有誰、擅長什麼。沒有這些資料，戰略分析只能給空泛的評分，建議書只能寫空泛的內容。

這個資料只有 Jin 能提供。

### 建議優先順序

1. **Jin 在 dev server (localhost:3000) 驗收最小展示版三件**——確認方向對不對
2. **填入知識庫真實資料**——團隊成員（00A）、公司實績（00B），至少各 3 筆
3. **決定知識庫儲存方案**——Notion 或 Supabase，解鎖 M02 開發
4. **用真實案例跑完整流程**——從情報搜尋到文件輸出，找出斷點
