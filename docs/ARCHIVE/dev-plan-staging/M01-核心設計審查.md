- **來源**：模型委員會三人審查（2026-02-20）
- **委員組成**：用戶代表（非技術老闆視角）、前端架構師、產品經理

## 審查結論摘要

M01 v1.0 的核心設計方向需要重新思考。技術規格寫得詳盡，但從產品和使用者角度有根本性的問題。

### 三人共識

1. **localStorage 設計方向錯誤**：應改用 IndexedDB（容量大、非同步不卡 UI）
2. **10 分鐘等待不可接受**：長時間分析應改為背景執行 + 完成通知，不是前台等待
3. **錯誤處理是核心功能**：使用者不知道資料不完整比沒有資料更危險
4. **分析結果不該自動消失**：使用者認為這是「做好的報告」，不是快取
5. **模組太大，MVP 需按使用者需求重新定義**：不是按技術分期拆

### 五個關鍵盲區

#### 盲區 1：沒有人驗證過 PCC 資料能不能支撐這些分析（用戶代表）

規劃預設資料是乾淨、完整、可以直接分析的。但政府採購資料有很多「看起來是資料但不能直接用」的情況。如果分析結果基於有問題的資料，情報反而會導致錯誤決策。

**行動項**：在寫第一行程式碼之前，拿一個真實標案手動跑一次分析，確認結果有意義。

#### 盲區 2：從資料出發而非從工作流程出發（產品經理）

規劃問的是「PCC 有 1422 萬筆，能做什麼分析」，應該問的是「企劃團隊在評估一個案子時，需要回答什麼問題」。6 個 Tab 是以分析功能為中心，不是以使用者任務為中心。

企劃的實際思路：「我下週要投 XXX，我需要知道什麼？」

#### 盲區 3：統編可能不在搜尋結果裡（前端架構師）

統編欄位可能只存在於 tenderDetail（逐案查），不在搜尋結果（批量查）裡。如果是這樣，整個交叉比對的 IO 量會乘以 N 倍。

**行動項**：用真實 API 回應確認統編在哪個端點回傳。

#### 盲區 4：分析結果沒有下游（產品經理）

情報模組的分析結果要怎麼進入 L2-L8 的提案流程？如果只是給人看、不進入後續工作流程，就是儀表板不是駕駛艙。

#### 盲區 5：長時間運算的架構根本不對（前端架構師）

- Vercel 預設 timeout 10 秒，10 分鐘在伺服器端跑不完
- 瀏覽器切 tab 或螢幕熄滅，前端運算會中斷
- 應改為：使用者點「開始分析」→ 背景跑（Web Worker / requestIdleCallback）→ 跑完通知
- 部分成功要能存，中斷要能繼續

### 建議的 MVP 重新定義

**產品經理方案（2 頁面取代 6 Tab）**：

1. **案件情報**：輸入案名或案號 → 這個案子的歷史（以前招過嗎、誰得標、投幾家、我們投過嗎）
2. **對手側寫**：點進一個廠商 → 跟我們撞過幾次、在哪些機關勝率高

延後：市場趨勢圖表、自我分析全景、P 偵察、評委分析

**前端架構師方案（按 IO 複雜度拆）**：

- Phase 1 只做自我分析 + 機關偏好（IO 量可預測）
- Phase 2 加競爭對手分析（改為背景 Job）
- Phase 3 加評委分析（IO 最密集，需要驗證可行性）

**判斷標準**：MVP 給企劃團隊用一個月。如果他們因此改變了某個標案的投標決策，模組就有價值。

### 前端架構師的技術建議

1. 儲存層從 localStorage 換 IndexedDB（用 `idb` 庫，1KB）
2. Route Handler 拆成多個 route（`/api/pcc/search`、`/api/pcc/detail`、`/api/pcc/batch`），不要用 body action 分流
3. 規劃 Server Component / Client Component 邊界
4. 競爭對手分析改為背景 Job + 斷點續傳

### 開工前必做（全員同意）— 已驗證 (A44T, 2026-02-21)

用真實案例手動驗證：
1. ✅ 統編在哪個 API 回傳？→ **搜尋結果就有**（`brief.companies.ids` 直接包含統編），不需逐案查 tenderDetail。盲區 3 不成立。
2. ✅ tenderDetail 是否包含評委資料？→ **有**（`最有利標:評選委員` 欄位，含姓名、職業、出席、學經歷）。但只有最有利標/準用最有利標的案子才有。
3. ✅ searchByCompany 單頁回應大小？→ Page 1: 200KB（~70 筆），Page 2: 65KB（~69 筆）。合計 265KB，前端可接受。
4. ✅ 真實標案分析？→ 竹北光節案（780 萬，4 家投標），大員洛川排名第 4，未得標原因「非最有利標或最優勝廠商」。評選序位明確，資料對投標決策有幫助。

**結論**：資料品質足夠支撐 Phase 3 分析。Phase 3 競爭分析核心已建成（commit 86fea86），包含自我分析 + 對手排行 + 機關分析 + 年度趨勢。
