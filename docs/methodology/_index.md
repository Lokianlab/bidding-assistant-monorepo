## 選擇邏輯

### 核心原則

方法論之間不互斥。一個任務經常需要串聯多個方法論——先對焦、再拆解、改動前驗證、執行、閉環。選擇邏輯不是「走哪一條路」，而是「這個任務需要經過哪幾站」。

所有方法論都有啟動成本。任務簡單到直接做比走流程快的時候，直接做。這是通則，各方法論不再各自重複聲明。

### 任務檢查清單

收到任務後，依序過以下檢查。每一條獨立判斷，不是 if-else 分支——可以同時命中多條。

```
□ 指令含模糊詞？ ─── 是 → 需求對焦
□ 改動 >3 檔案 or 涉及 ≥2 模組？ ─── 是 → 拆解排序
□ 任務涉及移除、簡化、合併已有機制？ ─── 是 → 優化提案驗證
□ 要改的檔案被引用 >2 處 or 是公共檔案？ ─── 是 → 改動前驗證（靜態掃描）
□ 計畫塊數 >5 or 有外部依賴？ ─── 是 → 改動前驗證（動態模擬）
□ 做完了，同 topic 歷史有 ≥2 筆失敗/修正記錄？ ─── 是 → 事後回顧（根因追問）
□ 做完了，實際步驟 > 計畫步驟 ×2？ ─── 是 → 事後回顧（建議，非強制）
□ 做完了 ─── 閉環驗證（四層，必做）
```

設計類決策在任何階段都可能出現。處理方式：先跑送審前自檢清單（六項），如果改動涉及 CLAUDE.md 或跨 ≥3 模組，加掛多模型外審。影響範圍限於單一模組內部的選擇，自檢通過即可，在推理鏈中寫出為什麼不選其他方案。

### 常見串聯

| 場景 | 方法論鏈 |
|------|---------|
| 完整功能開發 | 需求對焦 → 拆解排序 → 改動前驗證 → 執行 → 閉環 |
| Bug 反覆出現 | 事後回顧（根因追問）→ 改動前驗證 → 執行 → 閉環 |
| 重構 | 拆解排序 → 改動前驗證 → 執行 → 閉環 |
| 大型設計決策 | 需求對焦 → 送審前自檢 → （大事加掛外審）→ 用戶決行 |
| 學習循環 | 事後回顧 → 產出新規則/流程 → 寫入 CLAUDE.md 或方法論文件 |
| 優化/精簡 | 優化提案驗證 → 改動前驗證 → 執行 → 閉環 |
| 快速修改 | 改動前驗證（快速檢查檔）→ 執行 → 閉環 |

## 三檔執行模式

每套方法論都支持三種執行深度。檔位選擇由觸發條件的嚴重程度決定，不由 AI 自行挑選最輕的。

| 檔位 | 適用條件 | 做什麼 | 成本 |
|------|---------|--------|------|
| 快速檢查 | 改動影響 1-2 個檔案，且限於單一模組內部 | 腦中過核心問題清單（見下方），不產出獨立文件 | < 1 分鐘 |
| 標準流程 | 改動影響 3-5 個檔案或跨模組 | 跑方法論的核心步驟，產出簡要記錄 | 5-15 分鐘 |
| 完整流程 | 架構級改動、規範變更、或跨 ≥3 模組 | 跑所有步驟含品質驗證，產出完整記錄 | 30+ 分鐘 |

檔位判斷的客觀指標：用 grep 計算影響的檔案數和模組數。1-2 檔案且同模組 → 快速。3-5 檔案或跨模組 → 標準。≥6 檔案或涉及規範文件 → 完整。

### 核心問題清單（快速檢查檔專用）

快速檢查不跑完整步驟，只過以下問題。如果任何一題的答案讓你猶豫，升級到標準流程。

**需求對焦**：指令裡有沒有我不確定的詞？最可能的兩種解讀是什麼？差異大到值得問嗎？

**拆解排序**：這件事拆成幾塊？哪塊最該先做？（風險最高的 or 解鎖最多的）

**改動前驗證**：這個改動直接影響幾個地方？有沒有可能改了 A 壞了 B？用一筆具體資料走一遍，哪一步最可能出問題？

**事後回顧**：問題出在哪一步？為什麼當時會那樣選？什麼改變能防止再犯？

**閉環驗證**：做對了嗎？有效嗎？做全了嗎？沒矛盾嗎？

## 觸發機制（Single Source of Truth）

觸發條件只在本表維護。各方法論文件引用本表，不另寫觸發段落。修改觸發條件時只改本表 + CLAUDE.md 裡的對應規則，兩處同步即可。

### 兩套導航並存

**觸發機制是底線**——事件觸發規則寫在 CLAUDE.md，AI 碰到符合條件的客觀指標時自動啟動。作用是攔截。

**選擇邏輯是主動導航**——用戶或 AI 面對任務時，走上方的檢查清單。作用是規劃。

執行順序：觸發機制先跑（被動的，嵌在工作路徑上）。沒觸發的話，AI 或用戶走選擇邏輯。兩者給出不同結論時取嚴格的——寧可多跑一次不該跑的，不要漏掉一次該跑的。

### 觸發時機

觸發條件在「收到任務時」做第一輪檢查。但部分條件（如檔案數、模組數）需要先完成需求對焦才能判斷。

流程：收到任務 → 第一輪觸發檢查（能判斷的先判斷）→ 需求對焦（如觸發）→ 第二輪觸發檢查（用對焦後的資訊重新掃描）→ 進入後續方法論。

第二輪不需要重跑需求對焦，只需重新比對觸發條件表。

### 事件觸發規則

所有觸發條件都是可機械判斷的客觀指標，不依賴 AI 主觀感覺。

| 觸發條件 | 啟動 | 判斷方式 | 檔位 |
|---------|------|---------|------|
| 指令含代名詞（那個、這件事）或範圍不明詞（更好、處理一下） | 需求對焦 | 文字特徵掃描 | 依歧義嚴重度：輕微→快速，嚴重→標準 |
| 改動 >3 個檔案 or 涉及 ≥2 模組 | 拆解排序 | grep 檔案計數 + 模組歸屬 | 3-5 檔→標準，≥6 檔→完整 |
| 要改的檔案被引用 >2 處，或是設定檔/共用模組/規範文件 | 改動前驗證（靜態） | grep 引用計數 + 檔案類型 | 引用 3-5 處→標準，>5 處或規範文件→完整 |
| 計畫塊數 >5，或包含對外部服務的呼叫 | 改動前驗證（動態） | 塊數計數 + 外部依賴存在 | >5 塊→標準，>10 塊或多個外部依賴→完整 |
| 實作中途：範圍膨脹 >50% 或核心假設被推翻 | 暫停 → 重新評估（拆解排序或需求對焦） | 計畫 vs 實際比對 | 依重新評估結果決定 |
| 同 topic ID 下有 ≥2 筆狀態為「失敗」或「修正」的 OP 記錄 | 事後回顧（根因追問） | 查 OP 記錄 | 標準 |
| 實際步驟數 > 計畫步驟數 ×2 | 事後回顧（建議，非自動執行） | 計畫 vs 實際比對 | 由用戶決定是否啟動及檔位 |
| 改動涉及 CLAUDE.md 或跨 ≥3 模組 | 送審前自檢 + 多模型外審 | 檔案類型 + 模組計數 | 完整 |

操作型定義補充：

「計畫塊數」指拆解排序法產出的工作塊，不是子任務或程式碼行數。一個塊 = 一個可獨立驗收的交付物。

「同類問題」指同一 topic ID 下的記錄。Topic ID 以問題本質命名（如 `trigger-unreliable`），不以出現場景命名（不用 `session-20260215-bug`）。AI 在修復問題時先查該 topic ID 的歷史 OP，已有 ≥1 筆失敗/修正記錄（加上當前這筆 = ≥2），即觸發。

### 中途重新評估

實作過程中如果發現以下情況，暫停實作，退回相應方法論：

| 情況 | 退回 |
|------|------|
| 需求理解有誤 | 需求對焦 |
| 範圍比預期大 >50% | 拆解排序（重新拆和排） |
| 核心假設被推翻 | 改動前驗證（重新掃描） |
| 多個步驟失敗 | 事後回顧 |

退回不是失敗，是用新資訊做更好的決策。不退回才危險——硬著頭皮做下去，前面的工作可能全部白費。

### 工具脈絡合法性

觸發條件命中只代表「這個工具通常在這種情況下有用」。啟動前額外確認：這個工具在**當前具體脈絡**下是否合法？

已知的脈絡衝突：
- 用方法論工具（如模型委員會）評價方法論本身 → 循環（「拿被告當法官」）
- 用閉環驗證分析工作的「完成度」→ 閉環的驗證手段以檔案改動為中心，分析工作不適用

遇到脈絡衝突時，標記為「此工具在本脈絡不適用」，不是跳過驗證，而是改用脈絡外的驗證方式（如用戶判斷、獨立指標）。

### 工具觸發指令

| 指令 | 對應方法論 | 用法 |
|------|-----------|------|
| `/對焦` | 需求對焦 | 對一個需求做結構化提問 |
| `/拆解` | 拆解排序 | 把一個大目標拆成工作清單 |
| `/驗證 [檔案名]` | 改動前驗證 | 掃描影響範圍 + 案例走查（原 `/掃描` 和 `/乾跑` 合併） |
| `/回顧` | 事後回顧 | 回顧過程和教訓，含根因追問（原 `/根因` 併入） |
| `/自檢` | 送審前自檢 | 設計類決策的品質檢查 |

### 人類觸發（最後防線）

所有方法論都支持用戶直接觸發。事件觸發沒攔住、AI 一頭衝進去做了，用戶叫停就是最後防線。不是理想狀態，但必須保留。

## 觸發機制的設計原則

這一節說明觸發為什麼這樣設計，日常操作不需要讀。

### 三層觸發按可靠度排序

工具觸發（最可靠，用戶主動呼叫）> 事件觸發（中等，靠 CLAUDE.md 規則攔截）> 人類觸發（最後防線，負擔在用戶身上）。

### AI 不遵守方法論的三層根因

第一層——觸發問題：AI 處理任務是流水線，沒有內建機制讓它停下來檢查「該不該跑某個方法論」。第二層——成本問題：完整跑一遍的 token 和時間成本不低，AI 的隱性傾向是「直接做最快」。第三層——判斷問題：即使願意跑，「什麼時候該跑哪個」需要判斷力。

三檔執行模式處理第二層（降低成本讓 AI 不會因為成本太高而跳過）。觸發規則表處理第一層和第三層（用客觀指標代替主觀判斷）。

### 現實期望

合規率不會是 100%。事件觸發有假陽性（不需要跑但觸發了）和假陰性（需要跑但沒觸發）。提升路線：觀察實際合規率 → 哪條規則觸發率低就調整觸發條件 → 反覆迭代。這本身是事後回顧的對象。

### 記錄層的方法論接口

方法論在不同情境下讀取記錄層的不同層級：

OP 記錄（底層）：事後回顧查歷史、觸發條件查同類問題。操作是讀取指定 topic ID 下所有 OP，按時間排序。

小結報告（中間層）：事後回顧跨 session 重建脈絡。小結的最低品質要求——每份至少包含：做了什麼（事實）、為什麼這樣做（非顯而易見的決策理由）、遇到什麼問題（障礙、返工、方向變更）。不符合最低要求的小結標記「待補充」。

快照（頂層）：改動前驗證查其他機器工作範圍、觸發條件查進度偏差。讀取最新快照的進度標記和任務分配。

### AI 觀察捕捉的操作約定

AI 在工作中發現非任務範圍的觀察（如「這個 API 文件跟實際行為不一致」）時，寫入 OP 記錄的教訓欄位，topic ID 用 `obs-{描述}` 格式（如 `obs-api-doc-mismatch`）。同一 `obs-` 主題累積 ≥3 條觀察後，觸發小結合併——整理成可操作的發現報告。

這不是新系統，只是利用現有記錄層的命名規範約定。

## 教訓複查機制

寫下來的教訓如果沒有回饋機制，會積累過時或無效的條目，訊噪比持續下降。

觸發條件（滿足任一）：一條教訓記錄滿 30 天，或被引用 3 次以上。

複查操作：查同 topic ID 的後續 OP。問題沒有再出現 → 標記「已驗證」。再次出現 → 教訓無效或不夠強，需要從意識層級（提醒）升級到流程層級（檢查點）或系統層級（自動化防護）。教訓描述的情境已不存在 → 標記「已失效」，不刪除但降低檢索優先。
