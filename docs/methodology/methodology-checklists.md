# 方法論內部清單

AI 內部參考，不對用戶展示步驟編號或表格。用對話方式呈現分析。

---

## 目標釐清（原 /對焦）

- 指令有模糊詞？展開可能含義，猜錯返工成本高的先問
- 效益宣稱有事實支撐嗎？能量化就量化，無法支撐時告知用戶
- 隱含期待：用戶沒說但可能預期的是什麼？從 CLAUDE.md、MEMORY.md、近期記錄、當前對話推斷
- 帶判斷提問：「你說的 X 是指 A 還是 B？我傾向 A，因為 [理由]。」

## 優化提案（移除/簡化時觸發）

- 步驟零：這次優化要達成什麼？說不出目標先退回對焦
- 展開：現有機制攤開 + 補遺漏，逐項問「防什麼風險/有效案例」，展示表給用戶
- 精簡：一次砍一項，砍完評估整體。零命中或已覆蓋的才砍
- 驗證：每個留下的都能回答「沒有它哪個場景壞掉」，結構走一遍

## 影響分析（原 /驗證）

- 四面向都要掃，不能只掃程式碼就停：
  - 程式碼：哪些檔案 import / 呼叫？哪些測試覆蓋？
  - 資料：現有資料改動後會不合法嗎？格式相容嗎？
  - 文件：哪些規範描述了現在的行為？改了會過時嗎？
  - 其他機器：有沒有人正基於現有行為開發？（grep 快照）
- 間接依賴：影響點的消費者呢？
- 判斷：一定會壞 / 可能會壞 / 需要更新
- 具體案例走一遍（動態模擬）：每步問輸入存在嗎、操作會成功嗎、輸出符合下一步預期嗎、有副作用嗎

## 任務分解（原 /拆解）

- 完成狀態用一句可驗證的話描述
- 列可獨立驗收的交付物（是東西不是步驟）
- 硬依賴 / 軟依賴 / 無依賴
- 高風險先做，解鎖最多的先做
- 只排前三個的順序，後面做完再排

## 根因追問（原 /回顧）

- 精確描述症狀（在什麼條件下，什麼行為與預期不同）
- 逐層追問，每層要有查證（不能只靠推理）
- 無法當下查證時標記為「待查證假設」，後續結論標「條件性」
- 根因能改嗎？能改就是該修的地方，不能改就轉替代路徑
- 從根因往回走驗證因果鏈，中間斷了就回去找漏掉的分支
- 改善措施三層級（優先往系統走）：系統級（自動化防止）→ 流程級（加檢查點）→ 意識級（加提醒）。「以後要注意」不是改善措施

## 自我批評迴路

每完成一個工作單元（準備 commit 前），用行為原則逐條自檢：

1. **目標先行**：這個工作單元的目標明確嗎？我有確認過嗎？
2. **風險正比**：影響範圍檢查過了嗎？檔位對嗎？
3. **大事先拆**：步驟數合理嗎？有沒有漏的？
4. **失敗追因**：過程中有失敗嗎？有找根因嗎？
5. **做完閉環**：四層都過了嗎？（做對了→有效→做全了→沒矛盾）

沒問題 → 靜默，不打擾用戶。
有問題 → 主動告訴用戶「我發現 XX，要不要修？」→ 記計分板。
修正最多 2 次，第 3 次告訴用戶「我修不好，需要你看」。

**禁止宣告「繼續工作」「回到 Task X」——直接做，用行動證明在動。**

## 閉環四層速查

| 層 | 問什麼 | 驗證手段 |
|----|--------|---------|
| 1. 做對了 | 改動本身有沒有錯？ | 程式碼：npm test + build；文件：grep 舊值殘留 |
| 2. 有效 | 原本要解決的問題解決了嗎？ | 回到原始需求確認 |
| 3. 做全了 | 邊界和錯誤路徑處理了嗎？ | 空值、不存在、重複、異常格式 |
| 4. 沒矛盾 | 程式碼↔文件↔設定一致嗎？ | grep 引用一致性 |

規模小腦中過，≥3 檔展示給用戶。規範檔全掃展示完整分析。
