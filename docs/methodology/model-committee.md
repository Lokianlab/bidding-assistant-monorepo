# 模型委員會（Model Committee）

> 送審前自檢的可選加掛步驟，用於重大設計類決策（改動涉及 CLAUDE.md 或跨 ≥3 模組）。不是獨立方法論——觸發和入口在 methodology-01 的送審前自檢段落。

## 定義

自檢通過後，把推理鏈交由不同模型的 agent 代替用戶做初步評估。

核心精神：**委員會是用戶的代理人——替用戶挑問題、替用戶磨提案，站在用戶的立場評判。**

委員會不是外部法官。委員找問題的動機是「我（代替用戶）要用這個東西，所以我要確保它不會出事」。

> 最終裁決權在用戶。委員會的評判是初評，不是定案。

---

## 適用情境

| 情境 | 為什麼適合 | 範例 |
|------|-----------|------|
| 設計類問題 | 有多種可行方案，需要取捨 | 記錄層架構、暫存索引機制 |
| 發現缺口 | 實際使用暴露出設計盲區 | 「查詢效率不如預期」→ 設計修正 |
| 方案影響範圍大 | 改動牽涉多個檔案或規範 | 修改 CLAUDE.md 的規則、跨模組重構 |
| 方法論級變更 | 改動影響所有機器的工作方式 | 修改方法論文件、改變記錄層規範 |

## 不適用情境

| 情境 | 該怎麼做 |
|------|---------|
| 執行類任務（方向已明確） | 直接做 |
| 用戶已給具體指令 | 照做，有疑問問用戶 |
| 簡單 bug 修復 | 定位 → 修 → 驗證 |
| 有唯一正確答案的問題 | 直接做 |

---

## 先備條件：需求脈絡

委員開始審查前，必須收到用戶的需求脈絡。沒有脈絡的委員會變成學術審查，偏離用戶真正在意的事。

需求脈絡至少包含：
- **原始問題**：用戶要解決什麼
- **背景**：為什麼現在要做這件事
- **限制**：時間、成本、風險承受度
- **「夠好」的標準**：用戶在意的是什麼（實用性？嚴謹度？速度？）

提案者負責在提交審查時附上這些脈絡。

---

## 流程

### 1. 提案者產出推理鏈

包含：原始問題（一句話）、推理步驟（編號列表）、結論。

### 2. 提案者附上需求脈絡

告訴委員：用戶是誰、要解決什麼、限制是什麼、「夠好」長什麼樣。

### 3. 啟動委員審查

使用 Task tool 啟動不同模型的 agent。委員收到的 prompt：

```
你是用戶的代理人，替用戶評估以下提案。

你的任務：
1. 站在用戶的立場，找出這個提案用了之後可能出事的地方
2. 對每個問題，說明出事的後果和嚴重程度
3. 如果你覺得提案需要修改，具體說明怎麼改

你不是學術審查員。不要要求統計證據或對照實驗，除非用戶的需求脈絡明確要求。
你的評判標準是：「如果我是用戶，我會不會放心用這個？」

---

**需求脈絡**：
{用戶背景、限制、「夠好」的標準}

**原始問題**：{問題}

**推理鏈**：
{推理步驟}

---

輸出格式：
1. 列出發現的問題（每個標明嚴重程度：致命/重要/建議）
2. 總體評價：一句話說明你的判斷
3. 建議：送給用戶看（可能需要先改某些地方）/ 需要大改
```

### 4. 打磨（如需要）

如果委員指出需要修改的地方：
1. 提案者針對每個問題修改推理鏈
2. 重新提交同一委員審查
3. 反覆直到委員認為可以呈給用戶

這個階段就是「替用戶磨提案」。

### 5. 呈核

委員認為可以了，提案者把以下內容呈給用戶：
1. 推理鏈（如經修改，附修改歷程）
2. 委員的評價摘要（找到什麼問題、怎麼處理的）
3. 委員認為還有風險但可接受的地方（讓用戶知道）

### 6. 用戶決行

用戶看完後：
- **同意** → 進入實作
- **退回再磨** → 回到步驟 4，可能加入更多委員（不同模型）
- **整個重來** → 回到步驟 1

用戶加入更多委員時，每次加入的必須是不同模型（不是同模型再看一遍）。

## 模型特性（來自 12 輪實測）

| 模型 | 審查風格 | 適合場景 |
|------|---------|---------|
| Haiku | 壓力測試（找最多問題，覆蓋面廣，含中低信度） | 快速初篩、成本敏感 |
| Sonnet | 精準外科（找最少問題，每個命中要害） | 邏輯嚴密度要求高 |
| Opus | 結構分析（中等數量，附正面評價，看全局） | 架構級決策、找結構性悖論 |

核心原則：**加人就加不同模型**。同模型多看幾遍沒有新發現。

---

## 品質標準

好的委員審查：
- 站在用戶立場，問的是「用了會不會出事」
- 每個問題標明嚴重程度和後果
- 有具體修改建議，不只說「有問題」
- 會指出提案的優點，不只找缺點

壞的委員審查：
- 用學術標準要求統計證據（除非用戶需求明確要求）
- 對每一步都質疑（過度審查）
- 只說「駁回」不說怎麼改
- 問題之間大量重疊

---

## 已知限制

1. **Agent 只看推理鏈**：沒有完整 context window，可能漏掉背景。需求脈絡盡量補充完整。
2. **同模型邊際遞減**：同模型第 3 次審查增量接近 0（Haiku 10 輪實測）。
3. **成本隨委員數增長**：每加一位委員 = 一次 API 呼叫。Haiku 最便宜。
4. **委員不能取代用戶**：委員找的是推理瑕疵，用戶帶入的是 AI 沒有的經驗和判斷。

## 自我迭代

每次執行完成後，觀察以下信號判斷委員會機制是否有效、是否需要調整：

| 觀察 | 健康信號 | 警示信號 | 調整方向 |
|------|---------|---------|---------|
| 委員覆蓋率 | 委員找到的問題涵蓋用戶在意的面向 | 用戶指出委員完全沒碰到的盲區 | 需求脈絡是否不夠完整？委員 prompt 需調整？ |
| 打磨效率 | 1-2 輪打磨即可呈核 | ≥3 輪打磨仍在來回 | 提案者初始品質太低還是委員標準太嚴？ |
| 模型互補性 | 不同模型找到不重疊的問題 | 多模型找到高度重複的問題 | 模型組合或 prompt 差異化需調整 |

同類警示信號累積 ≥3 次，觸發事後回顧追查根因。

## 實測數據

### 同模型 vs 跨模型（2026-02-19）

| 指標 | Haiku 看 10 遍 | Haiku+Sonnet+Opus 各 1 遍 |
|------|---------------|--------------------------|
| 總問題數 | ~6（第 5 遍後不再增加） | 每個模型各 2-6 個，含獨特發現 |
| 獨特發現 | 「遵守率≠品質」 | Sonnet:「兩種遵守不同難度」、Opus:「遞迴悖論」 |
| 邊際價值 | 第 5 遍後幾乎為零 | 每個新模型都有新視角 |

### 與舊方法對比

同一條推理鏈：
- 自問自答法：13 輪迭代仍未自行發現問題
- 模型委員會一審（1 agent）：一次找到 4 個問題
