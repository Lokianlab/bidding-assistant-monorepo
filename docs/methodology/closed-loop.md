# 閉環原則

所有工作的收尾標準。不是獨立方法論，是每件事做完都必須過的驗證關卡。

四層驗證，有順序，缺一不可。第一層沒過就不用談後面的。

---

## 四層架構

### 第一層：做對了（正確性）

修改本身無錯誤、無遺漏。

| 修改類型 | 驗證手段 | 通過標準 |
|----------|----------|----------|
| 程式碼 | `npm test` + `npm run build` | 測試全過、build 零錯誤 |
| 文件 / 設定檔 | grep 舊值 | 殘留為零、所有副本一致 |
| 環境變數 / 欄位改名 | grep 全 codebase 所有引用處 | 無斷鏈 |
| 刪除功能 / 移除依賴 | grep 確認無殘留引用 | 零殘留 |

### 第二層：有效（目的性）

做對了不等於有效。回到原始問題：這個修改要解決什麼？那個問題現在真的解決了嗎？

驗證方式因場景而異：改了環境變數 → 程式能實際連上服務嗎？修了 bug → 異常行為消失了嗎？加了功能 → dev server 上實際操作一遍，行為符合預期嗎？改了文件 → 照著文件走一遍，步驟正確嗎？

無法當下驗證時（需要外部服務、需要用戶操作），必須標註「待驗證」及具體驗證方式，不可默認有效。

### 第三層：做全了（完備性）

主流程通了，不代表所有情況都通了。

四個面向逐一確認：

| 面向 | 檢查什麼 | 方法 |
|------|-----------|------|
| 輸入邊界 | 空值、極端值、異常格式、特殊字元 | 對每個輸入欄位列出非正常值，逐一確認有處理或有不處理的理由 |
| 狀態邊界 | 資料不存在、已刪除、併發修改、重複操作 | 模擬這些狀態下的行為 |
| 錯誤路徑 | 外部服務不可用、網路中斷、權限不足 | 確認每個可能錯誤都有 error handling |
| 使用者路徑 | 非預期順序操作、中途取消、重複提交 | 想像一個「不看說明書的用戶」會怎麼用 |

不需要窮舉所有邊界。優先檢查兩類：最可能發生的（頻率高）和後果最嚴重的（發生一次就出大問題）。

通過標準：每個邊界都有明確的決定——「有處理」或「不處理，理由是 XX」。最差的是某個邊界根本沒想到。

### 第四層：沒矛盾（一致性）

你改了一個地方，系統其他地方可能還在描述舊行為。這種矛盾最危險——程式碼正常運作（前三層都過），但文件或設定給的資訊是錯的，下一個接手的人或機器會被誤導。

三個對齊：

| 對齊項目 | 確認什麼 | 方法 |
|---------|-----------|------|
| 程式碼 ↔ 文件 | 實際行為跟文件描述一致嗎？ | 找到所有描述這個功能的文件（README、CLAUDE.md、API 文件、註解），逐一比對 |
| 設定檔 ↔ 實際行為 | 設定檔裡的值跟系統實際使用的值一致嗎？ | grep 設定項的 key，確認所有引用處用對了 |
| 模組 ↔ 模組 | 不同模組對同一件事的假設一致嗎？ | 改了模組 A 的行為，找出所有依賴 A 的模組，確認假設還成立 |

如果改動前做了改動前驗證（靜態掃描），第四層只需要拿掃描結果當檢查清單，不需要重掃。如果沒跑靜態掃描（改動太小不需要），第四層至少做 grep 確認改動涉及的文件和程式碼描述一致。

閉環驗證發現問題時，視嚴重程度決定是當場修還是記下來。如果同類問題反覆出現（同 topic ≥2 筆失敗/修正 OP），應觸發事後回顧追查根因（見 methodology-04.md）。

## 快速入口捷徑

不知道從哪一層開始時，依改動類型找入口：

| 改動類型 | 從哪層進 | 理由 |
|----------|---------|------|
| 改程式碼（邏輯/功能） | 第 1 層（正確性）→ 往下 | 先確認本身沒錯，再問有沒有效 |
| 改文件/設定/規範 | 第 4 層（一致性）→ 往上 | 文件改動最容易的錯是和其他地方矛盾 |
| 修 bug | 第 2 層（有效性）→ 往外 | 先確認 bug 消失，再確認沒引入新問題 |
| 刪功能/移除依賴 | 第 3 層（完備性）→ 往外 | 先確認邊界都清乾淨，再確認本身沒錯 |

## 三檔執行

| 檔位 | 適用場景 | 做法 |
|------|---------|------|
| 快速 | 改動影響 1-2 個檔案，邏輯簡單 | 腦中過四層問題：對了嗎？有效嗎？全了嗎？矛盾嗎？不產出文件 |
| 標準 | 改動跨檔案或有外部依賴 | 四層逐一驗證，簡要記錄結果 |
| 完整 | 架構級改動或規範變更 | 四層完整驗證，產出驗證清單，標記每個檢查項的結果 |

## 自我迭代

每次執行完成後，觀察以下信號判斷閉環驗證是否有效、是否需要調整：

| 觀察 | 健康信號 | 警示信號 | 調整方向 |
|------|---------|---------|---------|
| 攔截有效性 | 四層通過後用戶未再發現問題 | 通過四層的交付物仍被用戶退回 | 哪一層沒攔住？通過標準太寬鬆還是驗證手段不匹配？ |
| 問題分佈 | 問題多在前兩層就攔住 | 問題集中在第三、四層才發現 | 前兩層的驗證是否太表面？ |
| 檔位準確率 | 選快速檔事後沒出問題 | 選快速檔但事後出問題，或完整檔浪費時間 | 檔位判斷的客觀指標需調整 |

同類警示信號累積 ≥3 次，觸發事後回顧追查根因。

## 專案教訓

**暫存檔寫「不裝」但實際已配好**：程式碼正常、功能正常、邊界條件也有處理——前三層全過。但文件跟實際狀態矛盾，另一台機器根據文件做了錯誤決策。這就是第四層的價值。

**/kb 只驗了十分之一的情境就宣布完成**：主流程通了（第一、二層過），但空值、不存在的 ID、特殊字元全沒測。這就是第三層的價值。
