# Workflow Engine — 12 階段自動化連結架構

> 文件等級：L3 架構決策｜狀態：規劃初版｜出處：3O5L 設計
> 優先序：P1（Phase 2b 核心）｜目標完成日：2026-04-03
> 前置條件：P1 全層驗收 + M02-M11 全規格交付完成

---

## 一、核心定位

**問題**：Jin 每年處理 46 個案件，當前全靠手動串接 12 個工具和環節，操作繁瑣、容易出錯

**解決方案**：建立 Claude Code SDK 驅動的自動化工作流引擎
- 機器自動觸發各階段任務
- 各模組間自動傳遞數據
- Jin 僅在關鍵節點做決策

**成功指標**：
- ✅ 12 階段全自動連結（無手動環節）
- ✅ 各階段數據流通順暢（無信息孤島）
- ✅ Agent 決策透明（所有建議帶理由）
- ✅ 平均案件處理時間 -60%（從手動 8h → 自動 3h）

## 三、各階段詳細設計

### 第 1-3 階段：案件入庫 + 情報評估

| 階段 | 模組 | 機器操作 | 決策點 | 數據流 |
|------|------|---------|--------|--------|
| **01** | W01 | 每日自動掃 PCC 新案 | — | RFQ → 案件庫 |
| **02** | M01 | 自動搜尋相似案歷史 | — | 案件 → 情報庫 |
| **03** | M07 | 推薦相關外包夥伴 | [決策點 1] | 需求 → 資源評估 |
| **決策點 1** | — | Jin 審閱推薦資源 | 接受 / 調整 | 資源清單 → KB |

**觸發條件**：
- 01：時間觸發（每天 06:00）
- 02：01 完成後自動觸發
- 03：02 完成 + KB 查詢完成後自動觸發
- 決策點 1：03 完成，Webhook 通知 Jin

**Agent 設計**：
- `agent-scan-case`：掃描 RFQ，提取關鍵字，檢查重複
- `agent-search-intelligence`：檢索知識庫，找相似案例
- `agent-recommend-resources`：根據需求推薦外包資源

### 第 4-5 階段：決策分析 + 建議書起草

| 階段 | 模組 | 機器操作 | 決策點 | 數據流 |

| **04** | M03 | 自動分析案件贏率 + 建議 | [決策點 2] | 案件 + 歷史 → 分析報告 |
| **05** | M06 P1-P3 | 自動勾勒方向 + 初稿 + 檢查 | [決策點 3] | 決策 → 初稿 → 審閱 |

**觸發條件**：
- 04：決策點 1 Jin 確認後觸發
- 05：04 完成或 Jin 跳過分析直接進行後觸發
- M06 P2：05 P1 完成 1 分鐘後自動觸發
- M06 P3 + M04：05 P2 完成後自動觸發

**Agent 設計**：
- `agent-strategy-analysis`：計算贏率、提出策略建議
- `agent-proposal-draft`：根據決策自動起草建議書初稿
- `agent-quality-check`：檢查品質、建議改進

**M04 品質閘門決策邏輯**：
```
如果品質評分 >= 8 分
  → 直接通過，進入決策點 3
如果品質評分 6-7 分
  → 提示 Jin「品質可接受，但建議改進」
如果品質評分 < 6 分
  → 拒絕通過，標記「需要重做」
```

## 四、Agent 通用設計原則

### Agent 職責清單

每個 Agent 應該遵循以下職責模式：

| Agent 類型 | 責任 | 輸入 | 輸出 | 失敗處理 |
|-----------|------|------|------|--------|
| 數據蒐集類 | 從外部系統抓數據，檢查完整性 | API/檔案路徑 | 結構化數據 | 重試 3 次，失敗標記 |
| 分析類 | 基於數據模型做決策，建議 | 結構化數據 | 分析報告 + 建議 | 降低精度，返回基線建議 |
| 生成類 | 基於決策生成文件/內容 | 決策 + 範本 | 文件 | 使用預設範本 |
| 檢查類 | 驗證質量/合規性 | 文件/數據 | 評分 + 反饋 | 保守評分（偏低）|
| 通知類 | 傳達決策點給 Jin | 階段完成信號 | Webhook / 儀表板更新 | 重試通知 3 次 |

### 故障恢復策略

```
Agent 執行失敗
  ├─ 重試 1 次（相同參數）
  ├─ 重試 2 次（降級參數，如移除可選過濾）
  ├─ 退而求其次（使用預設值或歷史結果）
  └─ 通知 Jin（標記為「需要人工介入」）

絕不允許：
  ❌ 沉默失敗（失敗但不通知）
  ❌ 部分數據傳遞（缺少必要欄位）
  ❌ 流程卡住（無法進行下一步）
```

## 五、決策點設計

### 6 個決策點詳細說明

#### 決策點 1：資源評估
**時機**：02 搜尋完成，03 推薦完成
**Jin 選擇**：
- ✅ 接受推薦 → 資源清單存入 KB
- 🔄 調整資源 → 手動編輯，存入 KB
- ❌ 不需要資源 → 跳過

**觸發下一步**：自動進入 04（戰略分析）

#### 決策點 2：投標決策
**時機**：M03 分析完成
**Jin 選擇**：
- ✅ 投標 → 進入 05（建議書起草）
- ❌ 不投標 → 案件標記「已決策」，結束

**觸發下一步**：進入 05 或 結案

#### 決策點 3：建議書審閱
**時機**：M06 P3 + M04 品質檢查完成
**Jin 選擇**：
- ✅ 通過 → 進入 06（報價驗算）
- 🔄 修改後重新檢查 → 回到 M06 P2
- ❌ 返工 → 重新分析（回到 04）

**觸發下一步**：進入 06 或 退回

#### 決策點 4：報價批準
**時機**：M05 估價完成
**Jin 選擇**：
- ✅ 批準 → 進入 07（資格文件）
- 🔄 調整報價 → 重新估算
- ❌ 返工 → 回到 05（重新擬定提案）

**觸發下一步**：進入 07 或 退回

#### 決策點 5：簡報確認
**時機**：M08 簡報生成完成
**Jin 選擇**：
- ✅ 同意方向 → 進入 10（議價分析）
- 🔄 建議調整 → 重新生成
- ❌ 推遲準備 → 暫存簡報，等待手動觸發

**觸發下一步**：進入 10 或 暫存

#### 決策點 6：談判策略決策
**時機**：M09 議價分析完成、評選結果出爐
**Jin 選擇**：
- ✅ 按建議談判 → 進入 11（履約追蹤）
- 🔄 自訂策略 → 手動編輯，進入 11
- ❌ 接受原報價（不談判）→ 進入 11

**觸發下一步**：進入 11

## 六、數據流設計

### 知識庫作為中樞

```
┌─────────────────────────────────┐
│   知識庫（KB）— 中樞數據倉庫      │
│  ├─ 案件歷史（勝率 / 失利原因）   │
│  ├─ 成本基線（單價 / 工人日）     │
│  ├─ 資源目錄（外包夥伴 + 評價）   │
│  ├─ 模板庫（提案 / 簡報 / 合約）  │
│  ├─ 品質規則（閘門標準）          │
│  └─ 市場數據（競爭對手 / 議價空間）│
└─────────────────────────────────┘
        ↑ 輸入來源
        │
   ┌─────────────────────────┐
   │ 各階段 Agent            │
   │ ├─ 提取（讀 KB）         │
   │ ├─ 計算（基於 KB 數據）  │
   │ ├─ 生成（套 KB 模板）    │
   │ └─ 回流（新經驗存 KB）  │
   └─────────────────────────┘
        ↓ 持續優化
```

### 階段間的數據傳遞

| 源階段 | 目標階段 | 數據內容 | 格式 |
|--------|---------|---------|------|
| 01 | 02 | 案件基本信息 | RFQ JSON |
| 02 | 03 | 需求分析 | 需求文本 + 關鍵字 |
| 03 | 04 | 資源清單 | 夥伴清單 |
| 04 | 05 | 戰略決策 | 投標決策 + 建議 |
| 05 | 06 | 提案內容 | Markdown / Word |
| 06 | 07 | 報價單 | Excel / JSON |
| 07 | 08 | 完整文件 | PDF + Zip |
| 09 | 10 | 簡報內容 | PPTX + 講稿 |
| 11 | 12 | 結果數據 | 勝敗 + 金額 + 反饋 |

## 八、風險與應對

| 風險 | 影響 | 應對方案 |
|------|------|--------|
| Agent 決策不可信 | 用戶拒絕採納 | 所有建議必須帶理由，提供反向決策路徑 |
| 決策點太多 | Jin 疲勞 | 決策點限制在 6 個，支援「快速批准」模式 |
| 數據流中斷 | 流程卡住 | 每個 Agent 故障時提供預設值或手動輸入 |
| KB 數據陳舊 | 決策質量下降 | 結案時強制回流，建立反饋迴圈 |
| 模組不相容 | 數據格式衝突 | 統一所有模組的 input/output 格式（JSON Schema） |
