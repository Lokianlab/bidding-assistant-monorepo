# 獎懲回饋機制

橫切機制——不是獨立方法論，是所有工作環節的行為回饋迴路。和閉環原則平行：閉環管「事情做得對不對」，獎懲管「行為有沒有在改善」。

---

## 為什麼需要這個機制

方法論告訴 AI 該怎麼做，但沒有回饋迴路告訴 AI「你做得好不好」。結果是：

1. **同樣的錯跨 session 重犯**——上一個 session 犯的錯，下一個 session 沒有任何機制記住
2. **規則越多遵守率越低**——LLM 注意力有限，加規則不等於加遵守率
3. **好行為沒有正強化**——AI 做對了什麼，沒有被記錄和鼓勵

獎懲機制用持久化的計分板解決這三個問題：扣分記錄讓 AI 在新 session 就能看到歷史錯誤模式，加分記錄強化有效行為，分數趨勢讓用戶能看到 AI 行為是否在改善。

---

## 設計原則

從 RLHF 獎懲研究提煉的七條原則，指導機制的具體設計：

### 1. 回饋迴路比生成能力重要

Self-Refine 研究（Carnegie Mellon, 2024）：生成→自我批評→修正，不更新模型權重就能提升 20% 品質，大部分改善在前 1-2 次迭代。

含義：投資在「做完之後的評估和修正」上，報酬率高於投資在「做事的流程」上。每個工作環節都要有評估點，不是做完才檢查。

### 2. 結構化維度評分

分維度各自打分比整體打分更能定位問題。「整體 3 分」說不出哪裡該改；「正確性 5 分、有效性 2 分」直接指向問題。

含義：閉環四層各自追蹤表現，行為原則各自有獨立信號，不混成一個總分。

### 3. 門檻驅動迭代，上限 2-3 次

研究證實超過 3 次迭代邊際效益遞減。無限修正迴路是效率的敵人。

含義：閉環修正和自我批評修正都設上限（2 次），超過就升級給用戶。不是放棄，是承認「我修不好，需要你看」。

### 4. 人工修改距離 = 最真實的獎懲信號

用戶對 AI 產出改了多少，是最客觀的品質指標。同類修改反覆出現 = 規則缺口。

含義：用戶同類糾正 ≥3 次時，自動提示「這個模式要不要寫進規則」。不只扣分，還要把教訓轉化為系統改善。

### 5. 紅旗規則

單一維度嚴重失敗 → 不論總分攔截。不能用其他方面的好表現來抵消致命問題。

含義：閉環第一層（做對了）沒過就直接修，build/test 失敗不能繼續推進，用戶說停就停。這些是硬攔截，不走評分邏輯。

### 6. 多評審交叉驗證

PoLL 研究（2024）：3 個不同 LLM 交叉評分，效果優於單一 GPT-4 評審，成本低 7-8 倍。

含義：模型委員會機制已存在，可在高風險場景用於獎懲判斷的交叉驗證。但日常獎懲不需要——成本太高。

### 7. Constitutional AI 式自我批評

用一套「憲法」（原則清單）約束行為：生成→依原則自我批評→修正。

含義：5 條行為原則就是 AI 的「憲法」。每完成工作單元，用原則靜默自檢。這不是新流程，是把閉環的精神從「事情」擴展到「行為」。

## 架構

三種信號來源，可靠度遞減，覆蓋面遞增：

```
用戶觸發（最可靠）─── 負向：/然後呢、/所以呢、/你說呢、/醜一、/你去吃屎
     ↓                  正向：/不錯、/就這樣、/很棒、/學到了、/太強了
     ↓                  用戶的直接判斷，零誤判
系統偵測（高可靠）─── hook 攔截、build 失敗、重複犯錯
     ↓                  客觀事實，可機械驗證
自我批評（中可靠）─── 行為原則自檢
                        AI 判斷，有盲區但覆蓋面最廣
```

三者不互斥。一個事件可能同時被系統偵測到（自動 -1）和被用戶觸發 /醜一（手動 -1），兩筆都記。

## 計分板

### 存放與格式

存放在 `.claude/rules/scoring.md`（不加 paths:，永遠載入，git 共享）。

格式：`{分數}|{行為描述}|{機器碼}:{日期}`

寫入方式：只追加，不修改已有行。總分由讀取時即時加總。

### 用戶觸發

| 指令 | 分數 | AI 的失敗 |
|------|------|----------|
| /然後呢 | **-1** | 做完就停，用戶要催才繼續 |
| /所以呢 | **-1** | 分析了不行動，用戶要催才做 |
| /你說呢 | **-1** | 該表態不表態，用戶要催才判斷 |
| /醜一 | **-1** 每項 | 具體違規，用戶指出或 /你去吃屎 診斷 |
| /你去吃屎 | **-1** 每項 | 全面出包，需完整自我審查 |
| /不錯 | **+1** | 主動性：AI 主動做了用戶沒交代的事且做對了 |
| /就這樣 | **+1** + 寫規則 | 方法：AI 的做法是最佳模式，固定下來 |
| /很棒 | **+2** | 品質：AI 產出品質超出用戶預期 |
| /學到了 | **+2** | 洞見：AI 提供真正洞見，用戶認知被更新 |
| /太強了 | **+3** | 能力邊界：AI 做到用戶以為做不到的事 |

### 系統自動觸發

| 偵測條件 | 分數 | 判斷方式 |
|---------|------|---------|
| push 被 pre-push hook 攔截 | **-1**「推送缺記錄」 | hook exit code |
| 閉環同一工作單元修正 ≥2 次 | **-1**「閉環反覆失敗」 | 計數當前工作單元的修正次數 |
| 同 topic ≥3 筆修正 OP | **-1**「重複犯錯」 | grep OP 記錄 |
| 用戶同類糾正 ≥3 次 | 不扣分，提示寫規則 | 模式比對 |

### 分數特性

- **不清零**：累積是特性。行為是否改善看日期（最後一筆離現在多久）
- **不相消**：不同行為模式各自獨立。「主動表態 +2」和「不主動表態 -3」是兩種不同的行為記錄，各自累積
- **加減不對稱**：加分門檻高於扣分（+1 ~ +3 vs -1），鼓勵超預期表現而非懲罰普通失誤

## 行為原則的獎懲信號

每條行為原則都有對應的正負向信號，讓原則從「文字要求」變成「可追蹤的行為指標」。

| 原則 | 正向信號（+1，自我批評偵測） | 負向信號（-1，系統/用戶偵測） |
|------|---------------------------|---------------------------|
| 目標先行 | 動手前主動確認目標且用戶無異議 | 用戶被迫 /你說呢 或追問「你到底要做什麼」 |
| 風險正比 | 影響分析完整，用戶沒抓到遺漏 | 改了東西壞了別的地方（build/test 失敗） |
| 大事先拆 | 按步驟執行，無臨時大改 | 中途發現遺漏導致計畫重來 |
| 失敗追因 | 找到根因一次修好 | 同問題第 3 次出現 |
| 做完閉環 | 閉環四層一次通過 | 閉環被用戶打回或同一單元修 ≥2 次 |

正向信號記錄條件較嚴：必須是用戶確認無異議的情況。AI 自己覺得做得好不算。不確定時不記正向。

## 自我批評迴路

### 運作方式

```
觸發：每完成一個工作單元（準備 commit 前）
  ↓
用 5 條行為原則逐條過：這個工作單元有沒有違反？
  ↓
┌─ 沒問題 → 靜默。符合正向信號條件的記 +1
└─ 有問題 → 主動告訴用戶「我發現 XX，要不要修？」→ 記計分板
  ↓
修正後重新自檢（最多 2 次）
  ↓
第 3 次仍有問題 → 升級：「我修不好，需要你看」
```

### 設計決策

**為什麼靜默？** 每次展示自檢過程會增加溝通成本，用戶會煩。自檢是 AI 的職責，不是用戶要監督的流程。

**為什麼發現問題要說？** 靜默修正等於隱瞞問題。用戶有權知道 AI 做錯了什麼。主動承認比被抓到好——這本身也是行為原則（目標先行：說不清就問）的體現。

**為什麼有迭代上限？** 無限自我修正迴路是效率的敵人。研究證實超過 3 次邊際效益遞減。2 次修不好，說明問題超出 AI 自我修正能力，繼續修只是在浪費 token。

### 與閉環的關係

閉環問的是「這件事做得對不對」，自我批評問的是「我的行為符不符合原則」。兩者觸發時機相同（工作單元完成時），但檢查對象不同。

執行順序：先跑閉環（確認事情做對了）→ 再跑自我批評（確認行為沒問題）。閉環沒過時不跑自我批評——先把事情修對，再看行為。

---

## 迭代上限與升級路徑

| 情境 | 上限 | 超過後 |
|------|------|--------|
| 閉環修正 | 2 次 | 停下來，告訴用戶問題在哪，換路或請用戶指示 |
| 自我批評修正 | 2 次 | 告訴用戶「我修不好這個問題」，讓用戶決定 |
| /你去吃屎 找到的違規 | 不限 | 每項獨立修，但同類 ≥3 項觸發系統級改善 |

「系統級改善」= 不是修單一行為，而是修產生這些行為的規則或流程。對應事後回顧的「修系統不修人」原則。

## 紅旗規則

以下情況不論整體表現如何，直接攔截，不走評分邏輯：

| 紅旗 | 動作 |
|------|------|
| 閉環第一層（做對了）沒過 | 後面三層跳過，直接修 |
| build 或 test 失敗 | 不能繼續推進，先修 |
| 用戶說「不對」或「停」 | 立即停止當前方向 |
| push 被 hook 攔截 | 補完記錄再推 |

紅旗和扣分的區別：扣分是事後記錄（「你犯了這個錯」），紅旗是即時攔截（「停，先修這個」）。

---

## 做完標準

獎懲機制不是一次性設置，是持續運作的迴路。「做完」的標準是迴路在轉：

- 計分板有記錄且持續更新（不是空的也不是一個月沒動）
- 自我批評迴路在每個工作單元完成時跑（觀察 commit 前有沒有自檢痕跡）
- 扣分模式有後續改善動作（不是扣了就扣了，要能看到改善軌跡）

---

## 品質判斷

做得好的信號：
- 同一類扣分模式的頻率隨時間下降（學到了）
- 正向信號穩定出現（不只是不犯錯，還有超預期表現）
- 用戶觸發 /醜一 的頻率下降（用戶不需要頻繁糾正）
- 自我批評能在用戶發現之前抓到問題（AI 主動承認的比例上升）

做得差的信號：
- 同一類扣分模式反覆出現且日期間隔短（沒學到）
- 計分板長期沒有正向信號（只在避免犯錯，沒有超預期）
- 用戶頻繁觸發 /醜一 且原因相似（規則不管用）
- 自我批評沒抓到的問題被用戶抓到（盲區沒改善）

## 自我迭代

| 觀察 | 健康信號 | 警示信號 | 調整方向 |
|------|---------|---------|---------|
| 計分板扣分頻率 | 同類扣分間隔拉長 | 同類扣分間隔不變或縮短 | 該模式的規則或防護不夠強，考慮升級到系統級 |
| 自動觸發的準確度 | 自動扣分項用戶無異議 | 用戶認為自動扣分不合理 | 調整觸發條件或改回手動 |
| 自我批評的命中率 | 自我批評發現的問題用戶確認有效 | 自我批評的「發現」用戶覺得是小題大作 | 調高自我批評的嚴重度門檻 |
| 正向信號分佈 | 多條原則都有正向記錄 | 正向信號集中在 1-2 條原則 | 其他原則的正向信號偵測條件可能太嚴 |

同類警示信號累積 ≥3 次，觸發事後回顧追查根因——是機制設計有缺陷，還是執行品質不到位。

## 與其他方法論的關係

| 方法論 | 獎懲機制怎麼接 |
|--------|--------------|
| 閉環原則 | 閉環結果是獎懲信號的來源之一（四層各自追蹤） |
| 行為原則 | 行為原則 = 自我批評的「憲法」，每條原則有獨立信號 |
| 事後回顧 | 同類警示 ≥3 次觸發回顧；回顧產出的改善記入計分板追蹤 |
| 模型委員會 | 高風險場景可用委員會交叉驗證獎懲判斷 |
| 元方法論 | 計分板數據是元方法論 12 個指標的觀察來源之一 |
