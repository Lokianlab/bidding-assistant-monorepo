# 決策框架 — 五層優先序與做歪信號

> 文件等級：L2 方法論｜狀態：v0.1｜出處：產品羅盤 v0.6 + Manual-Adjustment-Checklist.md

---

## 核心原則：第零原則

**成果正確性 > 過程沉沒成本**

所有決策、技術選型、優先序判斷的終極標準是：產出是否準確服務需求。

---

## 五層優先序決策框架

當面臨決策時，按以下層級判斷。上層永遠推翻下層：

| 層 | 標準 | 判斷點 | 違反示例 |
|---|---|---|---|
| **1** | **成果正確性** | 產出是否解決實際問題？ | 系統輸出需要 Jin 大量重寫 |
| **2** | **Jin 的時間** | 是否真的減少 Jin 人工投入？ | 新功能沒有減少 Jin 工作量 |
| **3** | **品質天花板** | 系統能達到的最佳品質上限 | 為了快速交付而砍掉核心驗證 |
| **4** | **開發速度** | 能多快交付可用功能 | 手動串接流程，無法自動化 |
| **5** | **技術優雅度** | 程式碼的美觀與架構完美性 | 為了代碼優雅引入複雜性 |

**鐵律**：第 5 層永遠不能推翻前 4 層。優雅度服從速度，速度服從品質，品質服從 Jin 時間，時間服從成果正確性。

---

## 決策流程（實際應用）

### 範例 1：新功能是否值得做？

```
Q1: 這個功能解決 Jin 的真實痛點嗎？（第 1 層）
  ├─ 否 → 不做
  └─ 是 → Q2

Q2: 這個功能能減少 Jin 的人工投入嗎？（第 2 層）
  ├─ 否 → 降級優先序或不做
  └─ 是 → Q3

Q3: 品質上限是什麼？（第 3 層）
  ├─ 高（需多次審查）→ 預留時間
  └─ 低（簡單功能）→ Q4

Q4: 多快能交付？（第 4 層）
  ├─ <3 天 → 快速模式
  └─ ≥3 天 → 拆分或分期

Q5: 代碼如何設計最優雅？（第 5 層）
  └─ 在滿足前 4 層後再優化
```

### 範例 2：技術選型

| 選項 | 成果正確性 | Jin 時間 | 品質 | 速度 | 優雅度 | 結論 |
|------|----------|--------|------|------|--------|------|
| A: 快速且髒 | ✅ | ⚠️ 高投入 | ❌ 低 | ✅ 快 | ✅ 優 | ❌ 不選（品質不達標）|
| B: 穩健且標準 | ✅ | ✅ 低 | ✅ 高 | ⚠️ 中等 | ⚠️ 中等 | ✅ **選** |
| C: 過度設計 | ✅ | ✅ 低 | ✅ 高 | ❌ 慢 | ✅ 優 | ❌ 不選（速度損失不值得）|

結論：**選 B**。B 滿足前 4 層（成果✅ + 時間✅ + 品質✅ + 速度⚠️），而 C 雖然優雅但速度損失超過收益。

## 七項做歪信號（檢查清單）

任何開發或決策出現下列跡象，表示偏離第零原則（成果正確性）。**應立即暫停並溝通澄清**：

### 1️⃣ 系統輸出需要 Jin 大量重寫

**信號**：完成的文件、分析、建議需要 Jin 手動重寫超過 20% 的內容

**違反層級**：第 1 層（成果正確性）

**根因**：模型定位或邊界設定有誤，違反自動駕駛定位

**修法**：停下來問 Jin「輸出遺漏了什麼？」重新定義需求

---

### 2️⃣ 新功能沒有減少 Jin 的工作量

**信號**：功能上線後，Jin 說「我的工作反而更多了」

**違反層級**：第 2 層（Jin 的時間）

**根因**：功能設計只考慮了「有沒有」，沒考慮「減少多少」

**修法**：重新評估功能範圍。如果無法減少 Jin 時間，降級優先序或不做

---

### 3️⃣ 結案資料沒有流入知識庫

**信號**：結案後 30 天內，沒有對應的知識庫條目或標案飛輪輸入

**違反層級**：第 1 層（成果正確性），損傷飛輪

**根因**：知識庫 + 結案流程沒有自動化串接

**修法**：檢查 W01/M06 → KB 的資料管道是否暢通；如遺漏則補上

---

### 4️⃣ 程式碼有硬編碼路徑或作業系統相關假設

**信號**：代碼中出現 `C:\\`、`./` 相對路徑、或 Windows 特定命令（cmd.exe 等）

**違反層級**：部署約束（無法上雲）

**根因**：開發時沒有考慮跨平台部署需求

**修法**：用 `path.join()` 或環境變數替換；無差別地在 macOS/Linux 上測試

---

### 5️⃣ 技術優雅度犧牲了前 4 層

**信號**：為了代碼美觀或架構完美而：
- 引入複雜性導致速度下降（層 4）
- 砍掉核心驗證導致品質下降（層 3）
- 增加 Jin 維護負擔（層 2）

**違反層級**：優先序層級

**根因**：本末倒置，過度工程化

**修法**：簡化設計；「好夠就行」優於「完美但慢」

---

### 6️⃣ 把使用說明或例子當成硬需求

**信號**：「文件中的例子」被當作「功能的強制要求」

**違反層級**：第 1 層（成果正確性），文件範圍混淆

**根因**：使用文件未清楚標示「例子」vs「規格」

**修法**：在使用文件中明確區分：
- 💡 例子（只為說明，不是要求）
- 📋 規格（強制要求）
- ⚙️ 可選參數

---

### 7️⃣ 仍需 Jin 人工複製貼上來連接流程階段

**信號**：S1 完成後，Jin 需手動複製輸出到 S2 的輸入；沒有自動化管道

**違反層級**：第 4 層（開發速度），違反管道自動化目標

**根因**：整合不完整，留下手動接線

**修法**：檢查模組是否遵循 Web App → SaaS → SDK 三層，每層是否有自動化接線點

## 做歪信號的檢查時機

| 時機 | 檢查者 | 方式 |
|------|--------|------|
| feature 分派前 | JDNE | 逐項 grep + 快照檢查 |
| 規格評審時 | 分派者 + 承接者 | 對照 SaaS Phase spec 檢查 |
| 上線驗收前 | Z1FV + Jin | 過 7 項清單 + 部署約束檢查 |

---

## 部署約束（四項硬性規定）

所有代碼上線前須符合以下四項。不符合 = 無法合併：

### 1. 無硬編碼路徑

**規則**：禁止 Windows 路徑（`C:\\`）、相對路徑（`./file`）、home 符號（`~`）

**改用**：環境變數、`path.join()`、API 抽象

**檢查**：`grep -r "C:\\|^\\.\/|^~\/" src/ app/`

---

### 2. 跨平台相容性

**規則**：代碼必須在 macOS/Linux 上執行無誤

**檢查清單**：
- [ ] 路徑用 `path.join()` 或 `/`
- [ ] 禁止 Windows 特定命令（`cmd.exe`、`wmic` 等）
- [ ] 檔案操作用 Node.js API，不用 shell
- [ ] 環境變數大小寫區分對待

**驗證**：開發機 macOS/Linux 跑 `npm test && npm run build`

---

### 3. API 資料存取

**規則**：任何 Notion/Supabase 存取經 MCP Server 或 API Route；禁止硬連庫

**檢查**：`grep -r "new.*Client\|\.from\|createClient" src/` 確認無直連

**例外**：`/api/` 路由內部可以用 createClient，但須經 middleware 租戶隔離

---

### 4. 環境變數機密

**規則**：`API_KEY`、`DB_PASSWORD`、`OAUTH_SECRET` 等全存 `.env`；禁止提交到 git

**檢查**：`git diff HEAD~5 | grep -i "key\|secret\|token\|password" || echo "ok"`

**例外**：公開的常數值（如 API 端點 URL）可以寫代碼

---

## 審批與驗收

| 階段 | 檢查者 | 產出 |
|------|--------|------|
| **規格評審** | 分派者 + JDNE | 確認規格符合 7 項做歪信號 + 部署約束初檢 |
| **代碼審查** | 非作者機器 | 確認部署約束 4 項全過 |
| **上線前** | Z1FV + Jin | 過 7 項做歪信號終檢 + SaaS Phase spec 終檢 |

## 常見誤區

| 誤區 | 為什麼錯 | 正確做法 |
|------|---------|--------|
| 層 5 最重要 | 優雅的爛產品沒人用 | 滿足層 1-4 後再優化層 5 |
| 我有好理由就能破例 | 例外堆積會變成新規則 | 異常先記錄，三級決策通過再破例 |
| 規格文件只是參考 | 導致理解偏差 | 規格是法律，必須逐項對上 |
| 部署約束是後考慮的 | 後改動成本 10 倍 | 早期設計階段就檢查 |
