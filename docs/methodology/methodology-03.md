# 改動前驗證（Pre-change Verification）

在動手改東西或執行多步驟計畫之前，用兩種工具提前找出漏洞：靜態掃描（從改動點往外找依賴）和動態模擬（用具體案例走過計畫）。視情況用一種或兩種都用。

觸發條件和檔位判斷見 methodology-index.md 觸發機制表。本文件只寫操作步驟。

---

## 判斷入口

改的是已有的東西（修改現有檔案、更換元件、改設定）→ 跑靜態掃描。

要執行一套多步驟計畫（新功能開發、資料遷移、流程部署）→ 跑動態模擬。

兩者都是 → 兩種都跑。先靜態（確認影響範圍），再動態（確認計畫跑得通）。

---

## 工具一：靜態掃描（原漣漪掃描）

從改動點往外一圈一圈掃描，找出所有會被影響的地方。

### 第一步：精確描述改動

回答三個問題：改什麼？從什麼改成什麼？為什麼要改？要具體——不是「調整 API 邏輯」，而是「把 /api/tenders 的回傳格式從陣列改成分頁物件，新增 total 和 nextCursor 欄位」。說不清楚要改什麼，你還沒準備好掃描。

### 第二步：畫第一圈——直接依賴

四個面向都要掃：

| 面向 | 要找什麼 | 怎麼找 |
|------|---------|--------|
| 程式碼 | 哪些檔案 import 這個模組？哪些函式呼叫這個 API？哪些測試覆蓋這個功能？ | grep、IDE 搜尋 |
| 資料 | 哪些資料結構需要遷移？現有資料改動後會不合法嗎？ | schema 比對 |
| 文件 | 哪些規範文件描述了現在的行為？改了之後文件會過時嗎？ | 手動搜尋 |
| 其他機器 | 有沒有其他機器正基於現有行為開發？ | 查小結、查 snapshot |

grep 能找到程式碼層面的引用，找不到「這個設計隱含地假設了 XX」。後者需要對系統整體架構的理解。

### 第三步：畫第二圈——間接依賴

對第一圈每個東西再問一次：「如果這個因為我的改動而改變了行為，還有什麼會受影響？」通常兩圈就夠。第二圈還發現大量影響，可能代表改動太大——考慮拆小。

### 第四步：判斷每個影響點

| 判斷項 | 內容 | 範例 |
|--------|------|------|
| 影響類型 | 一定會壞 / 可能會壞 / 不會壞但需要更新 | 直接呼叫改了格式的 API → 一定會壞 |
| 嚴重程度 | build 失敗（馬上發現）/ 行為微妙改變（很久才發現）/ 資料毀損（不可逆） | 測試 mock 寫死舊格式 → 測試過但行為不對 |
| 處理方式 | 改程式碼 / 加測試 / 更新文件 / 通知其他機器 / 不需處理 | 文件過時 → 更新文件 |

### 第五步：評估總體風險並制定行動計畫

三個問題：總影響範圍跟預期一樣嗎（比預期大很多要重新考慮改法）？有不確定的風險嗎（不確定的比已知的危險，考慮先做小實驗確認）？最壞情況是什麼，能退回去嗎？

處理順序：先處理「一定會壞」→ 再處理「需要更新」→ 最後處理「可能會壞」（加防禦或監控）。影響點跨多台機器的，決定是你一起處理還是通知各自處理。

## 工具二：動態模擬（原乾跑驗證）

在真正動手之前，用腦子（而不是鍵盤）把計畫從頭到尾走一遍。

### 第一步：把計畫攤成線性步驟

重新攤的不是「要做什麼」，而是「系統在每一步之後的狀態」。格式：步驟 N → 系統狀態變成 XX → 步驟 N+1 用 XX 作輸入 → 狀態變成 YY。沒辦法描述某一步之後的系統狀態，代表你對那一步的理解不夠，先搞清楚。

### 第二步：挑一個具體案例走進去

不要用「假設有個 request 進來」這種抽象方式。挑一筆真實資料（或盡量接近真實的假資料），一步一步走。資料在每一步長什麼樣？欄位值是什麼？哪個函式處理它？產出什麼？

具體案例會逼你處理「如果這個欄位是空的怎麼辦」「日期格式不是預期的呢」這類抽象模擬容易忽略的邊界條件。

### 第三步：在每個步驟問四個問題

輸入存在嗎（這一步需要的所有輸入，此時都可取得嗎）→ 抓步驟順序錯誤、缺少前置步驟。操作會成功嗎（什麼條件會導致失敗）→ 抓權限不夠、資源不足、格式不對。輸出符合下一步預期嗎 → 抓型別不匹配、缺少欄位。有沒有副作用 → 抓意外寫入、觸發通知、改變全域狀態。

### 第四步：刻意走歪路

正常路徑走完後，故意改輸入：空值、超長字串、特殊字元、不存在的 ID、已刪除的資料。不需窮舉——挑最可能發生的和後果最嚴重的。

### 第五步：記錄發現並做決策

發現分三類：阻塞（不解決計畫跑不動，必須執行前修）、缺陷（計畫能跑但會出錯，可以實作時順手修）、待確認（需要更多資訊才能判斷）。

三個選項：按計畫執行（沒有阻塞）、修正後執行（有阻塞但知道怎麼修）、推翻重設計（阻塞太多或太根本）。

## 做完標準

靜態掃描完成：能列出所有已知影響點，每個都有影響類型、嚴重程度、處理方式的判斷。有行動計畫。

動態模擬完成：正常路徑跑通（能用自己的話說出從頭到尾的資料流）。邊界條件已識別。明確說了「照做」「修後做」或「推翻重來」。

---

## 品質判斷

做得好的信號：靜態掃描四個面向（程式碼、資料、文件、其他機器）都掃到了。動態模擬用了具體案例而不是抽象描述。至少找到一個計畫階段沒想到的問題——一個都沒找到，要麼計畫已經完美（不太可能），要麼驗證太表面。

做得差的信號：只掃了程式碼，忘了資料、文件、其他機器。動態模擬太抽象（「先讀資料，然後處理，然後存起來」——每一步具體狀態都沒看到）。只走正常路徑，沒測邊界。掃描深度跟改動大小不成比例——小改動花了太多時間驗證。

---

## 已知限制

靜態掃描的隱性依賴問題：如果 A 和 B 的關係靠約定而非程式碼引用，grep 找不到。需要對系統整體架構的理解，不能只靠工具。

動態模擬的黑箱步驟問題：不知道某個函式內部怎麼運作時，模擬在那個點就是黑箱。標記為「待確認」，實作到那裡時特別小心。

併發難模擬：多個非同步操作的時序問題很難用腦子跑。Race condition 通常需要實際跑才找得到。

第二圈準確度低：越遠的影響越難判斷。如果第二圈出現很多「可能」，考慮讓不同模型交叉驗證。

## 自我迭代

每次執行完成後，觀察以下信號判斷方法是否有效、是否需要調整：

| 觀察 | 健康信號 | 警示信號 | 調整方向 |
|------|---------|---------|---------|
| 掃描遺漏率 | 實作時未撞到掃描沒找到的依賴 | 實作時發現掃描漏掉的影響點 | 遺漏來自四面向的哪一個？工具能力不足還是面向本身有缺口？ |
| 模擬準確率 | 模擬走通的路徑實作也走通 | 模擬沒問題但實作撞牆 | 案例是否太理想化？歪路測試是否涵蓋最可能的失敗模式？ |
| 驗證比例 | 驗證深度跟改動風險匹配 | 小改動花太多時間驗證，或大改動驗證太淺 | 檔位判斷的客觀指標需校準 |

同類警示信號累積 ≥3 次，觸發事後回顧追查根因。

## 專案實例

### 靜態掃描案例：PCC API 影響分析

改動：PCC（政府採購網）的 API 端點要更換。

第一圈：呼叫 PCC API 的地方——爬蟲服務、資料同步排程、測試 mock、環境變數設定、API 文件。

第二圈：爬蟲服務的產出被哪些功能使用？→ 標案列表頁面、搜尋功能、通知服務。

判斷：「一定會壞」3 個（直接呼叫端點）、「需要更新」2 個（環境變數和文件）、「可能會壞」1 個（測試 mock 寫死舊格式）。

行動計畫：改環境變數 → 改呼叫程式碼 → 更新測試 mock → 全套測試 → 更新文件。

### 動態模擬案例：/kb 模擬時找到 4 個漏洞

背景：/kb 指令的 8 步流程，準備實作。拿一筆具體知識條目（「碰撞處理策略」），從用戶輸入 `/kb save '碰撞處理策略'` 開始一步一步走。

| # | 漏洞 | 類別 | 處理 |
|---|------|------|------|
| 1 | 步驟 3 需查索引避免重複，但索引格式未定義 | 阻塞 | 實作前先定義索引格式 |
| 2 | 標題含特殊字元時檔名不合法 | 缺陷 | 實作時加 sanitize |
| 3 | 沒有處理「存了但反悔想刪」的場景 | 缺陷 | 加進實作計畫 |
| 4 | 兩台機器同時存不同條目到同一索引檔會衝突嗎？ | 待確認 | 觸發了碰撞處理策略的設計 |

漏洞 1 如果沒有乾跑，會在實作到步驟 3 時才發現，前面兩步的工作可能白費。

### 靜態掃描案例：多機器同時推 _index.md

這不是程式碼改動，是協作流程的風險。影響點是「_index.md 的內容一致性」。git merge conflict 會馬上發現（build 失敗），但靜默的內容丟失不會——這是更危險的風險。這個案例催生了碰撞處理策略。
