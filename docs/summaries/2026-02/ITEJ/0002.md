## A. 議題列表

A1. 擴充 /暫存 指令新增 issue type
  背景：用戶嘗試用 /暫存 捕捉「待辦清單沒有 SSOT」的觀察，發現 plan/rule 兩種類別無法承接只有問題沒有結論的情境——「寫入內容」和「整合指南」欄位無法填寫，硬套會產出空殼文件。
  經過：用自問自答法確認現有機制（OP 記錄、快照 [?]、小結 C 章、MEMORY.md）都無法承接「帶完整上下文的設計觀察」。比較三方案（加 issue type / 放寬格式 / 另建機制），選定新增 issue type。建立首個 issue 暫存檔「待辦清單缺乏SSOT」作為實際驗證。
  結論：/暫存 新增 issue 類別，格式比 plan/rule 輕（現況分析 + 可能方向取代結論三節），生命週期為建立→討論→升級或關閉。

A2. 分析並解決待辦查詢不確定性
  背景：用戶問「待辦清單資料從哪來？」，揭露待辦查詢沒有標準程序。A44T 機器實際查待辦時漏了「知識庫建置精靈」（該項存在於三份暫存檔但不在快照或進行中 topic 裡）。
  經過：AI 初始判斷「稱不上問題」，被用戶以實際案例挑戰後修正。經多輪自問自答迭代：從「缺 SSOT」→「查詢不穩定」→「AI 沒有標準程序」→「需要固定掃描來源的指令」。方案從 4 項改動（改寫入端 + 讀取端）逐步精簡為 1 項（只建 /待辦 指令）。用戶三次追問「目標是什麼？解法有效嗎？」推動收斂。最後確認：查詢端有標準程序就夠了，寫入端的 AI 判斷在決策當下是可靠的。
  結論：建立 /待辦 指令，定義 5 個固定掃描來源（快照待做/未決項、進行中 topic、暫存索引、暫存檔內容的具體交付物、小結 C 章），issue 暫存檔已關閉。

A3. 撰寫 ITEJ-0001 小結報告
  背景：ITEJ 機器首次執行 /小結。
  經過：涵蓋 6 個議題和 JDNE-0001 待決策項目盤點。過程中用戶指出 C 章項目必須有 A 章對應（「要，但有 C 就要有 A，記得嗎？」），補加了 A7 盤點段落。
  結論：ITEJ-0001 存檔完成（6 項已定案、2 項待決策承繼自 JDNE-0001）。

A4. 盤點 ITEJ-0001 待決策項目現況
  背景：每份小結需檢查前一份的待決策項目是否已解決。
  現況：ITEJ-0001 的 C1（知識庫存放方案）和 C2（CLAUDE.md 拆檔）在本次範圍內均未被討論，仍未決。

## B. 已定案

B1. /暫存 新增 issue type，用來捕捉有問題沒結論的設計觀察——原有 plan/rule 格式有必填欄位（寫入內容、整合指南），強套會產出空殼文件，issue type 用更輕的格式（現況分析 + 可能方向）避免此問題。
B2. 待辦查詢不確定性由 /待辦 指令解決——根因是 AI 每次即興選擇查詢來源，解法是建立 5 個固定掃描來源的標準程序。
B3. 解決待辦查詢只需建 /待辦 指令一項改動，不需同時改 /暫存 或 /小結 的寫入端——寫入端的 AI 判斷在決策當下是可靠的，問題出在事後查詢時缺標準程序。
