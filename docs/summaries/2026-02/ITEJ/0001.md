## A. 議題列表

A1. 設計並升級 MEMORY.md 維護規則
  背景：多機器環境中 MEMORY.md 不透過 git 同步，需要明確規範什麼該寫、什麼不該寫。
  經過：初版透過自問自答法確定定位為「本機踩坑筆記」，設定寫入和清理觸發條件。第二版發現初版缺少「提升」機制（知識進得去出不來），改為「寫入前分流」方案：用 8 類分類表在寫之前就決定該去哪裡。驗證本機現有 3 條內容全部不該在 MEMORY.md，清空並搬到共享位置。
  結論：MEMORY.md 只收三類（首次觀察、本機限定偏好、本機環境問題），其他直接寫共享位置。分流表已寫入 CLAUDE.md。

A2. 提煉自問自答法的實踐經驗
  背景：設計 MEMORY.md 規則過程中多次使用自問自答法，累積了可重複的教訓。
  經過：整理出四個關鍵教訓：空檔案不需複雜協議、先查現有機制是否已覆蓋、每輪目標是收斂不是發散、需要主觀判斷的規則不是好規則。
  結論：教訓已記錄，自問自答法的共享文件由 JDNE 維護（docs/methodology/）。

A3. 清理 GitHub Actions PCC 監控殘留
  背景：PCC API 封鎖 GitHub Actions 雲端 IP（403），監控方案已改為 Google Apps Script，殘留 workflow 需清理。
  經過：刪除 `.github/workflows/pcc-monitor.yml`，用戶手動完成 GAS 每小時監控設定。
  結論：GitHub Actions 監控已刪除，GAS 監控已上線。

A4. 設計多機器碰撞處理策略
  背景：JDNE 和 ITEJ 同時編輯 CLAUDE.md 和 _index.md，發現預防性規則（禁止同時編輯）依賴人的紀律，實際無法遵守。
  經過：用自問自答法重新設計，從「預防碰撞」改為「碰撞是常態，系統靠容錯」。為每種碰撞類型制定處理方式：_index.md 靠 append-only auto-merge、Topic ID 重複靠 /更新 掃描、CLAUDE.md 矛盾靠 diff 語義檢查。
  結論：CLAUDE.md 協作規則改寫為容錯模式，/更新 新增碰撞偵測。

A5. 釐清重啟、壓縮、更新三套流程的分工
  背景：/更新指令和 CLAUDE.md 啟動流程功能重疊但各缺一塊，需要釐清邊界。
  經過：逐項對比步驟，討論 npm 驗證、碰撞偵測、上下文恢復的配置位置，用自問自答確認收斂。
  結論：重啟=完整驗證（含 npm）；壓縮=輕量恢復（補完 hook 遺漏）；/更新=手動同步（不跑 npm）。

A6. 改版 /暫存 指令並整合 stop hook 同步提醒
  背景：/暫存 原本只能存 dev-plan 結論、依賴 context window（compact 後消失）、無類別區分。30 分鐘定時同步規則依賴 AI 自律，不可靠。
  經過：7 檔同步改動：/暫存 改為記錄層驅動（含增量更新）、新增 plan/rule 分類、/修改計畫 加 type 過濾、stop hook 加 30 分鐘提醒、pre-compact 寫時間戳、CLAUDE.md 刪除定時同步段落、暫存索引 4 欄遷移至 5 欄。
  結論：全部改動完成並推送，索引格式驗證通過（19 行全部 5 欄）。

A7. 盤點 JDNE-0001 待決策項目現況
  背景：JDNE-0001 列出 5 項待決策，需確認哪些已被後續工作解決。
  經過：逐項比對。C1（/小結品質標準）已由 JDNE 補撰寫指南解決；C5（/暫存改版）已由 ITEJ 完成 infra-staging-upgrade 解決；C2（自問自答法主題合併）索引已合併但 ITEJ 實踐經驗是否併入 self-qa.md 未確認。
  現況：C3（知識庫存放方案）和 C4（CLAUDE.md 拆檔）仍未決

## B. 已定案

B1. MEMORY.md 只收三類內容，其他在寫入前分流到共享位置——不分流的話知識會卡在本機。
B2. 多機器碰撞處理改為容錯模式——預防性規則靠人的紀律，多台機器同時工作時無法遵守。
B3. 重啟/壓縮/更新三套流程各有明確分工——重啟最重（含 npm），壓縮最輕（補 hook），/更新介於中間。
B4. /暫存 改為記錄層驅動、支援 plan/rule 分類——解決 compact 後資訊消失，擴大適用範圍。
B5. 30 分鐘同步提醒由 stop hook 驅動，取代 AI 自律的定時規則——hook 機制比 AI 計時可靠。
B6. GitHub Actions PCC 監控移除，改用 GAS——GitHub Actions 雲端 IP 被封鎖。
