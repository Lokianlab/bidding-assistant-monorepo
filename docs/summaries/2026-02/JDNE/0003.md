## A. 議題列表

A1. 改進 /待辦 指令，加入交叉驗證機制
  背景：用戶要求待辦清單拉出來後，要逐項確認進度和記錄是否一致，再列給用戶看。
  經過：指令改了三次。第一版只標註過時項目；用戶說「不一致的記錄一直保留著不太對吧」，改成當場修正；用戶再問「OP 記錄一定為真嗎？」，最終改為不預設任何來源為真——遇到矛盾就去看實際狀態（檔案存不存在、程式碼改了沒、git 歷史），確認事實後當場改正所有不一致的記錄。
  結論：/待辦 指令現在會自動交叉驗證，發現矛盾就修。

A2. 執行交叉驗證，修正多處記錄不一致
  背景：用 A1 改好的指令跑待辦清單，逐項核實。
  經過：發現三處矛盾——標案 API 工具的程式碼專案建立時就已寫好（快照寫「待做」）、方法論索引已由另一台機器更新好（快照寫「待完成」）、待辦盤點已由另一台機器完成（主題索引寫「進行中」）。全部依實際狀態當場改正。
  結論：交叉驗證有效，後續又發現更多需改的（見 A6）。

A3. 嘗試在記憶壓縮前提醒更新快照
  背景：發現快照沒捕捉到壓縮前正在討論的內容，用戶問「壓縮前不是要快照嗎？」。
  經過：在壓縮前的自動存檔 hook 裡加了快照提醒（用 exit 2 擋住壓縮）。但另一台機器後來移除了這個機制——理由是壓縮發生時 AI 記憶已快滿，沒有餘裕做更新，而且旗標檔會觸發別的 hook 製造混亂。
  結論：嘗試失敗，只保留自動存檔功能。壓縮前的快照問題目前無解。

A4. 完成模型委員會方法論的部分收尾
  背景：上份小結列出收尾步驟：舊方法文件加「已被取代」標記、更新方法論索引。
  經過：舊方法文件頂部加了標記。方法論索引在驗證中發現另一台機器已經更新完了。剩下暫存檔還沒推進開發計畫。
  現況：只剩暫存檔待推。

A5. 安裝建置標案 API 的查詢工具
  背景：交叉驗證時發現程式碼早就寫好了（6 個查詢工具、217 行），但從未安裝過。
  經過：安裝依賴、編譯成功，AI 工具設定也早已配好。
  現況：技術上可以跑了，待用戶驗收。

A6. 建立驗收規則，新增「待驗收」狀態
  背景：用戶發現我把標案 API 工具標為「完成」，但用戶根本沒看過。接著又指出知識庫精靈也是另一台機器建的、用戶沒驗收。用戶明確要求：「所有還沒確認驗收通過的，全都不能當作完成」「只有我能驗收，任何機器都不能驗收。」
  經過：在共用規範裡加入驗收規定，快照標記新增「待驗收」（從四種變五種），主題索引也新增對應狀態。回頭檢查所有被標為「完成」的項目，把用戶沒驗收過的全部改回「待驗收」。
  結論：驗收規則已定案並廣播（見 B 章）。

A7. 識別結論層設計、知識庫精靈、待辦盤點為待驗收
  背景：建立驗收規則後，回頭清查各台機器快照中所有標「完成」的項目。
  經過：結論層設計是 ITEJ 做的暫存改版機制；知識庫精靈 /kb 指令是 A44T 建的；待辦盤點和 /待辦 指令分別是 A44T 和 ITEJ 完成的。三項都沒經過用戶驗收，全部從「完成」改為「待驗收」。
  現況：連同標案 API 工具，共 4 項待用戶驗收。

A8. 待辦盤點
  背景：本次共跑了四次待辦清單，逐次修正記錄。
  經過：最終結果 12 項——3 項待做、4 項待驗收、4 項待決、1 項等待中。「待驗收」是新增的分類。
  現況：清單穩定。

## B. 已定案

B1. 驗收規則——「完成」只能在用戶明確說驗收通過後才標。不確定就問用戶。任何機器都不能替用戶驗收。已寫入共用規範。
B2. 新增「待驗收」狀態——代表工作已做完但用戶尚未確認，快照和主題索引都支援。
B3. /待辦 指令加入交叉驗證——掃完後逐項看實際狀態，矛盾當場修正，不預設任何來源為真。
