## A. 議題列表

A1. 清理方法論文件中看不懂的段落
  背景：用戶覺得方法論文件不好讀，要求審閱
  經過：發現「回應前分流」那段在分高中低三級，但 self-qa.md 裡已經有一模一樣的適用情境表了，等於寫了兩次，用戶也說看不懂。順手把 CLAUDE.md 的多機器協作規則補完——機器之間是平等的，可以互相挑毛病，數量不限
  結論：回應前分流整段砍掉，機器對等關係寫進規範

A2. 把不可靠的觸發規則換掉
  背景：記錄層原本有 4 條觸發規則，其中「每 15 分鐘更新」和「session 結束前寫入」根本做不到——AI 沒有時鐘，也猜不到 session 什麼時候斷。「討論有結論時」定義也很模糊
  經過：砍到只剩 2 條，全部綁在 git push 上——只要推程式碼就順便更新記錄，簡單明確。同時把 Stop hook 的檢查範圍從只看記錄資料夾擴大到所有有改動的檔案
  結論：不再靠 AI 自己記得要寫記錄，改由系統自動擋——推之前沒寫記錄就不讓你推

A3. 設計並建立 /kb 知識庫精靈指令
  背景：知識庫維護精靈的 prompt 早就寫好了（914 行，功能完整），但一直只能手動複製貼進 Claude.ai 的 Project 裡用，很麻煩。盤點待辦時才發現這件事根本沒人在追蹤
  經過：先暫存了一份精靈的現況盤點，釐清哪些做好了、哪些沒做。然後進入計畫模式設計怎麼做——決定不把 914 行全塞進指令檔，而是寫一個薄薄的調度層（134 行）去引用那份大 prompt。計畫被用戶打回來兩次：第一次嫌寫入格式沒標準化，第二次問「啟動前不用先 pull 嗎？」。修完之後用戶要求模擬跑一遍自證可行，跑的過程中抓到 4 個漏洞，修完才放行實作
  結論：/kb 指令已經可以用了。打 `/kb 成員` 就能開始建人員資料，打 `/kb 實績` 建案件資料，精靈會引導對話、檢查品質、自動寫入本地檔案

A4. 待辦清單盤點
  背景：用戶問現在還剩什麼沒做
  經過：原本 8 項待辦，逐項比對三台機器的進度。第一次算完剩 4 項，pull 之後發現 ITEJ 已經做完暫存改版，漏算了，再修正為 3 項
  現況：還有 3 件事要做（建 PCC MCP、建 Notion MCP、更新開發計畫），另外 PCC API 延遲分析在等監控數據進來
