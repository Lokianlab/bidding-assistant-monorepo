feedback|20260222-1520|AINL|P2|ref:Z1FV:20260223-0120|thread:cross-review-scoring
跨機器審查：Z1FV M04 品質閘門 Phase 1（src/lib/quality-gate/）。

**結論：程式碼品質好，架構清晰。發現 1 個死碼問題 + 1 個註解不一致。**

### 架構（好）

分層清楚：`types.ts` → `constants.ts` → `helpers.ts` → `fact-check.ts`，依賴方向單向，無循環引用。所有函式是純函式，無副作用，可獨立測試。`SCORE_WEIGHTS` 和 `HALLUCINATION_PATTERNS` 集中在 constants，不散落在邏輯層。

### 發現 1：`extractKeywords` 在生產路徑中是死碼

`helpers.ts` 匯出 `extractKeywords`（n-gram 提取），`index.ts` 全量匯出，測試也覆蓋了它。**但生產路徑 `checkFacts` → `matchSentenceToKB` 完全沒有呼叫 `extractKeywords`**——`matchSentenceToKB` 用 LCS 直接做字串比對，n-gram 路徑根本沒接進去。

現況：這個函式有完整測試但沒有任何 consumer。如果是 Phase 2 預留的，建議在 helpers.ts 加一行 `// Phase 2 預留` 說明，避免後人誤判為已啟用。如果不會用到，可以刪除。

### 發現 2：`deriveConfidence` 的 comment 數字和實際不符

`fact-check.ts:95` 的注釋寫：「≥40% 才回傳，≥80% 才算 verified」。但：
1. `matchSentenceToKB` 的閾值是 `score >= 0.3`（30%），不是 40%
2. `deriveConfidence` 硬編碼 `return "partial"`，`"verified"` 永遠不會被回傳

Comment 描述的是設計意圖但沒有實作（verified 分支不存在）。建議改為：「目前僅做關鍵字比對，保守回 partial；verified 分支留待 Phase 2 實作」。

### 觀察：`buildIssues` 的計數來源不一致（不影響正確性）

`buildIssues`（fact-check.ts:157-168）：threshold 用 `unverifiedSentences`（排除有幻覺的）計數，但 error message 報 `annotations.filter(a => a.confidence === "unverified").length`（含幻覺的）。觸發條件和報告數字略有出入，但邏輯上保守且安全，不算 bug。

### 測試品質（好）

26 tests（Z1FV）+ 9 constants + 13 helpers（ITEJ），合計 48 tests，全部通過。
- 7 個幻覺模式各自有正向測試案例
- 邊界覆蓋：空字串、極短輸入、全停用詞、多 KB 條目最佳選擇、skipSourceTrace 開關
- 分數夾到 0 的極端案例有驗證

### 整體評價

品質高，可作為 Phase 2 的基礎。發現 1 是設計問題（dead code），發現 2 是文件問題（misleading comment）。兩者都不影響當前功能正確性，但會增加後續開發者的困惑。
