## 關鍵決策與設計

### 信任度計分公式
```
信任度 = (評分 / 5 × 60%) + (合作次數 / 100 × 40%)
```
**理由**：評分來自歷史平均（穩定），合作次數可能累積偏差（次要參考）

### KB 搜尋架構
**選擇**：向量搜尋 + 語義匹配 + 多層篩選
**理由**：平衡準確度與性能，支持複雜查詢

### 多租戶認證策略
**選擇**：OAuth + Supabase RLS 雙層防禦
**理由**：標準安全實踐，支持水平擴展

### 評分權重分配
- 技術方案：35%
- 成本效益：35%
- 風險控制：20%
- 進度可行性：10%

**理由**：根據客戶優先序（得標率最高）調整

## 協調過程與經驗

### 多機並行開發
- 6 個機器同時進行不同模組
- 使用快照宣告制避免衝突
- Git 衝突出現時優先檢查設計決策而非代碼衝突

**經驗**：早期决策鎖定 > 後期衝突解決

### 品質管理（三層制度）
- **L1 自主**：bug、測試、清理（無需審查）
- **L2 展示**：新功能（需非作者審查）
- **L3 先議**：架構決策（/暫存 → JDNE → Jin）

**成效**：品質穩定，決策透明，交付加速

### 信息同步機制
- 快照作為機器狀態單一來源
- 每次 push 自動更新快照
- 決策點清晰標記 ([ ] [ >] [x] [?])

**成效**：避免重複工作，實時掌握進展

### OP 記錄規範
- 只記「例外」（失敗、決策、教訓）
- 成功靠 commit message
- 複雜情況寫詳細OP
