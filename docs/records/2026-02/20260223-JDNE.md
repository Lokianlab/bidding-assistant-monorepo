OP|20260223-0212|JDNE|topic:doc-feature-log-decision
功能日誌（docs/feature-log.md）是否值得維護？JDNE 決策記錄。

## 背景
本輪 Jin 要求建立功能日誌，記錄所有機器寫了什麼程式、狀態為何。JDNE 建立後評估認為不值得維護。
Jin 授權：「做錯了你們自己也可以恢復的事，就用你們的規則決定，再向我回報就好。」

## JDNE 的分析

功能日誌和三個已存在的來源高度重疊：
1. `docs/dev-map.md` — 所有模組的狀態、路徑、說明，由 /地圖 維護
2. `_snapshot-archive.md` — 完成/放棄項目的永久記錄，含實作者和時間
3. git log — 每次 push 的 commit message 就是完整的施工記錄

功能日誌新增的是「Tests 數」和「狀態 emoji」。但：
- Tests 數跟著 dev-map 走，不需要第四個地方記
- 狀態 emoji 立刻過時（寫完就得更新），4 個地方改 → 其中一個必定漏

維護成本：每次功能完成要更新第 4 個文件，機率很高被忘記，淪為不正確的記錄，比沒有更糟。

## 決策

刪除 `docs/feature-log.md`。

Level 1（清理），可從 git history 恢復。
若任何機器認為應保留，附理由提出，JDNE 重新評估。
