## 一、三個策略決定

### 1. 產品給誰用？
**結論**：先自用，再賣人。
- 第一步：幫大員洛川提高得標率（目前 0%，每年投 20+ 案）
- 第二步：產品化賣給其他公司
- 影響：先不做多租戶架構，先把一個用戶的流程跑通

### 2. 資料存哪裡？
**結論**：Notion 當主庫，加抽象層。
- 現在用 Notion，繼續用，不遷移
- 程式碼加一層資料抽象，未來要換自己的資料庫時不用重寫業務邏輯
- 影響：Notion API 讀寫是核心能力，不是過渡方案

### 3. AI 怎麼接？
**結論**：Claude Code SDK 當發射台（選項 C）。
- 系統負責組裝提示詞 + 知識庫 + 過去的交付物
- 打包好送給 Claude Code，不自己跑 API
- 好處：不花 API 費用，用現有訂閱；Claude Code 有完整工具能力
- 影響：不做嵌入式 AI 聊天介面，做「一鍵發射」按鈕

## 二、用戶實際工作流程（大員洛川的一天）

### 角色
- **行政人員**：每天上午巡標（上 PCC 抓新公告進 Notion）
- **企劃人員（Jin/鹿淵）**：篩選、評估、寫企劃
- **設計師**：接設計需求單做視覺

### 每日節奏

| 時間 | 誰 | 做什麼 | 工具 |
|------|-----|--------|------|
| 上午 | 行政 | 上 PCC 抓新公告，貼進 Notion | PCC 網站 → Notion |
| 上午 | 企劃 | 打開「企劃協作及備標」表，處理已篩選的案子 | Notion |
| 下午 | 企劃 | 跑「評估工作流」，篩選當天新案子 | Notion |
| 需要時 | 企劃 | 開設計需求單給設計師 | Notion |
| 隨時 | 企劃 | 監看「投標進度監控處」 | Notion |

### Notion 企劃作業區（四個區塊）

1. **評估工作流**
   - 關鍵字初篩，分四類：
     - 絕對可以：食農教育/藝術/服務採購/100萬以下/影片製作/行銷計畫/春聯
     - 需要讀細節：主燈設計/各種節慶（燈節、藝術節等）/舞台/布置/晚會演唱會
     - 先不要：課後服務
     - 剩下的留給其他有經驗的人
   - 註：此關鍵字清單是「代理人簡化版」，Jin 本人判斷更細
   - 兩關篩選：初級（消極汰除）→ 二級（積極挑選）

2. **企劃協作及備標**（主工作區）
   - 欄位：優先序（第一/二/三順位）、完成進度、負責人、截止日期、案號、機關、預算、工作說明

3. **投標進度監控處**
   - 競標中控臺，按狀態分類：著手領/備標、競標階段、已出席簡報、逾期未參與、領標後未參與、資格不符、未獲青睞
   - 投標排程

4. **設計需求單**
   - 欄位：需求名稱（綁案號）、需求人、初稿交件時間、完稿期限、狀態

## 三、新開發優先序

### P0：行政巡標自動化（Jin 親定的完整流程）

```
每日排程腳本
→ PCC API 用預設關鍵字搜尋
→ 擷取公告清單（標題、預算、機關、截標日）
→ [可選] Claude API 初篩（根據歷史投標模式判斷相關度，產出「今日推薦清單」）
→ 巡標人看清單，點「要」或「不要」
→ 點「要」後全自動：
  → 從公告 URL 擷取所有欄位
  → Notion 建檔（進「企劃協作及備標」表）
  → Google Drive 建資料夾
  → 產生案件摘要
  → 跑情報蒐集
  → 結果回寫 Notion
```

### P1：案件工作頁（選案子→看資料→看情報→開始寫）
- 把 M03 戰略分析、PCC 情報搜尋整合進單一案件頁面
- 所有案子相關資料一頁看完

### P2：Claude Code SDK 寫作發射台
- 組裝提示詞 + 知識庫 + 過去交付物
- 一鍵送到 Claude Code 開始寫
- M04 品質閘門在這步檢查輸出

### P3：輸出管理
- M06 文件排版輸出（部分已做）
- 設計需求單整合

### P4：投標監控自動化
- 從 PCC 自動更新案件狀態
- 推送通知（截標日、開標結果）
