# 記錄與交接機制 v2

**設計日期**：2026-02-24（v2 重寫）
**問題根源**：v1 在 session 1 結束時丟失了一段重要討論（「三種指令」的問題），/clear 後 AI 無法恢復上下文。
**設計原則**：Jin 不管怎麼開場，AI 都能在 30 秒內接回上次的位置。

---

## 核心機制：三層記憶

| 層 | 檔案 | 記什麼 | 讀的時機 | 成長方式 |
|---|---|---|---|---|
| 現在在哪 | `進度總覽.md` | 當前狀態、開放的線、阻塞 | 每次啟動 | 覆寫，不累積 |
| 一路走來 | `累積記憶.md` | 決策、修正、教訓、原則 | 每次啟動 | 慢慢長，超 150 行精簡 |
| 每次聊了什麼 | `journal-{日期}-s{N}.md` | 完整對話過程 | 需要回溯時才搜 | 每 session 一個 |

啟動只讀前兩個（短），日誌不讀（長）。這樣 token 成本固定，但記憶持續累積。

### 1. 狀態檔（恢復點）：`進度總覽.md`

**這是唯一的恢復入口。** /clear 後 AI 第一個讀、也只需要讀這個。

內容結構：

```markdown
# 進度總覽
最後更新：{日期} {時間} session{N}

## 上次結束在哪裡
{一段話，把最後在討論什麼、討論到什麼程度、有什麼未完成的想法寫清楚}

## 開放的線（未完成的討論）
- {議題}：{討論到哪裡、下一步是什麼}
- {議題}：{同上}

## 開發進度
- [狀態] 工作項目：一句話

## 阻塞中（需要 Jin 操作）
- {什麼事}：{Jin 需要做什麼}

## 下次 session 第一件事
{具體的，不是「等 Jin 確認」，而是「報告狀態然後繼續 X」}
```

**關鍵區別**：v1 的進度總覽只有開發進度。v2 多了「上次結束在哪裡」和「開放的線」。這兩個區塊解決了「/clear 後不知道從哪開始」的問題。

**大小限制（硬性）**：進度總覽永遠維持在 80 行以內。做法：
- 「上次結束在哪裡」每次覆寫，不累積歷史
- 「開放的線」只留活的，結束的直接刪掉
- 「開發進度」只留進行中和阻塞的；已完成的項目收進當次日誌，總覽裡刪除
- 如果逼近 80 行，優先砍已完成項目和已解決的阻塞

### 2. 日誌（過程紀錄）：`journal-{日期}-s{N}.md`

每個 session 一個檔案。記錄對話中的重要內容。

```markdown
# Journal — {日期} Session {N}

## 時間線

### {時間} {主題}
{發生了什麼、討論了什麼、決定了什麼}

### {時間} {主題}
{同上}

## 認知修正
| AI 原本的判斷 | Jin 的修正 | 修正後的認知 |
|---|---|---|

## 本次產出
- {檔案或成果}：{一句話說明}
```

日誌是**過程記錄**，用來回溯「為什麼做了這個決定」。日常恢復不需要讀日誌，只讀進度總覽。

## 存檔規則

### 什麼時候更新進度總覽

1. **Session 結束前**（必做）：把「上次結束在哪裡」和「開放的線」寫到最新狀態
2. **重要決策後**：改變方向、得到新指示
3. **長時間討論切換話題時**：確保前一個話題的結論不丟

### 什麼時候寫日誌

- Session 進行中，每完成一個有意義的討論段落就補一條
- 不用等 session 結束才一次寫完（v1 的失敗就是因為等到最後才存）

### 什麼不用記

- 純技術執行過程（commit message 就夠了）
- AI 的中間搜尋過程
- 沒有產生決策或洞見的討論

---

## Session 啟動 SOP（AI 自動執行）

```
1. 讀 docs/各種輸出文件/進度總覽.md
2. 一次報完：
   - 上次結束在哪裡
   - 有哪些開放的線
   - 有什麼阻塞
   - 建議下一步
3. 等 Jin 說往哪走，就往哪走
```

Jin 不需要知道該問什麼問題。不管說「上次到哪了」「我們聊到哪」「開發怎樣了」「繼續」甚至只打一個字，AI 都給同一個完整回報。

```
docs/各種輸出文件/

  journal-20260224-s2.md               ← 每個 session 一個
  journal-{日期}-s{N}.md               ← 持續累積
  備標自動化系統設計-20260224.md        ← 產出（設計文件）
  協作模組定義-20260224.md             ← 產出（模組定義）
  標案資料庫深度分析報告-20260224.md    ← 產出（分析報告）
  記錄與交接機制-20260224.md           ← 本文件（元規則）
```

## 防丟失：自動存檔觸發點

「段落式即時存檔」的風險是 AI 判斷不了什麼時候該存。以下情況**自動觸發**更新進度總覽 + 日誌，不靠 AI 判斷：

1. **Jin 給了明確指示或修正**（「不要這樣做」「改成 X」「記住 Y」）→ 立刻存
2. **話題切換**（從 A 議題跳到 B 議題）→ 先存 A 的結論再開始 B
3. **Jin 說「存一下」「記起來」或任何暗示要保存的話** → 立刻存
4. **連續對話超過 10 輪沒存** → 自動存一次
5. **AI 準備回應 /clear 或 /完成** → 先存再執行

第 4 點是兜底機制：即使前面的觸發都沒命中，至少每 10 輪對話保底存一次。

## Jin 的輸入方式（AI 怎麼接收指示）

Jin 不會用結構化格式給指示。AI 要能從自然對話中辨識以下幾類訊號：

| Jin 可能說的話 | 代表什麼 | AI 要做的事 |
|---|---|---|
| 「不要這樣」「這樣不好」「不對」 | 方向修正 | 停下來，理解問題，調整方向，記進累積記憶 |
| 「記住 X」「以後都要 Y」 | 永久指示 | 立刻寫進累積記憶 |
| 「這個不急」「先做那個」 | 優先序調整 | 更新進度總覽的開放的線順序 |
| 「存一下」「先記起來」 | 要求保存當前討論 | 更新進度總覽 + 日誌 |
| 隨口一句觀察或想法 | 可能是重要洞見 | 記進日誌，如果有跨 session 價值就也記進累積記憶 |
| 貼截圖或外部資訊 | 補充上下文 | 摘要重點記進日誌 |
| 一個字或很短的回應 | 可能是確認、否定、或等 AI 繼續 | 根據上下文判斷，不確定就當確認 |

**核心原則**：Jin 不需要學 AI 的語言，AI 要學 Jin 的語言。

## v1 → v2 的改動摘要

| v1 的問題 | v2 怎麼解決 |
|-----------|------------|
| 進度總覽只有開發進度，沒有「聊到哪」 | 加「上次結束在哪裡」+「開放的線」 |
| Session 結束才一次存檔，尾巴的討論會丟 | 改為段落式即時存檔 |
| 三種不同的問法才能問到不同的資訊 | 不管怎麼問，AI 一次講完全部 |
| AI 需要讀 5 個檔案才能拼出全貌 | 只讀進度總覽就能恢復 |
| 「等 Jin 確認方向後開始工作」太被動 | 改為主動報告 + 建議下一步 |

_v1 失敗案例：2026-02-24 session 1 結束後的「三種指令」討論未被存檔，導致下次 /clear 後 AI 完全找不到這段對話。v2 的「開放的線」機制專門解決這個問題。_
