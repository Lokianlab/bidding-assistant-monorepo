## 系統定位

**高階幕僚系統**——人負責決定和提議，機器負責規劃和執行。

不是自動駕駛（機器取代人），不是駕駛艙（人操作機器）。是整個團隊的支援層：降低每個人的工作負擔，讓人專注在判斷和創意上。

### 角色分工

| 角色 | 負責 | 系統幫他做什麼 |
|------|------|---------------|
| 行政人員 | 點選、確認、基本操作 | 一點就自動匯入建檔，不用手動抄寫 |
| 企劃人員 | 提議創意、精準表達 | 需要什麼情報上系統就有，不用自己找 |
| Jin（決策者） | 投不投、選方向、確認送出 | 整理好的選項和報告，快速決策 |
| 機器 | 分類、篩選、分析、規劃、執行、歸檔 | 自動跑，不需要人盯 |

```
        Jin（決策者）
         ↑ 整理好的選項和報告
    ─────────────────
    ↑                ↑
 行政人員          企劃人員
（點選操作）     （提議創意）
    ↑                ↑
    └── 機器支援層 ──┘
    （分類/篩選/分析/規劃/執行/歸檔）
```

### 核心原則

**企劃的品質取決於情報的紮實度。**

企劃人員的時間極度緊湊，不應花在補充情報、驗證正確性、分析策略上。企劃人員想要什麼資訊、想解答什麼問題，上系統就能找到答案。情報部門的產出不是原始資料，是**已分析、有結論、可直接用的行動指引**。

這決定了開發優先序的因果邏輯：
```
情報部門的紮實度 → 決定 → 企劃部門的產出品質 → 決定 → 投標命中率
```

## 系統三區塊

系統分成三個區塊，各自解決不同的問題：

```
設定區塊（地基）
  公司資料、知識庫、範本、串接
      ↓ 提供資料和設定
功能區塊（房子）
  情報、企劃、管理、首頁
      ↓ 需求驅動
開發區塊（施工隊）
  建設、測試、部署、維護

版本控制區塊（管家）── 橫切所有區塊
  誰改的、改了什麼、什麼時候、能不能回去
```

### 功能區塊 — 系統能做什麼

使用者每天碰的就是這一塊。

| 分區 | 包含 |
|------|------|
| 首頁（每日簡報） | 新案推薦、待決策、進行中、時限警示、最新結果 |
| 情報部門 | 巡標與發現、情報蒐集與分析、勝算評估、行政自動化、服務企劃、記錄歸檔（34 項，詳見下方） |
| 企劃部門 | 策略規劃、內容生產線、評選準備線、報價與資格、送件管理（24 項，詳見下方） |
| 管理層 | 投標績效總覽、敗因集中分析、情報準確度回顧、案件類型勝率分析、Ticket 追蹤總覽 |


### 設定區塊 — 系統認識誰、知道什麼

讓系統變成「我們公司的」幕僚。設好了就不用天天動。

| 分區 | 包含 |
|------|------|
| 公司基本資料 | 公司名稱、統編、資格證照、可投標領域、團隊成員角色 |
| 知識庫（00A-00E） | 00A 團隊成員、00B 公司實績、00C 時程範本、00D 應變 SOP/Q&A、00E 案後檢討 |
| 知識庫初始化與更新 | **初始化**：AI 讀 Drive 過去所有檔案，逐頁/逐段建卡片（文字摘要+分類標籤+縮圖），存為可搜尋資料庫。**增量更新**（不靠 AI 自動觸發）：(1) 事件觸發——新檔案放進 Drive 或 Notion 案件狀態改為結案時自動索引 (2) 手動觸發——「索引新檔案」按鈕 (3) 狀態看板——顯示已索引數量、最後索引時間、待索引數量，隨時知道有沒有加進去 |
| 範本管理 | Drive 資料夾結構範本、企劃書範本、簡報範本、文件範本 |
| 串接設定 | Notion API、Google Drive OAuth、PCC API、Perplexity（未來） |
| 巡標設定 | 關鍵字組合、排除條件、推薦靈敏度 |
| 使用者與權限 | 角色定義（決策者/行政/企劃）、各角色可見範圍和操作權限 |


### 開發區塊 — 系統怎麼進化

使用者不碰，開發者碰。

| 分區 | 包含 |
|------|------|
| 開發基礎建設 | Monorepo 架構、測試框架（Vitest）、CI/CD（GitHub Actions） |
| 開發工具 | 25 Skill 指令、5 Hook、記錄層、機器分工協調 |
| 品質保障 | 三級品質制度、閉環驗證、方法論體系 |
| 部署 | 三階段遷移（本地→上線→營運）、環境變數、資料庫安全 |

### 輸入輸出（I/O）區塊 — 進出自由

系統裡的任何東西——成品或半成品——都要能匯出和匯入。系統不是牢籠。

**匯出場景**：
- 情報報告寫到一半，需要寄給同事看
- 企劃書初稿要給外部顧問審
- 勝算評估要印出來開會用
- 簡報草稿要先在別的地方修改

**匯入場景**：
- 同事在外面寫好一段內容要放回系統
- 收到甲方補充的文件要加進案件
- 過去的企劃書要匯入知識庫

**格式支援**：PDF、DOCX、PPTX、純文字（至少）。

### 存檔機制 — 確保不會因中斷而損失

**核心目的**：不管是斷網、當機、不小心關掉、還是隔了三天才回來——打開就是上次離開的狀態，一個字都不丟。

具體技術方案待實作階段設計，但需求明確：
- 工作中的半成品要即時自動保存
- 案件的所有產出物要自動歸檔到正確位置
- 跨案件的知識要在結案時自動回流知識庫
- 使用者永遠不需要按「存檔」按鈕

### 版本控制區塊 — 誰改了什麼、能不能回去

橫切所有區塊的統一能力。核心邏輯一致：誰改的、改了什麼、什麼時候改的、能不能回去。

| 對象 | 說明 |
|------|------|
| 案件交付物 | 企劃書、簡報等每次修改自動存版本，可比對差異、回溯 |
| 知識庫與範本 | 更新時自動記錄變更歷史，可查閱、可還原 |
| 情報紀錄 | 情報蒐集的結果和分析，保留歷史版本 |
| 系統程式碼 | Git（已有） |

## 三層使用架構

系統同時服務三個層面的需求：

### 執行層（每天）

Jin 打開系統就像看報紙——今天有什麼案子、哪些值得看、挑了就開始做。

六個進入動機：
1. **有沒有新案子？** — 掃描發現新機會
2. **這個案子能不能做？** — 評估決策
3. **手上的案子做到哪了？** — 繼續工作
4. **什麼快到期了？** — 時限管理
5. **結果出來了，為什麼？** — 覆盤學習
6. **我要開始做這個案子了** — 啟動新專案

執行層是主角，系統預設入口。

### 管理層（定期）

定期回頭看全局——方向對不對、成績怎麼樣、哪裡要調整。

- 今年/去年/這個月/上個月的投標數量、勝率、趨勢
- 落標敗因集中分析（實績不足？方向偏了？遇到老對手？）
- 情報準確度回顧（勝算評估 vs 實際結果）
- 哪些類型的案子勝算較高，應該多投？
- 需要檢討或改進的地方
- **Ticket 追蹤總覽**：所有進行中案件的 Ticket 狀態——誰在等誰、哪裡卡住、哪些 Ticket 超時未完成。一眼看出瓶頸在哪

### 行政層（持續運作）

不面對 Jin，但支撐前兩層能運作。做兩件事：

**記錄與歸納**
- 每個案件的過程自動記錄（情報、決策、企劃、結果）
- 結案後自動歸納教訓
- 零散過程整理成結構化知識

**歸檔與儲存**
- 文件自動歸到正確位置（Notion、Google Drive）
- 知識庫持續更新（00A-00E）
- 歷史資料可搜尋、可引用

行政層標準：**Jin 永遠不需要自己整理、歸檔、找資料。**

### 三層如何循環

```
執行層（每天）
  做案子、產出成果
      ↓ 自動記錄
行政層（持續運作）
  記錄歸納、歸檔儲存
      ↓ 餵養
管理層（定期）
  看成績、看趨勢、找改善點
      ↓ 調整方向
執行層
  下一個案子做得更好
```

## 兩部門架構

### 情報部門

**使命**：讓企劃部門在動手之前，就知道該打哪裡、怎麼打、值不值得打。

#### 情報三層次

情報部門的工作本質是三個層次的遞進：

| 層次 | 做什麼 | 產出 | 對應功能 |
|------|--------|------|---------|
| **理解** | 搞清楚狀況是什麼 | 事實：案子、甲方、對手、評審、市場 | #1-8, #11, #14 |
| **預測** | 推斷接下來會怎樣 | 判斷：會不會贏、評審怎麼評、對手怎麼打 | #9-10, #12-13, #15-19 |
| **控制** | 用理解和預測來影響結果 | 行動指引：怎麼定位、強調什麼、避開什麼 | #25-28, #30 |

三層有嚴格的先後順序：理解不夠 → 預測就是猜的 → 控制就是亂打。

#### 偵察模式（Jin 決定前）

主動搜尋、分析、評估，產出三層情報：

| 層級 | 內容 | 舉例 |
|------|------|------|
| 表面資料 | 招標文件、預算、時程、資格門檻 | RFP 摘要、投標須知整理 |
| 分析洞見 | 競爭者關係、評委結構、機關偏好、實績壁壘 | 「得標者跟甲方合作 6 年、累計 394 萬」 |
| 潛在價值 | 市場空白、策略意涵、未被注意的機會或風險 | 「二二八高中教案在教育部資源網幾乎空白」 |

具體工作：
- 自動巡標（PCC API 關鍵字搜尋）
- 關鍵字推薦與優化
- 標案推薦（值得投的案件主動推送）
- 競爭者與甲方關係調查
- 評審委員結構分析
- 實績壁壘比對
- 案件策略意涵評估
- 勝算評估報告產出（5 項檢核表）

**→ Jin 決定：投 / 不投 / 有條件投**

#### 行政模式（Jin 決定後，全自動）

Jin 一鍵按下「投」，機器自動完成所有行政設置：
- 在資料庫（Notion）建案，填入所有已知資料
- 在 Google Drive 建立案件資料夾
- 套用範本、填入基本資訊
- 建立文件結構框架（時程、檢查清單、文件目錄）
- 情報持續補充（新資訊自動更新到案件頁）

**RFP 上傳處理**：行政人員上傳 RFP 一次（到 Notion），系統自動複製到 Google Drive 案件資料夾，同時解析並餵給兩個部門——情報部門產出 RFP 摘要與解讀（#14），企劃部門啟動評分項目對齊分析（#1）。後續任何環節需要 RFP，系統自動從對應位置拉取，不需要人再傳一次。

```
行政人員上傳 RFP
       ↓ 系統自動解析
   ┌───┴───┐
   ↓       ↓
情報部門  企劃部門
RFP 摘要  評分對齊
解讀分析  分析啟動
```

Jin 不需要碰任何設置工作。打開案件資料夾，一切就位，直接進入企劃階段。

### 企劃部門

**使命**：基於情報，產出對症下藥、有創意、能打動評審的企劃與評選簡報。

**企劃人員與情報部門的關係**：企劃人員高度依賴情報部門提供的資訊和建議。企劃工作非常緊湊，企劃人員的時間應該花在創意和精準表達上，不應再花在補充情報、驗證正確性、分析及制定策略。情報部門必須確保：企劃人員想要什麼資訊、想解答什麼問題，上系統都找得到。

**追問機制**：企劃過程中發現缺資料時，不需要離開系統或口頭聯繫——透過 Ticket 系統向情報部門發出追問需求（標明需要什麼、為什麼需要、什麼時候要）。情報部門補充完成後，成果自動回到企劃流程的對應位置，不中斷企劃人員的工作節奏。

**情報與企劃並行**：情報不需要全部蒐集完才開始企劃。截標日不等人，兩邊可以同時推進。當情報端發現重大更新（例如發現對手與甲方有深層關係），系統立刻通知企劃端，標出哪些已完成的內容可能受影響。由 Jin 決定：調整方向、繼續原路線、或放棄。

| 企劃人員的問題 | 情報部門應該備好的答案 |
|---|---|
| 這案子在要什麼？ | RFP 重點摘要、評分項目權重、甲方真正在意的事 |
| 誰是對手？ | 競爭者清單、實績、與甲方關係深度 |
| 評審是誰？ | 評委名單/推測、背景、偏好 |
| 我們的優勢？ | 自家 vs 對手實績比對、差異化空間 |
| 怎麼打？ | 寫作策略建議、重點配置 |
| 有沒有參考？ | 過去類似案件的企劃書、成功/失敗案例 |

#### 策略模式（Jin 決定前）

根據情報部門提供的資訊及機關需求：
- 評分項目對齊分析（三層解讀：字面要求 / 甲方真正在意 / 評審怎麼看）
- 產出 3-5 個企劃方向，各有策略邏輯及優劣勢比較
- 多輪收斂：人類挑選偏好方向並補充意見，機器產出新一輪提案，反覆直到方向確定
- 企劃方向確定後，若 RFP 有視覺交付物需求才觸發視覺方向提案（Mood Board → 主視覺）

**→ Jin 決定：經多輪收斂選定方向（+ 視覺方向，如需要）**

#### 流水線模式（Jin 決定後，機器接管）

**內容生產線**（並行分流 → 合流驗證）：
```
策略方向確定（含預算粗估）
    ↓ 分流
┌─────────┬─────────┬─────────┐
│  文字線  │  視覺線  │  數據線  │
│ 找材料   │ 找材料   │ 找材料   │
│ 組裝     │ 組裝     │ 組裝     │
│ 線內驗證 │ 線內驗證 │ 線內驗證 │
└────┬────┴────┬────┴────┬────┘
     └─────────┼─────────┘
               ↓ 合流
        預算細編 → 合流組裝
               ↓
        兩層驗證（零件 + 整體）
           ╱          ╲
        通過          不通過→退回對應的線
         ↓
      排版輸出（M06）
```

**評選準備線**（與內容生產線同步或接續）：
```
評選策略（根據情報部門的評委分析）
  → 簡報策略（重點配置、時間分配）
    → 簡報方向（視覺風格、敘事邏輯）
      → 簡報內容組裝（80% 知識庫 + 20% 創作，M08）
        → 簡報輸出（PPT/PDF）
          → 講稿生成
            → Q&A 預測 + 建議回答
```

**→ Jin 最終確認送出**

## 兩部門的互動

```
情報部門                                企劃部門

[偵察模式]
 巡標 + 推薦
 情報分析（含 RFP 摘要）
 勝算評估
    ↓
 ★ Jin: 投/不投
    ↓
[行政模式]
 一鍵建案 + RFP 上傳解析 ─────→ 評分對齊分析（同步餵入）
 Drive 資料夾
 範本 + 框架
    ↓
 情報持續補充 ──────→  [策略模式]
                       3-5 方向提案
                       多輪收斂
                           ↓
                       ★ Jin: 選定方向
                           ↓
                       [流水線模式]
                        ┌─ 內容生產線（並行分流）
                        │   文字線 / 視覺線 / 數據線
                        │   各線：找材料→組裝→線內驗證
                        │   需人介入 → Ticket 派工
                        │   合流 → 預算細編
                        │   兩層驗證（零件+整體）
                        │   排版輸出
                        │
                        └─ 評選準備線
                            評選策略 → 簡報組裝
                            簡報輸出 → 講稿 → Q&A
                                ↓
                            ★ Jin: 確認送出
                        │
 ◄── Ticket 追問 ──────┘  企劃缺資料 → Ticket 發追問
 情報補充 ────────────────→ 成果自動回到企劃流程
```

企劃與情報是**雙向互動**：情報主動餵養企劃，企劃也透過 Ticket 系統向情報發追問需求。

## 未來擴展：執行部門

等情報和企劃穩定運作後，第三個部門處理得標後的工作：

## 三種決標方式與流程差異

系統在案件進來時辨識決標方式，自動啟用對應的流程，不需要的步驟直接跳過：

| 決標方式 | 特徵 | 啟用的流程 | 不需要的流程 |
|---------|------|-----------|------------|
| 最有利標（須評選） | 評審打分+口頭簡報 | 完整流程：情報→策略→內容→評選簡報→送件 | — |
| 最有利標（書審） | 評審打分，只看書面 | 情報→策略→內容→送件（企劃書品質要求更高） | 評選準備線（不用簡報） |
| 最低標 | 合格區間內最低價得標 | 簡化流程：情報（報價導向）→資格文件→報價→送件 | 策略規劃、內容生產線、評選準備線 |

### 最低標專用流程

最低標不拼企劃品質，拼的是：(1) 文件 100% 合格 (2) 報價在合格區間內最低。

**機制**：
- 預算公開、底價不公開（開標才知道）
- 合格區間：底價的 80%～100%
- 高於底價 → 出局；低於底價八成 → 出局
- 合格區間內最低價得標

**情報部門（報價導向）**：

| 項目 | 做什麼 | 來源 |
|------|--------|------|
| 預算確認 | 公開預算金額 | RFP |
| 底價推估 | 查同機關/同類案件歷史底價折數（勞務案通常 93-99%） | PCC 歷史決標 |
| 合格區間推估 | 基於推估底價算出 80-100% 區間 | 計算 |
| 競爭者報價推估 | 過去同類案件的得標金額折數 | PCC 歷史決標 |
| 投標家數推估 | 同類案件通常幾家投標（越多價格越殺） | PCC 歷史資料 |
| 成本底線計算 | 做下來最少要花多少 | 知識庫 00B + 詢價 |

**企劃部門（簡化）**：

| 項目 | 做什麼 |
|------|--------|
| 資格文件盤點 | 確保 100% 合格（文件不齊直接出局） |
| 報價策略 | 三層折扣法：預算 × 底價折數 × 目標得標折數 |
| 成本 vs 報價分析 | 報價必須高於成本底線（不虧錢） |
| 風險評估 | 報太低（低於八成出局 / 虧錢）vs 報太高（輸給對手） |

**最低標決策流程**：

```
發現案件 → 辨識為最低標
    ↓
情報：預算確認 → 底價推估 → 合格區間 → 競爭分析 → 成本底線
    ↓
★ Jin 決定：投不投（勝算 + 利潤空間）
    ↓
資格文件盤點 + 確保合格
報價策略（三層折扣法 + 風險分析）
    ↓
★ Jin 決定：報多少
    ↓
送件（資格文件 + 報價單）
```

**實證參考**（漁村市集案，45 萬勞務案）：
- 預算 450,000、底價 445,000（預算的 98.9%）
- 5 家投標，各佔一個生態位：文件不合格出局、報太低出局、報太高出局、合格但非最低、合格且最低得標
- 得標價 427,200（底價的 96%），第二低 443,571（底價的 99.7%），差 16,371 元
- 詳細研究見 `docs/各種輸出文件/知識庫/報價策略/`

### 最有利標（書審）差異

跟須評選的差別：
- 評選準備線（第 9-12 項）整段跳過——沒有口頭簡報
- 企劃書品質要求更高——沒機會口頭解釋、沒有視覺衝擊補強、沒有 Q&A 回答追問
- 交叉驗證的第二層（整體驗證）更重要——評審不會給第二次機會
- 書審不是「少做一段」，是「企劃書那段要做得更狠」

## 企劃部門功能清單（以最有利標須評選為基準，20 項）

以下為完整流程。書審跳過評選準備線（第 9-12 項），最低標使用上方專用流程。

### 一、策略規劃（3 項，Jin 決定前）

| # | 功能 | 說明 | 自動化 |
|---|------|------|--------|
| 1 | 評分項目對齊分析 | 先搞清楚戰場。三層解讀每個評分項目：(1) RFP 字面要求 (2) 甲方真正在意什麼（來自情報 #7 機關檔案）(3) 評審會怎麼看（來自情報 #11-13 評委分析）。交叉比對競爭態勢與我方優劣勢，產出「我們要回答的問題」+差異化切入點 | 半自動 |
| 2 | 企劃方向提案（多輪收斂） | 基於分析結果產出 3-5 個企劃方向，各有策略邏輯及優劣勢比較。人類挑選偏好方向（可複選）並補充意見（選用），機器根據選擇+意見產出新一輪提案，反覆收斂直到方向確定 | 半自動 |
| 3 | 視覺方向提案 | 企劃方向確定後，RFP 有視覺交付物需求時觸發（AI 讀 RFP 判斷，不確定標出讓人確認）。分兩階段：選方向時產出 3 組 Mood Board（色彩、調性、參考圖），多輪收斂選定風格；方向確定後生成主視覺 2-3 版，選定後帶入內容生產線的排版輸出。技術用 AI 圖像生成，具體模型待實作定 | 半自動 |

使用者動作：Jin 看完選項，經多輪收斂後選定方向，再選定視覺方向（如需要）。

### 二、內容生產線（5 項，Jin 決定後機器接管）

兩個階段：內容循環（反覆到確定）→ 排版輸出。各階段的修改不互相干擾。

視覺只分兩種：**是內容的視覺**（海報、主視覺、圖表等交付物/說明素材）→ 進內容循環一起組裝和驗證；**是版面的視覺**（頁面背景、字體配色、模板）→ 排版階段處理。

**視覺線的外部工具路徑**：AI 生成的視覺素材不一定能直接用。當品質不夠好時，系統支援匯出→人在外部工具（Illustrator、Canva 等）修改→匯回系統的流程。匯出時產出 Ticket（標明規格需求和參考資料），完成後上傳回系統，自動放回視覺線的組裝流程。這結合了 I/O 區塊和 Ticket 機制。

內容循環不是單線進行，方向確定後可以並行，各線各自找材料→組裝→線內驗證，通過後再合流：

```
策略方向確定（含預算粗估，確認方向做不做得起）
    ↓ 分流
┌─────────────┬─────────────┬─────────────┐
│   文字線     │   視覺線     │   數據線     │
│ 找材料       │ 找材料       │ 找材料       │
│ 組裝         │ 組裝         │ 組裝         │
│ 線內驗證 ✓   │ 線內驗證 ✓   │ 線內驗證 ✓   │
└──────┬──────┴──────┬──────┴──────┬──────┘
       └─────────────┼─────────────┘
                     ↓ 內容大致穩定
               預算細編（詢價、行情、利潤、競爭力）
                ╱            ╲
           預算合理          超標→回去調整內容
                ↓
          合流組裝（接合處理）
                ↓
          全文交叉驗證（含預算 vs 內容一致性）
              ╱            ╲
           通過            不通過→退回對應的線
            ↓
         排版輸出
```

**預算兩階段**：
- **粗估**（策略規劃時）：抓大概、確認方向做不做得起、影響方向選擇
- **細編**（內容穩定後）：詢價、確認行情、評估利潤和競爭力、組裝預算表

預算細編的驗證重點：
- 數字正確性（有沒有算錯）
- 雙向對應（每個預算項目對應計畫裡的活動、每個活動都有編預算）
- 利潤率合理性
- 報價競爭力（參考 PCC 情報的類似案件得標金額）
- 格式符合 RFP 要求

**內容循環（第 4-7 項，反覆到內容確定為止）：**

| # | 功能 | 說明 | 自動化 |
|---|------|------|--------|
| 4 | 章節骨架生成 | 兩層結構：(1) 評分項目驅動的章節（權重高篇幅多）(2) 標準附件（經費預算表、進程規劃表、交付項目列表、風險管理方案、品質管控、人力配置、溝通機制等，不管 RFP 有沒有提都該有）。標準附件從知識庫 00C/00D 自動帶入 | 全自動 |
| 5 | 知識庫材料注入 | AI 讀 Drive 所有過去簡報/企劃書，逐頁/逐段建卡片（文字摘要+分類標籤+縮圖），存成搜尋資料庫。三種找法：(1) 自動推薦（根據當前案件類型和章節）(2) 關鍵字搜尋 (3) 按分類瀏覽。含文字和視覺素材 | 半自動 |
| 6 | 內容組裝 | AI 不主動寫，在人的決定下組裝：70% 組裝 + 20% 改寫 + 10% 創作。文字和視覺都在這裡組裝——海報、主視覺、識別系統、圖表、示意圖等視覺交付物也是內容 | 半自動 |
| 7 | 交叉驗證（兩層+攻防） | **第一層 零件驗證**：逐條對照 RFP 每條要求、評分每個指標、KPI、全文一致性（數字、時程、人名）。漏的標出、偏的標出。**第二層 整體驗證**：AI 模擬評審視角讀整份文件——有沒有回應甲方真正在意的事、有沒有完整的敘事邏輯（理解問題→解決方案→執行能力）、評審看完會不會覺得「這家公司懂我」。**攻防分析**：（優先）哪些是我方強項，能不能打磨更利；（其次）哪些是弱點，能不能補救→回去修企劃。兩層+攻防都通過才算過。不通過→回到第 5-6 項修改 | 半自動 |

**Ticket 派工機制（貫穿內容循環）：**

需要人介入的工作（視覺設計、詢價報價、文字撰寫等），系統自動辨識並產出 Ticket。

Ticket 結構：
- **派工區**：做什麼、什麼規格、什麼時候交、參考資料連結（案件頁的情報和策略）
- **驗收區**：完成者填「我完成了什麼」，系統自動產出驗收清單（數量、規格、對齊 RFP 第 X 條、對齊評分第 Y 項、跟其他部分有無矛盾），驗收者逐項確認，判定通過或需修改

驗收清單由系統根據本案 RFP 和策略分析自動生成。不預設驗收者全程參與——任何人接手都能照清單驗收。完成的成果上傳後自動放回組裝流程。

**排版輸出（內容確定後）：**

| # | 功能 | 說明 | 自動化 |
|---|------|------|--------|
| 8 | 排版輸出 | 內容確定後，M06 套用公司模板，文字+視覺+格式合一，產出 PPT/PDF（服務建議書也用 PPT/PDF，不用 Word——PPT 排版能力強太多）。排版/美學/格式問題在此階段迭代調整（改的是版面，不是內容） | 半自動 |

### 三、評選準備線（4 項）

Q&A 攻防在企劃階段就已完成（內容生產線第 7 項交叉驗證的攻防分析）：強項打磨更利、弱點能補就補→回去修企劃。到簡報階段運用這些結論。

| # | 功能 | 說明 | 自動化 |
|---|------|------|--------|

## 開發順序：三刀切法

不按功能模組切，按**系統成熟度**切。每一刀交付的是兩個部門的能力提升。

### 第一刀：情報部門 MVP

**核心目標**：做對投/不投的決定。就算後續不再擴充，光是這一點就能改善整體的時間投入和命中率。

**定位**：情報部門的完整 MVP。企劃部門（寫企劃、做簡報）留在第二刀。

**三塊建設內容**：

**A. 情報自動拉取**（解決痛點：每案手動查情報要數小時）
- 情報 #7 機關檔案：PCC 自動拉該機關採購歷史
- 情報 #8 競爭者分析：自動拉過去得標者、連續年數、金額
- 情報 #14 RFP 摘要：上傳 RFP 後 AI 自動摘要評分項目和重點
- 情報 #16 Perplexity 研究鏈：系統生成假設驗證式提示詞，結果回存
- 情報 #17-19 勝算評估：五項檢核，能自動填的填、不能的標紅，產出紅黃綠燈

**B. 一鍵建案**（解決痛點：手動建案+開檔+填資料要 30 分鐘）
- 情報 #20 一鍵建案：PCC 資料灌入 Notion
- 情報 #21 Drive 資料夾：自動建結構、套範本
- 情報 #23 範本預填：已知資料自動填入

**C. 知識庫卡片索引**（解決痛點：找過去素材翻找 20 分鐘）
- 企劃 #5 知識庫材料注入（只做「找到」，不做「搬入」）
- AI 讀 Drive 過去檔案、逐頁建卡片
- 三種找法：自動推薦、關鍵字搜尋、分類瀏覽
- 找到後點開原檔手動複製（第二刀再做自動搬入）

**驗收標準**：

| 區塊 | 痛點 | 現狀 | 第一刀後 |
|------|------|------|---------|
| A 情報拉取 | 每案手動查情報 | 數小時 | 30 分鐘 |
| B 一鍵建案 | 手動建案+開檔+填資料 | 30 分鐘 | 1 分鐘 |
| C 知識庫索引 | 翻找過去的素材 | 20 分鐘/次 | 10 秒/次 |

**驗收方式**：拿一個真實案件，Jin 從頭走一遍。三個場景都比現在快才算通過。

**部署方案**：

```
Jin 的電腦（永遠開著）
  ├─ Next.js Web App
  ├─ Claude Code（用現有訂閱登入，AI 費用 $0）
  └─ Cloudflare Tunnel → 對外提供 https 存取

Supabase（免費）— 知識庫卡片索引 + 情報快取 + 勝算評估
Notion — 案件管理（現有）
Google Drive — 檔案存放（現有）
```

四層定位：
- Jin 的電腦：運算（Web App + Claude Code）
- Notion：案件管理
- Drive：檔案存放（鷹架範本也在這裡，可隨時更新）
- Supabase：搜尋和快取

月費：$0。AI 費用：$0（走 Claude 訂閱）。需要：域名約 $10/年 + Cloudflare 免費帳號。

⚠️ Supabase 免費方案限制：7 天無使用自動暫停。解法：定時 ping 保活。
⚠️ 電腦重開機/斷網時服務中斷。解法：開機自動啟動設定。

**鷹架和結構都是設定，不是程式碼**：
- 鷹架範本放在 Drive 固定位置，隨時可改，系統建案時讀取當下最新版
- 案件內部結構可更新
- 命名規則：({案件唯一碼})({截標日期}) {案名}
- 程式碼只記位置（Drive folder ID），不記內容

### 第二刀：企劃部門 MVP

情報部門跑順之後：
- 寫作策略生成（情報 → 企劃方向選項）
- Claude Code SDK 串接（策略 + 知識庫 + 情報 → 企劃骨架）
- 品質閘門串接（自動檢查對齊度）
- 評選準備線（簡報策略 → 簡報生成 → 輸出）

驗收標準：Jin 選完方向後，企劃書初稿 + 評選簡報初稿自動產出，Jin 只需微調。

### 第三刀：深化 + 飛輪

- 情報全自動化（Perplexity 研究鏈無需人工）
- 知識庫 00A-00E 完整初始化
- 結案回流機制
- 勝算模型自我校準（歷史數據驗證）

## 設計原則

1. **人做決策，機器做其他一切**
2. **情報品質 > 產出速度**——方向錯了，做再快也沒用
3. **先跑通再做深**——薄但完整的流程 > 深但斷裂的模組
4. **驗證驅動**——每一刀都用真實案件驗證，不做沒人用的功能
