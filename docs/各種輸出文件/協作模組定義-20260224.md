# 協作模組定義

**來源**：2026-02-24 Jin 與 Claude 實際互動過程提煉
**原則**：每個模組是一個可重複使用的人機協作單元，有明確的輸入、過程、產出。
**用途**：下次啟動不從頭聊，直接調用對應模組。

---

## 模組清單

### M-ENV｜環境診斷與修復

**觸發**：系統連不上、功能不正常
**輸入**：錯誤訊息或症狀描述
**過程**：
1. AI 查設定檔（.env.local、middleware、API route）
2. AI 直接測試連線（curl / node script）
3. 定位根因
4. 修復並驗證
**產出**：問題修復 + 根因紀錄
**今日實例**：Notion token 差一個字、OAuth 擋住開發模式、API fallback 缺失

---

### M-SCAN｜資料庫全景掃描

**觸發**：需要了解整體資料狀態
**輸入**：資料庫連線資訊
**過程**：
1. AI 拉取全量或取樣資料
2. 統計分布（狀態、類型、預算帶、機關、地區）
3. 識別模式與異常
4. 產出結構化報告
**產出**：數據分析報告（.md）
**今日實例**：Notion 8000 筆標案資料庫深度分析

---

### M-FUNNEL｜決策漏斗分析

**觸發**：需要理解轉換率和瓶頸
**輸入**：M-SCAN 的原始數據
**過程**：
1. 定義漏斗各層（進入 → 篩選 → 決策 → 行動 → 結果）
2. 計算每層轉換率
3. 識別最大流失點
4. 交叉分析流失原因（預算、類型、時間、地區）
**產出**：漏斗分析 + 瓶頸識別 + 改善建議
**今日實例**：8000 → 791 → 136 → 16 → 0 的斷崖分析

---

### M-ASSET｜資產盤點

**觸發**：開始新階段前，需要知道手邊有什麼
**輸入**：專案目錄
**過程**：
1. AI 掃描檔案結構
2. 分類（提示詞、程式碼、文件、設定）
3. 評估每個資產的狀態（可用 / 需更新 / 已過時）
4. 識別資產之間的關係（依賴、衝突、重複）
**產出**：資產清單 + 狀態評估
**今日實例**：22 個提示詞檔案 + 8 階段遞進鏈 + Notion 模板內 9 階段 SOP

---

### M-COMPARE｜工具/方案比較

**觸發**：有兩種以上做法，需要決定用哪個
**輸入**：兩個方案 + 同一個測試案例
**過程**：
1. 用同一個案例分別跑兩個方案
2. 逐維度對比產出品質
3. 識別各自的強項和盲區
4. 得出結論：取代 / 互補 / 各有適用場景
**產出**：比較報告 + 決策建議
**今日實例**：PCC API vs Perplexity 情報品質比較（史多禮案例）
**Jin 修正**：AI 原判「可取代」→ 修正為「互補」

---

### M-PROBE｜技術可行性驗證

**觸發**：不確定某個技術方案是否可行
**輸入**：想驗證的假設
**過程**：
1. 寫最小測試程式
2. 直接跑，看結果
3. 不猜，測了再說
**產出**：可行 / 不可行 + 具體限制
**今日實例**：Notion API 能不能讀頁面區塊內容？→ 可以，讀到了完整結構

---

### M-CASE｜案例搜尋與選取

**觸發**：需要具體案例來做分析或測試
**輸入**：搜尋條件（關鍵字、公司、機關、時間範圍）
**過程**：
1. PCC API 搜尋標案
2. 交叉比對（我方 vs 競爭者的共同案件）
3. Notion 搜尋確認內部紀錄
4. 選定最適合的案例
**產出**：案例基本資訊 + 雙方投標紀錄 + Notion 頁面 ID
**今日實例**：找到二二八教案（序位第 2 未得標）+ 人權走讀（三方競爭）

---

### M-READ｜Notion 頁面深度讀取

**觸發**：需要讀取某個案件的完整 Notion 資料
**輸入**：Notion page ID
**過程**：
1. 讀取頁面屬性（欄位值）
2. 遞迴讀取所有區塊（heading、paragraph、file、toggle、code 等）
3. 識別附件 PDF 的 URL
4. 提取結構化摘要
**產出**：頁面完整內容 + 附件清單 + 結構摘要
**今日實例**：二二八教案頁面（招標文件、摘要、指令集、評選結果）

---

### M-INTEL｜競爭者情報組裝

**觸發**：需要了解特定競爭者的實力
**輸入**：公司名稱
**過程**：
1. PCC API 搜尋全部投標紀錄（search_by_company）
2. PCC API 搜尋得標紀錄（search_by_winner）
3. 統計：投標數、得標數、主要機關、金額範圍、案件類型
4. 與 Perplexity 深度報告結合（如有）
**產出**：競爭者情報摘要
**今日實例**：史多禮 107 筆投標紀錄 + Perplexity 579 行深度報告

---

### M-SAVE｜進度存檔

**觸發（AI 主動，不等人類提醒）**：
- 完成一個有意義的段落
- 做了重要決策或認知修正
- 討論主題切換時
- 每 20-30 分鐘（即使沒有明確段落結束）
- 要結束 session

**輸入**：當前 session 的對話內容（推理過程、修正過程、決策脈絡）
**過程**：
1. 紀錄對話的實際推理鏈（不只是結論，包含怎麼推到這個結論的）
2. 紀錄認知修正（AI 原本想什麼、人類怎麼修正、修正後的認知是什麼）
3. 紀錄決策脈絡（為什麼選 A 不選 B，當時的考量是什麼）
4. 列出待討論事項
5. 寫入文件

**產出**：session 紀錄（.md），存到 docs/各種輸出文件/
**核心原則**：結論可以重推，但推理過程不可重現。存的是過程，不只是結果。
**Jin 指示**：「固定時間或固定階段就要存起來，中斷損失很大」

**今日實例**：人機協作紀錄-20260224-session1.md

---

### M-ASSEMBLE｜偵察模板組裝

**觸發**：確定要對某個案件進行情報分析
**輸入**：M-READ 的 Notion 頁面內容 + M-INTEL 的 PCC 數據
**過程**：
1. 從 Notion 提取案件基本資訊（案名、機關、預算、截標日）
2. 從 PCC 提取機關歷史標案、得標者、投標家數
3. 填入 P 偵察報告模板
4. 輸出給用戶，用戶貼到 Perplexity
**產出**：填好的 Perplexity 偵察模板

### M-L1｜戰略分析執行

**觸發**：用戶貼回 Perplexity 報告
**輸入**：Perplexity 報告 + Notion 招標文件 + PCC 數據 + L1 提示詞
**過程**：
1. 系統自動識別是哪個案件（從報告中的案名/案號比對）
2. 組裝完整 L1 輸入（偵察報告 + RFP + 系統核心 + 階段索引）
3. 執行 L1 戰略分析 SOP（Step 1-8）
4. 產出戰略分析報告 + 投標決策建議
**產出**：L1 戰略分析報告（.docx）

## 模組調用順序（典型場景）

### 場景 A：新 session 啟動
```
M-SAVE（讀取上次紀錄）→ 接回工作
```

### 場景 B：系統出問題
```
M-ENV → 修好 → 繼續原本工作
```

### 場景 C：想了解某個競爭者
```
M-INTEL（PCC 搜尋）→ M-COMPARE（對比我方）→ M-SAVE
```

### 場景 D：對一個新標案做情報分析
```
M-CASE（確認案件）→ M-READ（讀 Notion）→ M-INTEL（查競爭者）
→ M-ASSEMBLE（組裝偵察模板）→ 用戶跑 Perplexity → M-L1（戰略分析）
→ M-SAVE
```

### 場景 E：全局盤點
```
M-SCAN → M-FUNNEL → M-ASSET → M-SAVE
```

---

## 設計原則

1. **每個模組可獨立運行**，不強制綁定順序
2. **模組之間通過文件傳遞**，不依賴 session 記憶
3. **人類在關鍵節點介入**（審閱、修正、決策），AI 不自作主張
4. **每次互動都存檔**，下次不從頭開始
5. **AI 的判斷會被修正**，修正本身也是模組的一部分（紀錄在 M-SAVE 裡）
