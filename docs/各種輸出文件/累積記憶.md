## Jin 的核心指示（永久有效）

1. 給判斷，不要丟選擇題
2. 不要套用爛模板，品質提升 > 速度提升
3. 每一步都要存，過程即產出
4. 我們互動的過程就是系統的原型
5. Google Drive 串接不能跳過（一鍵上新 = Notion + Drive 同時）
6. Notion 模板 Step 0-9 是舊版參考，正式體系用 L1-L8
7. 每一個環節都是瓶頸，不只是某一段——時間太多、效益太少（0225）
8. 系統定位 = 高階幕僚系統：人做決策，機器做決策前的準備和決策後的執行（0225）
9. 記錄順序（硬性）：先更新日誌（記錄討論過程），再更新指引文件（修改結論）。日誌先行，不能反過來（0225）

## 系統架構決策（0225 確立）

**高階幕僚系統**，兩部門架構，取代 P0-P4：
- **情報部門**：偵察模式（巡標+分析+勝算評估）→ 行政模式（一鍵建案+Drive+範本）
- **企劃部門**：策略模式（方向選項）→ 流水線模式（內容生產線+評選準備線）
- **Jin 只決策三次**：投不投、選方向、確認送出
- **未來擴展**：執行部門（履約+結案+知識回流）
- 詳見 `docs/各種輸出文件/系統架構指引-幕僚系統.md`
- 開發順序：第一刀情報部門 MVP → 第二刀企劃部門 MVP → 第三刀深化+飛輪

## Perplexity 使用方法論（從二二八教案案例提煉）

### 核心原則

提示詞結構：已知線索 → 假設驗證 → 追蹤指令 → 意外發現區。
不要平行提問，要設計成調查鏈。假設驗證讓 Perplexity 從「列事實」升級到「建構因果鏈」。

### 三個使用階段，三種不同的提示詞

**階段一：L1 偵察（投標前）— 問「要不要投」**
- 目的：評估勝算，決定投不投
- 種子資料：案號、機關名、預算、過去得標者（PCC API 先跑）
- 假設方向：「我們懷疑這個機關有固定合作廠商」「我們懷疑某競爭者在此領域有不可取代的實績」
- 關鍵輸出：競爭者與甲方關係深度、實績壁壘、勝算評估（投/不投/有條件投）
- 時機：拿到招標公告後、決定投標前

**階段二：L5 評選準備（投標後）— 問「怎麼贏」**
- 目的：針對評審可能的偏好調整簡報策略
- 種子資料：評審須知的評分項目、已知的外聘委員背景（從同類標案的決標公告推測）
- 假設方向：「我們懷疑外聘委員來自 X 領域，他們會在意 Y」「甲方最近的重點業務方向是 Z」
- 關鍵輸出：委員可能的偏好、甲方近期策略方向、簡報重點建議
- 時機：投標截止後、簡報前

**階段三：檢討（落標後）— 問「為什麼輸、下次怎麼判斷」**
- 目的：理解敗因，提煉可複用的判斷框架
- 種子資料：評選結果（分數、序位）、委員名單、得標者資訊、我方企劃書的方向
- 假設方向：「我們懷疑得標者與甲方有 N 年合作關係」「我們懷疑我方企劃偏離了評分重點」「我們懷疑評審團組成結構性偏向甲方既有夥伴」
- 關鍵輸出：敗因拆解（實績壁壘/關係網絡/企劃方向/評審結構）、可複用的勝算評估檢核表
- 時機：收到評選結果通知後
- 追蹤指令重點：「追查得標者與甲方所有公開合作紀錄」「追查評審委員的專業網絡」「找出得標者的未得標紀錄（反駁過度推論）」

## 投標勝算評估檢核表（從二二八教案案例提煉）

投標前完成以下五項，任一項紅燈就要慎重考慮不投：

1. **查甲方近三年同類案件得標者** — 同一廠商連續 3 年以上得標 → 替換成本偏好極高（紅燈）
2. **查過去決標公告的評審委員名單** — 有些機關會公開，有些不會。有公開的 → 整理「常見評審委員庫」，推測本次可能的評審組成，據此調整企劃書方向和簡報策略。沒公開的 → 本身就是風險因子，代表你對評審偏好完全盲目（黃燈）
3. **查評審團結構** — 機關內部委員 > 50% → 評選高度受既有偏好主導（黃燈）
4. **比對競爭者特化實績** — 對手在精確交叉領域有 ≥3 個直接案例而我方為 0 → 勝算不到兩成（紅燈）
5. **判斷案件策略意涵** — 對甲方具先驅性意義 → 傾向選「最安全」而非「最有創意」的廠商（黃燈）

### 評審委員情報的時間點（Jin 指示 0225）

**法規依據**：《採購評選委員會組織準則》——「機關公開委員名單者，公開前應予保密；未公開者，於開始評選前應予保密。」公不公開由機關決定。政府電子採購網有「採購評選委員名單」查詢功能。

**三個情報來源，由早到晚**：
1. **招標公告/招標文件**：部分機關直接公開委員名單（最早、最有價值）
2. **同機關過去的決標公告**：列出過去案件的委員名單（可推測本次組成，因內部委員幾乎不換）
3. **決標後公告**：本案的委員名單（最晚，只能用於檢討）

**L1 偵察必做事項**：
- 查政府電子採購網，看本案招標公告有沒有公開委員名單
- 查同機關過去 3 年的決標公告，整理「常見評審委員庫」
- 有名單 → 對委員做背景調查，據此調整企劃書方向
- 沒名單 → 列為風險因子，盲投風險更高
- **不是等送出建議書之後才開始想評委是誰**

## 技術發現（踩過的坑）

- **Notion formula/rollup 欄位**不能放進 API 的 filter_properties，會報錯。解法：resolvePropertyIds 自動排除 + 失敗降級重試。（s1 0224）
- **Notion token** 差一個字（82→86）導致長期連不上，排查時要逐字比對。（s1 0224）
- **Middleware** 在開發模式需要跳過 OAuth，否則本機測試全擋。（s1 0224）

## 案例分析洞見

### 二二八教案（11402-39，序位第 2）
- 輸在「執行計畫與預期成果」（30%，最高權重）：企劃書偏學術分析，缺乏具體成品預覽
- 企劃書的問題不是寫得不夠多，而是方向偏了——分析教科書侷限花了太多篇幅，評審想看的是「做出來長什麼樣」
- 廠商實績不對口：文化活動 ≠ 教案開發，評審中有 2 位高中老師
- 甲方執行長只給 80 分（差一分淘汰），唯一給第 1 的是一位高中老師
- 啟示 1：AI 輔助備標的重點 = 確保每一頁都對著評分項目寫，不是快速產出更多內容
- 啟示 2：備標時要優先研究機關首長最在意什麼，他的分數往往決定生死
- 啟示 3：史多禮（故事 StoryStudio）跟基金會有既有合作（出版、共生音樂節、後門咖啡對談），2018 年就做過人權教案（175 萬），兩座金鼎獎，臺史博限制性招標常客。信任基礎 + 硬實績 > 企劃品質
- 啟示 4：L1 戰略分析應包含「競爭者與甲方既有關係」調查，這決定要不要投
- 啟示 5：Perplexity 能挖到的東西（共生音樂節交叉、後門咖啡對談、行政院手冊案）比 PCC API + 網頁搜尋更深。在 L1 偵察報告中 Perplexity 的角色不可替代
- 補充：基地文化核心是原住民族文史，跟二二八教案不對口，這解釋了第 3 名
- 詳見 `二二八教案評選分析-20260224.md`
