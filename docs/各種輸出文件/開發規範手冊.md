## 二、模組化原則

> **來源**：以下原則來自 `bidding-assistant/CLAUDE.md`，是現有規範。

### 2.1 單一來源原則（Single Source of Truth）

所有常數、狀態值、欄位名稱**只在一個地方定義**，其他地方一律透過 import 使用。

| 資料類型 | 唯一定義位置 |
|----------|-------------|
| 標案狀態值 | `src/lib/constants/bid-status.ts` |
| Notion 欄位名稱 | `src/lib/constants/field-mapping.ts` |
| Notion API 欄位清單 | `src/lib/constants/notion-fields.ts` |
| 功能模組定義 | `src/lib/modules/feature-registry.ts` |
| AI 階段定義 | `src/data/config/stages.ts` |
| 知識庫矩陣 | `src/data/config/kb-matrix.ts` |
| 應用程式設定 | `src/lib/settings/types.ts` + `defaults.ts` |

**禁止**：在元件中硬編碼字串常數（狀態名稱、欄位名稱、顏色值、路徑、設定值）。

### 2.2 Feature Registry Pattern

所有功能模組在 `FEATURE_REGISTRY`（`src/lib/modules/feature-registry.ts`）中註冊。每個功能有：

```typescript
interface FeatureDefinition {
  id: string;              // 唯一識別碼
  name: string;            // 中文名稱
  description: string;     // 功能描述
  icon: string;            // emoji 圖示
  routes: string[];        // 對應的 URL 路徑
  section: "core" | "tools" | "output";  // 分類
  defaultEnabled: boolean; // 預設是否啟用
  dependencies?: string[]; // 依賴的其他功能 ID
}
```

功能開關狀態由 `isFeatureEnabled()` 統一判斷（結合 registry 預設值 + 使用者設定）。側欄、路由守衛、設定頁面全部從 registry 讀取。

凡是「可新增同類項目的集合」都用 Registry 陣列模式：
- 功能模組 → `FEATURE_REGISTRY`
- 儀表板卡片 → `CARD_REGISTRY`
- 統計指標 → `METRIC_REGISTRY`
- AI 階段 → `STAGES`

新增項目只需在 Registry 陣列中加一筆，不需修改渲染邏輯。

### 2.3 三層分離

```
src/lib/          ← 資料層（商業邏輯、Hook、工具函式、型別）
src/components/   ← 展示層（UI 元件，只負責渲染，不含商業邏輯）
src/app/          ← 頁面層（串接資料層和展示層）
```

**資料層** `src/lib/{feature}/`：
- `types.ts` — 介面定義（必要）
- `constants.ts` — 功能專屬常數（選用）
- `helpers.ts` — 純函式，可獨立測試（選用）
- `use{Feature}.ts` — React Hook，資料取得與計算（選用）
- `index.ts` — 匯出彙整（選用）
- `__tests__/` — 測試檔案（必要）

**展示層** `src/components/{feature}/`：
- 元件只透過 props 或 Hook 接收資料
- 不直接呼叫 API 或操作 localStorage

**頁面層** `src/app/`：
- 串接資料層和展示層
- API Route 放在 `src/app/api/`

**禁止**：在頁面檔中寫商業邏輯或硬編碼設定。

### 2.4 新模組資料夾結構規範

每個新功能模組**必須**遵循以下結構：

```
src/lib/{feature}/
  types.ts        ← 介面與型別定義（必要）
  index.ts        ← 匯出彙整（選用）
  constants.ts    ← 功能專屬常數（選用）
  helpers.ts      ← 純函式（選用）
  use{Feature}.ts ← React Hook（選用）
  __tests__/      ← 測試檔案（必要）

src/components/{feature}/
  {Component}.tsx  ← UI 元件（只透過 props 接收資料）

src/data/config/   ← 靜態設定資料（選用）
```

### 2.5 設定系統

- **型別定義**：`src/lib/settings/types.ts` 的 `AppSettings` 介面
- **預設值**：`src/lib/settings/defaults.ts` 的 `DEFAULT_SETTINGS`
- **存取方式**：`useSettings()` Hook（`src/lib/context/settings-context.tsx`）
- **儲存位置**：localStorage key `"bidding-assistant-settings"`

新增設定時必須：
1. 在 `AppSettings` 介面中加型別（用 `?` optional，讓舊資料自動 fallback）
2. 在 `DEFAULT_SETTINGS` 中加預設值
3. 透過 `updateSettings()` 或 `updateSection()` 更新

### 2.6 命名規則

| 類型 | 規則 | 範例 |
|------|------|------|
| 資料夾 | kebab-case | `case-setup/`, `knowledge-base/` |
| 型別檔 | `types.ts` | `src/lib/intelligence/types.ts` |
| 常數檔 | `constants.ts` | `src/lib/intelligence/constants.ts` |
| Hook | `use{Feature}.ts`，camelCase | `useIntelligence.ts` |
| 元件 | PascalCase | `PccPanel.tsx`, `WinAssessment.tsx` |
| API Route | `route.ts` 放在路徑資料夾中 | `src/app/api/intelligence/[caseId]/route.ts` |
| 測試檔 | `*.test.ts` 放在 `__tests__/` | `__tests__/helpers.test.ts` |

## 三、第一階段新模組規範

> **來源**：以下為第一階段新增規範。

### 3.1 四個新模組的職責邊界

| 模組 | 職責 | 不做的事 |
|------|------|---------|
| `intelligence` | PCC 情報拉取、RFP 解析、勝算檢核、Perplexity 提示詞 | 不做案件建立、不做知識庫搜尋 |
| `case-setup` | 一鍵建案（Notion 建案 + Drive 建資料夾 + 鷹架複製 + 範本預填） | 不做情報分析、不做文件生成 |
| `knowledge-base` | Drive 檔案掃描、文字提取、AI 卡片生成、全文搜尋 | 不做案件相關操作 |
| `decisions` | 記錄投/不投決策和理由 | 不做分析判斷 |

### 3.2 模組間依賴規則

```
intelligence → case-setup    （情報頁的「投」按鈕觸發建案）
intelligence → decisions     （情報頁的決策面板記錄投/不投）
knowledge-base 獨立          （不依賴其他新模組）
所有新模組 → supabase        （資料存取）
```

依賴方向是**單向**的：`case-setup` 和 `decisions` 依賴 `intelligence`，反過來不行。`intelligence` 不 import `case-setup` 或 `decisions` 的任何東西。

### 3.3 Supabase 表的使用規範

| Supabase 表 | 存取模組 | 用途 |
|-------------|---------|------|
| `intelligence_cache` | `intelligence`（透過 `supabase/intel-client.ts`） | PCC 情報快取，7 天有效 |
| `knowledge_cards` | `knowledge-base`（透過 `supabase/cards-client.ts`） | 知識卡片（摘要、標籤、來源連結） |
| `decisions` | `decisions`（透過 `supabase/decisions-client.ts`） | 投/不投決策記錄 |

**規則**：
- 每張表只有一個模組的 client 檔負責 CRUD
- 其他模組需要讀取，透過該 client 檔的函式，不直接寫 SQL
- Supabase client 的 `supabase/intel-client.ts`、`supabase/cards-client.ts`、`supabase/decisions-client.ts` 是資料存取的唯一入口

### 3.4 API Route 命名規範

```
/api/intelligence/[caseId]     ← PCC 情報拉取（GET）+ 快取更新（POST）
/api/intelligence/rfp-parse    ← RFP 上傳解析（POST）
/api/case-setup                ← 一鍵建案（POST）
/api/knowledge-base/search     ← 知識庫搜尋（GET）
/api/knowledge-base/scan       ← 手動觸發索引（POST）
/api/knowledge-base/status     ← 索引狀態（GET）
/api/decisions                 ← 決策記錄 CRUD（GET/POST）
/api/dashboard                 ← 首頁資料彙整（GET）
```

命名規則：
- 用 kebab-case
- 資源名在前，動作在後（`/knowledge-base/search`，不是 `/search/knowledge-base`）
- 動態路由用 `[param]` 語法（`/intelligence/[caseId]`）

## 四、程式碼規範

### 4.1 TypeScript 嚴格模式

> **來源**：現有規範。

TypeScript 設定為嚴格模式。所有新檔案必須有型別定義，禁止 `any`（除非有明確理由並加註釋說明）。

```typescript
// 正確
function calculateScore(items: WinCheckItem[]): number { ... }

// 錯誤
function calculateScore(items: any): any { ... }
```

### 4.2 UI 文字使用繁體中文

> **來源**：現有規範。

所有面向使用者的文字必須使用繁體中文。包括：
- 按鈕文字
- 標籤文字
- 錯誤訊息
- placeholder
- toast 通知

```typescript
// 正確
<Button>開始分析</Button>
toast.error("情報拉取失敗，請稍後再試");

// 錯誤
<Button>Start Analysis</Button>
toast.error("Failed to fetch intelligence");
```

### 4.3 匯入路徑

> **來源**：現有規範。

一律使用 `@/` 路徑別名，不用相對路徑：

```typescript
// 正確
import { useSettings } from "@/lib/context/settings-context";
import { FEATURE_REGISTRY } from "@/lib/modules/feature-registry";
import { logger } from "@/lib/logger";
import type { IntelligenceReport } from "@/lib/intelligence/types";

// 錯誤
import { useSettings } from "../../lib/context/settings-context";
import { logger } from "../../../lib/logger";
```

### 4.4 禁止硬編碼

> **來源**：現有規範 + 第一階段補充。

以下項目禁止直接寫在程式碼中：

| 項目 | 正確做法 |
|------|---------|
| 字串常數（狀態名稱、欄位名稱） | 定義在 `constants.ts` |
| 顏色值 | 用 Tailwind class 或 `constants.ts` |
| API endpoint URL | 定義在模組 `constants.ts` |
| 設定值（閾值、比率） | 放 `AppSettings` + `DEFAULT_SETTINGS` |
| 環境相關路徑 | 用 `process.env` |
| 魔術數字 | 給有意義的常數名稱 |

```typescript
// 正確
// src/lib/intelligence/constants.ts
export const WIN_CHECK_THRESHOLDS = {
  GREEN: 70,
  YELLOW: 40,
} as const;

// src/lib/intelligence/win-assessment.ts
import { WIN_CHECK_THRESHOLDS } from "./constants";
if (score >= WIN_CHECK_THRESHOLDS.GREEN) { ... }

// 錯誤
if (score >= 70) { ... }
```

### 4.5 設定系統的使用方式

> **來源**：現有規範。

讀取設定：

```typescript
import { useSettings } from "@/lib/context/settings-context";

function MyComponent() {
  const { settings } = useSettings();
  const threshold = settings.qualityGate?.overallPassThreshold ?? 70;
  // ...
}
```

更新設定：

```typescript
const { updateSection } = useSettings();

// 更新某個區段
updateSection("qualityGate", {
  ...settings.qualityGate,
  overallPassThreshold: 80,
});
```

新增設定欄位時，**必須**用 `?` optional 並在 `DEFAULT_SETTINGS` 加預設值，這樣舊使用者的 localStorage 不會壞掉。

### 4.6 日誌記錄規範

> **來源**：現有規範。

關鍵操作必須透過 `logger` 記錄：

```typescript
import { logger } from "@/lib/logger";

// API 呼叫
logger.info("api", "PCC 情報拉取完成", `案件: ${caseId}, 耗時: ${ms}ms`);
logger.error("api", "PCC 情報拉取失敗", error.message);

// 快取操作
logger.debug("cache", "情報快取命中", `案件: ${caseId}`);

// 設定變更
logger.info("settings", "知識庫設定已更新");

// 使用者操作
logger.info("system", "使用者決定投標", `案件: ${caseName}`);
```

| 級別 | 使用時機 |
|------|---------|
| `debug` | 快取命中/未命中、內部流程細節 |
| `info` | API 呼叫成功、設定變更、使用者關鍵操作 |
| `warn` | 非致命問題（快取過期、部分資料缺失） |
| `error` | API 呼叫失敗、解析失敗、不可預期的錯誤 |

### 4.7 Hydration 安全

> **來源**：現有規範。

任何讀取 localStorage 的程式碼都必須用 `useEffect` + `hydrated` flag，避免 SSR/CSR hydration 不一致：

```typescript
const [hydrated, setHydrated] = useState(false);

useEffect(() => {
  setHydrated(true);
}, []);

if (!hydrated) return <Skeleton />;
```

### 4.8 錯誤處理

> **來源**：第一階段新增。

所有外部服務呼叫必須有錯誤處理：

```typescript
// API Route 中
export async function GET(request: Request) {
  try {
    const data = await fetchPccData(caseId);
    return Response.json(data);
  } catch (error) {
    logger.error("api", "PCC 情報拉取失敗", String(error));
    return Response.json(
      { error: "情報拉取失敗，請稍後再試" },
      { status: 500 }
    );
  }
}
```

外部服務呼叫需要重試機制：

```typescript
async function fetchWithRetry(url: string, maxRetries = 3): Promise<Response> {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      const response = await fetch(url);
      if (response.ok) return response;
      if (attempt === maxRetries) throw new Error(`HTTP ${response.status}`);
    } catch (error) {
      if (attempt === maxRetries) throw error;
      await new Promise((r) => setTimeout(r, 1000 * attempt)); // 指數退避
    }
  }
  throw new Error("不應到達此處");
}
```

## 六、外部服務存取規範

### 6.1 Notion API

> **來源**：現有規範 + 第一階段補充。

- **讀取**：可以讀取使用者的原始 Notion 資料庫（案件清單、案件詳情）
- **寫入**：一鍵建案時寫入 Notion。第一階段只有 `case-setup` 模組會寫入
- **禁止**：直接修改現有案件的 Notion 屬性（除非明確設計為此用途）
- **實驗/測試**：一律用沙盒或複製的資料庫

### 6.2 Google Drive API

> **來源**：第一階段新增。

- **讀取**：知識庫掃描（列出資料夾內所有檔案）
- **建立**：一鍵建案時建立案件資料夾
- **複製**：鷹架範本複製到新案件資料夾
- **配額注意**：每次 API 呼叫之間加 200ms delay，避免觸發限速

```typescript
// Drive API 呼叫間隔
const DRIVE_API_DELAY_MS = 200;

async function batchDriveOperation(items: string[]) {
  for (const item of items) {
    await driveClient.copy(item);
    await new Promise((r) => setTimeout(r, DRIVE_API_DELAY_MS));
  }
}
```

### 6.3 Supabase

> **來源**：第一階段新增。

- **用途**：搜尋 + 快取 + 結構化資料存取
- **存取方式**：透過模組專屬的 client 檔（`intel-client.ts`、`cards-client.ts`、`decisions-client.ts`）
- **禁止**：元件直接 import Supabase client 做查詢
- **免費方案注意**：7 天無活動會暫停，需 cron job 每 3 天 ping 一次

```typescript
// 正確：透過 client 函式存取
import { getIntelCache } from "@/lib/supabase/intel-client";
const cached = await getIntelCache(caseId);

// 錯誤：直接使用 supabase client
import { supabase } from "@/lib/supabase/client";
const { data } = await supabase.from("intelligence_cache").select("*");
```

### 6.4 PCC API（政府採購網）

> **來源**：第一階段新增。

- **存取方式**：直接 HTTP 呼叫，不走 MCP Server
- **原因**：Web App 的 API Route 可以直接發 HTTP request，不需要 MCP 中間層
- **快取策略**：查詢結果存 `intelligence_cache`，7 天有效
- **不穩定處理**：重試 3 次、失敗時顯示「部分情報拉取失敗」不阻塞其他功能

```typescript
// src/lib/intelligence/pcc-client.ts
import { PCC_API_BASE } from "./constants";

export async function fetchTenderInfo(tenderId: string) {
  const response = await fetchWithRetry(`${PCC_API_BASE}/tender/${tenderId}`);
  return response.json();
}
```

### 6.5 Claude Code（本機呼叫）

> **來源**：第一階段新增。

- **用途**：RFP 解析（從 PDF 提取結構化資訊）、知識庫 AI 分類（生成摘要和標籤）
- **呼叫方式**：透過 CLI 呼叫本機的 Claude Code
- **重試機制**：失敗時重試，最終失敗時降級（保留原始檔案連結，標記「未解析」）
- **不做即時對話**：這不是聊天機器人功能，是批次處理

## 七、資料安全

### 7.1 核心規則

> **來源**：現有規範（硬性規定）。

1. **禁止直接修改原始資料庫**（Notion 等）。實驗、測試、開發一律使用沙盒或複製的資料庫
2. **API token 一律走環境變數**。禁止硬編碼或存入 localStorage
3. **`.env.local` 不進 git**。已在 `.gitignore` 中排除
4. **鷹架範本是設定不是程式碼**——Jin 在 Drive 裡修改範本，程式碼不需要動

### 7.2 環境變數

所有環境變數放在 `.env.local`，變數名稱列在 `.env.example`：

```bash
# 現有
NOTION_TOKEN=                         # Notion API token
NOTION_DATABASE_ID=                   # Notion 案件資料庫 ID
GOOGLE_CLIENT_ID=                     # Google OAuth client ID
GOOGLE_CLIENT_SECRET=                 # Google OAuth client secret
GOOGLE_REFRESH_TOKEN=                 # Google OAuth refresh token
GOOGLE_SHARED_DRIVE_FOLDER_ID=        # Drive 備標集中區 folder ID
NEXT_PUBLIC_SUPABASE_URL=             # Supabase URL
NEXT_PUBLIC_SUPABASE_ANON_KEY=        # Supabase anon key
SUPABASE_SERVICE_ROLE_KEY=            # Supabase service role key

# 新增（第一階段）
GOOGLE_SCAFFOLD_FOLDER_ID=            # Drive「BZ. 製標鷹架」folder ID
GOOGLE_KB_FOLDER_IDS=                 # 知識庫索引範圍（逗號分隔的 folder IDs）
```

### 7.3 API Route 安全

> **來源**：現有規範。

API 路由（`/api/*`）必須加入基本認證檢查。具體實作視部署環境決定（Cloudflare Tunnel 層級或應用層級）。

### 7.4 部署前檢查清單

> **來源**：現有規範。

PR 提交前必須檢查：

- [ ] 無硬編碼路徑（`grep -r "C:\\|^\\.\/|^~\/" src/`）
- [ ] 跨平台相容性（`npm test && npm run build`）
- [ ] 資料存取經 API（確認無 createClient 直連）
- [ ] 機密存 .env（`git diff HEAD~1 | grep -i "key\|secret\|token"`）

## 八、Git 規範

### 8.1 Commit 格式

> **來源**：現有規範。

```
[type] 一句話描述（機器碼）
```

type 值：
| type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修 bug |
| `test` | 測試 |
| `review` | 程式碼審查 |
| `infra` | 基礎設施 |
| `admin` | 管理性變更 |

範例：
```
[feat] 新增 PCC 情報快取 Supabase schema（A44T）
[fix] 修正 Drive API 配額限速問題（Z1FV）
[test] 補 win-assessment 邊界條件測試（ITEJ）
```

### 8.2 Commit 規則

- 先 `npm test` + `npm run build` 通過再 commit
- 不 force push（`git push --force` 禁止）
- 每個 commit 做一件事，不混雜不相關的改動

### 8.3 推送

- 推送用 `bash .claude/hooks/push-retry.sh`（自動 fetch + rebase + push，3 次重試）
- session 最多推送 2-3 次（中段 + 收尾），admin 改動攢一批

## 九、部署

### 9.1 架構

> **來源**：第一階段新增。

```
Jin 的電腦（本機）
  ├─ Next.js dev server（port 3000）
  ├─ Cloudflare Tunnel → 外部可存取的 HTTPS URL
  └─ Claude Code CLI（RFP 解析、知識庫分類用）

外部服務
  ├─ Supabase（PostgreSQL + 全文搜尋）
  ├─ Notion API
  ├─ Google Drive API
  └─ PCC API
```

### 9.2 開機自動啟動

用 Windows Task Scheduler 設定：
1. `cloudflared tunnel run` — Cloudflare Tunnel
2. `npm run dev`（在 `bidding-assistant/` 目錄）— Next.js dev server

### 9.3 Supabase 保活

Supabase 免費方案 7 天無活動會暫停。設定 cron job 每 3 天 ping 一次：

```bash
# 簡單的 ping（curl Supabase REST endpoint）
curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/" \
  -H "apikey: $NEXT_PUBLIC_SUPABASE_ANON_KEY" > /dev/null
```

暫停後手動進 Supabase console 喚醒即可，資料不會丟失。

## 十、常見陷阱

### 10.1 Supabase 免費方案 7 天暫停

**問題**：7 天無 API 呼叫，Supabase 會自動暫停專案。知識庫搜尋和情報快取全部失效。

**解法**：cron job 每 3 天 ping。失效時手動進 console 喚醒。

### 10.2 PCC API 偶爾不穩

**問題**：政府採購網 API 偶爾回應慢或 timeout。

**解法**：
- 快取機制（`intelligence_cache` 表，7 天有效）
- 重試 3 次（指數退避）
- 失敗時顯示「部分情報拉取失敗」，不阻塞其他功能

### 10.3 PPTX 解析品質不一

**問題**：PowerPoint 特效多的頁面，文字提取結果可能不完整或格式混亂。

**解法**：
- 只提取文字，不保證格式
- 掃描 PDF 如果無法解析，標記「無法解析」顯示在狀態看板
- 保留原始檔案連結，讓使用者可以手動查看

### 10.4 Claude Code CLI 呼叫需重試機制

**問題**：本機 Claude Code 呼叫可能因各種原因失敗（程序未啟動、回應超時等）。

**解法**：
- 重試機制（最多 3 次）
- 最終失敗時降級為不解析（保留原始檔案連結，標記「未解析」）
- 用 `logger.error` 記錄失敗原因

### 10.5 localStorage hydration 安全

**問題**：Next.js SSR 階段 `localStorage` 不存在，直接存取會導致 hydration mismatch。

**解法**：所有讀取 localStorage 的地方都用 `useEffect` + `hydrated` flag 包裹（見 4.7 節）。

### 10.6 Drive API 配額限制

**問題**：短時間大量呼叫 Drive API 會觸發限速。知識庫初始化掃描特別容易中招。

**解法**：
- 每次 API 呼叫間加 200ms delay
- 初始化分批跑
- 背景執行 + 進度看板顯示進度

### 10.7 知識庫初始化耗時

**問題**：第一次建索引需要掃描所有 Drive 檔案、提取文字、AI 分類，可能需要數小時。

**解法**：
- 背景執行（不阻塞 UI）
- 進度看板即時顯示
- 已完成的部分可以立即搜尋，不需等全部完成
