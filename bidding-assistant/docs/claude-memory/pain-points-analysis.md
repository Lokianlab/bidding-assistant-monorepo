# 痛點深度分析（2026-02-17 討論成果）

## 商業背景

- 公司：大員洛川，15 人，16+ 年經驗
- 核心業務：政府標案企劃（課程/導覽/走讀/傳統服飾/綠色照護）
- 年營收目標：NT$3,000 萬
- 去年實績：投標 46 案，總預算金額 NT$105,748,770（平均 NT$2,298,886，中位數 NT$1,128,305）
- **得標 0 件，0 元** — 0% 得標率
- 標案類型：大部分是**最有利標**（評分比並，建議書品質重要）
- 收入公式：收入 = 投標案件數 × 得標率 × 平均金額 × 毛利率

## 現行工作流程（痛苦的）

```
招標文件
  → Perplexity 查資料
  → Gemini 想創意
  → Claude 寫初稿
  → Nanobanana 生主視覺
  → 人肉縫合各工具產出
  → 拉舊案當底稿，修修補補
  → 排版（建議書 + 評選簡報）
  → 送件
  → 落標
  → 沒有系統性地學習
  → 下次重來...

## 完整痛點清單（23 條）

### 選案階段
1. **篩案靠感覺和興趣** — 沒有可驗證的篩選機制，可能投了不該投的案

### 戰略階段
21. **沒有深入的戰略分析就開始寫** — 不了解機關潛台詞、競爭優勢、贏的切入角度
22. **缺乏戰略性的企劃架構** — 沒有核心論述、故事線、決勝點判斷，建議書變成資料堆疊

### 拆解階段
2. **需求書沒有系統拆解** — 寫漏評分項目，有些要求沒寫到
3. **沒有結構化的寫作順序** — 不知道先寫什麼後寫什麼，亂寫一通

### 寫作階段
4. **知識庫是空的（00A-00E 未建立）** — AI 只能寫空話，沒有公司實績/團隊/時程/SOP 可用，缺乏信賴感
5. **在 4-5 個 AI 工具間跳來跳去** — Perplexity/Gemini/Claude/Nanobanana，不知做到哪了
6. **多案並行，大腦不斷切換** — 同時寫 2-3 案，每案在不同工具的不同對話裡，要自己記住所有狀態
7. **每個 AI 都要重新調校** — 花大量時間「教 AI 認識你」，調好了又到對話上限
8. **對話上限是黑箱** — 不知道何時會爆，爆了交接文件又不準（記憶已被多次壓縮），工作成果消失
9. **各工具產出不一致** — 要自己當人肉膠水縫合，Gemini 的創意跟 Claude 的初稿對不上
10. **AI 幻覺抓不完** — 校稿時很難全部發現，被評委問到答不出來
11. **AI 不自檢、死不認錯** — 生完就辯護，要修很難修，你變成 AI 的品管員
23. **AI 的「創意」沒有常識** — 聽起來厲害但不切實際（如：社區活動用 AR、50 人活動說大數據），缺乏實務判斷

### 產出階段
12. **寫完還是像初稿** — 品質上不去
13. **主視覺/示意圖跟文字脫節** — 圖文不搭，不夠專業。需要圖文並茂讓評委在腦海中建構執行實況
14. **排版很痛** — 建議書和評選簡報的排版都是痛點

### 結案階段
15. **落標意見回收不系統** — 只能泛泛地問「哪裡要精進」，評審也只能泛泛回答，參考價值低
16. **每次從頭來，經驗沒累積** — 做完的案件經驗沒有回流到系統

### 策略層問題
17. **實績冷啟動困境** — 沒得標 → 沒實績 → 評分被扣 → 繼續沒得標（雞生蛋問題）
18. **質 vs 量策略問題** — 46 案全投但品質都不夠，是否應該集中火力在少數案件？
19. **不夠了解評委真正在看什麼** — 不知道評委 30 分鐘翻完建議書時的閱讀行為和關注重點
20. **不了解競爭對手** — 不知道贏的是誰、他們的建議書長什麼樣

## 關鍵洞察

### 原開發計畫書的問題
1. **是功能清單不是產品** — 列了 10 個批次幾十個功能，但沒有回答「使用者的一件事是什麼」
2. **想取代太多工具** — Notion 的案件管理、pcc 監控、Google Calendar、文件管理、儀表板... 15 人公司不需要自建這些
3. **核心價值被埋沒** — L1-L8 提示詞系統是公司護城河，但在計畫裡只是 Batch D 的一個小項目
4. **結案回饋閉環排在最後（Batch J）** — 但它應該是最早做的，因為飛輪越早轉越好

### 系統的真正定位
- ❌ 不是儀表板
- ❌ 不是看板
- ❌ 不是 RAG 聊天機器人
- ❌ 不是專案管理工具
- ✅ **是一個提案寫作駕駛艙：從招標文件進去，一份可以上場的建議書出來**

### 系統要做的事
1. 一個地方拆解需求
2. 一個地方做戰略分析和企劃架構
3. 一個地方規劃寫作順序
4. 一個地方調度所有 AI（研究/創意/寫作/圖像）
5. 一個地方存放公司真實資料（團隊/實績/時程/SOP）
6. 一個地方追蹤進度（哪些寫了、哪些沒寫）
7. 一個地方做品質檢查（每句話能追溯來源、抓幻覺、抓不切實際的承諾）
8. 一個地方處理排版（建議書 + 評選簡報）
9. 一個地方回收落標經驗
10. 永遠記得你的公司、團隊、過去經驗，不需要重新調校

## 6 個系統能力（對症下藥，2026-02-17 定版）

### 能力 1：戰略分析引擎
從招標文件到作戰計畫，一條龍：
① 拆解：評分項目、交付要求、資格條件、應備文件
② 分析：適配度、決勝點、紅旗、評委關注點
③ 決策：投 or 不投（可驗證的依據）
④ 作戰計畫：核心論述 + 攻略策略 + 寫作順序 + 每項所需資料
→ 解決：#1, #2, #3, #19, #20, #21, #22（7 條）

### 能力 2：公司知識庫
飛輪的燃料，結構化存放 + 向量化索引：
- 00A 團隊、00B 實績、00C 時程、00D 應變、00E 檢討、00F 創意庫
- AI 寫作時自動注入最相關的知識
- 案件結束後自動回流更新
→ 解決：#4, #16, #17, #23（4 條）

### 能力 3：統一 AI 調度中心
- 每案一個工作區（對話/產出/進度都在一起）
- 自動選 AI、帶入知識庫 + 本案脈絡 + 作戰計畫
- 不需重新調校，系統永遠記得
- 對話上限自動處理精準交接
- 多案並行：獨立工作區，一鍵切換
→ 解決：#5, #6, #7, #8, #9（5 條）

### 能力 4：AI 品質閘門
每段產出經過三道檢查：
- 閘門 1 事實查核：有知識庫來源嗎？
- 閘門 2 需求對照：回應了哪個評分項目？
- 閘門 3 實務檢驗：預算做得到？對象用得了？公司有能力？
→ 解決：#10, #11, #12, #23（4 條）

### 能力 5：視覺生成整合
- 寫到需要圖的地方，系統建議並生成對應圖像
- 一案一套視覺風格，圖文同源
→ 解決：#13（1 條）

### 能力 6：排版輸出 + 落標學習
排版：建議書/簡報一鍵匯出，套用公司模板
落標學習：結構化分析每案結果，累積後校準篩案機制
→ 解決：#14, #15, #18（3 條）

### 核心結論
> **L1-L8 在 prompt 層設計了約一半痛點的解法，但因缺乏系統支撐，
> 實際上幾乎沒有一條被真正解決。**
>
> 系統是「手動流程的管理介面」—— 能讓你看到資料、複製 prompt，
> 但不能幫你思考、幫你寫、幫你檢查。
>
> 就像設計了完美的健身課表（L1-L8），但沒有教練盯著做（系統不追蹤執行）、
> 沒有器材（知識庫是空的）、沒有場地（每次自己找地方搬器材）、
> 做完沒記錄（沒有回饋循環）。課表再好，這樣練不出肌肉。

## 行政流程原貌（2026-02-18 庭瑜流程，修正版）

### pcc 採購網費用結構
- 預覽包：NT$600/月/帳號，可看 200 件
- 公司買多個帳號（200*N），因為每月都用滿額度
- 預覽包文件 = 完整文件集，但法律上不能用來投標
- 正式領標：另外付費/件，取得投標資格（內容相同）

### 完整漏斗（數據確認版 v2）

```
pcc 瀏覽（多帳號 200*N）
    ↓ 行政初篩（標準：案名 vs 公司業務範圍）
600-1200 件/月 通過初篩
    ↓ 行政第一輪（每件都做）：
    │   ├─ 貼表：案件資訊填入 Notion
    │   └─ 貼檔：下載 → 清理文件 → 製作備標評估文件 → 嵌入 Notion
    ↓
    ↓ 企劃初選（標準：看 Notion 上的案件摘要）
    ↓
~200 件/月 選入
    ↓ 行政第二輪（選入的才做）：
    │   ├─ 開 Drive 資料夾結構
    │   └─ 將備標評估文件貼入 Drive
    ↓
    ↓ 企劃 + AI 深度分析
    ↓
~?? 件/月 做 L1 深度分析
    ↓ 決策會議
~4 件/月 決定投標（46件/年）→ 正式領標 → L2-L8
```

### 行政每日工作流（四動作，每件都做）

每件案子（600-1200 件/月）庭瑜都執行四個動作：
1. **貼表** — 案件資訊（案號/案名/機關/預算/截標日等）填入 Notion

所有情報都彙整在 Notion 案件頁面中，供企劃初選用。

⚠️ **關鍵發現：情蒐 = 手動版 P 偵察報告（查決標記錄 + 地點資訊），每月做 600-1200 次。**

## 企劃端深度診斷（2026-02-18）

### 企劃端現行做法
- 企劃認領案子後，各自想辦法在截標前交出建議書
- 沒有統一流程，每人用不同 AI（Gemini/Claude/GPT）
- 做法：把備標評估文件整份丟給 AI →「寫一份完整建議書」→ 拿到初稿 → 自己改
- 完全跳過 L1 戰略分析、L2 備標規劃，直接衝到「寫完」
- 沒有審稿、沒有品質閘門，寫完就送

### 資源衝突問題
- 每天有新案子出現，企劃可能認領新案
- 新案 vs 在跑的案子搶時間 → 進程打架
- 三種結局：半途放棄（浪費投入）/ 硬投卡位（品質不足）/ 趕出來但品質只是交差
- 團隊長期爭論「精 vs 量」：精做怕全滅成本高、量衝怕品質不夠
- 實際結果：精也不精、量也不量，不上不下 → 46 案 0 得標

### 根因分析
```
量衝不起來 ← 每案手工作業太重，流程無法規模化
質提不上去 ← 跳過分析直接寫，AI 產出的是沒有靈魂的初稿
兩頭卡住 ← 流程本身不支援任何一種策略
```

兩種策略都需要但都缺乏的基礎建設：
- 數據驅動的篩選機制（L1 投標決策矩陣）
- 公司知識庫（00A-00E，AI 寫作的彈藥）
- 競爭情報（P 偵察報告）
- 結構化流程（L1-L8，而非一次性讓 AI 生完）
- 品質閘門（審稿、檢核清單）

### 解法：案件分級 + 對應策略

```
200 件/月（企劃選入）
    ↓ L1 快速分析（每案 30 min）
    ↓
├── A 類（高勝算）→ 全力做 L1-L8 → 精
├── B 類（中勝算）→ 精簡流程 → 量
├── C 類（可能獨家）→ 最低限度 → 撿漏
└── D 類（低勝算）→ 放棄 → 省時間
```

精和量不是二選一，是依案件分級同時進行。

## 執行端深度診斷（2026-02-18）

### 執行端現況

- Notion 上有完整的「執行企劃作業流程工作手冊」（30 個節點、5 階段）
- **但未貫徹，不確定可行性** — 跟 L1-L8 一樣：設計完整，落地為零

- 目前在跑的案子來自**老客人主動回頭**，不是新標案得標

### 執行流程設計（Notion 原始結構）

```
階段零：前置啟動（0A 交付清單+期程, 0B 創意發想）
階段一：規劃/報價/打樣
  ├── Part A: 概念與報價（1A 設計→1B 訪價→1C 報價→1D 提案→1E 決策閘門）
  └── Part B: 打樣與驗收（1F 打樣→1G 審核→1H 決策閘門→1J 簽約）
階段二：量產與前置準備（雙軌平行）
  ├── 物資流：2A 下單→2B 追蹤→2C 驗收→5A 理貨→5B 打包→5C 物流
  └── 人力流：3A 需求→3B 招募→4 工作手冊→3C 教育訓練
階段三：現場執行
  6 通告單→7A 進場→7B 執行→7C 突發處理→8 核銷確認→7D 清點→7E 撤場
階段四：結案與核銷
  9A 物資歸位→9B 費用結算→9C 專案檢討→9D 核銷報告
```

設計品質很高（有決策閘門、雙軌並行、Key Output），問題不是設計不好，是沒有落地。

### 三端全貌

| 端 | 設計 | 落地 | 產出 | 人力 |
|----|:----:|:----:|------|------|
| 行政端 | ✅ | ✅ 貫徹 | 穩定（600-1200 件/月） | 5 位助理 |
| 企劃端 | ✅ L1-L8 | ❌ 未部署 | 46 案 0 得標 | 同一批人 |
| 執行端 | ✅ 30 節點 | ❌ 未貫徹 | 差強人意（靠老客戶） | 同一批人 |

### 根因：同一批人扛企劃+執行

```
同一批人的時間
├── 寫新案建議書（企劃端）← 每天有新案搶注意力
├── 跑已得標案的交付（執行端）← 有硬 deadline
└── 哪邊著火救哪邊 → 兩邊都做不好
```

### 飛輪斷裂點

```
執行端未貫徹
  → 9C 專案檢討不做 → 00E 案後檢討永遠空
  → 執行品質差強人意 → 00B 實績寫不出好案例
  → 知識庫沒有好彈藥 → 下次建議書更難寫
  → 得標更難 → 惡性循環
```

### 黃偉誠成功建制行政端的模式

```
混亂 → 設計流程 → 找到對的人 → 反覆調校 → 穩定運作
```

企劃端和執行端需要走同樣的路，但更難：
- 行政端 = 規則性工作（看案名分類、轉檔），流程化直覺
- 企劃端 = 判斷性工作（投不投、怎麼寫才贏），需要外化決策邏輯
- 執行端 = 交付性工作（專案管理、供應商、現場），需要外化管理經驗

### 關鍵洞察

1. **設計能力不缺**——三端都有完整設計（行政SOP/L1-L8/30節點流程）
2. **落地能力是瓶頸**——只有行政端成功落地
3. **行政端成功的關鍵因素**：有專職人力（5位助理）、工作可切割、結果可量化
4. **企劃+執行失敗的關鍵因素**：人力共用、互搶資源、結果難量化、救火模式

    └── 簡報官 Agent（Claude, L5-L8 prompt + context）
          ↓
    ┌─────┴─────┐
    │  Notion DB │  知識庫（00A-00F）
    └───────────┘
```

#### 每案的 Agent 上下文

每案在 Forum 帖開始時建立一個 Case Context：
- 招標文件內容
- P 偵察報告（如有）
- L1 分析結果（完成後加入）
- L2 規劃結果（完成後加入）
- 知識庫相關片段（知識官每次查詢後更新）
- 所有 Agent 共享這個 Case Context，不需重新調校

#### 成本控制

- 所有 Agent 走 Claude Max 訂閱（CLI），不走 API 計費
- 同一案件的 Agent 共享 Case Context，避免重複傳送知識庫
- 輕量查詢（/kb search, /case info）不啟動 Agent，直接查資料庫回傳

## Discord Bot 規格（2026-02-18 初版，持續更新）

### 定位
- **Discord = 日常操作介面（駕駛座），Web app = 管理後台（儀表板）**
- 團隊已在使用 Discord，bot 嵌入既有環境
- 支援：上傳文件 → 下指令 → 分析/寫作 → 輸出文件，全流程在 Discord 完成

### 頻道結構

```
📂 標案系統（新增分類）
├── #標案公告        一般頻道｜bot 單向推播（新案通知、截標提醒）
├── #標案指令台      一般頻道｜通用指令（查知識庫、系統操作、上傳新案文件）
└── 📋 標案工作區    Forum 頻道｜每案一帖
         標籤：🔵新案 🟡分析中 🟠寫作中 🔴審查中 ✅已投標 ⚫結案
```

- 每案一個 Forum 帖（自動建立），所有分析/寫作/文件都在帖內
- Forum 天然支援標籤過濾、格狀瀏覽、自動歸檔 → 輕量案件看板
- 結案帖標籤改 ⚫結案，不用搬頻道

### Discord vs Web app 分工

| 功能 | Discord | Web app |
|------|---------|---------|
| 上傳/輸入文件 | **主要入口** | 備用 |
| 下指令（L1-L8） | **主要** | 不做 |
| 查案件/進度 | **快速查** | 全局儀表板 |
| 知識庫查詢 | **快速查** | 完整編輯 |
| 輸出文件 | **主要出口** | 備用下載 |
| 統計/分析報表 | ❌ | **主要**（圖表） |
| 系統設定 | ❌ | **主要** |
| 知識庫維護（00A-00E） | ❌ | **主要** |
