### 1.2 三大業務主軸

1. **課程、導覽及走讀服務**：公衛教育、人權教育、食農教育、國家語言教育、藝文展覽策劃
2. **臺灣傳統服飾製作與推廣**：型制史研究、復原設計、品牌轉譯
3. **綠色照護與永續發展**：環境教育、森林保健、親子互動教育、長照新模式

### 1.3 收入公式（第一性原理）

```
收入 = 得標案件數 × 平均案件金額 × 毛利率

其中：
  得標案件數 = 投標案件數 × 得標率
  投標案件數 = 發現的機會數 × 投標決策通過率
  得標率 = f（建議書品質、簡報品質、價格競爭力、團隊適配度）
  毛利率 = f（報價精準度、履約成本控制、風險管理）
```

系統的所有功能都必須能回答：**這個功能推動了公式中的哪一項？**

### 1.4 完整業務鏈

```
標案發現 → 投標決策 → 備標 → 企劃撰寫 → 定稿排版 → 送件
    → 評選準備 → 評選 → 得標 → 履約執行 → 結案 → 收錢
                                                      ↓
                    ←──── 經驗累積回饋（知識庫更新） ←──┘
```

### 2.3 人機協作斷點機制

提示詞系統已內建斷點設計。AI 在關鍵決策點暫停，等待人類確認後才繼續：

```
斷點 A：輸入確認 — 「前序結論整理如上，確認無誤後開始」
斷點 B：結構確認 — 「章節架構如上，有無需要調整？」
斷點 C：內容審閱 — 「企劃稿全文如上，請審閱」
斷點 D：預算審閱 — 「報價表與品質檢查結果如上，請確認」
斷點 E：最終確認 — 「一致性檢核結果如上，確認後產出文件」
```

**系統的角色**：把這些斷點變成可追蹤的狀態（待處理/已通過），讓團隊所有人可以看到每個案件卡在哪個斷點。

### 2.4 知識庫維護循環

T1 知識庫維護精靈（915 行）定義了完整的知識庫更新流程：

```
案件結案
  ↓
PM 填寫 00E 案後檢討筆記
  ↓
上傳給 T1 精靈
  ↓
Phase D：精靈掃描 00E，建立待辦清單
  ├→ 更新 00B（新增實績）
  ├→ 更新 00A（成員新經歷/新進人員建檔）
  ├→ 修正 00C（時程範本偏差 >30% 時修正）
  ├→ 修正 00D（新風險情境 / SOP 步驟修正）
  └→ AI 工具回饋整理
  ↓
知識庫更新完成 → 下一個案件的 L1-L8 品質更好
```

**系統的角色**：在驗收日自動提醒 PM 填寫 00E → 追蹤知識庫更新狀態 → 版本控制。

## 3. 五個角色的需求與痛點分析

### 3.1 管理者（黃偉誠）

| 需求 | 痛點 | 因果鏈 |
|------|------|--------|
| 掌握公司投標全貌 | 無法即時看到得標率、金額趨勢、案件分布 | → 策略決策靠直覺而非數據 → 可能投了不該投的案、錯過該投的案 |
| 評估企劃人員效益 | 沒有量化指標（每人負責案件數、得標率、產出速度） | → 無法公平評估 → 團隊管理靠感覺 |
| 控制公司獲利 | L3 做完利潤分析後，履約階段的實際成本沒有人追蹤回來比對 | → 不知道哪種案件實際利潤高 → 無法優化投標策略 |
| 年度策略調整 | 缺少歷史數據分析（哪些類型案件得標率高、哪些機關常合作） | → 每年重新摸索 |

### 3.2 行政（庭瑜）

| 需求 | 痛點 | 因果鏈 |
|------|------|--------|
| 發現新標案 | 手動刷 pcc 網站、怕漏掉 | → 錯過投標機會 → 直接影響「發現的機會數」 |
| 建立案件 | 手動將標案資訊打到 Notion | → 重複勞動、可能打錯 |
| 追蹤截標日 | 腦記 + 手動提醒團隊 | → 漏提醒 → 來不及投標 → 廢標 |
| 送件確認 | 文件齊全度 checklist 每次重做 | → 遺漏文件 → 廢標風險 |
| 協調團隊進度 | 要一個個問企劃「做到哪了」 | → 浪費溝通時間 |

### 3.3 企劃人員

| 需求 | 痛點 | 因果鏈 |
|------|------|--------|
| 快速啟動 AI 協作 | 開新 Claude 對話要重新組裝提示詞、貼知識庫、附前序文件 | → 每次啟動浪費 15-30 分鐘 → 累積後吃掉大量有效工時 |
| 系統故障時能繼續工作 | Claude Code 掛掉就停擺 | → 需要 prompt 模板庫作為容災方案，切到 Claude.ai 網頁版繼續 |
| 管理多案件並行 | 同時進行 3-4 個案件，容易混淆哪案到哪階段 | → 拿錯文件、漏做步驟 |
| 前序文件串接 | L3 要用 L1/L2 的產出，散落在不同對話和資料夾 | → 找檔案浪費時間 → 可能用到舊版本 |
| 團隊審閱協作 | L3 斷點 C/D 需要報價人員、主管審閱 | → 用 LINE 傳 .docx、等回覆 → 流程不透明 |
| 知識庫查詢 | 寫建議書時要查過往實績、團隊資料 | → 翻 .md 檔案不方便 |

### 3.4 設計師

| 需求 | 痛點 | 因果鏈 |
|------|------|--------|
| 清楚知道要做什麼 | 收到一大份文件，要自己找 `【美編製作】` 標記 | → 容易漏做 |
| 知道優先序 | 不確定哪個設計先做、截標日是哪天 | → 問企劃 → 增加溝通成本 |
| 交付設計稿 | 完成的設計稿不知道放哪、怎麼回傳 | → LINE 或 email → 版本混亂 |
| 理解設計需求 | 文字描述的設計需求不夠具體 | → 反覆修改 |

### 3.5 執行組（PM + 現場人員）

| 需求 | 痛點 | 因果鏈 |
|------|------|--------|
| 履約進度追蹤 | 各人用各自的 Excel/Notion，格式不統一 | → 管理者看不到全貌 |
| 成本控制 | L3 的內部預算控管摘要是靜態文件，實際支出用另一張表 | → 超支時發現太晚 |
| 現場應變 | 出事時翻 00D 的 .md 檔，手機不好查 | → 延誤應變黃金時間 |
| **結案檢討** | 要填 00E 筆記，格式繁瑣、容易拖延 | → 拖著不填 → 經驗沒有回到知識庫 → 下次犯同樣的錯 |
| 經驗保存 | 新進人員不知道前案的教訓 | → 重複踩坑 |
| 知識庫更新 | T1 精靈產出格式化區塊後，要手動複製貼入 .md 檔 | → 麻煩 → 不做 → 知識庫逐漸過時 |

## 4. 系統定位與邊界

### 4.1 系統定位

**L1-L8 提示詞系統的指揮中心**——不是取代 AI 或取代人，而是讓「人 + AI + 工具」的協作流程更順暢。

### 4.2 三層架構的分工

| 層 | 負責什麼 | 工具 |
|----|---------|------|
| AI 層 | 依 L1-L8 提示詞產出分析、企劃、報價、策略、講稿 | Claude Code（正常）/ Claude.ai 網頁版（容災） |
| 美學層 | 建議書排版設計、簡報排版設計 | Canva（Claude Code 透過 MCP 調用） |
| 管理層 | 案件追蹤、團隊協作、知識累積、績效分析 | **本系統（Web App）** |

### 4.3 系統不做的事

- **不做 AI 推論**：不調用 AI API。AI 工作在 Claude Code 中完成。
- **不做排版**：建議書和簡報的美學設計在 Canva 中完成。
- **不取代人的決策**：所有斷點仍由人類確認。系統只是讓斷點狀態可見。
- **不取代 Notion**：Notion 是資料主庫，系統是 Notion 的「專業視圖」。

### 4.4 prompt 模板庫的定位

prompt 模板庫是系統的基本功能，不是附加功能：

| 場景 | 模板庫的角色 |
|------|------------|
| 正常運作（Claude Code） | 提示詞索引 — 查閱每階段載入哪些檔案、為什麼 |
| 系統故障（切 Claude.ai） | 容災工具 — 一鍵複製正確的提示詞組合 |
| 新人訓練 | 教學參考 — 理解 L1-L8 流程和知識庫結構 |

功能需求：
- 按階段瀏覽提示詞（L1-L8 + T1/T3）
- 顯示每階段需要的檔案組合（依 `STAGE_KB_RULES`）
- 一鍵複製完整提示詞組合（系統核心 + 階段提示詞 + 對應知識庫）
- 知識庫引用矩陣的視覺化呈現（00-3 的表格互動版）

## 9. 設計原則

### 9.1 人機協作，不是無人自動化

> 「我要的是人機協作的系統，不是無人自動化的系統。人機『如何』協作是你要關注的。」

- 系統不取代人的判斷，只讓判斷所需的資訊更容易取得
- AI 的斷點（A/B/C/D/E）在系統中可見、可追蹤，但決策權仍在人手上
- 容災設計：任何工具掛了，人都能切到替代方案繼續工作

### 9.2 需求三層結構

分析每個功能時使用：

```
需求：角色想要達成什麼
  ↓
痛點：目前為什麼做不到或做得痛苦
  ↓
因果鏈：這個痛點不解決會導致什麼後果（連結到收入公式）
```

### 9.3 從收入公式逆推優先序

每個功能都要能回答：**這推動了「發現機會數 × 投標決策通過率 × 得標率 × 案件金額 × 毛利率」的哪一項？**

### 9.4 技術原則

- **SSOT**（Single Source of Truth）：常數只定義一次
- **Feature Registry Pattern**：所有模組在註冊表中管理
- **閉環開發**：寫碼 → 審查 → 測試 → build → 驗證
- **Hydration-safe**：所有 localStorage 操作用 useEffect
- **中文優先**：所有 UI 文字使用繁體中文
