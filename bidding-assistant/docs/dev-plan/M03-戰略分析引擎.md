# M03 戰略分析引擎（Strategic Analysis Engine）

## 定位

戰略分析引擎 = **從標案機會到投標決策的決策中樞**。

M01（情報模組）是眼睛——蒐集硬數據；M03 是大腦——把數據變成判斷。

對應能力 1（00-總論）。解決痛點 7 條：#1 篩案靠感覺、#2 需求沒拆解、#3 沒寫作順序、#19 不懂評委、#20 不了解對手、#21 沒戰略分析、#22 缺企劃架構。

## 核心功能

### 功能 1：案件適配度評分（Fit Scoring）

輸入一筆標案，自動算出「這案子適不適合我們」的分數。

**評分維度**（5 維，各 0-20 分，總分 100）：

| 維度 | 權重 | 資料來源 | 計算邏輯 |
|------|------|---------|---------|
| 領域匹配 | 20 | 案名關鍵字 × 00B 實績庫 | 案名關鍵字命中實績數量和相關度 |
| 機關熟悉度 | 20 | M01 機關情報 × 00B | 在這個機關的投標/得標紀錄 + 在位者分析 |
| 競爭強度 | 20 | M01 市場趨勢 | 同類案件平均投標家數 + 紅海/藍海判斷 |
| 規模適合度 | 20 | 預算金額 × 00B 實績規模 | 案件預算 vs 歷史案件金額分布 |
| 團隊可用性 | 20 | 00A 團隊資料 × 案件需求 | 所需專長是否有人選（人 × 資格 × 經驗） |

**輸出**：

```typescript
interface FitScore {
  total: number;                  // 0-100
  dimensions: {
    domain: DimensionScore;       // 領域匹配
    agency: DimensionScore;       // 機關熟悉度
    competition: DimensionScore;  // 競爭強度
    scale: DimensionScore;        // 規模適合度
    team: DimensionScore;         // 團隊可用性
  };
  verdict: "建議投標" | "值得評估" | "不建議" | "資料不足";
  reasons: string[];              // 2-3 句判斷理由
  redFlags: string[];             // 紅旗警示
}

interface DimensionScore {
  score: number;          // 0-20
  confidence: "高" | "中" | "低";
  evidence: string;       // 依據什麼算出來的
}
```

**門檻建議**（可調）：
- ≥ 70 → 建議投標
- 50-69 → 值得評估
- < 50 → 不建議
- 任一維度 confidence = "低" 且占比 > 40% → "資料不足"

### 功能 2：投標決策報告（Bid Decision Report）

把適配度評分擴展為完整的決策依據文件。

**報告結構**：

```
一、案件概覽
  - 基本資訊（案名、機關、預算、截標日、決標方式）
  - 適配度總分 + 雷達圖

二、競爭態勢
  - 已知潛在對手（M01 競爭分析）
  - 各對手的歷史戰績（對手得標率、擅長領域）
  - 我方 vs 對手的優劣勢比較

三、機關分析
  - 機關偏好分析（M01 機關情報）
  - 在位者分析（連續得標者、我方在此機關的紀錄）
  - 評委分析（如有評委資料：常見評委、偏好風格）

四、風險評估
  - 紅旗列表（資格不符、實績不足、預算異常、在位者太強）
  - 每個紅旗的嚴重程度和應對建議

五、決策建議
  - verdict + 理由
  - 如果投：勝算評估、建議切入角度
  - 如果不投：放棄理由、是否值得追蹤
```

### 功能 3：作戰計畫（Battle Plan）

決定投標後，自動產出結構化的作戰計畫。這是 L1 prompt 的系統輔助——不取代人 + AI 的 L1 對話，而是提前組裝好硬數據讓 L1 更快更準。

**作戰計畫結構**：

```
一、核心論述（1-2 句）
  - 我們為什麼是這案子的最佳選擇
  - 從適配度分析的最強維度推導

二、三大攻略點
  - 攻略點 1：{從適配度分析推導}
  - 攻略點 2：{從競爭態勢推導}
  - 攻略點 3：{從機關偏好推導}
  - 每個攻略點附上：來源依據 + 可用的知識庫素材

三、評分項目清單（從招標文件拆解）
  - 自動提取不屬於 M03 範圍（需要讀 PDF = L1 的工作）
  - M03 提供：該機關過去案件的常見評分模式（M01 數據）

四、團隊配置建議
  - 從 00A 團隊庫匹配適合人選
  - 各成員最相關的經歷/實績

五、知識庫調度清單
  - 00A：需要哪些人的哪些資料
  - 00B：最相關的實績案例
  - 00C：適用的時程範本
  - 00D：需要準備的應變方案
  - 00E：過去類似案件的檢討

六、P 偵察補充需求
  - M03 已有的硬數據
  - 還需要 Perplexity 查的軟情報清單
  - 自動組裝 P 偵察 prompt（呼叫 M01 的 generateScoutPrompt）
```

### 功能 4：投標組合分析（Portfolio Optimizer）

同時評估多案時，提供組合建議。

**輸入**：N 筆候選案件（各自已有適配度評分）
**處理**：

1. 時程衝突偵測（截標日、人力重疊）
2. 資源負載計算（同時投 M 案需要多少企劃人力）
3. 期望值排序（適配度分數 × 預算金額 × 歷史得標率近似值）
4. 人力分配建議（誰負責哪案）

**輸出**：

```typescript
interface PortfolioRecommendation {
  recommended: TenderWithPriority[];    // 建議投的，按優先序
  conflicts: ConflictPair[];            // 時程/人力衝突
  dropped: TenderWithReason[];          // 不建議投的
  totalLoad: number;                    // 預估總人力需求（人天）
  capacityWarning: string | null;       // 超載警示
}
```

### 3. 適配度評分引擎（核心演算法）

純函式，不碰 API，輸入結構化資料、輸出分數。

#### 3a. 領域匹配（domain）

```typescript
function scoreDomain(
  tenderTitle: string,
  tenderKeywords: string[],     // 從案名提取的關鍵字
  portfolio: KBEntry00B[],       // 實績庫
): DimensionScore
```

邏輯：
1. 從案名提取關鍵字（斷詞：按頓號/逗號/常見分隔 + 業務類型詞彙表）
2. 對每個關鍵字，在 00B 實績庫搜尋匹配的案件
3. 匹配數量 + 匹配案件的規模和成果 → 分數
4. 0 匹配 → 0 分（全新領域）；3+ 匹配且含主要實績 → 滿分

#### 3b. 機關熟悉度（agency）

```typescript
function scoreAgency(
  agencyIntel: AgencyIntel | null,  // M01 機關情報
  portfolio: KBEntry00B[],           // 實績庫（機關名稱匹配）
): DimensionScore
```

邏輯：
1. 在此機關有得標紀錄 → 高分
2. 在此機關有投標紀錄但沒得標 → 中分（至少認識）
3. 在位者弱（無連續得標者）→ 加分
4. 在位者強（連續 3+ 次得標）→ 減分
5. 從未接觸此機關 → 低分但不是 0（新機關也有機會）

#### 3c. 競爭強度（competition）

```typescript
function scoreCompetition(
  marketTrend: MarketTrend | null,  // M01 市場趨勢
  bidderCount: number | null,        // 本案已知投標家數
): DimensionScore
```

邏輯：
1. 藍海（平均投標家數 < 3） → 高分
2. 一般（3-6 家）→ 中分
3. 紅海（> 6 家）→ 低分
4. 趨勢減少 → 加分（市場在冷卻）
5. 實際投標家數優先於市場平均

#### 3d. 規模適合度（scale）

```typescript
function scoreScale(
  budget: number | null,
  portfolio: KBEntry00B[],  // 實績庫的金額分布
): DimensionScore
```

邏輯：
1. 計算公司歷史案件金額中位數和 IQR
2. 預算在 IQR 內 → 滿分
3. 預算在 1.5×IQR 內 → 中分
4. 預算超出 2×IQR → 低分（案子太大或太小）
5. 無預算資料 → confidence = "低"

#### 3e. 團隊可用性（team）

```typescript
function scoreTeam(
  tenderKeywords: string[],
  team: KBEntry00A[],
): DimensionScore
```

邏輯：
1. 關鍵字匹配團隊成員的專長/認證/經歷
2. 有 3+ 人匹配 → 滿分
3. 有 1-2 人匹配 → 中分
4. 無人匹配 → 低分
5. 匹配的人中有主管級（計畫主持人資格）→ 加分

### 4. 知識庫匹配器

從五個知識庫中找出與特定標案最相關的素材。

```typescript
function matchKB(
  tenderTitle: string,
  tenderKeywords: string[],
  agencyName: string,
  kb: KnowledgeBaseData,
): KBMatchResult

interface KBMatchResult {
  team: { entry: KBEntry00A; relevance: string }[];
  portfolio: { entry: KBEntry00B; relevance: string }[];
  templates: { entry: KBEntry00C; relevance: string }[];
  risks: { entry: KBEntry00D; relevance: string }[];
  reviews: { entry: KBEntry00E; relevance: string }[];
}
```

匹配策略（簡單但有效）：
- 關鍵字匹配：案名關鍵字 vs 各庫 entry 的文字欄位
- 機關匹配：機關名稱 vs 00B 的 client 欄位
- 類型匹配：案件類型 vs 00C 的 applicableType
- 結果匹配：00E 中同機關/同類型案件的檢討
- 不用向量搜尋（知識庫 < 200 筆，關鍵字就夠）

## 設計決策

### 為什麼適配度評分用確定性演算法，不用 AI？

1. **可解釋**：每個分數都能說出來源，不是黑箱
2. **一致性**：同樣的輸入永遠同樣的輸出
3. **可測試**：純函式 + 單元測試
4. **速度**：瞬間完成，不需等 API
5. **成本**：$0

AI 在哪裡？在 L1（戰略分析 prompt）。M03 把硬數據準備好，L1 的 AI 對話負責軟判斷。硬歸硬、軟歸軟。

### 為什麼分數用 0-100 而不是 A/B/C？

- 可以排序、加權、組合計算
- verdict 標籤（建議投標/值得評估/不建議）是從分數衍生的
- 使用者可以調整門檻值，不需要改程式

### 為什麼 KB 匹配用關鍵字不用向量搜尋？

- 五個知識庫 entry 總量預估 < 200 筆
- 關鍵字匹配在此規模已夠準
- 不增加依賴（無需向量資料庫或 embedding API）
- 符合設計原則：全部走現有訂閱，額外成本 = $0

### 為什麼 M03 不取代 L1？

M03 是**系統輔助**，L1 是**人 + AI 對話**。兩者互補：

| | M03（系統） | L1（AI 對話） |
|---|---|---|
| 輸入 | 結構化數據 | 招標文件 PDF + M03 產出 |
| 產出 | 數據驅動的評分/報告 | 策略性的分析洞見 |
| 風格 | 客觀、可重複 | 創意、脈絡感知 |
| 人參與 | 低（看結果、調權重） | 高（3-5 輪對話） |

M03 做完再進 L1 → L1 不用從頭查數據，直接從策略面切入。

## 技術約束

1. 所有評分邏輯是純函式（`fit-scoring.ts`、`helpers.ts`），不碰 React state
2. Hook 層負責呼叫 M01 hooks 取得資料，再傳入純函式
3. 知識庫資料透過 `useKnowledgeBase()` hook 取得
4. 快取：適配度結果快取在 state 中（非 localStorage），因為依賴即時資料
5. 評分權重透過 Settings 可調
6. 所有 UI 文字中文
7. 遵循 SSOT：常數定義在 `constants.ts`，不硬編碼
8. Feature Registry 註冊，dependencies 包含 intelligence
