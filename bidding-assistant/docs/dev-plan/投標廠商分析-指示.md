# 投標廠商分析——執行指示

當使用者要求分析某家投標廠商時，依以下流程執行。

---

## 流程

### Step 1：確認輸入
使用者提供公司名稱（必要）和統一編號（選填）。
如果只有名字沒有統編，先用 PCC API 搜尋確認。

### Step 2：產出 Perplexity 提示詞
用下方模板，填入公司名稱和統編，交給使用者去 Perplexity Deep Research 跑。

### Step 3：等使用者回傳 Perplexity 研究結果
使用者會貼回文字或提供檔案路徑。

### Step 4：拉 PCC 數據
同步執行以下工作：

**4a. 全部投標記錄**
- MCP 工具 `search_by_company`，逐頁拉完（每頁檢查 total_pages）
- 用 `brief.companies.name_key` 判斷每筆得標/未得標
- 統計：總投標數、得標數、未得標數、得標率

**4b. 得標金額明細**
- WebFetch `pcc.mlwmlw.org/merchants/{統編}`
- 解析頁面 `__NEXT_DATA__` JSON，取出每筆標案名稱、日期、金額
- 彙整年度營收表

**4c. 投標行為分析**
- 從 openfun 資料計算：
  - 獨家投標 vs 競爭投標件數
  - 各競爭家數下的得標率
  - 回頭客比例（同機關重複得標）
  - 獨家案構成（回頭客指定/契約變更/無人競爭）
  - 預算捕獲率

**4d. 市場規模**
- 從 Perplexity 研究識別公司的核心能力關鍵字
- 用 `search_by_title` 查各關鍵字的全國標案 total

**4e. 關聯企業（如有）**
- 從 Perplexity 研究識別關聯企業
- 用關聯企業的名稱/統編重複 4a-4b

**4f. 競爭者（如有統編）**
- 從 Perplexity 研究取得競爭者統編
- 拉競爭者的標案記錄做交叉比對

### Step 5：整合分析，輸出 11 章報告

---

## 報告架構（11 章）

### 第 1 章：公司定性
- 一句話定性
- 維度表：本質、品牌資產、商業模式、成長引擎、護城河、天花板、風險、類比
- 定性依據（引用 Perplexity 的具體證據）

### 第 2 章：團隊結構
- 創辦人/代表人背景表
- 核心成員辨識表
- 運作模式（固定團隊 vs 核心+外包）
- 徵才紀錄 → 推斷規模和能力缺口
- 關鍵人物依賴度

### 第 3 章：營收與商業模式
- 年度標案營收表（年度/件數/金額/均價）
- 非標案收入盤點
- 營收結構——按服務類型（佔比、單案均價）
- 營收結構——按客戶（佔比、合作模式）
- 商業模式特徵表

### 第 4 章：品質實證
- 獎項紀錄表
- 出版品/館藏紀錄
- 展覽成果
- 品牌影響力指標（粉絲、流量、媒體報導）
- 外部評價（正面+負面）

### 第 5 章：利基市場定位
- 核心市場表（能力/全國標案數/競爭強度/公司位置）
- 上位市場表
- 產業結構（參與者分類）
- 利基本質洞見
- 產業慣例（聯合投標/分包/評選方式）
- 政府預算趨勢

### 第 6 章：競爭力評估（SWOT）
- S：得標率趨勢 + 品質證據 + 品牌 + 回頭客 + 無系統性對手
- W：競爭家數 vs 得標率 + 能力邊界 + 團隊/資本限制
- O：捕獲率 + 獨家案來源 + 非標案成長 + 關聯企業協同
- T：客戶集中 + 獨家案下降 + 預算政策 + 競爭者動態

### 第 7 章：競爭者比較
- 競爭者名冊表（公司/統編/資本額/規模/能力/關係）
- 能力對比表
- 標案交叉比對（如有數據）
- 威脅排序

### 第 8 章：關聯企業分析
- 關聯圖譜
- 各企業標案數據（分開算）
- 資源共享判斷
- 聯合投標紀錄
- 風險傳導
- 合併能力圈

> 併入標準：共同負責人 → 必須。業務重疊/互補 → 必須。同一標案出現 → 必須。純投資且不相關 → 不需要。無關聯企業則省略本章。

### 第 9 章：競爭策略研判
- 差異化切入點表（策略/邏輯/適用情境）
- 老客戶滲透路徑（逐客戶分析裂縫+進入方式）
- 通用滲透邏輯（從旁門進 → 建關係 → 等週期 → 展示差異）

### 第 10 章：風險訊號
- 風險表（類型/狀態/說明）
- 涵蓋：履約爭議、訴訟、負面新聞、政策風險、法規、AI 衝擊、關聯企業外溢

### 第 11 章：關鍵數字速查
- 單一彙總表

---

## 寫作規則

1. **數據表格為主**——每個觀點必須有表格支撐
2. **自然語氣**——禁用：戰場、群狼、天敵等戲劇化用語
3. **精簡**——更多資訊/更少文字
4. **一致人稱**——用公司名或「他們」，不用「你」
5. **標註證據等級**——硬數據 / 有依據的推論 / 推測
6. **洞見方法論**——全樣本觀察 → 結論 → 同樣本深挖一層。不拆成年度小切片（<10 筆不下結論）

## Perplexity 提示詞模板

```
我正在分析一家台灣公司的政府標案競爭力，需要你做深入研究。
公司名稱：【填入】
統一編號：【填入，如知道】

我已經有這家公司完整的政府採購投標與決標數據，不需要你查標案記錄。
我缺的是標案資料庫裡查不到的東西。

---

## A. 公司真實面貌

### A1. 基本資料
- 成立日期、資本額、登記地址
- 經濟部商工登記的營業項目
- 官網、社群帳號（FB/IG/LinkedIn/YouTube）
- 是否有關係企業、母子公司

### A2. 團隊結構與規模
- 創辦人/代表人完整背景（學歷、經歷、專業領域、過往機構）
- 可辨識的核心成員（報導、LinkedIn、講者資訊、書籍作者欄）
- 104人力銀行、LinkedIn、CakeResume 上的徵才紀錄（職缺內容、數量、時間）
- 運作模式：固定團隊 or 核心小組＋案案外包？
- 常合作的外包夥伴或協力廠商

### A3. 收入結構
- 政府標案以外的收入來源？（民間案、版稅、顧問、教學、演講）
- 自有品牌或產品？
- 公開的財務資訊、獲投紀錄、營收披露？

---

## B. 作品與口碑

### B1. 代表作品清單
- 所有能找到的已完成作品（出版品、展覽、網站、影片、活動）
- 每件附：名稱、年份、委託單位、簡述

### B2. 作品客觀成果（請盡量挖）
- 出版品：國家圖書館館藏？館藏數？再版？銷量？書評？
- 展覽：參觀人次、展期延長、媒體報導量
- 得獎：金鼎獎、國家出版獎、文化部獎項、設計獎
- 學術引用
- 其他可量化成果

### B3. 外部評價
- 媒體報導
- 業界評價
- 負面評價或爭議

---

## C. 同業競爭者（盡可能完整）

### C1. 競爭者名冊
每家提供：
- 公司名稱、統一編號（務必提供，我需要交叉比對標案記錄）
- 主要業務與專長
- 代表作品（2-3 件）
- 大致規模
- 與目標公司的關係（純競爭/合作/各有地盤/師出同門）

### C2. 產業結構
- 參與者分幾種類型？
- 人才從哪來？好不好找？
- 標案以外怎麼賺錢？

### C3. 產業慣例
- 聯合投標常見嗎？常見組合？
- 分包/協力慣例
- 最有利標 vs 最低標趨勢

---

## D. 產業環境與政策

### D1. 產業趨勢
- 成長/持平/萎縮？有數據嗎？
- 結構性變化（數位轉型、AI、新進者）
- 相關產業報告或學術研究

### D2. 政策與預算
- 相關部會近三年預算趨勢（盡量找細項）
- 相關中長期政策計畫（名稱、期程、總預算）
- 採購量趨勢

### D3. 採購端決策邏輯
- 評審委員怎麼組成？
- 公開的評選紀錄？
- 業界共識的得標關鍵因素？

### D4. 產業 benchmark
- 典型年營收、團隊規模、得標率？
- 可對比的標竿公司？

---

## E. 深挖項目

### E1. 公司真實規模
- 徵才紀錄（104/LinkedIn/CakeResume）
- 財務資訊、獲投紀錄

### E2. 作品客觀成果
- 國圖館藏數、展覽人次、得獎、學術引用

### E3. 產業 benchmark
- 產業報告（文創產業年報等）

### E4. 採購決策
- 評審組成、最有利標佔比、公開評選紀錄

### E5. 同業標案表現
- 競爭者的統一編號（務必提供）
```

## PCC 數據處理備忘

### API 端點
| 用途 | 端點 |
|------|------|
| 廠商全部投標 | `/api/searchbycompanyname?query={名稱}&page={N}` |
| 標案名稱搜尋 | `/api/searchbytitle?query={關鍵字}&page={N}` |
| 標案詳情 | `/api/tender/{unit_id}/{job_number}` |
| 決標資料 | `/api/award/{unit_id}/{job_number}` |

### 得標金額
`pcc.mlwmlw.org/merchants/{統編}` — `__NEXT_DATA__` JSON

### 判斷得標/未得標
`brief.companies.name_key` → 值含 "得標廠商" or "未得標廠商"

### 已知問題
- `/searchbyaward` 和 `/searchbyunit` 偶爾回 HTML
- openfun 記錄數和 mlwmlw.org 可能不一致 → 金額用 mlwmlw，投標行為用 openfun
- 預算欄位名不固定，用 `:預算金額` 結尾匹配
- 大量數據寫 .mjs 腳本跑，不用 node -e 內聯
