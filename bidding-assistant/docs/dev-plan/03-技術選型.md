# 03 技術選型決策紀錄

每個決策記錄：背景、選項、決定、理由、否決原因。

## ADR-01：前端框架 — Next.js 16

- **決定**：Next.js 16 + App Router + React 19 + TypeScript + Tailwind CSS 4
- **理由**：已有完整專案基礎（560 tests, 10 features, 23 routes），無理由換框架
- **否決**：Remix（生態系小）、Nuxt（Vue 學習成本）、純 React SPA（失去 SSR 和 API Routes）

## ADR-03：向量資料庫 — 不需要

- **決定**：不建向量資料庫
- **理由**：
  - 知識庫（00A-00E）預估 ~72,000 tokens = 1M context 的 7%，直接塞進 context 即可
  - Notion 資料庫有 10,000+ 筆，但查詢用 Notion API structured filter，不需要向量搜尋
  - 「找相似歷史案件」目前無意義（0 得標，無 00E 檢討回饋）
- **否決**：Pinecone / pgvector / Chroma — 架構過度複雜，收益不明

## ADR-05：公司識別方式 — 統編優先

- **決定**：所有交叉比對、統計、分組以統編（8位數字）為 key
- **理由**：PCC API 中公司名稱有多種寫法（如有/無英文名、空白差異），但統編永遠不變
- **否決**：模糊名稱比對（不可靠、需維護同義詞表）
- **名稱正規化**：僅用於 UI 顯示和搜尋，不用於身份判斷

## ADR-06：情報資料來源 — g0v PCC API

- **決定**：使用 g0v PCC API (`https://pcc-api.openfun.app`) 作為核心情報來源
- **理由**：免費、無認證、開放 CORS、資料涵蓋完整（得標者、未得標者、評委、金額）
- **否決**：直接爬政府電子採購網（違反 ToS、維護成本高、反爬機制）
- **補充**：API 可取代 P 偵察報告 ~80% 的人工搜尋，Perplexity 只需補軟情報

## ADR-07：Discord Bot 定位

- **決定**：Discord = 日常操作介面（駕駛座），Web app = 管理後台（儀表板）
- **理由**：團隊已用 Discord 溝通，學習成本最低
- **否決**：LINE Bot（API 限制多）、Slack（額外訂閱費）、純 Web App（切換成本高）
