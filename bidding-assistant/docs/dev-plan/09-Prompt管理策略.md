## 二、版控策略

### 版號規則

```
{major}.{minor}
major：結構性大改（新增/刪除章節、改變輸出格式）
minor：措辭微調、修正、補充
```

範例：L1 v2.3 = 第 2 次結構改版 + 第 3 次微調

### 儲存結構

```
public/prompts/
  stages/
    01.md          ← L1 戰略分析（生產版）
    02.md          ← L2 備標規劃
    ...
  tools/
    T1.md          ← T1 知識庫精靈
    P-template.md  ← P 偵察報告模板
  knowledge/
    00A.md ~ 00E.md  ← 知識庫內容（由 T1 產出）
```

### 版本歷史記錄

每份 prompt 檔案開頭包含：

```markdown
---
id: L1
version: "2.3"
updated: "2026-02-21"
changelog:
  - "2.3: 加入 PCC API 數據交叉比對步驟"
  - "2.2: 評委分析段落重寫"
  - "2.0: 結構改版，從 5 步改為 8 步"
---
```

## 三、演進流程

```
觀察問題（AI 產出品質不佳、用戶反饋、結案檢討）
  → 定位到哪個 prompt 的哪個步驟
  → 修改 prompt（版號 +0.1）
  → 用下一個案件實測
  → 有效 → 保留（版號確認）
  → 無效 → 回滾到上一版
```

### 測試方式

| 階段 | 測試方法 | 判斷標準 |
|------|---------|---------|
| Phase 1（自用） | 每個新案件是一次實測 | 用戶主觀判斷 + 結案回饋 |
| Phase 2（團隊） | A/B 測試（新舊版同時跑） | 品質評分 + 使用者滿意度 |

Phase 1 不需要複雜的 A/B 框架。用戶只有一個人，每個案件手動對比即可。

## 四、Prompt 組裝機制

`src/data/config/prompt-assembly.ts` 管理 prompt 組裝：
- 每個 L 階段 prompt = 基礎指令 + 知識庫片段 + 案件上下文
- 知識庫注入量由 `kb-matrix.ts` 定義（哪個階段需要哪些 00 系列）

```
L1 prompt = stages/01.md + 00A（團隊） + 00B（實績） + 案件招標文件 + P 報告
L3 prompt = stages/03.md + 00C（時程） + 00D（應變） + L2 規劃書
```

## 五、優先序

1. **P 偵察報告模板建檔**：目前手動組裝，應建成 `tools/P-template.md` 讓系統自動組裝
2. **L2-L8 撰寫**：按投標流程順序，每個案件實際走到才寫（不預寫）
3. **00A-00E 建立**：靠 T1 精靈引導，Layer 0 任務
4. **版控自動化**：Phase 2 再考慮（Phase 1 手動管理版號即可）
