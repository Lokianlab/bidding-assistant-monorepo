# 08 風險與待決事項

## 必須先決定（阻塞開發）

### #1 Agent SDK = API 計費，不是 $0

**問題**：開發計畫假設「全走訂閱，額外成本 = $0」，但 Agent SDK 走 API 計費。
5 個 Agent（戰略官/企劃官/品管官/知識官/簡報官）每次呼叫都要付 API 費用。

**可能方案**：
- A) 接受 API 成本（估算月費？）
- B) 包裝 CLI 呼叫（走訂閱但開發複雜度高）
- C) 簡化為單一 Agent + 不同 system prompt（降低成本但功能退化）

**狀態**：待決定

### #2 團隊採用風險

**問題**：15 人公司，目前工作流是 Perplexity+Gemini+Claude+Nanobanana 跳來跳去。
新系統（SaaS 提案駕駛艙）需要團隊改變習慣。

**風險**：
- 建好了沒人用
- 學習曲線太陡
- 只有老闆在用，其他人回到舊方法

**對策建議**：
- 先做最痛的點（L1 戰略分析 + P 偵察自動化）
- 一個人先用，證明有效再推廣
- 系統設計要比現有流程更簡單，不是更複雜

**狀態**：需持續關注

## 需要討論但不阻塞

### #3 Vercel 部署限制

**原問題**：Discord Bot 主機（24/7 運行跑在哪？）
**風險轉變**：改為 SaaS 後不需自管主機，但 Vercel 有自己的限制：

- Hobby plan：每月 100GB 頻寬、Serverless 執行時間上限、每次部署 250MB
- 冷啟動：Serverless Function 閒置後首次請求有 500ms-2s 延遲
- 費用跳變：Hobby → Pro 是 $0 → $20/月，非平滑增長

Phase 1 自用大概率在免費額度內。Phase 2 多人使用時需評估是否升 Pro。

**狀態**：Phase 1 不阻塞，Phase 2 前需評估

### #4 Vercel Serverless 逾時

**原問題**：Discord 15 分鐘逾時（L1 可能跑 10-15 分鐘）
**風險轉變**：SaaS 環境下限制更嚴格，不是更寬鬆：

- Vercel Hobby：Serverless Function timeout = 10 秒
- Vercel Pro：最多 60 秒（部分 region 300 秒）
- Discord Interaction timeout 是 15 分鐘，反而更寬裕

PCC 批次搜尋、Notion 全量抓取等長時間操作在 Route Handler 跑不完。與 M01 模型委員會審查盲區 5 同一問題。

**可能方案**：
- 前端分段請求（每次只查一頁/一家廠商）
- 背景 Job 架構（Web Worker / requestIdleCallback）
- 部分成功可存、中斷可繼續

**狀態**：待設計，與 M01 v2.0 合併處理

### #5 招標文件上傳

**原問題**：Discord 25MB 檔案上傳限制
**風險轉變**：Vercel body size 限制比 Discord 更嚴格：

- Vercel Serverless Function body size：4.5MB（硬限制，Hobby 和 Pro 相同）
- 招標文件 PDF 通常 5-30MB，超過限制
- 走 API Route 上傳行不通

**可能方案**：
- Presigned URL 直傳雲端儲存（S3 / GCS / Supabase Storage）
- Vercel Blob Storage
- Phase 1 不需上傳（手動處理），Phase 2 再設計方案

**狀態**：Phase 1 不阻塞，Phase 2 前需設計

## 業務背景與待決策略

### #8 企劃端根因

**現況**：
- 完全跳過 L1-L8，直接把文件丟 AI「寫完整建議書」
- 沒有審稿、沒有品質閘門
- 46 案 0 得標

**根因**：

```
量衝不起來 ← 每案手工作業太重
質提不上去 ← 跳過分析直接寫
兩頭卡住 ← 流程不支援任何一種策略
```

**狀態**：系統核心要解決的問題，驅動整體開發優先序

### #9 案件分級策略

```
200 件/月（企劃選入）
    ↓ L1 快速分析（每案 30 min）
├── A 類（高勝算）→ 全力 L1-L8 → 精
├── B 類（中勝算）→ 精簡流程 → 量
├── C 類（可能獨家）→ 最低限度 → 撿漏
└── D 類（低勝算）→ 放棄 → 省時間
```

**狀態**：策略方向明確，分級標準待定義

### #10 三端全貌與人力共用

| 端 | 設計 | 落地 | 產出 |
|----|:----:|:----:|------|
| 行政端 | ✅ | ✅ 貫徹 | 穩定（600-1200 件/月） |
| 企劃端 | ✅ L1-L8 | ❌ 未部署 | 46 案 0 得標 |
| 執行端 | ✅ 30 節點 | ❌ 未貫徹 | 差強人意 |

**根因**：同一批人扛企劃+執行，哪邊著火救哪邊。

**關鍵洞察**：
1. 設計能力不缺（三端都有完整設計）
2. 落地能力是瓶頸（只有行政端成功）
3. 行政端成功因素：專職人力、可切割、可量化
4. 企劃+執行失敗因素：人力共用、互搶資源、救火模式

**狀態**：需要討論人力配置策略
