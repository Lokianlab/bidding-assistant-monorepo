# 08 風險與待決事項

## 必須先決定（阻塞開發）

### #1 Agent SDK = API 計費，不是 $0

**問題**：開發計畫假設「全走訂閱，額外成本 = $0」，但 Agent SDK 走 API 計費。
5 個 Agent（戰略官/企劃官/品管官/知識官/簡報官）每次呼叫都要付 API 費用。

**可能方案**：
- A) 接受 API 成本（估算月費？）
- B) 包裝 CLI 呼叫（走訂閱但開發複雜度高）
- C) 簡化為單一 Agent + 不同 system prompt（降低成本但功能退化）

**狀態**：待決定

### #2 團隊採用風險

**問題**：15 人公司，目前工作流是 Perplexity+Gemini+Claude+Nanobanana 跳來跳去。
新系統（Discord Bot）需要團隊改變習慣。

**風險**：
- 建好了沒人用
- 學習曲線太陡
- 只有老闆在用，其他人回到舊方法

**對策建議**：
- 先做最痛的點（L1 戰略分析 + P 偵察自動化）
- 一個人先用，證明有效再推廣
- Discord Bot 設計要比現有流程更簡單，不是更複雜

**狀態**：需持續關注

## 需要討論但不阻塞

### #3 Discord Bot 主機

**問題**：Bot 需要 24/7 運行，跑在哪裡？
- 自家電腦？（不穩定）
- VPS？（月費）
- Serverless？（冷啟動延遲）

**狀態**：待討論

### #4 Discord 15 分鐘逾時

**問題**：Discord Interaction 有 15 分鐘回應時限。
L1 戰略分析可能跑 10-15 分鐘，接近上限。L3 企劃草稿可能更久。

**可能方案**：
- Deferred response + webhook callback
- 拆分成多步驟，每步獨立回應
- 改用非 Interaction 的 message 模式（無時限但 UX 不同）

**狀態**：待討論

### #5 招標文件上傳（25MB 限制）

**問題**：Discord 檔案上傳限制 25MB（Nitro 500MB）。
招標文件 PDF 通常 5-30MB，可能超過限制。

**可能方案**：
- Nitro 訂閱
- 改用 Google Drive 連結
- Web App 處理上傳，Bot 處理指令

**狀態**：待討論

### #6 Web App 定位

**問題**：Discord Bot 成為主介面後，現有 Web App（Next.js）的角色是什麼？
- 凍結開發？
- 轉為管理後台？
- 兩邊並行維護？

**建議**：Web App 負責情報模組（視覺化分析）+ 設定管理 + 知識庫維護。
Bot 負責日常投標流程（L1-L8）。

**狀態**：傾向 Web App = 管理後台，待確認

### #7 開發資源與時程

**問題**：誰來開發？時程多長？
- 純靠 Claude Code Agent Team？
- 需要外部開發者？
- 能否在下一輪投標季前完成核心功能？

**狀態**：待討論
