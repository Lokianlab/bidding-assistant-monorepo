# 00 總論

## 系統定位

> **從招標文件到得標，越投越強、越投越準的 AI 提案系統。**

- 不是儀表板
- 不是看板
- 不是 RAG 聊天機器人
- 不是專案管理工具
- **是一個提案寫作駕駛艙：從招標文件進去，一份可以上場的建議書出來**

### 產品介面

- **SaaS 網頁 = 主要操作介面**（具體方案待定案）
- Web app 管理後台 = 設定、知識庫管理、績效分析
- 技術棧：Next.js + React + Claude Code SDK

> 注：Discord Bot 方案曾被提出（2026-02-18），後因 SaaS 網頁能提供更完整的操作體驗而放棄。

## 商業背景

- 公司：大員洛川，15 人，16+ 年經驗
- 核心業務：政府標案企劃（課程/導覽/走讀/傳統服飾/綠色照護）
- 年營收目標：NT$3,000 萬
- 去年實績：投標 46 案，總預算 NT$105,748,770（平均 NT$2,298,886，中位數 NT$1,128,305）
- **得標 0 件** — 0% 得標率
- 標案類型：大部分是最有利標（建議書品質重要）

### 收入公式

```
收入 = 得標案件數 x 平均案件金額 x 毛利率
  得標案件數 = 投標案件數 x 得標率
  投標案件數 = 發現的機會數 x 投標決策通過率
```

**每個功能都必須能回答：推動了公式中的哪一項？**

## 現行工作流程

```
招標文件
  → Perplexity 查資料
  → Gemini 想創意
  → Claude 寫初稿
  → Nanobanana 生主視覺
  → 人肉縫合各工具產出
  → 拉舊案當底稿，修修補補
  → 排版（建議書 + 評選簡報）
  → 送件
  → 落標
  → 沒有系統性地學習
  → 下次重來...
```

## 完整痛點清單（23 條）

### 選案階段
1. **篩案靠感覺和興趣** — 沒有可驗證的篩選機制，可能投了不該投的案

### 戰略階段
21. **沒有深入的戰略分析就開始寫** — 不了解機關潛台詞、競爭優勢、贏的切入角度
22. **缺乏戰略性的企劃架構** — 沒有核心論述、故事線、決勝點判斷，建議書變成資料堆疊

### 拆解階段
2. **需求書沒有系統拆解** — 寫漏評分項目，有些要求沒寫到
3. **沒有結構化的寫作順序** — 不知道先寫什麼後寫什麼，亂寫一通

### 寫作階段
4. **知識庫是空的（00A-00E 未建立）** — AI 只能寫空話，沒有公司實績/團隊/時程/SOP 可用
5. **在 4-5 個 AI 工具間跳來跳去** — Perplexity/Gemini/Claude/Nanobanana，不知做到哪了
6. **多案並行，大腦不斷切換** — 同時寫 2-3 案，每案在不同工具的不同對話裡
7. **每個 AI 都要重新調校** — 花大量時間「教 AI 認識你」，調好了又到對話上限
8. **對話上限是黑箱** — 不知道何時會爆，爆了交接文件又不準，工作成果消失
9. **各工具產出不一致** — 要自己當人肉膠水縫合
10. **AI 幻覺抓不完** — 校稿時很難全部發現，被評委問到答不出來
11. **AI 不自檢、死不認錯** — 生完就辯護，要修很難修
23. **AI 的「創意」沒有常識** — 聽起來厲害但不切實際（社區活動用 AR、50 人活動說大數據）

### 產出階段
12. **寫完還是像初稿** — 品質上不去
13. **主視覺/示意圖跟文字脫節** — 圖文不搭，不夠專業
14. **排版很痛** — 建議書和評選簡報的排版都是痛點

### 結案階段
15. **落標意見回收不系統** — 只能泛泛地問「哪裡要精進」，參考價值低
16. **每次從頭來，經驗沒累積** — 做完的案件經驗沒有回流到系統

### 策略層問題
17. **實績冷啟動困境** — 沒得標 → 沒實績 → 評分被扣 → 繼續沒得標
18. **質 vs 量策略問題** — 46 案全投但品質都不夠，是否應該集中火力？
19. **不夠了解評委真正在看什麼** — 不知道評委 30 分鐘翻完建議書時的關注重點
20. **不了解競爭對手** — 不知道贏的是誰、他們的建議書長什麼樣

## 飛輪三齒輪

1. **選對的案**（策略）— 系統輔助分析
2. **寫出好的建議書**（執行）— 系統的核心價值
3. **從每次結果學習**（回饋）— 系統幫助累積

## 五角色痛點對照

| 角色 | 核心痛點 | 對應痛點編號 |
|------|---------|-------------|
| 管理者 | 無得標率/趨勢數據、無法量化企劃效益、利潤未追蹤 | #15, #16, #18 |
| 行政 | 每月 600-1200 件手動四動作（貼表/貼檔/摘要/情蒐）、Notion AI 摘要格式不穩、情蒐靠人工查 PCC | （不在 23 條範圍，見工作流程） |
| 企劃 | AI 啟動慢、工具間跳轉、多案混淆、前序文件散落 | #5, #6, #7, #8, #9 |
| 設計師 | 找不到美編製作標記、不知優先序/截標日 | #13, #14 |
| 執行組(PM) | 履約格式不統一、成本超支晚發現、現場查 SOP 不便 | #16, #17 |

## 6 大系統能力

### 能力 1：戰略分析引擎
從招標文件到作戰計畫，一條龍：
- 拆解：評分項目、交付要求、資格條件、應備文件
- 分析：適配度、決勝點、紅旗、評委關注點
- 決策：投 or 不投（可驗證的依據）
- 作戰計畫：核心論述 + 攻略策略 + 寫作順序 + 每項所需資料
- 解決痛點：#1, #2, #3, #19, #20, #21, #22（7 條）

### 能力 2：公司知識庫
飛輪的燃料：
- 00A 團隊、00B 實績、00C 時程、00D 應變、00E 檢討、00F 創意庫
- AI 寫作時自動注入最相關的知識
- 案件結束後自動回流更新
- 解決痛點：#4, #16, #17, #23（4 條）

### 能力 3：統一 AI 調度中心
- 每案一個工作區（對話、產出、進度都在一起）
- 自動選 AI、帶入知識庫 + 本案脈絡 + 作戰計畫
- 不需重新調校，系統永遠記得
- 對話上限自動處理精準交接
- 多案並行：獨立工作區，一鍵切換
- 解決痛點：#5, #6, #7, #8, #9（5 條）

### 能力 4：AI 品質閘門
每段產出經過三道檢查：
- 閘門 1 事實查核：有知識庫來源嗎？
- 閘門 2 需求對照：回應了哪個評分項目？
- 閘門 3 實務檢驗：預算做得到？對象用得了？公司有能力？
- 解決痛點：#10, #11, #12, #23（4 條）

### 能力 5：視覺生成整合
- 寫到需要圖的地方，系統建議並生成對應圖像
- 一案一套視覺風格，圖文同源
- 解決痛點：#13（1 條）

### 能力 6：排版輸出 + 落標學習
- 排版：建議書/簡報一鍵匯出，套用公司模板
- 落標學習：結構化分析每案結果，累積後校準篩案機制
- 解決痛點：#14, #15, #18（3 條）

## 設計哲學

### 原開發計畫的教訓

過去的開發計畫（v3.0 批次 A-J）有 4 個問題，促成了現在的「6 大能力」架構：

1. **是功能清單不是產品** — 列了 10 個批次幾十個功能，但沒有回答「使用者的一件事是什麼」
2. **想取代太多工具** — Notion 的案件管理、pcc 監控、Google Calendar、文件管理、儀表板... 15 人公司不需要自建這些
3. **核心價值被埋沒** — L1-L8 提示詞系統是公司護城河，但在計畫裡只是 Batch D 的一個小項目
4. **結案回饋閉環排在最後（Batch J）** — 但它應該是最早做的，因為飛輪越早轉越好

### 設計原則

1. **人機協作，不是無人自動化** — 所有關鍵斷點仍由人類確認
2. **需求三層思考：需求 → 痛點 → 因果鏈** — 每個功能要能連回收入公式
3. **從收入公式逆推優先序** — 直接影響得標率的先做
4. **系統要做的 10 件事**：
   - 一個地方拆解需求
   - 一個地方做戰略分析和企劃架構
   - 一個地方規劃寫作順序
   - 一個地方調度所有 AI（研究/創意/寫作/圖像）
   - 一個地方存放公司真實資料（團隊/實績/時程/SOP）
   - 一個地方追蹤進度（哪些寫了、哪些沒寫）
   - 一個地方做品質檢查（每句話能追溯來源）
   - 一個地方處理排版（建議書 + 評選簡報）
   - 一個地方回收落標經驗
   - 永遠記得你的公司、團隊、過去經驗，不需要重新調校

## 系統不做什麼

- 不取代 Notion 做專案管理
- 不自建 Google Calendar
- 不做通用文件管理
- 不訓練自己的模型
- 不建向量資料庫（知識庫 ~72,000 tokens，直接塞進 1M context）

### 決策 2：PCC API 情報基礎設施（2026-02-18 定版）

- **g0v PCC API**：`https://pcc-api.openfun.app`（免費、無認證）
- 角色：戰略分析的硬數據來源，取代 Perplexity 80% 的搜尋工作
- 能力遠超預期：評委、投標金額、未得標原因都有
- 詳細整合策略見 `pcc-api-integration.md`
- PCC Monitor 已上線（GAS 每小時監控 API 更新延遲）

### 決策 3：Agent 架構 — 雙層策略（2026-02-18 定版）

| 層 | 用途 | 技術 | 誰用 |
|----|------|------|------|
| **施工層** | 開發系統時多 agent 並行寫程式 | Claude Code Agent Team | 開發者 |
| **產品層** | SaaS 網頁背後的 AI 編制 | Claude Code SDK（TypeScript） | 企劃團隊（透過 SaaS 網頁） |

全部走現有訂閱，額外成本 = $0。

#### 產品層：5 個 AI Agent

| Agent | 角色 | 負責 | 使用的 prompt |
|-------|------|------|-------------|
| **戰略官** | 分析與決策 | L1 戰略分析、投標決策、案件拆解 | L1 v2.3 + P 報告 |

## 產品方向（Jin 確認 2026-02-22）

1. **先自用，再產品化** — 先幫大員洛川跑通流程，先不做多租戶
2. **Notion 當主庫，加抽象層** — 現有 Notion 繼續用，程式碼加資料抽象層
3. **Claude Code SDK 當發射台** — 系統組裝材料，一鍵送 Claude Code 執行，不做嵌入式聊天

## 優先序（P0-P4，按實際工作流程排序）

按用戶真實操作順序（打開→選案→看資料→寫企劃→輸出→監控）排列，不按技術層級。

| 階段 | 做什麼 | 對應能力 | 狀態 |
|------|--------|---------|------|
| P0 | 行政巡標自動化（PCC→篩選→建案） | PCC API + Notion | ✅ 完成 |
| P1 | 案件工作頁（選案→看資料→看情報） | 能力 1 + 2 | 待開始 |
| P2 | 寫作發射台（組裝材料→一鍵送 Claude Code） | 能力 3 + 4 | 待開始 |
| P3 | 輸出管理（排版 + 設計需求單） | 能力 6 | 待開始 |
| P4 | 投標監控自動化（狀態追蹤 + 通知） | — | 待開始 |

能力 5（視覺生成）目前用 Nanobanana 手動做，不在 P0-P4 範圍內。
