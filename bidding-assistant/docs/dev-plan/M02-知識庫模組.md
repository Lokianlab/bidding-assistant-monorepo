# M02 知識庫模組（Knowledge Base Module）

## 定位

知識庫 = **提案寫作的彈藥庫**。沒有知識庫，AI 只能寫空話；有了知識庫，AI 能寫出有公司實績、團隊經歷、具體數據的建議書。

五大知識庫（00A-00E）是提案寫作駕駛艙的核心資產，從 L1 選案到 L8 檢討，每個階段都要從中提取不同內容。

## Supabase Schema

### 資料表設計

採「一表 + 類別欄位」模式（不拆 5 張表），理由：
- 跨類別搜尋不需 UNION
- 類別結構已穩定（00A-00E 不會頻繁增減）
- 各類別的差異用 JSON `data` 欄位吸收

```sql
-- 知識庫主表
CREATE TABLE kb_entries (
  id          UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  tenant_id   UUID NOT NULL REFERENCES auth.users(id),    -- 多租戶隔離
  category    TEXT NOT NULL CHECK (category IN ('00A','00B','00C','00D','00E')),
  entry_id    TEXT NOT NULL,       -- 人類可讀 ID（M-001, P-2025-001, T-EXH 等）
  status      TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active','draft','archived')),
  data        JSONB NOT NULL,      -- 各類別的完整欄位（結構由 TypeScript types 保證）
  search_text TEXT GENERATED ALWAYS AS (
    COALESCE(data->>'name','') || ' ' ||
    COALESCE(data->>'projectName','') || ' ' ||
    COALESCE(data->>'templateName','') || ' ' ||
    COALESCE(data->>'riskName','') || ' ' ||
    COALESCE(data->>'title','') || ' ' ||
    COALESCE(data->>'client','')
  ) STORED,                        -- 全文搜尋用
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now(),
  UNIQUE (tenant_id, category, entry_id)
);

-- 索引
CREATE INDEX idx_kb_tenant_category ON kb_entries(tenant_id, category);
CREATE INDEX idx_kb_search ON kb_entries USING gin(to_tsvector('simple', search_text));

-- RLS
ALTER TABLE kb_entries ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can only access own entries"
  ON kb_entries FOR ALL
  USING (tenant_id = auth.uid());
```

### 知識庫元資料表

```sql
-- 各租戶的 KB 設定與統計

```

### 附件儲存

```sql
-- 原始文件（Word/PDF/Excel）的索引
CREATE TABLE kb_attachments (
  id          UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  tenant_id   UUID NOT NULL REFERENCES auth.users(id),
  entry_id    UUID REFERENCES kb_entries(id) ON DELETE SET NULL,
  filename    TEXT NOT NULL,
  storage_path TEXT NOT NULL,      -- Supabase Storage path
  mime_type   TEXT NOT NULL,
  size_bytes  BIGINT NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE kb_attachments ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can only access own attachments"
  ON kb_attachments FOR ALL
  USING (tenant_id = auth.uid());
```

Supabase Storage bucket: `kb-files`，按 `{tenant_id}/{category}/{filename}` 組織。

## 核心設計

### 1. API Routes（/api/kb）

POST method，body 中 `action` 分流：

| action | 用途 | 輸入 | 輸出 |
|--------|------|------|------|
| `list` | 列出某類別所有條目 | `{ category, status? }` | `KBEntry[]` |
| `get` | 取得單一條目 | `{ category, entryId }` | `KBEntry` |
| `create` | 新增條目 | `{ category, data }` | `{ id, entryId }` |
| `update` | 更新條目 | `{ category, entryId, data }` | `{ ok }` |
| `delete` | 刪除條目 | `{ category, entryId }` | `{ ok }` |
| `updateStatus` | 變更狀態 | `{ category, entryId, status }` | `{ ok }` |
| `stats` | 統計數量 | `{}` | `Record<KBId, { total, active, draft, archived }>` |

### 2. 搜尋 API（/api/kb/search）

```typescript
POST /api/kb/search
{
  query: string;         // 搜尋關鍵字
  categories?: KBId[];   // 限定類別（不傳=全部）
  status?: KBEntryStatus; // 限定狀態
  limit?: number;        // 預設 50
  offset?: number;
}
→ { results: KBEntry[], total: number }
```

搜尋使用 PostgreSQL `to_tsvector` 全文搜尋 + `ILIKE` 模糊匹配 fallback。

### 3. 批量匯入 API（/api/kb/import）

```typescript
POST /api/kb/import
{
  entries: Array<{
    category: KBId;
    data: object;        // 對應類別的完整欄位
  }>;
  mode: "append" | "replace";  // append=追加, replace=清空再匯入
}
→ { imported: number, errors: Array<{ index: number, error: string }> }
```

單次上限 500 筆，超過用分頁。replace 模式需額外確認。

### 4. 匯出 API（/api/kb/export）

```typescript
POST /api/kb/export
{
  format: "json" | "markdown";
  categories?: KBId[];    // 不傳=全部
  status?: KBEntryStatus; // 不傳=只 active
}
→ json: KnowledgeBaseData
→ markdown: { text: string, filename: string }
```

Markdown 格式用現有 `renderKBToMarkdown()` 生成——這是 KB → AI context 的核心輸出。

### 5. 離線策略

```
                                  ┌─────────────────┐
                                  │   Supabase DB    │  ← SSOT
                                  └────────┬────────┘
                                           │
                                  ┌────────▼────────┐
                                  │   /api/kb/*      │  ← API layer
                                  └────────┬────────┘
                                           │
                     ┌─────────────────────┼─────────────────────┐
                     │ online              │ offline              │
                     ▼                     ▼                     │
              直接打 API           localStorage cache             │
              回寫 cache           (readonly, stale ok)          │
                     │                     │                     │
                     └─────────────────────┼─────────────────────┘
                                           ▼
                                    useKnowledgeBase()
                                    (統一介面，對 UI 透明)
```

- 上線時：每次操作都走 API，API 回應後更新 localStorage 快取
- 離線時：讀快取（唯讀），寫入操作排隊等上線後同步
- 快取 key：`bidding-assistant-kb-cache`（與 v1 的 `bidding-assistant-knowledge-base` 不同，避免衝突）
- 快取內容：完整 `KnowledgeBaseData`，TTL 24 小時

### 6. 遷移路徑（v1 → v2）

```
v1 localStorage 資料  ──→  匯出 JSON  ──→  /api/kb/import  ──→  Supabase
                                                    ↑
                                            自動偵測：
                                            首次登入 Supabase 且
                                            localStorage 有 v1 資料
                                            → 提示一鍵遷移
```

遷移完成後：
1. localStorage 的 v1 key 標記為 `migrated`
2. 新的 cache key 接手
3. UI 完全不變（Hook 介面相同）

### 7. 與其他模組的整合

| 消費者 | 怎麼用 KB | 目前狀態 |
|--------|----------|---------|
| **提案組裝（assembly）** | `computeFileList()` 查詢各階段需要哪些 KB → Markdown 注入 prompt | ✅ 已透過 `kb-matrix.ts` 定義 |
