# M06 排版輸出模組（Export & Layout Module）

## 定位

排版輸出模組 = **從 AI 寫作成果到可交件文件的最後一哩路**。

M03 決定要不要投標，L1-L8 產出文字內容，M06 把內容變成客戶看得到的實體交件。

對應能力 6（00-總論）。解決痛點 #15 排版耗時、#16 格式不一致、#17 每次從零開始。

## 現有基礎

M06 **不是從零開始**。已有三個已實作的底層模組：

| 模組 | 位置 | 做什麼 | 狀態 |
|------|------|--------|------|
| docgen | `lib/docgen/` | Markdown → DOCX 轉換（封面、目錄、頁首頁尾、表格、列表） | 完成，1026 行測試 |
| assembly | `lib/assembly/` | 提示詞檔案組裝（token 估算、檔案清單、內容合併） | 完成，31 個測試 |
| 文件設定 | `lib/settings/` 的 DocumentSettings | 字型、字級、紙張、邊距、行距、頁首頁尾範本 | 完成 |

**M06 的角色**：在這三個底層模組之上，加入「文件組裝管線」——把 L1-L8 的 AI 文字、知識庫資料、公司資訊自動組裝成結構化章節，然後交給 docgen 排版輸出。

```
現有底層（不動）          M06 新增（本模組）
─────────────          ──────────────
docgen（排版引擎）   ←   文件組裝管線（章節組裝 + 範本套用）
assembly（提示詞組裝） ←   內容來源橋接（讀取 L1-L8 產出）
DocumentSettings     ←   範本系統（多套預設 + 品牌規範）
```

## 核心功能

### 功能 1：文件組裝管線（Document Assembly Pipeline）

把散落在各處的 AI 產出組裝成一份結構化文件。

**問題**：目前用戶必須手動複製 L3/L4 的 AI 文字、貼到文件生成工具、調章節順序。M06 把這個流程自動化。

**輸入來源**（三種，用戶選一）：

| 來源 | 操作方式 | 適用時機 |
|------|---------|---------|
| 手動貼入 | 在 UI 逐章輸入/貼上 Markdown 內容 | 現階段主要方式（L3/L4 產出複製過來） |
| 檔案匯入 | 拖入 .md/.txt 檔案，自動辨識章節 | 有完整 Markdown 檔時 |
| Notion 拉取 | 從案件資料庫拉對應案件的 L3/L4 產出 | 未來 Notion MCP 建好後（Phase 4） |

**組裝邏輯**：

```typescript
interface AssemblySource {
  type: "manual" | "file" | "notion";
  chapters: ChapterSource[];
}

interface ChapterSource {
  title: string;              // 章節標題（如「第壹章 專案理解」）
  content: string;            // Markdown 內容
  sourceStage?: string;       // 來源階段（L3/L4/manual）
  kbRefs?: string[];          // 引用的知識庫 ID（00A/00B/...）
}

interface AssemblyResult {
  chapters: ChapterInput[];   // 餵給 docgen 的結構
  metadata: {
    projectName: string;
    totalChars: number;
    chapterCount: number;
    kbRefsUsed: string[];
    assembledAt: string;      // ISO timestamp
  };
  warnings: AssemblyWarning[];
}

interface AssemblyWarning {
  type: "empty_chapter" | "missing_kb_ref" | "long_chapter" | "duplicate_content";
  chapter: number;
  message: string;
}
```

**組裝步驟**：

1. **讀取來源** → 取得各章原始內容
2. **結構驗證** → 檢查空章節、異常長度、重複段落
3. **知識庫注入**（選用）→ 將 00A-00E 的特定條目嵌入指定位置
4. **品牌文字替換** → 公司名稱、聯絡資訊等佔位符替換
5. **產出 ChapterInput[]** → 交給 docgen

### 功能 2：範本系統（Template System）

不同類型的文件有不同的章節結構和格式要求。

**內建範本**：

| 範本 ID | 名稱 | 章節數 | 說明 |
|---------|------|--------|------|
| `proposal-standard` | 標準建議書 | 5-10 | 政府採購標案建議書，五章制或依需求書自訂 |
| `proposal-simplified` | 簡式建議書 | 3-5 | 小型標案/勞務採購，精簡版 |
| `plan-brief` | 企劃摘要 | 2-3 | L2 備標規劃產出的精簡版 |
| `custom` | 自訂 | 不限 | 用戶自行定義章節結構 |

**範本定義**：

```typescript
interface DocumentTemplate {
  id: string;
  name: string;
  description: string;
  chapters: TemplateChapter[];
  defaultSettings: Partial<DocumentSettings>;  // 範本專屬預設（覆蓋全域設定）
  variables: TemplateVariable[];               // 範本內佔位符
}

interface TemplateChapter {
  defaultTitle: string;       // 預設標題（如「第壹章 專案理解與服務理念」）
  required: boolean;          // 是否必填
  suggestedLength: string;    // 建議字數（如「2000-3000」）
  kbSuggestions: string[];    // 建議引用的知識庫（如 ["00B", "00D"]）
  guideText: string;          // 撰寫指引（顯示在 UI 的提示文字）
}

interface TemplateVariable {
  key: string;                // 如 "projectName", "companyName"
  label: string;              // 中文標籤
  source: "settings" | "manual";  // 從設定自動帶入 or 用戶填
  defaultValue?: string;
}
```

**範本儲存**：

- 內建範本存 `src/data/config/document-templates.ts`（靜態設定，同 stages.ts 模式）
- 用戶自訂範本存 localStorage（同現有設定系統模式）
- 不另建資料庫

### 功能 3：多格式輸出（Multi-format Export）

在 DOCX 基礎上擴充其他格式。

| 格式 | 狀態 | 實作方式 | 優先序 |
|------|------|---------|--------|
| DOCX | 已實作 | `docgen/generate-docx.ts`（docx 套件） | — |
| PDF | Phase 3 | 瀏覽器 `window.print()` + CSS print stylesheet | 中 |
| Markdown 匯出 | Phase 2 | 直接打包原始 Markdown | 高 |
| 簡報大綱 | Phase 4 | 產出 L6 簡報架構的結構化文字 → Canva MCP | 低 |

**PDF 策略決策**：選擇 CSS print stylesheet + 瀏覽器列印，不引入 puppeteer/headless chrome。理由：

- 零額外依賴（不增加 node_modules）
- Vercel serverless 不適合跑 headless browser
- 政府標案以 Word 為主，PDF 是次要需求
- 用戶可從 DOCX 直接轉 PDF（Word 內建功能）

```typescript
interface ExportOptions {
  format: "docx" | "markdown" | "print";
  template: string;           // 範本 ID
  chapters: ChapterInput[];
  documentSettings: DocumentSettings;
  companySettings: CompanySettings;
  coverPage?: boolean;
  tableOfContents?: boolean;
  includeMetadata?: boolean;   // Markdown 匯出時是否包含 YAML frontmatter
}

// 匯出結果
type ExportResult =
  | { format: "docx"; blob: Blob; filename: string }
  | { format: "markdown"; text: string; filename: string }
  | { format: "print"; html: string };  // 注入到 iframe 觸發列印
```

## 資料流

```
用戶操作（選範本 + 填/匯入內容）
    ↓
AssemblyPipeline（組裝 + 驗證 + KB 注入 + 佔位符替換）
    ↓
ChapterInput[]（結構化章節）
    ↓
[選用] 品質檢查（M04 閘門，Phase 3）
    ↓
ExportEngine（依格式分發）
    ├── DOCX → generateDocx()（現有 docgen）
    ├── Markdown → 打包純文字
    └── Print → 生成 HTML + CSS print stylesheet
    ↓
downloadBlob() / 瀏覽器列印
```

## 知識庫整合

文件中經常需要嵌入知識庫內容（如團隊簡介、過往實績、時程範本）。

**整合方式**：在章節內容中使用佔位符，組裝管線自動替換。

```markdown
## 執行團隊

{{kb:00A:PM}}

本團隊具備以下相關實績：

{{kb:00B:recent:3}}
```

**佔位符語法**：

| 佔位符 | 說明 |
|--------|------|
| `{{kb:00A:角色}}` | 插入 00A 團隊資料中指定角色的簡介 |
| `{{kb:00B:recent:N}}` | 插入 00B 實績庫中最近 N 筆相關實績 |
| `{{kb:00C:標準}}` | 插入 00C 時程範本庫中指定範本 |
| `{{kb:00D:類別}}` | 插入 00D 應變 SOP 庫中指定類別 |
| `{{company:name}}` | 公司名稱（從設定帶入） |
| `{{company:taxId}}` | 統一編號 |
| `{{project:name}}` | 案件名稱 |
| `{{date:roc}}` | 民國年月日 |

**KB 資料讀取**：透過現有 `lib/knowledge-base/` 的型別和載入機制。Phase 1 只支援手動貼入的 KB 內容（localStorage），Phase 4 支援從 Notion 即時拉取。

```typescript
interface KBInjector {
  resolve(placeholder: string, kbData: KnowledgeBaseData): string;
  findPlaceholders(content: string): KBPlaceholder[];
  validate(placeholders: KBPlaceholder[], kbData: KnowledgeBaseData): KBValidationResult;
}

interface KBPlaceholder {
  raw: string;              // 原始佔位符文字
  category: "00A" | "00B" | "00C" | "00D" | "00E";
  selector: string;         // 角色/類別/篩選條件
  params?: Record<string, string>;
}

interface KBValidationResult {
  valid: boolean;
  unresolved: KBPlaceholder[];  // 找不到對應資料的佔位符
}
```

## 設計決策

### 決策 1：新模組 vs 擴充 docgen

**選擇**：新建 `lib/output/`，不改動 `lib/docgen/`。

理由：
- docgen 是純排版引擎（Markdown → DOCX），職責單一且穩定
- M06 管線邏輯（範本、KB 注入、驗證）不屬於排版引擎的責任
- docgen 已有 1026 行測試，改動有破壞風險
- `lib/output/` 透過 import 使用 docgen，是消費者而非修改者

否決方案：直接在 docgen 加管線邏輯 → 模組膨脹、職責混亂。

### 決策 2：PDF 生成方式

**選擇**：瀏覽器 `window.print()` + CSS print stylesheet（Phase 3）。

理由：
- 零依賴（不裝 puppeteer、wkhtmltopdf）
- Vercel serverless 不適合 headless browser
- 政府標案文件以 Word 為主，PDF 需求次要
- 用戶可從 Word 另存 PDF（品質更好）

否決方案：
- puppeteer → 佈署限制 + 500MB+ 依賴
- react-pdf → 排版引擎完全不同，雙倍維護成本
- 付費 API（Gotenberg 等）→ 違反 $0 成本原則

### 決策 3：範本儲存方式

**選擇**：內建範本存 `src/data/config/`，自訂範本存 localStorage。

理由：
- 同 stages.ts、kb-matrix.ts 的模式（靜態設定檔）
- 自訂範本數量少（預計 < 10 個），localStorage 足夠
- 不需要另建資料庫或檔案系統

否決方案：
- Notion 儲存 → 增加外部依賴，離線不可用
- 本地 JSON 檔案 → Next.js 前端無法直接寫檔

### 決策 4：知識庫佔位符語法

**選擇**：雙大括號 `{{kb:00A:PM}}`。

理由：
- 不會與 Markdown 語法衝突
- 可以用正則簡單解析
- 同頁首頁尾的佔位符風格（`{{案名}}`），使用者學一次就好

### 決策 5：與 M04 品質閘門的整合時機

**選擇**：Phase 3 整合。Phase 1-2 先不加品質檢查。

理由：
- M04 尚在草案階段，介面未定
- 文件輸出的核心價值是「快速生成」，不是「品質把關」
- Phase 3 時 M04 應已有穩定介面，屆時加入「匯出前自動檢查」

### Phase 2：知識庫注入 + Markdown 匯出

**目標**：文件中可嵌入知識庫內容，支援 Markdown 格式匯出。

**新建檔案**：
- `lib/output/kb-injector.ts` — 佔位符解析與替換
- `lib/output/markdown-export.ts` — Markdown 匯出
- `components/output/KBInsertDialog.tsx` — KB 插入 UI
- 測試：kb-injector + markdown-export

## 與其他模組的關係

```
M01（情報）→ M03（戰略）→ L1-L8 → M06（輸出）
                                      ↑
M04（品質閘門）──── Phase 3 整合 ──────┘
                                      ↑
KB（00A-00E）──── 內容注入 ───────────┘
```

- **M01 → M06**：無直接依賴。M01 的情報餵給 M03 做決策，不直接進文件。
- **M03 → M06**：間接關係。M03 決定投標後，L2-L4 的產出才需要 M06 排版。
- **M04 → M06**：Phase 3 整合。M06 匯出前呼叫 M04 做品質檢查（可選）。
- **KB → M06**：直接依賴。佔位符語法讀取 00A-00E 內容嵌入文件。
- **docgen → M06**：M06 是 docgen 的消費者，透過 `generateDocx()` 做最終排版。
