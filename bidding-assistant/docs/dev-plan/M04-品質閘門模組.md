# M04 品質閘門模組（Quality Gate Module）

## 定位

品質閘門 = **每段 AI 產出經過三道檢查才放行**。

現有的品質檢查模組（`lib/quality/`）是 M04 的**第零道**——文字表面品質。M04 在此基礎上加三道深層檢查，從「有沒有錯字」提升到「有沒有說謊、有沒有回應需求、做不做得到」。

對應能力 4（00-總論）。解決痛點 4 條：#10 AI 幻覺抓不完、#11 AI 不自檢死不認錯、#12 寫完像初稿、#23 AI 創意沒常識。

## 與現有品質模組的關係

```
現有 lib/quality/         M04 品質閘門（新增）
─────────────────         ──────────────────
閘門 0：文字品質           閘門 1：事實查核
 - 禁用詞                  - 知識庫來源追溯
 - 用語修正                - 數據來源驗證
 - 鐵律（數字/日期/預算）   - 幻覺偵測
 - 重複句
 - 承諾風險               閘門 2：需求對照
 - 段落/句子長度            - 評分項目覆蓋率
 - 公司名稱一致性           - 招標條件對應
                           - 遺漏偵測

                          閘門 3：實務檢驗
                           - 預算可行性
                           - 執行能力驗證
                           - 常識檢查
```

閘門 0 已實作（`lib/quality/rules.ts`，13 條規則，55 個測試）。M04 專注於閘門 1-3。

## 三道閘門設計

### 閘門 1：事實查核（Fact Check）

> 這段話有來源嗎？數字從哪來的？

**目的**：偵測 AI 幻覺和無來源的宣稱。

**檢查項目**：

| 檢查 | 方法 | 嚴重度 |
|------|------|--------|
| 知識庫來源追溯 | 每個事實宣稱必須對應到 00A-00E 的具體 entry | error |
| 數據來源標記 | 涉及數字的句子必須標記來源（KB/PCC/招標文件） | error |
| 實績引用驗證 | 提到的實績必須在 00B 中存在 | error |
| 團隊資歷驗證 | 提到的人員資歷必須與 00A 一致 | error |
| 虛構偵測 | 模式比對常見 AI 幻覺特徵（見下方） | warning |

**AI 幻覺特徵偵測**（確定性規則，不靠 AI）：

```typescript
const HALLUCINATION_PATTERNS = [
  // 虛構的具體數字（沒有來源但非常具體）
  { pattern: /提升了?\s*\d{2,3}%/, flag: "具體百分比需要來源" },
  // 虛構的調查/研究
  { pattern: /根據.*調查|研究(指出|顯示|發現)/, flag: "引用的研究需要具體出處" },
  // 過度修飾的無來源主張
  { pattern: /業界(首創|領先|第一|唯一)/, flag: "業界定位宣稱需要佐證" },
  // 虛構的統計
  { pattern: /每年(服務|接待|處理).*[萬千百]\s*(人次?|件|場)/, flag: "服務量數據需要來源" },
];
```

**知識庫追溯機制**：

```typescript
interface SourceAnnotation {
  sentenceIndex: number;      // 句子序號
  claim: string;              // 宣稱內容
  source: KBSourceRef | null; // 來源引用（null = 無來源）
  confidence: "verified" | "partial" | "unverified";
}

interface KBSourceRef {
  kbId: KBId;         // "00A" | "00B" | ...
  entryId: string;    // "M-001" | "P-2025-001" | ...
  field: string;      // 哪個欄位
}
```

### 閘門 2：需求對照（Requirement Tracing）

> 這段回應了哪個評分項目？有沒有漏掉的？

**目的**：確保建議書覆蓋所有招標需求。

**需求追溯矩陣**：

```typescript
interface RequirementMatrix {
  requirements: Requirement[];         // 從招標文件提取的需求
  coverage: CoverageResult[];          // 每個需求的覆蓋狀態
  uncoveredCount: number;              // 未覆蓋數量
  coverageRate: number;                // 覆蓋率 0-100%
}

interface Requirement {
  id: string;                   // "R-01"
  source: string;               // "評分項目第3條" | "資格條件"
  description: string;          // 需求描述
  weight: number | null;        // 評分權重（如有）
  category: "評分" | "資格" | "交付" | "格式";
}

interface CoverageResult {
  requirementId: string;
  status: "covered" | "partial" | "missing";
  coveredBy: string[];          // 哪些章節/段落回應了這個需求
  gap: string | null;           // 缺口描述
}
```

**運作方式**：

1. **需求提取**（半自動）：L1 戰略分析產出需求清單 → 使用者確認/修改 → 存入案件 context
2. **覆蓋偵測**（自動）：比對建議書各章節與需求清單的關鍵字匹配
3. **缺口報告**（自動）：列出未覆蓋的需求 + 嚴重程度（評分權重高的優先）

**限制**：需求提取依賴 L1 產出，M04 本身不做招標文件解析。如果沒有需求清單，閘門 2 跳過（不擋人）。

### 閘門 3：實務檢驗（Feasibility Check）

> 預算做得到嗎？時間夠嗎？公司有能力嗎？

**目的**：攔截「聽起來厲害但做不到」的方案。

**檢查項目**：

| 檢查 | 方法 | 嚴重度 |
|------|------|--------|
| 預算可行性 | 承諾事項的估算成本 vs 案件預算 | error |
| 時程可行性 | 承諾時程 vs 合約期間 + 00C 時程範本經驗值 | warning |
| 團隊能力 | 承諾的人力配置 vs 00A 實際團隊 | warning |
| 規模合理性 | 承諾規模 vs 00B 歷史實績的規模 | warning |
| 常識檢查 | 模式比對不合理的方案（見下方） | warning |

**常識檢查規則**（確定性規則）：

```typescript
const COMMON_SENSE_RULES = [
  // AI 常見的不切實際建議
  { pattern: /AR|VR|元宇宙|區塊鏈/, context: "budget < 500000",
    message: "預算低於 50 萬的案子不太可能負擔 AR/VR/元宇宙" },
  { pattern: /大數據分析/, context: "participants < 100",
    message: "參與者不到 100 人不需要大數據分析" },
  { pattern: /AI\s*(模型|系統|平台).*開發/, context: "budget < 1000000",
    message: "自建 AI 系統通常需要百萬以上預算" },
  { pattern: /國際(連線|合作|交流)/, context: "budget < 300000",
    message: "國際合作涉及旅費/翻譯，低預算案件需謹慎" },
];
```

**預算可行性引擎**：

```typescript
interface BudgetFeasibility {
  totalBudget: number;               // 案件預算
  estimatedCosts: CostItem[];        // 從建議書提取的成本項目
  totalEstimate: number;             // 估算總成本
  margin: number;                    // 預算餘裕 %
  verdict: "充裕" | "合理" | "緊繃" | "超支";
  warnings: string[];
}

interface CostItem {
  description: string;
  estimatedAmount: number;
  source: "explicit" | "inferred";   // 明確寫出 vs 從描述推算
  confidence: "高" | "中" | "低";
}
```

## 整合設計

### 品質報告

三道閘門 + 閘門 0 的結果合併為一份品質報告：

```typescript
interface QualityReport {
  // 閘門 0（現有）
  textQuality: QualityScore;
  textIssues: CheckResult[];

  // 閘門 1
  factCheck: {
    verifiedClaims: number;
    unverifiedClaims: number;
    hallucinations: SourceAnnotation[];
    score: number;        // 0-100
  };

  // 閘門 2
  requirementTrace: {
    coverageRate: number;
    uncovered: CoverageResult[];
    matrix: RequirementMatrix;
  } | null;               // null = 沒有需求清單

  // 閘門 3
  feasibility: {
    budget: BudgetFeasibility | null;
    timeline: TimelineFeasibility | null;
    team: TeamFeasibility | null;
    commonSense: CheckResult[];
    score: number;
  };

  // 總評
  overallScore: number;   // 四道閘門加權平均
  verdict: "通過" | "有風險" | "不建議提交";
  criticalIssues: string[];
}
```

### 品質報告的使用時機

```
L1 戰略分析 → 需求清單（閘門 2 的輸入）
     ↓
L2 備標規劃 → 預算分配（閘門 3 的輸入）
     ↓
L3 企劃草稿 → 【閘門 0+1+2+3 全跑】
     ↓
修正 → 再跑 → 通過
     ↓
L4 定稿撰寫 → 【閘門 0+1+2+3 全跑】
     ↓
提交
```

每個 L 階段產出後都能跑品質報告，但完整四道閘門在 L3/L4 最有價值。

### 與 M03 的關係

- M03 提供作戰計畫 → 閘門 2 用它作為需求覆蓋的基準
- M03 的適配度評分中的團隊/規模資料 → 閘門 3 用來驗證可行性
- 品質報告的結果可以回饋到 M03 的適配度校準（未來飛輪）

## 實作分期

### Phase 1：閘門 1 事實查核

**產出**：`types.ts` + `constants.ts` + `fact-check.ts` + `helpers.ts` + 測試

**驗收標準**：
- 幻覺模式偵測能正確抓到常見 AI 幻覺（≥ 5 個測試）
- 知識庫來源比對能匹配到正確的 entry（空 KB 不壞）
- 來源追溯標記格式正確

### Phase 2：閘門 2 需求追溯 + 閘門 3 可行性

**產出**：`requirement-trace.ts` + `feasibility.ts` + 測試

**驗收標準**：
- 需求矩陣能正確計算覆蓋率（含 partial）
- 預算可行性能偵測超支
- 常識檢查能正確觸發（低預算 + AR = 警告）

### Phase 3：品質報告 + UI

**產出**：`report.ts` + `useQualityGate.ts` + 全部 UI 元件 + 頁面

**依賴**：Phase 1-2 + 現有 lib/quality/

**驗收標準**：
- 四道閘門結果整合到一份報告
- 總評分數正確計算
- 品質不佳時能列出關鍵問題

### Phase 4：與 L 階段整合

**產出**：在 L3/L4 產出流程中嵌入品質閘門觸發點

**依賴**：Phase 3 + L3/L4 prompt 系統

**驗收標準**：
- L3 產出後能自動觸發品質報告
- 品質不通過時能引導修正方向

## 設計決策

### 為什麼新建 `lib/quality-gate/` 而不是擴展 `lib/quality/`？

1. **關注點不同**：`lib/quality/` 做文字表面檢查（regex 為主），`quality-gate/` 做語義層檢查（需要知識庫上下文）
2. **依賴不同**：閘門 0 不依賴任何其他模組，閘門 1-3 依賴知識庫和案件上下文
3. **可獨立開關**：使用者可能只想用閘門 0（快速文字檢查），不需要載入整個品質閘門
4. **不破壞現有**：lib/quality/ 已有 55 個測試，不碰它

### 為什麼閘門 1-3 盡量用確定性規則而不是全交給 AI？

同 M03 的理由：可解釋、可測試、$0。AI 做的是 L 階段的產出，品質閘門負責在 AI 產出後做客觀檢查——讓 AI 檢查 AI 會陷入同樣的盲區。

確定性規則能抓到的問題（佔 70-80%）：
- 無來源的數字宣稱 → regex
- KB 中不存在的實績引用 → 字串比對
- 預算超支 → 數學計算
- 常識違規 → 模式比對

需要 AI 判斷的問題（佔 20-30%），Phase 4 才做：
- 邏輯不一致（前後矛盾）
- 策略層的遺漏（有覆蓋但深度不夠）
- 創意方案的可行性（不在規則庫中的新方案）

### 為什麼閘門 2 允許沒有需求清單時跳過？

需求清單來自 L1，如果使用者只是想快速檢查一段文字（沒有走完整 L1 流程），閘門 2 不應該擋住。品質閘門的原則是**有多少資料做多少檢查**，不因缺少資料而阻止使用。
