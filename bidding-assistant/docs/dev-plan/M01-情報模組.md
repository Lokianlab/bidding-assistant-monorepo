# M01 情報模組（Intelligence Module）

## 定位

情報模組 = **戰略分析的數據引擎**，供給 L1 所需的所有硬數據。
系統的「眼睛」——看不清對手就打不準。

## 核心設計

### 1. API Route（/api/pcc）

POST method，body 中 `action` 分流：
- `searchByTitle` / `searchByCompany` / `listByUnit` / `tenderDetail` — 單次查詢
- `fetchAllPages` — 伺服器端自動翻頁到底（交叉比對基礎）
- `batchCompanySearch` — 批次查多家廠商

翻頁間 300ms 延遲，上限 20 頁。

### 2. 統編優先原則

- **主要識別**：統編（8位數字）→ 所有交叉比對、統計、分組用統編做 key
- **輔助顯示**：名稱正規化（去英文括號、空白）→ 只用於 UI 顯示
- 大員洛川統編：`89170941`
- `companyId → displayName` 映射表在分析過程中自動累積

### 3. 角色解析

從 `brief.companies.name_key` 和 `id_key` 解析：
- `得標廠商N:得標廠商` → winner
- `未得標廠商N:未得標廠商` → loser
- `投標廠商:投標廠商N` → bidder

### 4. 五大交叉比對分析

#### 4a. 競爭對手分析
- 輸入：大員洛川所有紀錄（139 筆）
- 處理：遍歷決標公告 → 以統編為 key 統計
- 產出：對手排行（撞案次數、勝率、擅長領域、偏好機關）
- 門檻：撞案 >= 2 次

#### 4b. 機關偏好分析
- 輸入：指定機關歷史標案
- 產出：在位者（連續得標 >= 2 次）、平均投標家數、我方紀錄

#### 4c. 市場趨勢分析
- 輸入：同類標案搜尋結果
- 產出：按年度彙整案件數、投標家數趨勢、紅海/藍海判斷

#### 4d. 自我分析
- 輸入：大員洛川所有紀錄
- 產出：按機關/年度/領域分類、勝率、對手排行、落選原因統計

#### 4e. 評委分析
- 輸入：逐案 tenderDetail（按需查詢）
- 產出：評委出現頻率、關聯機關、專長

### 5. P 偵察報告整合

輸入案件資訊 + 分析結果 → 輸出可直接貼入 Perplexity 的 prompt：
- 硬數據已預填
- Perplexity 只需補軟情報

### 6. 快取策略

| 資料類型 | TTL | 理由 |
|---------|-----|------|
| 搜尋結果 | 4 小時 | 每天有新案 |
| 自家紀錄 | 24 小時 | 更新頻率低 |
| 標案詳情 | 7 天 | 決標後不變 |
| 分析報告 | 12 小時 | 衍生計算 |

儲存：localStorage + 自動清理過期 + 手動清除按鈕。
