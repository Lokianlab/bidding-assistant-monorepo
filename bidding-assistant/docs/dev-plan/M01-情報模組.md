# M01 情報模組（Intelligence Module）

## 定位

情報模組 = **戰略分析的數據引擎**，供給 L1 所需的所有硬數據。
系統的「眼睛」——看不清對手就打不準。

## 資料來源

| 來源 | 狀態 | 用途 |
|------|------|------|
| g0v PCC API（詳見 A01） | 已就緒 | 標案搜尋、競爭分析、評委比對 |
| Notion 案件資料庫 | **暫緩整合** | 案件管理、知識庫查詢（目前用 Claude 內建 Notion MCP 即可） |

暫緩原因：Claude.ai 內建的 Notion MCP 已能滿足目前需求，自建版本延後到多 agent 協作（GPT/Gemini 也需要存取 Notion）的階段再啟動。

## 核心設計

### 1. API Route（/api/pcc）

POST method，body 中 `action` 分流：
- `searchByTitle` / `searchByCompany` / `listByUnit` / `tenderDetail` — 單次查詢
- `fetchAllPages` — 伺服器端自動翻頁到底（交叉比對基礎）
- `batchCompanySearch` — 批次查多家廠商

翻頁間 300ms 延遲，上限 20 頁。

### 2. 統編優先原則

- **主要識別**：統編（8位數字）→ 所有交叉比對、統計、分組用統編做 key
- **輔助顯示**：名稱正規化（去英文括號、空白）→ 只用於 UI 顯示
- 大員洛川統編：`89170941`
- `companyId → displayName` 映射表在分析過程中自動累積

### 3. 角色解析

從 `brief.companies.name_key` 和 `id_key` 解析：
- `得標廠商N:得標廠商` → winner
- `未得標廠商N:未得標廠商` → loser
- `投標廠商:投標廠商N` → bidder

### 4. 五大交叉比對分析

#### 4a. 競爭對手分析
- 輸入：大員洛川所有紀錄（139 筆）
- 處理：遍歷決標公告 → 以統編為 key 統計
- 產出：對手排行（撞案次數、勝率、擅長領域、偏好機關）
- 門檻：撞案 >= 2 次

#### 4b. 機關偏好分析
- 輸入：指定機關歷史標案
- 產出：在位者（連續得標 >= 2 次）、平均投標家數、我方紀錄

#### 4c. 市場趨勢分析
- 輸入：同類標案搜尋結果
- 產出：按年度彙整案件數、投標家數趨勢、紅海/藍海判斷

#### 4d. 自我分析
- 輸入：大員洛川所有紀錄
- 產出：按機關/年度/領域分類、勝率、對手排行、落選原因統計

#### 4e. 評委分析
- 輸入：逐案 tenderDetail（按需查詢）
- 產出：評委出現頻率、關聯機關、專長

### 5. P 偵察報告整合

輸入案件資訊 + 分析結果 → 輸出可直接貼入 Perplexity 的 prompt：
- 硬數據已預填
- Perplexity 只需補軟情報

### 6. 快取策略

| 資料類型 | TTL | 理由 |
|---------|-----|------|
| 搜尋結果 | 4 小時 | 每天有新案 |
| 自家紀錄 | 24 小時 | 更新頻率低 |
| 標案詳情 | 7 天 | 決標後不變 |
| 分析報告 | 12 小時 | 衍生計算 |

儲存：localStorage + 自動清理過期 + 手動清除按鈕。

## 附錄 A：Notion 案件資料庫結構

> 來源：Notion API 回應解析，供未來 Notion 整合參考。

**Collection ID**: `14cc71c7-7278-81cf-8521-000b97d017d1`

### 資料庫架構

| 資料庫 | Views 數量 | 用途 |
|--------|-----------|------|
| 主資料庫（Main DB） | 2 | 資料檢查表、全欄位瀏覽 |
| 子資料庫 DB1 | 5 | 備標中/第二順位/第三順位/時間軸/待確認協作 |
| 子資料庫 DB2 | 4 | 看板/行事曆（應啟動日/備標期限/截標時間） |

### 主要 View 功能摘要

- **資料檢查表**：找出資料不完整的案件（空欄位或標記「有問題」）
- **備標中案件**：截標日未到 + 第一順位 + 確定協作 + 未完稿
- **看板**：按備標決策分組的卡片（入選→領標→封標→各順位）
- **三個行事曆**：分別按應啟動日、備標期限、截標時間顯示

### 已知欄位清單

**核心欄位**：標案名稱(title)、招標機關(text)、案號(text)、案件唯一碼、歸檔號、預算金額(number)

**時間欄位**：截標時間(date)、備標期限、公告日、應啟動日

**狀態欄位**：
- 標案進程(status): To-do / In progress / Complete / 競標階段 / 已出席簡報 / 已投標
- 備標決策(select): 入選 / 第一順位案件 / 第二順位案件 / 第三順位案件 / 請行政人員領標填印 / 請行政人員封標交寄
- 決標公告(select): 決標歸檔 / ...
- 備標進度(multi_select): 建議書完稿 / ...

**分類欄位**：標案類型(multi_select)、檔案型態(select)、評審方式、決標方式、電子投標

**人員/協作欄位**：企劃主筆、企劃註記(select)、確定協作(checkbox)、檢新人員、資料有問題(checkbox)

**標記欄位**：逾期、不參與投標、第一/二/三順位、企劃完成、標單已交遞、分析作業完成、投標地址

共 89 個欄位，以上為從 views 的 filter/display 中提取的部分。完整欄位需從 Notion API 的 `retrieve database` 端點取得。

## 附錄 B：自建 Notion MCP Server 規格（暫緩）

> **狀態：暫緩**。Claude.ai 內建的 Notion MCP 已能滿足現階段需求。此規格保留供未來啟動參考。
>
> **啟動條件**：產品層多 agent 協作上線，GPT/Gemini 也需要存取 Notion 資料時。

### 為什麼要建

- 雲端 Notion MCP（Claude.ai 連結）只有 Claude 能用
- Agent Team 的品管官（GPT）、知識官（Gemini）也需要存取 Notion
- 本地 MCP 走標準協議，三家 CLI 都能用
- 封裝標案業務邏輯，agent 不需要懂 Notion schema

### 技術規格

```
notion-mcp/
  src/index.ts     ← MCP server（@modelcontextprotocol/sdk + @notionhq/client）
  dist/index.js    ← 編譯後
  package.json
  tsconfig.json
```

結構同 smugmug-mcp，認證用 env vars（NOTION_TOKEN + 各 database ID），註冊到 `.mcp.json`。

### Notion 資料結構

```
Notion
├── 案件資料庫（主庫）
│   ├── 案號、案名、機關、預算、截標日
│   ├── 狀態（追蹤中/分析中/寫作中/已投標/結案）
│   ├── 企劃認領人
│   ├── L1-L8 進度
│   └── 備標評估文件（PDF 嵌入）
│
├── 知識庫
│   ├── 00A 團隊資料庫
│   ├── 00B 實績資料庫
│   ├── 00C 時程範本庫
│   ├── 00D 應變 SOP 庫
│   ├── 00E 案後檢討庫
│   └── 00F 創意庫
│
└── 執行企劃作業流程工作手冊（30 節點）
```

### 工具清單（15 個）

**一、案件管理（5 個）**：list_cases、get_case、create_case、update_case、list_deadlines

**二、知識庫讀取（5 個）**：kb_search、kb_get_team、kb_get_portfolio、kb_get_sop、kb_get_review

**三、知識庫寫入（2 個）**：kb_add_entry、kb_update_entry

**四、案件上下文存取（3 個）**：save_artifact、get_artifact、list_artifacts

### 與雲端 Notion MCP 的關係

| | 雲端 Notion MCP | 本地 Notion MCP |
|---|---|---|
| 定位 | 通用 Notion 操作 | 標案系統專用 |
| 可用 provider | 只有 Claude | Claude / GPT / Gemini |
| 業務邏輯 | 無（通用 CRUD） | 有（案件管理、KB 分類、產物存取） |
| 使用時機 | 開發/除錯時直接操作 Notion | 產品層 agent 日常使用 |

兩者並存不衝突。雲端版用於臨時操作，本地版用於系統自動化。
