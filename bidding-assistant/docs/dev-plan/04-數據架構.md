# 04 數據架構

## 一、資料來源與角色

| 來源 | 角色 | 讀/寫 | 對應模組 |
|------|------|-------|---------|
| Notion API | 標案資料庫（主資料） | 讀+寫 | 指揮部、績效、案件看板 |
| PCC API（政府採購網） | 標案公告搜尋 | 唯讀 | 偵察報告（P 系列） |
| localStorage | 應用程式設定 + 快取 | 讀+寫 | 全域設定、儀表板佈局 |
| 檔案系統（`public/prompts/`） | Prompt 模板 | 唯讀 | AI 階段（L1-L8）、工具（T1） |
| SmugMug API | 專案照片 | 唯讀 | 實績展示 |

## 二、Notion API 資料流

### 欄位對照機制

系統用內部 key 操作資料，透過欄位對照表映射到 Notion 實際欄位名稱：

```
系統內部 key（FIELD_KEYS）
  → DEFAULT_FIELD_MAP（預設對照）
  → resolveFieldMap()（合併使用者自訂覆蓋）
  → 實際 Notion API 欄位名稱
```

- 定義：`src/lib/constants/field-mapping.ts`（21 個欄位）
- 使用者可在設定頁覆蓋欄位名稱（`AppSettings.fieldMapping`），不改程式碼
- 覆蓋值存 localStorage，空時用預設

### 查詢最佳化

每個頁面只查需要的欄位（`filter_properties`），加速回應：

| 頁面 | 欄位數 | 定義 |
|------|--------|------|
| 指揮部 | 17 | `FIELDS_DASHBOARD` |
| 績效中心 | 13 | `FIELDS_PERFORMANCE` |
| KPI 統計 | 5 | `FIELDS_DASHBOARD_KPI` |

定義位置：`src/lib/constants/notion-fields.ts`

### API Route 架構

所有 Notion 操作透過 `/api/notion` 代理（避免前端暴露 token）：

| action | 用途 | 回傳 |
|--------|------|------|
| `schema_and_query` | 首次載入（schema + 資料） | schema + pages + cursor |
| `continue_query` | 背景續抓（分頁） | pages + cursor |
| `query` | 一般查詢 | pages |
| `update` | 更新案件屬性 | success |
| `schema` | 純 schema（設定用） | schema |

載入策略：`quickLoad` 模式先回傳第一批（100 筆），前端背景續抓剩餘分頁。
分頁上限：每次 API call 最多串接 20 頁（防止 Serverless timeout）。

### Notion 資料解析

`parsePage()` 統一處理 Notion 的 12 種屬性類型（title、rich_text、number、select、multi_select、date、checkbox、url、status、people、unique_id、formula、rollup），輸出為簡化的 key-value。前端不直接處理 Notion 原始格式。

## 三、localStorage 儲存策略

### 設定存取

| key | 內容 | 大小估計 |
|-----|------|---------|
| `bidding-assistant-settings` | 全部 `AppSettings`（JSON） | ~5-15 KB |

- 存取方式：`useSettings()` Hook → `SettingsProvider` context
- Hydration 安全：SSR 時回傳 `DEFAULT_SETTINGS`，client 端 mount 後從 localStorage 載入
- 合併策略：`{ ...DEFAULT_SETTINGS, ...JSON.parse(saved) }`（新增欄位自動 fallback）

### 其他 localStorage 使用

| key | 模組 | 用途 |
|-----|------|------|
| 知識庫自訂選項 | `useCustomOptions` | 團隊成員、機關等自訂下拉選項 |
| 知識庫內容 | `useKnowledgeBase` | 00A-00E 知識庫本地存放 |
| 日誌 | `logger` | 操作日誌暫存 |
| 儀表板佈局 | `useCardLayout` | 卡片排列偏好 |
| 側欄狀態 | `Sidebar` | 展開/收合 |

### 快取策略（Phase 1）

Phase 1 不做獨立快取層。Notion 資料每次載入即時查詢，localStorage 只存設定不存資料快取。

原因：
- 自用階段資料量小（<500 筆），Notion API 回應快（~1-3 秒）
- 快取失效邏輯的複雜度不值得在 Phase 1 引入

Phase 2 考慮方向：
- 短期快取（5 分鐘 TTL）避免重複查詢
- SWR 模式（stale-while-revalidate）改善感知效能

## 四、PCC API 資料流

透過 PCC MCP Server（`pcc-api-mcp/`）存取，6 個工具：

| 工具 | 輸入 | 輸出 |
|------|------|------|
| search_tender | 標題關鍵字 | 標案清單 |
| search_company | 公司名稱 | 公司投標紀錄 |
| list_agency_tenders | 機關 ID | 機關所有標案 |
| get_tender_detail | unit_id + job_number | 標案完整資訊（含評委） |
| list_agencies | — | 機關列表 |
| get_api_info | — | 系統資訊 |

資料流向：
```
PCC 政府採購網 API
  → PCC MCP Server（轉譯）
  → Claude / AI Agent（分析 + 產出 P 偵察報告）
  → 使用者（透過 SaaS 網頁或 chat 介面）
```

PCC 資料不存入 Notion，僅作為 AI 分析的輸入素材。分析結果（P 偵察報告）是一次性產出，隨案件走。

## 五、檔案系統

### Prompt 模板（`public/prompts/`）

```
public/prompts/
  stages/
    01.md          ← L1 戰略分析（生產版，v2.3）
    02.md ~ 08.md  ← L2-L8（待寫）
  tools/
    T1.md          ← T1 知識庫精靈（v2.0，914 行）
    P-template.md  ← P 偵察報告模板（待建）
  knowledge/
    00A.md ~ 00E.md  ← 知識庫（由 T1 產出，Phase 1 存本地檔案）
```

透過 `/api/prompts` route 讀取（安全檢查：只允許 `.md`、禁止路徑穿越）。

### Prompt 組裝

每個 L 階段的實際 prompt = 基礎模板 + 知識庫片段 + 案件上下文：
- 知識庫注入矩陣：`src/data/config/kb-matrix.ts`（8 階段 × 5 知識庫，三種需求等級）
- 撰寫規則：L3、L4 額外套用撰寫規範（`WRITING_RULES_STAGES`）

## 七、設定型別全貌

`AppSettings` 包含 6 個主區塊 + 4 個頂層選項：

| 區塊 | 型別 | 內容 |
|------|------|------|
| `document` | `DocumentSettings` | 文件字型/字級/版面/頁首頁尾 |
| `connections` | `ConnectionSettings` | Notion/Google Drive/SmugMug 連線資訊 |
| `company` | `CompanySettings` | 公司名稱/統編/品牌 |
| `modules` | `ModuleSettings` | 知識庫矩陣/品質規則/計價 |
| `workflow` | `WorkflowSettings` | 階段設定/自動狀態規則/視圖 |
| `featureToggles` | `Record<string, boolean>` | 功能模組開關 |
| `fieldMapping` | `Partial<Record<...>>` | Notion 欄位名稱覆蓋 |
| `dashboardLayout` | `DashboardLayout` | 儀表板卡片佈局 |
| `yearlyGoal` | `number` | 年度目標金額 |
| `monthlyBidTarget` / `weeklyBidTarget` | `number` | 投標件數目標 |

所有設定都有預設值（`DEFAULT_SETTINGS`），新增欄位用 `?` optional，向後相容。

## 八、已知限制與 Phase 2 方向

| 限制 | 影響 | Phase 2 方向 |
|------|------|-------------|
| 知識庫存 localStorage | 換瀏覽器/清快取就不見 | Notion 或 Supabase（待決） |
| 無資料快取層 | 每次載入都打 Notion API | SWR 短期快取 |
| Token 存 localStorage | 安全性不足（XSS 風險） | 環境變數 + 後端 session |
| 單一 Notion database | 表格愈長效能愈差 | 考慮分庫（歷史案件歸檔） |
| Prompt 檔案在 `public/` | 部署後不可修改 | Phase 2 移至資料庫或 CMS |
