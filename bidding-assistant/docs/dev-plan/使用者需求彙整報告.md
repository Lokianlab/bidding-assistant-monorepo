# 使用者需求彙整報告
**大員洛川顧問有限公司 - 全能標案助理系統**

---

## 一、使用者背景與痛點

### 1.1 公司與團隊背景
- **公司名稱**: 大員洛川顧問有限公司
- **業務類型**: 政府採購標案投標與執行
- **團隊規模**: 4-8 人
- **團隊角色**: 行政、企劃、美編、監審
- **技術能力**: 使用者自述「完全小白」「完全沒有任何程式經驗」

### 1.2 核心痛點

#### 痛點 1: Claude 網頁版 Context 爆掉
- 使用 21 個提示詞檔案 (總計約 123KB)
- 需要在每個對話中手動貼上所有相關提示詞
- 經常超過 Claude 的 context 限制而報錯
- 不知道什麼時候該開新對話以避免對話太長

#### 痛點 2: 管理多個案子很亂
- 同時進行多個政府標案
- 每個案子進度不同、資料分散
- 不知道哪個案子進行到哪個階段
- 需要手動追蹤每個案子的截標時間、剩餘天數

#### 痛點 3: 知識庫維護困難
- 需要維護 5 個知識庫：
  - 00A: 團隊成員資料庫
  - 00B: 公司實績資料庫
  - 00C: 時程範本庫
  - 00D: 應變SOP範本庫
  - 00E: 案後檢討筆記
- 每次更新都要手動修改檔案
- 不確定資料格式是否正確

#### 痛點 4: 團隊協作困難
- 需要 4-8 人共用系統
- 不想用指令介面 (「我不喜歡指令介面，什麼要打 python main.py 之類的，記不住」)
- 希望能放在 Google Drive 或 Notion 上共用

#### 痛點 5: 操作不確定性
用戶原話:
> "我對目前狀況的不確定感在於，我是把二個助理 project 架好了，但操作上有很強的不確定感，不知道他什麼時候會報錯，什麼時候要開新對話以免對話太長會報錯，不知道修改和討論的對話能有多長，也不知道能不能順利的產出文件，以及不確定文件的品質夠不夠好，夠不夠穩定，能不能用，而且我很想請他自己去我的 notion 撈資料，但又常報錯。"

## 六、全域約束與品質規範

### 6.1 GC-01: 零模糊語言 (24 個禁用詞)
禁止使用模糊不清的詞彙，例如:
- 相關、有關、相應、適當、合適
- 必要時、如有需要、視情況
- 等、及其他、諸如此類
- 儘量、盡可能、力求
- 充分、完善、良好、優質
- 有效、高效、及時
- 以上、如下、前述、後續

### 6.2 GC-02: 政府用語規範
使用台灣政府標準術語

### 6.3 GC-10: 文件格式規範
- 格式: .docx
- 紙張: A4
- 字型: 標楷體 12pt (可設定)
- 邊距: 1 吋 (可設定)

### 6.4 六大鐵律
1. **Show, don't tell** - 用實績數據證明，不用空泛形容詞
2. **分數導向撰寫** - 對應評選標準配分
3. **零模糊語言** - 遵守 GC-01
4. **N+1/N+2 選項法** - 提供多樣化方案
5. **數字交叉驗證** - 確保所有數字一致
6. **政府公文語體** - 正式、專業、精準

系統應具備以下 AI 增強功能 (但全部功能必須在沒有 API Key 時也能運作):

### 7.1 需要 AI 的功能

5. **企劃草稿** - AI 輔助撰寫

2. **專案管理** - Notion 資料同步

## 八、關鍵技術需求

### 8.1 必須整合的服務
1. **Notion API**
   - 現有資料庫: 備標評估文件庫
   - 需要: 讀取、更新、自動狀態變更

2. **Google Drive API**
   - 位置: 共同雲端硬碟 → 專案執行中心 → 備標集中區
   - 需要: 自動上傳 .docx 檔案
   - 資料夾命名: `BID-xxxxx(民國紀年截標日期)案名`

3. **Claude API (選用)**
   - 用於 AI 增強功能
   - 必須能在沒有 API Key 時正常運作

### 8.2 技術棧選擇
- **前端框架**: Next.js 14+ (TypeScript, App Router)
- **UI 元件**: Tailwind CSS + shadcn/ui
- **部署平台**: Vercel (免費 Hobby Plan)
- **資料持久化**: localStorage (設定) + Notion (專案資料)

### 8.3 安全性與權限
使用者要求:
> "安全性問題，你要幫我規劃好"

需要考慮:
- Notion Integration Token 安全儲存
- Google Drive OAuth 權限管理
- Claude API Key 保護
- 團隊成員存取控制

## 十二、關鍵洞察

### 12.1 使用者真正需要什麼?

**不是**更多 AI 功能，**而是**:
1. 讓現有工作流程更順暢
2. 減少手動重複操作
3. 降低出錯機率
4. 讓非技術團隊也能輕鬆使用

### 12.2 成功的關鍵指標

系統是否成功的判斷標準:
1. ✅ 團隊成員不用記指令
2. ✅ Context 不再爆掉
3. ✅ 知識庫維護變簡單
4. ✅ 多案件管理不再混亂
5. ✅ 文件品質穩定可靠
6. ✅ 能自動從 Notion 撈資料

### 12.3 設計哲學

根據使用者需求，系統應該:
1. **簡單優於複雜** - 網頁介面勝過命令列
2. **可靠優於炫技** - 先確保基本功能穩定
3. **自動優於手動** - 能自動化就不要手動
4. **中文優於英文** - 所有介面都要中文化
5. **引導優於假設** - 逐步指導勝過假設使用者會操作
