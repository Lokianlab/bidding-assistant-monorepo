## 一、功能概述

### 目標
自動產出結案文件，並把關鍵學習資料回流知識庫，建立「得標→學習→下一案更強」的飛輪。讓每一次成功或失敗的案件都成為下一次的經驗累積。

### 核心使用場景
1. 案件完成（已履約完成狀態）→ 用戶點擊「結案」
2. 系統 AI 自動整理本案重點：做了什麼、學到什麼、下次注意
3. 結案學習點自動分類存入知識庫
4. 下一次相同類型標案時，KB 搜尋能取出本案的成功要點和風險提示

### 時間節點
- 觸發時機：案件進入「已履約完成」或「已結案」狀態
- 執行時機：案件完成後 1-3 天內進行結案評分與回流
- 回流窗口：案件結案後永久可搜尋

## 三、功能清單

### 3.1 結案文件生成（AI 輔助）

**觸發**：用戶點擊「進入結案程序」
**輸入**：案件 ID → 從 case-work 讀取全程履約日誌（M10 里程碑、進度報告、客戶反饋等）

**處理**：
- 呼叫 `/api/ai/generate-case-summary`
- Claude 根據履約日誌產生三段落結案摘要

**輸出**：
```json
{
  "case_id": "uuid",
  "case_name": "[案件名稱]",
  "sections": {
    "what_we_did": "本案完成了...",
    "what_we_learned": "我們學到...",
    "next_time_notes": "下次需要注意..."
  },
  "tags": ["時程管理", "客戶溝通", "技術風險", "成本控制"],
  "satisfaction_score": 8,  // 1-10
  "strategy_score": 7,
  "execution_score": 9
}
```

### 3.2 結案評分

**維度**（各 1-10 分）：

| 維度 | 說明 | 評分標準 |
|-----|------|--------|
| **策略評分** | 前期策略分析與定位是否準確 | 1-3: 偏離度大; 4-6: 部分達成; 7-10: 高度準確 |
| **執行評分** | 履約過程是否按計畫進行、品質如何 | 1-3: 延誤多; 4-6: 基本達成; 7-10: 超預期 |
| **客戶滿意度** | 客戶對最終成果與服務的評價 | 由客戶評分或內部估評 |

**總評分** = (策略 + 執行 + 滿意度) / 3

### 3.3 知識庫回流

**回流內容**：
- 結案摘要（三段落 + 標籤）
- 案例屬性（標案類型、規模、週期、成果、評分）
- 關鍵聯絡人與夥伴資訊

**回流目的**：
- 下一次同類型標案搜尋時，KB 能自動推薦本案的成功要點
- 搜尋「客戶溝通」等關鍵字時，能找到相關案例和經驗

**實裝方式**：
```
POST /api/kb/items/
{
  "title": "[案件名] - 結案經驗",
  "content": "[三段落摘要 + 標籤]",
  "case_id": "uuid",
  "category": "case_closing",
  "tags": ["..."],
  "metadata": {
    "bid_type": "工程",
    "case_size": "中",
    "duration_months": 6,
    "strategy_score": 7,
    "execution_score": 9,
    "satisfaction_score": 8
  }
}
```

### 3.4 成功模式識別（簡易版）

**功能**：根據多份結案資料，自動提取重複出現的成功模式

**實裝方式**（v1 簡易版）：
- 收集所有 case_learnings 表中的標籤
- 統計各標籤出現次數
- 將高頻標籤（≥ 3 次）標記為「成功模式」

**輸出**：
```json
{
  "success_patterns": [
    {
      "pattern": "提早3週與業主確認變更需求",
      "frequency": 5,
      "avg_satisfaction": 8.2
    },
    {
      "pattern": "團隊知識共享會議每週一次",
      "frequency": 4,
      "avg_satisfaction": 7.8
    }
  ]
}
```

**後期擴展**（v2）：支援更複雜的文本分析與模式識別

### 3.5 結案 UI

**頁面**：`/app/case/[id]/closing/`

**佈局**：
1. **結案摘要**（頂部）
  - AI 自動產生的三段落文本（可編輯）
  - 快速標籤選擇（預填常用標籤，支援自訂）

2. **評分表單**（左側）
  - 三個 1-10 滑塊（策略 / 執行 / 滿意度）
  - 實時計算總評分

3. **知識庫預覽**（右側）
  - 展示結案摘要將寫入 KB 的樣貌
  - 可見的標籤與搜尋關鍵字

4. **操作面板**（底部）
  - 「預覽 PDF」 → 生成結案報告 PDF
  - 「寫入 KB」 → 確認寫入知識庫
  - 「完成結案」 → 更新案件狀態為「已結案」

## 四、技術規格

### 4.1 後端結構

**新增資料夾**：`/src/lib/case-closing/`

```

  types.ts            ← 型別定義（CaseSummary, CaseLearning, SuccessPattern）
  helpers.ts          ← 純函式（評分計算、標籤生成、模式提取）

  __tests__/          ← 測試檔案（≥60 tests）
```

### 4.2 API 路由

**新增路由**：`/src/app/api/cases/[id]/close/`

```
POST   /api/cases/[id]/close/generate-summary
       → 生成結案摘要（輸入：case_id）

PATCH  /api/cases/[id]/close/save-assessment
       → 保存評分（輸入：strategy_score, execution_score, satisfaction_score, tags）

POST   /api/cases/[id]/close/write-to-kb
       → 寫入知識庫（輸入：summary_text, tags, scores）

GET    /api/cases/[id]/close/summary
       → 查詢結案摘要

GET    /api/kb/success-patterns
       → 查詢系統級成功模式（超過 3 次的高頻學習點）
```

### 4.3 Supabase 資料結構

**新增表**：`case_learnings`（結案學習點）

```sql
CREATE TABLE case_learnings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id UUID NOT NULL REFERENCES cases(id),
  tenant_id UUID NOT NULL REFERENCES users(id),

  title TEXT NOT NULL,
  what_we_did TEXT,
  what_we_learned TEXT,
  next_time_notes TEXT,

  tags TEXT[] DEFAULT '{}',

  strategy_score INT,  -- 1-10
  execution_score INT,
  satisfaction_score INT,

  kb_item_id UUID REFERENCES kb_items(id),  -- 寫入 KB 後的引用

  created_at TIMESTAMP DEFAULT now(),
  updated_at TIMESTAMP DEFAULT now(),

  INDEX idx_case_learnings_tenant (tenant_id),
  INDEX idx_case_learnings_tags (tags)
);

-- 新增欄位至 kb_items，標記來源為結案摘要
ALTER TABLE kb_items ADD COLUMN source_type TEXT;  -- 'manual', 'case_closing', 'import'
ALTER TABLE kb_items ADD COLUMN related_case_id UUID REFERENCES cases(id);
```

### 4.4 RLS 策略

```sql

CREATE POLICY "Users can view own tenant's learnings"
  ON case_learnings
  USING (tenant_id = auth.uid());

CREATE POLICY "Users can insert own tenant's learnings"
  ON case_learnings
```

## 七、API 詳細規格

### 生成結案摘要
```
POST /api/cases/[id]/close/generate-summary

Request:
{ "case_id": "uuid" }

Response (200):
{
  "case_id": "uuid",
  "case_name": "XXX 系統開發案",
  "what_we_did": "完成了系統的...",
  "what_we_learned": "本案中我們學到...",
  "next_time_notes": "下次需要注意...",
  "suggested_tags": ["時程管理", "客戶溝通", "技術風險"]
}
```

### 保存評分與標籤
```
PATCH /api/cases/[id]/close/save-assessment

Request:
{
  "strategy_score": 7,
  "execution_score": 9,
  "satisfaction_score": 8,
  "tags": ["時程管理", "溝通良好", "成本控制"],
  "custom_notes": "..."
}

Response (200):
{ "success": true, "total_score": 8 }
```

### 寫入知識庫
```
POST /api/cases/[id]/close/write-to-kb

Request:
{
  "title": "XXX 系統開發案 - 結案經驗",
  "content": "[三段落摘要]",
  "tags": ["時程管理", "..."],
  "metadata": { "strategy_score": 7, "execution_score": 9, "satisfaction_score": 8 }
}

Response (201):
{
  "kb_item_id": "uuid",
  "kb_item_url": "/app/tools/kb/[id]",
  "success": true
}
```

### 查詢成功模式
```
GET /api/kb/success-patterns?min_frequency=3

Response (200):
{
  "patterns": [
    {
      "pattern": "提早與業主確認變更需求",
      "frequency": 5,

    },
    {
      "pattern": "每週知識共享會議",
      "frequency": 4,
      "avg_scores": { ... }
    }
  ]
}
```

## 九、依賴與風險

**依賴**：
- ✅ M02（知識庫）- 已完成

- AI 生成的摘要可能不精確或語意不清

  → **緩解**：結案項目加 `source_type: 'case_closing'` 標籤，可單獨篩選
