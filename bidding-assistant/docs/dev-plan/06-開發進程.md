# 06 開發進程

## 策略前提（Jin 確認 2026-02-22）

三個決定，影響所有後續開發：

1. **先自用，再產品化** — 先幫大員洛川跑通整套流程，先不做多租戶
2. **Notion 當主庫，加抽象層** — 現有 Notion 繼續用，程式碼加資料抽象層，未來換庫不重寫業務邏輯
3. **Claude Code SDK 當發射台** — 系統組裝提示詞+知識庫+交付物，一鍵送 Claude Code 執行。不做嵌入式 AI 聊天，不花額外 API 費用

## P0-P4 開發路線

按實際工作流程排序（打開→選案子→看資料→看情報→寫企劃→輸出→監控），不按技術層級。

| 階段 | 目標 | 對應能力 | 狀態 |
|------|------|---------|------|
| P0 | 行政巡標自動化 | PCC API 搜尋 + Notion 建案 | ✅ 完成 |
| P1 | 案件工作頁 | 能力 1（戰略）+ 能力 2（知識庫）+ PCC 情報 | 待開始 |
| P2 | Claude Code SDK 寫作發射台 | 能力 3（調度）+ 能力 4（品質）| 待開始 |
| P3 | 輸出管理 | 能力 6（排版 + 設計需求單） | 待開始 |
| P4 | 投標監控自動化 | PCC 狀態追蹤 + 推送通知 | 待開始 |

### 為什麼這個順序

舊計畫按技術依賴排（基礎設施→核心→效率→錦上添花），但實際使用流程是：

```
行政巡標（發現機會）
  → 企劃選案（看資料、看情報）
    → 寫企劃（AI 撰寫 + 品質檢查）
      → 輸出（排版、設計）
        → 監控（狀態追蹤、結果回收）
```

每一步做完，用戶就能用到那一步。不需要等整個系統蓋好。

## P2：Claude Code SDK 寫作發射台

系統組裝好材料，一鍵送 Claude Code 寫企劃。

### 做什麼

- 組裝提示詞（L2 備標規劃 + L3 章節撰寫）
- 注入知識庫（00A-00E）+ 本案情報 + 過去交付物
- 「發射」按鈕：打包送 Claude Code SDK 執行
- M04 品質閘門檢查輸出（已有 114 測試）

### 架構重點

```
SaaS 網頁（UI）
  → 選案件、確認材料
  → 一鍵「發射」
    → Claude Code SDK（背景執行）
      → L2 備標規劃
      → L3 章節撰寫
      → L4 定稿
    → 品質閘門自動檢查
  → 結果回到案件頁
```

不做嵌入式聊天。系統的價值在「組裝正確的輸入」，不在「即時對話」。

## P4：投標監控自動化

- 從 PCC API 自動更新案件狀態（開標、決標、流標）
- 推送通知（截標日提醒、開標結果）
- 落標資料自動回收 → 知識庫更新（飛輪回饋）

## PCC API 的戰略價值

PCC API 不只在 P0 用，它貫穿整個流程：

| 階段 | PCC API 做什麼 | 價值 |
|------|---------------|------|
| P0 巡標 | 自動搜尋新公告 | 發現更多機會 |
| P1 案件頁 | 競爭對手分析、機關偏好、評委背景 | 投標決策更準 |
| P2 寫作 | 評委背景→調整寫作策略 | 得標率更高 |
| P4 監控 | 開標結果、決標資訊 | 自動追蹤、回饋學習 |

### 收入公式衝擊

```
收入 = 得標案件數 × 平均案件金額 × 毛利率
```

| 公式變數 | PCC API 的作用 | 影響 |
|---------|---------------|------|
| 發現機會數 ↑ | P0 自動監控新標案 | 中 |
| 投標決策通過率 ↑ | P1 競爭對手分析、機關偏好 | **高** |
| 得標率 ↑ | P1/P2 評委背景→寫作策略、價格分析 | **高** |
| 平均案件金額 ↑ | P0/P1 篩選高價值案件 | 中 |
