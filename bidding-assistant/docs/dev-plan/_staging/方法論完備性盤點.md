### 觀察

- 「做之前」6 個情境全空——只有 #1 多方案取捨有模型委員會，其他 5 個都靠 AI 臨場發揮
- 「做的過程」5 個全空——模擬驗證（#7）在 /kb 時用過但沒有成文的方法
- 「做完之後」只有閉環原則覆蓋了 #12 和 #13，但完備性（#14）和一致性（#15）沒有
- 「學習改善」幾乎全空——記錄系統有，但經驗萃取、事後檢討、流程改善都沒有方法
- 「溝通」靠指令工具（/小結、/待辦）覆蓋了部分，但向用戶解釋和提案呈現沒有

## 可能方向

1. **不需要 26 個都寫方法論**——有些情境靠常識就能處理（如 #10 替代路徑、#24 向用戶解釋），硬寫方法論反而多餘
2. **優先補高頻 + 出過事的**——模擬驗證（#7，/kb 出過漏洞）、完備性檢查（#14，/kb 只驗十分之一）、一致性檢查（#15，暫存檔和實際不一致）
3. **有些可以合併**——例如 #14 完備性 + #15 一致性可以合成一個「交付前檢查清單」
4. **有些是系統問題不是方法論問題**——例如 #16 決策同步靠 git 和記錄層就行，不需要額外方法論
5. **用戶需要決定投入程度**——是認真寫成文件級方法論，還是在 CLAUDE.md 加幾條規則就好
