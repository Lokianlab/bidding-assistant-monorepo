## 背景

方法論 v2 系統已整合至 docs/methodology/，定義了 5 個工具觸發指令（/對焦、/拆解、/驗證、/回顧、/自檢），但對應的 skill 檔案還不存在。用戶手動打指令時沒有東西會載入。

CLAUDE.md 的事件觸發規則讓 AI 自動跑方法論，但自動觸發是靜默的——用戶看不到 AI 正在走哪套流程。Skill 的價值是讓用戶主動呼叫時，AI 把每一步的思考過程攤開來展示。

## 結論

### 設計決策

**1. 4 個 skill，砍掉 /自檢**

/自檢 內容只有 6 項清單，已嵌在 methodology-01（需求對焦）後半段。做成獨立 skill 使用頻率太低。自檢邏輯嵌在 /對焦 裡——當產出是設計決策時自動帶入。CLAUDE.md 觸發規則表的 /自檢 指令改為提示「執行 /對焦 的送審前自檢段落」。

**2. 核心步驟內嵌，不引用外部檔案**

Skill 內容直接注入對話指令，AI 遵守度高。若只寫「Read methodology-XX.md 照做」，AI 要多一步讀檔，可能跳過細節或按自己理解做。

內嵌範圍：操作步驟（怎麼做）、產出格式、完成標準。
不內嵌：品質判斷、專案實例、已知限制（這些是參考，不是指令）。

代價：改了方法論要同步改 skill。但操作步驟相對穩定，品質判斷和實例才常更新。

**3. 手動觸發 vs 自動觸發的差異**

自動觸發（CLAUDE.md 規則）：AI 在腦中過方法論，用戶只看到最終結果（例如一個問題）。

手動觸發（/skill）：AI 把每一步的中間產出展示給用戶——列出了哪些模糊點、為什麼選這個先問、每個選項的差異。

Skill 裡的規則：「每一步的分析和判斷都要展示給用戶，不能只在腦中過。」

**4. 每個 skill 結尾帶下一步建議**

方法論經常串聯（對焦→拆解→驗證→執行→閉環），但不做串聯 skill（過度設計）。每個 skill 結尾提示下一步，用戶決定要不要接著跑。

### 4 個 Skill 規格

#### /對焦（對焦.md，~70 行）

觸發場景：用戶意圖不明確，需要結構化提問收斂需求。
參數：可選——需求描述。

檔位判斷：
- 快速（歧義輕微）：心中過——有不確定的詞嗎？兩種解讀差異大嗎？直接問一個問題收斂。
- 標準（歧義嚴重或涉及多選項）：走完四步。
- 完整（設計決策）：四步 + 送審前自檢六項 +（大事加掛多模型外審）。

核心步驟：
1. 逐詞掃描指令，標記模糊詞（代名詞、範圍不明詞、多義動詞）
2. 展開可能含義，找猜錯返工成本最高的 1-2 個歧義
3. 帶選項提問：「你說的 X 是指 A 還是 B？我傾向 A，因為 [理由]。」一次一個。
4. 用戶回答後一句話重述：「所以你要的是 [具體描述]，對嗎？」
5. 多條可行路徑時列比較表（做法/效果/成本/風險），標明傾向，等用戶選。
6.（設計決策時）自動跑自檢六項：問題定義/方案比較/自我否定/數字有據/回頭驗算/薄弱環節。

完成標準：用戶說「對」或做小幅修正。知道下一步該做什麼。
下一步建議：任務需要拆解 → /拆解。

#### /拆解（拆解.md，~70 行）

觸發場景：大任務需要拆成可獨立執行的小塊。
參數：可選——目標描述。

檔位判斷：
- 快速（改動 3-5 檔）：拆幾塊？哪塊先做？心中排好不產文件。
- 標準（改動 3-5 檔跨模組）：走八步，簡要記錄。
- 完整（≥6 檔或架構級）：八步完整走，產出帶依賴關係的工作清單。

核心步驟：
拆解五步——
1. 定義完成狀態（具體到可驗證）
2. 列交付物（可獨立驗收的產出，不是步驟）
3. 找依賴（硬依賴/軟依賴/無依賴）
4. 找可平行塊（不改同一檔案的無依賴塊）
5. 拆到一口大小（一個 session 能完成和驗證）

排序三步——
6. 標屬性（影響範圍×風險程度×獨立價值）
7. 排順序（高影響高風險先做）
8. 切第一批（前三個，不排完所有）

產出格式：編號清單，每塊標「交付物 | 依賴 | 優先序」。
完成標準：每塊有驗收條件，至少前三個順序確定，用戶同意排法。
下一步建議：開始動手前 → /驗證。

#### /驗證（驗證.md，~80 行）

觸發場景：動手改東西或執行計畫之前。
參數：可選——要改的檔案名。

入口判斷：
- 改既有的東西 → 靜態掃描
- 多步驟計畫 → 動態模擬
- 兩者都是 → 先靜態再動態

檔位判斷：
- 快速（1-2 檔同模組）：直接影響幾個地方？用一筆資料走一遍，哪步最可能出問題？
- 標準（3-5 檔或跨模組）：跑核心步驟，簡要記錄。
- 完整（≥6 檔或規範文件）：所有步驟含品質驗證。

靜態掃描五步——
1. 精確描述改動（改什麼、從什麼改成什麼、為什麼）
2. 第一圈：四面向掃描（程式碼/資料/文件/其他機器）
3. 第二圈：間接依賴
4. 判斷每個影響點（一定會壞/可能會壞/需要更新）
5. 行動計畫（先處理一定會壞的，再處理需要更新的）

動態模擬五步——
1. 把計畫攤成線性步驟（每步之後的系統狀態）
2. 拿一筆具體資料走進去
3. 每步問四問題（輸入存在嗎？操作會成功嗎？輸出符合下一步嗎？有副作用嗎？）
4. 刻意走歪路（空值、特殊字元、不存在的 ID）
5. 記錄發現（阻塞/缺陷/待確認），做決策（照做/修後做/推翻）

產出格式：影響點表（影響類型 | 嚴重度 | 處理方式）。
完成標準：能列出所有已知影響點，每個有判斷。正常路徑跑通，邊界已識別。
下一步建議：驗證通過 → 執行。做完 → 閉環驗證（自動，不需手動觸發）。

#### /回顧（回顧.md，~80 行）

觸發場景：做完了但過程比預期困難很多、同問題反覆出現。
參數：可選——問題描述或 topic。

模式判斷：
- 同 topic ≥2 筆失敗/修正 → 獨立根因追問（精簡版）
- 過程異常困難、方法被證實無效 → 完整四階段回顧

獨立根因追問五步——
1. 精確描述症狀（在什麼條件下，什麼行為與預期不同）
2. 逐層追問（每層必須有查證，不能只靠推理）
3. 判斷到底（能改變但再追碰到超出控制的 → 這就是該修的地方）
4. 驗證因果鏈（從根因往回走，修了根因症狀會消失嗎？）
5. 根因不可改 → 往回退一層找替代路徑

完整回顧四階段——
1. 還原事實：建時間線 + 標轉折點
2. 分析原因：對每個轉折點跑根因追問 + 區分個案/系統性
3. 設計改善：三級措施（意識→流程→系統），優先系統級
4. 落實：寫進 CLAUDE.md / 方法論 / 建任務

產出格式：根因結論 + 改善方案 + 具體落實位置。
完成標準：根因能用一句話說出，因果鏈每步說得通且有查證。改善有具體載體。
下一步建議：改善措施如果是新規則 → /暫存。

### 共通結構

每個 skill 檔案的結構：

```
一句話描述

## 檔位判斷
（基於改動檔案數/模組數的客觀指標）

## 快速檢查
（1-2 檔同模組時用，3-5 個核心問題）

## 標準/完整流程
（核心操作步驟，直接內嵌）

## 產出格式

## 完成標準

## 下一步建議
```
