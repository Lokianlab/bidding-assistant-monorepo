# 元方法論框架（Meta-Methodology）

目標：docs/methodology/meta-methodology.md（新建）

相關 OP：待建

## 背景

系統方法論 v2 已建成（4 套方法論 + 閉環 + 模型委員會，10 份文件），但缺少一個上位框架來回答：「憑什麼說這套方法論是好的？用什麼標準評價和改進它？」

在微調個別方法論文件之前，需要先確立元方法論——對方法論的評價與反思理論。否則微調只能靠體感，沒有客觀依據。

本框架參照 Thomann/Hutchinson 操作性元方法論（1973/1984），結合本專案人機協作的實際情境建構。

## 結論

### 一、四層遞進架構

```
高元方法論 ── 元方法論的開發標準及評價工具
  ↓
元方法論   ── 方法論的開發標準及評價工具
  ↓
方法論     ── 方法的開發標準及評價工具
  ↓
方法       ── 具體操作步驟
```

每一層為下一層的開發標準及評價工具。建構順序由上而下：先完成上層，再以上層開發和評價下層。

### 零、高元方法論——基礎審視原則

高元方法論為元方法論的開發標準及評價工具。這一層回答：「我們用來評價方法論的那套標準，它自己站得住腳嗎？」

#### 循環評價問題

元方法論的評價工具不能來自被評價的方法論。否則是拿被告當法官——用閉環四層來評價閉環四層、用觸發規則來評價觸發規則，都是循環論證。

操作化指標必須獨立於被評價的方法論工具。可用的是更底層的觀察手段：用戶行為（接受/退回）、結果與預期的差距、時間序列趨勢——這些不依賴任何特定方法論工具。

#### 基礎公設風險

放進元方法論基礎層的任何概念，都會滲透到所有後續建構的方法論和方法中，且極難被事後評估或移除。原因：評估工具本身就依賴這些基礎概念，系統無法用自己偵測自己的基礎錯誤。

例：如果在元方法論中採用貝式定理作為迭代機制的基礎，那貝式思維將成為整個系統的一部分。貝式定理本身是數學定理（邏輯上無誤），但我們對它的「應用方式」可能有誤（選錯先驗、誤判證據、設錯閾值）。如果應用方式有系統性偏差，整個系統都會繼承這個偏差，且系統自己偵測不到。

#### 基礎選擇準則

放進元方法論基礎層的概念，應滿足：

1. **獨立於被評價對象**——評價工具不是被評價系統的一部分
2. **可外部驗證**——不依賴系統內部判斷來確認正確性，可用系統外部的觀察驗證
3. **最小假設**——基礎公設越少越好，每多一個假設就多一個無法自我偵測的風險點
4. **失敗可辨識**——對每個基礎假設，要能說清楚：如果這個假設是錯的，我們會看到什麼現象。答案是「看不出來」的假設不該放在基礎層

#### 標準即修正方向

任何定義了目標狀態的標準，同時定義了從現狀到目標的修正方向。此性質適用於所有層級的所有標準，不依賴特定評價框架。

#### 回歸終止

四層架構在高元方法論終止，不再無限回歸。原因：高元方法論的原則（獨立、可外部驗證、最小假設、失敗可辨識、標準即修正方向）是接近自明的認識論原則，不需要更上一層的理論來證成。它們的正當性來自邏輯和經驗的基本共識，而非來自另一套後設框架。

### 二、定義

系統方法論：一套規範人機協作中 AI 行為的方法體系。

三個層次構成完整的方法論：
- **方法**：怎麼做（操作步驟）
- **原則**：為什麼這樣做（讓 AI 能舉一反三）
- **自我迭代**：怎麼知道方法有沒有用、怎麼改進（貝式更新機制）

缺任何一層，都不算方法論——只有方法是手冊，只有原則是哲學，沒有迭代是教條。

不是規範手冊（照做就好），是方法論（理解了才能用好）。這個命名定義了系統的層級。

### 三、根本問題

AI 有結構性缺陷：跨 session 無記憶、沒有內建時鐘不會主動停、自己檢查自己盲區共享、天性傾向直接行動而非先想清楚。多機器協作放大這些問題。

方法論的存在是用制度補結構性缺陷——不靠 AI 自律，靠系統設計。

### 四、目的體系——效率為終極標準

「效率」不是與品質、透明並列的一個維度，而是所有維度最終服務的目標：

```
品質穩定     → 減少返工      → 效率
過程可見     → 便於檢討      → 效率
穩定有信心   → 不用額外準備   → 效率
能自己成長   → 減少外部介入   → 效率
```

效率 = 協作產出的總價值 / 協作消耗的總成本（時間 + 溝通 + 返工 + 認知負擔 + 維護成本）

#### 第一層：協作品質（每次都要達到的）

1. **穩定 95 分**——靠制度不靠臨場發揮，每次都交出可預期的品質
2. **降低溝通往返**——AI 交出的東西用戶不用反覆退回重做
3. **複雜任務有信心**——步驟再多，每一步都有檢查點，做到第八步不用擔心第三步

#### 第二層：過程性質（系統必須具備的特性）

4. **過程可見**——關鍵步驟和關鍵判斷攤開，粒度以用戶能有效監督和決策為準
5. **出錯可回溯**——出了問題，能沿著步驟精準找到哪一步斷的
6. **結果可歸因**——答對了也能分辨是蒙對的、亂用方法碰巧對的、還是正確運用方法

#### 第三層：成長能力（讓系統越來越好）

7. **培養判斷力**——從照做規則 → 理解原則 → 舉一反三，能應對規則沒覆蓋的情況。掌握了方法就能自學、能知何能為及知何不能為，區分 known known 和 known unknown
8. **貝式自我迭代**——每次執行都是證據（成功/失敗/跳過），證據持續更新對方法有效性的信念，累積到閾值時觸發強化或修改。不只等別人指出問題，系統自己能監控、觀察、反思

#### 數學課比喻（貫穿全目的體系）

老師要求學生：不跳步驟、計算紙從左上角開始、小結論標星號、5 和 8 寫清楚、不確定先驗算再解題。

這些規範的效果：
- 降低錯誤率，提高解題效率（穩定 95 分）
- 學生交卷時已自己驗算過，老師不用一直退回（降低往返）
- 題目越長越難，按步驟寫的學生越穩（複雜任務有信心）
- 老師能從計算紙看到怎麼算的（過程可見）
- 答錯了翻計算紙就知道哪步錯（出錯可回溯）
- 答對了也能看出是真懂還是蒙的（結果可歸因）
- 學生掌握了方法就能自學、舉一反三（培養判斷力）
- 為老師的批改和檢討教學帶來參考資料，持續改善教法（貝式迭代）

### 五、設計原則

用來判斷「這條規則寫得好不好」「這段內容該不該放」的準則：

**P1 — 穩定可遵守**：規則基於客觀指標，不依賴 AI 自覺或記憶。能每次做到的規則才是好規則。

**P2 — 步驟可定位**：每步有編號或命名，出問題時能說「第三步的判斷有誤」。

**P3 — 原則可遷移**：不只告訴 AI「做什麼」，也告訴它「為什麼」。區分：培養判斷力的原則（留）vs 設計歷史紀錄（移出到記錄層）。

**P4 — 單一來源**：同一件事只在一個地方定義。10 份文件之間不重複、不矛盾。

**P5 — 證據雙向**：成功、失敗、跳過都是證據。不只收集問題，也收集「方法有效」的正面證據。

**P6 — 比例原則**：方法論的力度與任務的複雜度和風險成正比。簡單任務用輕量方法，複雜任務用完整流程。條件受限時，降低力度但不放棄——部分遵守的價值遠大於完全不遵守。

### 六、內容區分原則

用來判斷方法論文件中「什麼該留、什麼該移、什麼該砍」：

| 內容性質 | 判斷標準 | 處置 |
|---------|---------|------|
| 操作步驟（怎麼做） | AI 執行時需要 | 留 |
| 設計原則（為什麼這樣做） | 理解了能幫助 AI 舉一反三 | 留，寫成培養判斷力的形式 |
| 設計歷史（從 A 改成 B 的過程） | 只對設計者回溯有用 | 移出到記錄層 |
| 設計驗證（覆蓋了幾個情境） | 設計完成後不再被查閱 | 移出到附錄 |
| 沒有執行機制的規則 | 寫了但不會被觸發 | 補機制或移到「待建」 |

### 七、文件品質標準

每份方法論文件都應滿足：

1. **可穩定遵守**——規則基於客觀指標
2. **過程攤開**——每步的中間產出有定義
3. **出錯可定位**——步驟有編號，能精準指出斷點
4. **原則可遷移**——告訴 AI「為什麼」，讓它能應對變體
5. **自帶改進機制**——包含「怎麼知道方法有沒有用」和「怎麼改進」

### 八、系統層級檢視

不只看單一文件，還要看 10 份文件作為一個系統：

- **完整性**——從接到任務到交付成果，每個關鍵時刻都有對應方法
- **連貫性**——方法論之間銜接清楚（A 的產出是 B 的輸入）
- **不矛盾**——文件之間沒有互相衝突的規則
- **不冗餘**——同一件事只在一個地方定義
- **可迭代**——系統本身有機制檢驗和改進自己

### 九、修正邏輯

評價工具找出問題後，怎麼修：

**修正方向**：每個標準定義了目標狀態，現狀跟目標狀態之間的差距就是修正內容。不需要額外判斷——標準本身隱含了修正方向。

| 診斷結果 | 修正方向 |
|---------|---------|
| 設計原則不通過 | 改寫到通過（P1 換客觀觸發、P2 加編號、P3 補原則、P4 去重複、P5 補正面證據、P6 加檔位） |
| 內容放錯位置 | 搬到該去的地方（設計歷史→記錄層、設計驗證→附錄） |
| 空規則（寫了但沒執行機制） | 補機制或移到「待建」 |
| 定義不完整（缺方法/原則/迭代） | 補上缺的層次 |
| 系統銜接斷裂 | 對齊文件之間的輸入輸出介面 |
| 文件間矛盾 | 回到目的體系判斷哪邊對，統一 |

**修正排序**（多個問題同時存在時）：
1. 目的體系第一層的問題（品質、交付）優先於第二層（過程可見）優先於第三層（成長）
2. 同層內，系統層級問題（矛盾、斷裂）優先於單一文件問題（編號缺失）

**副作用檢查**：修完後用同一套評價工具再掃一遍。改 A 可能壞 B——合併重複內容可能讓另一份文件變不完整；搬走設計歷史可能讓上下文斷裂。修了沒驗不算修完。

### 十一、自我迭代機制（貝式更新）

```
執行方法論
  ↓ 產生
一筆執行紀錄（成功 / 失敗 / 跳過）
  ↓ 累積
對該方法有效性的信念更新
  ↓ 達到閾值
行動：夠信任 → 強化（簡化、內化）
      夠質疑 → 修改（調整規則、改觸發條件、換方法）
  ↓ 修改後
新的先驗 → 繼續收集證據 → 循環
```

主動迭代需要三個能力：
- 監控——執行過程中觀察自己的行為
- 發現——辨識出偏差或異常
- 反思——判斷問題是個案還是系統性的

### 十二、操作化——把「效率」變成可衡量指標（V2，已修正循環引用）

> V1 犯了循環評價錯誤（用閉環四層評價閉環四層、用觸發規則評價觸發規則等 6 處循環引用）。
> V2 原則：所有指標只基於不依賴任何方法論工具的底層觀察。

### 基本原則

所有指標只基於三種獨立可觀察的事物：

1. **用戶行為**——接受、退回、追問、花多少時間做決定
2. **產出與預期的差距**——做出來的東西跟要求的一不一樣
3. **時間序列**——同類事情隨時間推移，表現的變化方向

這三者不依賴閉環四層、觸發規則、三檔機制或任何方法論工具。無論方法論怎麼設計，這些觀察都成立。

效率有兩面——產出和成本。方法論要讓產出變大、成本變小。只看結果不夠，還要看過程——結果好可能是運氣，過程好才能穩定重複。指標分四類：

- **A. 成果指標**——協作產出了什麼品質的東西
- **B. 過程指標**——做事的方式是否透明、可追溯
- **C. 成本指標**——達成結果消耗了多少資源
- **D. 成長指標**——系統有沒有隨時間變好

A 和 C 合在一起衡量當下的效率。B 解釋為什麼效率高或低。D 衡量趨勢。

#### A. 成果指標

**A1. 用戶首次接受率**
- 定義：AI 交付後，用戶不要求修改就接受的比例
- 觀察方式：用戶說了「好」「可以」「通過」而非「改一下」「不對」「重做」
- 為什麼獨立：只看用戶的反應，不預設用什麼流程產出
- 基準：趨勢向上。絕對值因任務類型而異（探索型任務首次接受率本來就低）

**A2. 返工次數**
- 定義：同一件事被退回修改的次數
- 觀察方式：同一個交付物被用戶要求修改幾次才最終接受
- 為什麼獨立：純粹計數用戶的退回行為
- 基準：目標 ≤ 1 次。超過 2 次說明第一次交付時的理解或品質有系統性問題

**A3. 事後發現率**
- 定義：交付被接受後，才發現問題需要回頭修的比例
- 觀察方式：一個 topic 標記完成後，又出現修正類的操作
- 為什麼獨立：看的是時間序列上「完成後又動」的事實，不預設用什麼驗證流程
- 基準：趨勢向下。這是最危險的效率損失——以為做完了其實沒完

#### B. 過程指標

**B1. 關鍵判斷可見率**
- 定義：AI 做出影響方向的判斷時，有沒有向用戶展示判斷理由
- 觀察方式：回顧對話記錄——遇到選擇點時，AI 是直接做了，還是展示了「我選 A 因為 X」
- 為什麼獨立：看的是 AI 的溝通行為，不預設用什麼方法論框架
- 基準：方向性決策（選 A 不選 B）100% 可見。執行性步驟不要求

**B2. 問題定位速度**
- 定義：出了問題時，從「發現有問題」到「定位是哪裡出問題」花多少輪對話
- 觀察方式：計算從用戶說「有問題」到雙方確認根因之間的對話輪次
- 為什麼獨立：計的是時間和輪次，不預設用什麼回溯工具
- 基準：≤ 3 輪。超過 5 輪說明過程紀錄不足以支撐回溯

**B3. 意外率**
- 定義：執行過程中出現「事前沒預料到」的障礙的頻率
- 觀察方式：AI 在執行中說「沒想到」「原來還要」「這裡跟預期不同」的次數
- 為什麼獨立：看的是預期與實際的落差，不預設用什麼事前驗證方法
- 基準：趨勢向下。意外率持續不降，說明事前準備的方法（無論是什麼方法）沒有效果

#### C. 成本指標

**C1. 任務總輪次**
- 定義：從接到任務到用戶接受交付，總共經歷多少輪對話
- 觀察方式：計數
- 為什麼獨立：純粹的量化觀察
- 基準：同類任務的總輪次趨勢向下（做同類事情越來越快）。不設絕對值，因為不同任務天然需要不同輪次

**C2. 用戶等待與介入頻率**
- 定義：一個任務中，用戶需要主動介入（回答問題、做決定、提供資訊）的次數
- 觀察方式：計數 AI 向用戶提問或等待用戶決策的次數
- 為什麼獨立：看的是用戶被打斷的頻率
- 基準：每個任務 ≤ 3 次主動介入（不含最終驗收）。更多說明 AI 的自主判斷能力不足

**C3. 同類任務效率趨勢**
- 定義：做過一次的同類任務，第二次做是否更快更順
- 觀察方式：比較同類 topic 的首次 vs 後續的總輪次、返工次數
- 為什麼獨立：同類比較是統計方法，不依賴任何方法論工具
- 基準：第二次做同類任務，總輪次至少降低 20%。沒有下降說明經驗沒有被有效累積

#### D. 成長指標

**D1. 同類問題復發率**
- 定義：解決過的問題再次出現的比例
- 觀察方式：同一類錯誤或障礙在不同 session 中重複出現
- 為什麼獨立：看的是「同一件壞事發生了幾次」，不預設用什麼改善機制
- 基準：趨勢向下。復發率平坦或上升＝系統沒有在學習

**D2. 用戶介入層級趨勢**
- 定義：用戶介入的性質隨時間的變化——從「糾正錯誤」到「確認方向」到「只做最終驗收」
- 觀察方式：分類用戶介入的性質。糾錯（「這裡錯了」）vs 選擇（「選 A 還是 B」）vs 確認（「可以」）
- 為什麼獨立：看的是用戶行為的性質變化
- 基準：糾錯類介入的比例趨勢向下，確認類比例趨勢向上。這表示 AI 的判斷力在成長

**D3. 新情境處理能力**
- 定義：面對之前沒遇過的任務類型時，品質下降的幅度
- 觀察方式：比較熟悉任務和新任務的 A1（首次接受率）和 A2（返工次數）差異
- 為什麼獨立：比較的是同一批獨立指標在不同情境下的表現
- 基準：新任務的品質指標不低於熟悉任務的 70%。低於此說明方法論只教了「做什麼」沒教「怎麼想」——判斷力沒有被培養起來

#### 指標診斷邏輯

```
A 好 + B 好 + C 低 = 方法論有效
A 好 + B 差 = 結果對但過程不可見，下次可能出問題
A 差 + B 好 = 過程做了但方法本身不對，需要換方法
A 差 + B 差 = 方法論沒在運作
D 向上 = 系統在學習
D 平坦 = 學習機制失靈
C3 沒下降 = 經驗沒被累積
```

#### 收集方式

| 方式 | 成本 | 覆蓋 |
|------|------|------|
| 對話自然產生 | 零 | A1、A2、B1、C1、C2（都是對話中可直接觀察的） |
| 每次 /小結 時統計 | 小結多 3 分鐘 | A3、B2、B3、C3 |
| 每月趨勢分析 | 一次性 30 分鐘 | D1、D2、D3 |
