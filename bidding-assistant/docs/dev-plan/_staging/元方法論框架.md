## 背景

系統方法論 v2 已建成（4 套方法論 + 閉環 + 模型委員會，10 份文件），但缺少一個上位框架來回答：「憑什麼說這套方法論是好的？用什麼標準評價和改進它？」

在微調個別方法論文件之前，需要先確立元方法論——對方法論的評價與反思理論。否則微調只能靠體感，沒有客觀依據。

本框架參照 Thomann/Hutchinson 操作性元方法論（1973/1984），結合本專案人機協作的實際情境建構。

## 結論

### 一、四層遞進架構

```
元元方法論（基礎審視）——元方法論的基礎假設站得住腳嗎
  ↓ 選擇及判定
元方法論（評價框架）——憑什麼說方法論好或壞
  ↓ 建構及評價
方法論（10 份文件組成的系統）——該選哪個方法、為什麼
  ↓ 建構及評價
方法（各文件中的具體操作步驟）——怎麼解決具體問題
```

每一層為下一層提供評價標準和建構依據。建構順序由上而下：先完成上層，再以上層的標準建構和評價下層。

### 零、元元方法論——基礎審視原則

元方法論用來評價方法論，但元方法論自身的基礎假設也需要被審視。這一層回答：「我們用來評價方法論的那套標準，它自己站得住腳嗎？」

#### 循環評價問題

元方法論的評價工具不能來自被評價的方法論。否則是拿被告當法官——用閉環四層來評價閉環四層、用觸發規則來評價觸發規則，都是循環論證。

操作化指標必須獨立於被評價的方法論工具。可用的是更底層的觀察手段：用戶行為（接受/退回）、結果與預期的差距、時間序列趨勢——這些不依賴任何特定方法論工具。

#### 基礎公設風險

放進元方法論基礎層的任何概念，都會滲透到所有後續建構的方法論和方法中，且極難被事後評估或移除。原因：評估工具本身就依賴這些基礎概念，系統無法用自己偵測自己的基礎錯誤。

例：如果在元方法論中採用貝式定理作為迭代機制的基礎，那貝式思維將成為整個系統的一部分。貝式定理本身是數學定理（邏輯上無誤），但我們對它的「應用方式」可能有誤（選錯先驗、誤判證據、設錯閾值）。如果應用方式有系統性偏差，整個系統都會繼承這個偏差，且系統自己偵測不到。

#### 基礎選擇準則

放進元方法論基礎層的概念，應滿足：

1. **獨立於被評價對象**——評價工具不是被評價系統的一部分
2. **可外部驗證**——不依賴系統內部判斷來確認正確性，可用系統外部的觀察驗證
3. **最小假設**——基礎公設越少越好，每多一個假設就多一個無法自我偵測的風險點
4. **失敗可辨識**——對每個基礎假設，要能說清楚：如果這個假設是錯的，我們會看到什麼現象。答案是「看不出來」的假設不該放在基礎層

#### 回歸終止

四層架構在元元方法論終止，不再無限回歸。原因：元元方法論的原則（獨立、可外部驗證、最小假設、失敗可辨識）是接近自明的認識論原則，不需要更上一層的理論來證成。它們的正當性來自邏輯和經驗的基本共識，而非來自另一套後設框架。

### 二、定義

系統方法論：一套規範人機協作中 AI 行為的方法體系。

三個層次構成完整的方法論：
- **方法**：怎麼做（操作步驟）
- **原則**：為什麼這樣做（讓 AI 能舉一反三）
- **自我迭代**：怎麼知道方法有沒有用、怎麼改進（貝式更新機制）

缺任何一層，都不算方法論——只有方法是手冊，只有原則是哲學，沒有迭代是教條。

不是規範手冊（照做就好），是方法論（理解了才能用好）。這個命名定義了系統的層級。

### 三、根本問題

AI 有結構性缺陷：跨 session 無記憶、沒有內建時鐘不會主動停、自己檢查自己盲區共享、天性傾向直接行動而非先想清楚。多機器協作放大這些問題。

方法論的存在是用制度補結構性缺陷——不靠 AI 自律，靠系統設計。

### 四、目的體系——效率為終極標準

「效率」不是與品質、透明並列的一個維度，而是所有維度最終服務的目標：

```
品質穩定     → 減少返工      → 效率
過程可見     → 便於檢討      → 效率
穩定有信心   → 不用額外準備   → 效率
能自己成長   → 減少外部介入   → 效率
```

效率 = 協作產出的總價值 / 協作消耗的總成本（時間 + 溝通 + 返工 + 認知負擔 + 維護成本）

#### 第一層：協作品質（每次都要達到的）

1. **穩定 95 分**——靠制度不靠臨場發揮，每次都交出可預期的品質
2. **降低溝通往返**——AI 交出的東西用戶不用反覆退回重做
3. **複雜任務有信心**——步驟再多，每一步都有檢查點，做到第八步不用擔心第三步

#### 第二層：過程性質（系統必須具備的特性）

4. **過程可見**——關鍵步驟和關鍵判斷攤開，粒度以用戶能有效監督和決策為準
5. **出錯可回溯**——出了問題，能沿著步驟精準找到哪一步斷的
6. **結果可歸因**——答對了也能分辨是蒙對的、亂用方法碰巧對的、還是正確運用方法

#### 第三層：成長能力（讓系統越來越好）

7. **培養判斷力**——從照做規則 → 理解原則 → 舉一反三，能應對規則沒覆蓋的情況。掌握了方法就能自學、能知何能為及知何不能為，區分 known known 和 known unknown
8. **貝式自我迭代**——每次執行都是證據（成功/失敗/跳過），證據持續更新對方法有效性的信念，累積到閾值時觸發強化或修改。不只等別人指出問題，系統自己能監控、觀察、反思

#### 數學課比喻（貫穿全目的體系）

老師要求學生：不跳步驟、計算紙從左上角開始、小結論標星號、5 和 8 寫清楚、不確定先驗算再解題。

這些規範的效果：
- 降低錯誤率，提高解題效率（穩定 95 分）
- 學生交卷時已自己驗算過，老師不用一直退回（降低往返）
- 題目越長越難，按步驟寫的學生越穩（複雜任務有信心）
- 老師能從計算紙看到怎麼算的（過程可見）
- 答錯了翻計算紙就知道哪步錯（出錯可回溯）
- 答對了也能看出是真懂還是蒙的（結果可歸因）
- 學生掌握了方法就能自學、舉一反三（培養判斷力）
- 為老師的批改和檢討教學帶來參考資料，持續改善教法（貝式迭代）

### 五、設計原則

用來判斷「這條規則寫得好不好」「這段內容該不該放」的準則：

**P1 — 穩定可遵守**：規則基於客觀指標，不依賴 AI 自覺或記憶。能每次做到的規則才是好規則。

**P2 — 步驟可定位**：每步有編號或命名，出問題時能說「第三步的判斷有誤」。

**P3 — 原則可遷移**：不只告訴 AI「做什麼」，也告訴它「為什麼」。區分：培養判斷力的原則（留）vs 設計歷史紀錄（移出到記錄層）。

**P4 — 單一來源**：同一件事只在一個地方定義。10 份文件之間不重複、不矛盾。

**P5 — 證據雙向**：成功、失敗、跳過都是證據。不只收集問題，也收集「方法有效」的正面證據。

**P6 — 比例原則**：方法論的力度與任務的複雜度和風險成正比。簡單任務用輕量方法，複雜任務用完整流程。條件受限時，降低力度但不放棄——部分遵守的價值遠大於完全不遵守。

### 六、內容區分原則

用來判斷方法論文件中「什麼該留、什麼該移、什麼該砍」：

| 內容性質 | 判斷標準 | 處置 |
|---------|---------|------|
| 操作步驟（怎麼做） | AI 執行時需要 | 留 |
| 設計原則（為什麼這樣做） | 理解了能幫助 AI 舉一反三 | 留，寫成培養判斷力的形式 |
| 設計歷史（從 A 改成 B 的過程） | 只對設計者回溯有用 | 移出到記錄層 |
| 設計驗證（覆蓋了幾個情境） | 設計完成後不再被查閱 | 移出到附錄 |
| 沒有執行機制的規則 | 寫了但不會被觸發 | 補機制或移到「待建」 |

### 七、文件品質標準

每份方法論文件都應滿足：

1. **可穩定遵守**——規則基於客觀指標
2. **過程攤開**——每步的中間產出有定義
3. **出錯可定位**——步驟有編號，能精準指出斷點
4. **原則可遷移**——告訴 AI「為什麼」，讓它能應對變體
5. **自帶改進機制**——包含「怎麼知道方法有沒有用」和「怎麼改進」

### 八、系統層級檢視

不只看單一文件，還要看 10 份文件作為一個系統：

- **完整性**——從接到任務到交付成果，每個關鍵時刻都有對應方法
- **連貫性**——方法論之間銜接清楚（A 的產出是 B 的輸入）
- **不矛盾**——文件之間沒有互相衝突的規則
- **不冗餘**——同一件事只在一個地方定義
- **可迭代**——系統本身有機制檢驗和改進自己

### 十、自我迭代機制（貝式更新）

```
執行方法論
  ↓ 產生
一筆執行紀錄（成功 / 失敗 / 跳過）
  ↓ 累積
對該方法有效性的信念更新
  ↓ 達到閾值
行動：夠信任 → 強化（簡化、內化）
      夠質疑 → 修改（調整規則、改觸發條件、換方法）
  ↓ 修改後
新的先驗 → 繼續收集證據 → 循環
```

主動迭代需要三個能力：
- 監控——執行過程中觀察自己的行為
- 發現——辨識出偏差或異常
- 反思——判斷問題是個案還是系統性的

### 十一、操作化——把「效率」變成可衡量指標（草稿，有已知問題待修正）

> **已知問題**：此草稿多處違反元元方法論的「獨立於被評價對象」原則。
> A3 使用閉環四層（循環評價）、B1-B4 依賴觸發規則和三檔機制（被評價對象）。
> 需重新建構：指標應基於不依賴任何方法論工具的底層觀察（用戶行為、結果差距、時間趨勢）。

效率有兩面——產出和成本。方法論要讓產出變大、成本變小。只看結果不夠，還要看過程——結果好可能是運氣，過程好才能穩定重複。指標分四類：

- **A. 成果指標**——協作產出了什麼品質的東西
- **B. 過程指標**——方法論有沒有被正確執行
- **C. 成本指標**——方法論本身花了多少代價
- **D. 成長指標**——系統有沒有隨時間變好

A 和 C 合在一起衡量當下的效率。B 解釋為什麼效率高或低。D 衡量趨勢。

指標之間的診斷邏輯：
- 成果好 + 過程好 + 成本低 = 方法論有效（當下）
- 成果好 + 過程差 = 運氣好，不可持續
- 成果差 + 過程好 = 方法本身有問題，需要修改方法
- 成果差 + 過程差 = 方法論沒被遵守，先解決合規問題
- 成長指標向上 = 系統在改善（長期）
- 成長指標平坦 = 迭代機制失靈

#### A. 成果指標

**A1. 一次通過率**：用戶對 AI 交付物第一次就接受的比例。觀察：snapshot 項目從 `[ ]` 直接到 `[v]`/`[x]`，中間無「修正」OP。基準：目標 ≥ 80%。

**A2. 同 topic 修正密度**：同一 topic 下「修正」或「失敗」OP 佔該 topic 全部 OP 的比例。觀察：OP 狀態欄。基準：> 30% 觸發事後回顧。

**A3. 交付完備度**：交付物通過閉環四層的比例，及問題在哪一層被發現。觀察：閉環驗證記錄。基準：第一層問題（build/test 失敗）趨近零；同類三四層問題不重複出現。

#### B. 過程指標

**B1. 觸發合規率**：應觸發方法論且實際有觸發的比例。觀察：對照觸發規則表回顧最近任務。基準：目標 ≥ 80%，< 50% 說明觸發機制本身有問題。

**B2. 檔位適當率**：觸發後選擇的檔位與任務複雜度匹配的比例。觀察：對照檔位判斷指標。基準：偏差率 < 20%。

**B3. 過程可見度**：方法論執行中關鍵判斷有無被記錄或展示。觀察：OP 有無「決策」欄、AI 有無展示中間推理。基準：設計決策的理由 100% 有記錄。

**B4. 步驟跳過率**：方法論觸發後規定步驟中被跳過的比例。觀察：對照方法論步驟列表。基準：同一步驟被不同 session 跳過 ≥ 3 次 → 視為規則問題，觸發修訂。

#### C. 成本指標

**C1. 方法論時間佔比**：方法論活動的訊息輪次 / 總輪次。基準：快速檔 < 10%，標準檔 15-30%，完整檔可到 40%，> 50% 要檢討。

**C2. 用戶決策負擔**：方法論要求用戶做的決策次數和品質。觀察：AI 提問次數、是否帶選項。基準：每任務提問 ≤ 3 次（對焦階段除外），每問題帶選項和傾向。

**C3. 無效方法論執行**：跑了方法論但結論是「沒問題」的次數（假陽性）。基準：同一觸發規則連續 5 次無效 → 收緊觸發條件。

#### D. 成長指標

**D1. 同類問題復發率**：已做過回顧並設計改善的問題再次出現的比例。觀察：比較回顧記錄與後續 OP。基準：< 20%，復發自動升級改善層級。

**D2. 方法論自發觸發比例**：AI 自行觸發（非用戶手動 /指令）的比例趨勢。觀察：OP 觸發方式。基準：趨勢向上。

**D3. 規則修訂的證據基礎**：規則修改是否基於累積證據。觀察：修訂 OP 引用的證據數量。基準：每次修訂至少 3 筆支持證據，單一事件觸發的標記「暫行」。

#### 收集機制

| 層級 | 怎麼收集 | 成本 | 覆蓋指標 |
|------|---------|------|---------|
| 已有的 | 從 OP、snapshot、git history 提取 | 零 | A2、B1（部分） |
| 輕量添加 | OP 增加「方法論」欄位（觸發/檔位/結論） | 每筆多一行 | B1、B2、B4、C3、D2 |
| 定期抽查 | 每次 /小結 回顧本 session 指標 | 多 5 分鐘 | A1、A3、B3、C1、C2 |
| 累積分析 | 每月或每 10 次小結做趨勢分析 | 一次性 | D1、D2、D3 |
