### 知識庫儲存：兩個選項待決

**選項 A：Notion**
- 優勢：不加新服務、員工可直接在 Notion UI 編輯
- 劣勢：巢狀資料要拆多個 linked database、API rate limit 3 req/s

**選項 B：Supabase（只用 PostgreSQL）**
- 優勢：完美處理巢狀結構、SQL 查詢、pgvector 未來語意搜尋
- 劣勢：多加一個服務（但免費方案 500MB 夠用）

### Supabase 費用參考
| 階段 | Vercel | Supabase | 月費 |
|------|--------|----------|------|
| Phase 1 | Free | Free | $0 |
| Phase 2 | Pro $20 | Pro $25 | ~$45/月 |
| Phase 3 | 按量計費 | 按量計費 | 看用量 |

## 系統功能分類（查 vs 改）

### 查（讀取/檢視）
- 查標案進度（案件看板、儀表板、狀態總覽）
- 查知識庫（00A 團隊、00B 實績、00C 範本、00D SOP、00E 檢討）
- 查績效（投標勝率、趨勢圖表、交叉分析）
- 查提示詞模板（L1-L8 prompt、KB 引用矩陣）
- 查競爭情報（PCC API 對手紀錄、評委資訊）
- 查文件（已產出的建議書、簡報、報告）

### 改（建立/編輯/操作）
- 編輯知識庫（新增/修改 00A-00E 各類條目）
- 更新標案狀態（改案件進度、階段、備註）
- 組裝提示詞（選階段 + 選知識庫 → 產出 prompt）
- 報價驗算（輸入數字、調整參數）
- 品質檢查（貼入文字 → 跑檢核）
- 生成文件（產出 Word/簡報）
- 跑 AI 工作流（L1-L8 各階段多輪對話）
- 系統設定（Notion 連線、功能開關、欄位對照）

## 開發環境架構（重要發現）

### 核心問題
用戶想從手機（Telegram）繼續開發工作，但 Claude Code 跑在本地電腦上，
手機不知道該呼叫哪台電腦。

### 解法：指定一台常開電腦作為開發中心

```
手機 ──Telegram Bot──→ 常開的那台電腦（辦公室桌機）
                        ├── 專案檔案
                        ├── MCP servers（SmugMug etc.）
                        ├── Claude API + tools
                        └── Telegram Bot 程式

其他電腦 ──git/SSH──→ 同一台
```

- **不需要買雲端伺服器**，用現有電腦就行（參考 OpenClaw 用 Mac Mini 的做法）
- Phase 1：辦公室/家裡一台常開的電腦
- Phase 2：同上，或視需求搬到雲端
- Phase 3 商業化：才真正需要雲端伺服器

### Telegram Bot 架構
- Telegram Bot 跑在常開電腦上
- 收到訊息 → 呼叫 Claude API（帶 tool use）→ 操作本地檔案 → 回傳結果
- 支援語音輸入（手機端語音轉文字 → Telegram 傳文字訊息）
- 跟 Phase 2 的 Discord Bot 是同一個後端架構，做一次兩個頻道都能用

### 影響
- Layer 1 存取方式更新：Phase 1 也包含手機（Telegram）
- 「上雲」的時間點推遲到 Phase 2/3，Phase 1 用本地電腦即可
- Telegram Bot 可以作為精靈的另一個入口（不只 Claude Code Skill）
