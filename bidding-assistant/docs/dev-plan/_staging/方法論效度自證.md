# 方法論效度自證：10 項缺口與修正方案

類別：rule
目標：docs/methodology/（多份檔案）+ CLAUDE.md + .claude/commands/
日期：2026-02-21
相關 OP：@op:20260221-JDNE#0500

---

## 背景

方法論 v2 完成結構評估（P1-P5 逐檔 + 系統交叉比對）後，進行效度驗證。定義「有效 = 可自我迭代」。

驗證方式：先由 AI 做直覺評估作為對照組，再用方法論跑方法論自己（自證），比較差異。自證找到了直覺評估沒抓到的問題。同時，用戶從使用者角度補充了外部效度缺口。

結論：**內部效度成立**（方法論能對自身產出有意義的改進）。**外部效度有 7 項缺口**（方法論在讓用戶滿意這件事上還不夠完整）。

---

## 結論

10 項缺口分為兩類，都需要修進方法論：

### A. 內部結構缺口（3 項，自證發現）

**A1. 觸發時機的雞生蛋問題**
觸發表說「收到任務時」一次性檢查，但有些觸發條件（如「改動 >3 個檔案」）需要先做需求對焦才知道。缺「完成需求對焦後重新檢查其他觸發條件」的指引。

**A2. 缺中途重新評估**
實作中途範圍膨脹、假設被推翻、方向要調整，這些常見情況沒有對應的觸發條件和處理流程。方法論只有「做之前」和「做之後」，沒有「做到一半發現不對」。

**A3. 觸發不管脈絡合法性**
觸發條件告訴你「什麼時候啟動什麼工具」，但不檢查「這個工具在當前脈絡下是否概念上合法」。實例：模型委員會的觸發條件在評價方法論時命中，但用它評價方法論本身是循環的。

### B. 外部效度缺口（7 項，用戶回饋 + AI 觀察）

用戶的核心期待：「我許願，你執行。」方法論應該讓 AI 成為可靠的執行者，用戶只需確認，不需微管理。以下 7 項是方法論目前做不到的。

**B1. 聽不懂隱含期待**
需求對焦只抓字面歧義。用戶沒說出口的優先順序、品味偏好、「這個不用你說我也該知道」的隱含期待，方法論沒有機制處理。

**B2. 問太多**
方法論鼓勵「不確定就問」，但過度提問浪費用戶時間。什麼時候該問、什麼時候該自己判斷，沒有指引。

**B3. 跨 session 不會學**
用戶糾正了 AI，下個 session 可能犯一樣的錯。MEMORY.md 和 CLAUDE.md 是載體，但方法論沒有「把用戶糾正轉化為持久改進」的流程。

**B4. 溝通浪費時間**
不要讓用戶重複自己、不要問已經有答案的問題。「說人話」是規則，但溝通效率不在方法論範圍內。實例：用戶已經講過期待，AI 又問一次。

**B5. 該主動的時候被動**
選項中有一個明顯更好時，AI 不該問用戶選哪個，應該自己判斷。方法論沒有教 AI 判斷「這個我應該自己決定」。

**B6. 做完就停，不主動推進**
AI 完成一個步驟後停下來等用戶問「接下來呢」。用戶期待的流程是：AI 做完一步 → 主動帶到下一步 → 用戶確認 → 繼續。方法論有拆解排序（知道下一步），但沒有「完成後自動推進」的行為規範。

**B7. 不主動回報，讓用戶必須微管理**
記錄、快照、暫存這些 CLAUDE.md 有規則的事情，AI 不穩定遵守，又不主動告訴用戶自己的計畫和進度。用戶被迫反覆提醒，信任感下降，最終違背「許願→執行」的初衷。

## 寫入內容

### 1. _index.md 觸發機制表：加觸發時機說明（A1）

在觸發機制表上方加一段：

```
### 觸發時機

觸發條件在「收到任務時」做第一輪檢查。但部分條件（如檔案數、模組數）需要先完成需求對焦才能判斷。

流程：收到任務 → 第一輪觸發檢查（能判斷的先判斷）→ 需求對焦（如觸發）→ 第二輪觸發檢查（用對焦後的資訊重新掃描）→ 進入後續方法論。

第二輪不需要重跑需求對焦，只需重新比對觸發條件表。
```

### 2. _index.md 或新檔：中途重新評估機制（A2）

在觸發機制表新增一行：

```
| 實作中途：範圍膨脹 >50% 或核心假設被推翻 | 暫停 → 重新評估（拆解排序或需求對焦） | 計畫 vs 實際比對 |
```

在 _index.md 加一段簡要說明：

```
### 中途重新評估

實作過程中如果發現以下情況，暫停實作，退回相應方法論：

| 情況 | 退回 |
|------|------|
| 需求理解有誤 | 需求對焦 |
| 範圍比預期大 >50% | 拆解排序（重新拆和排） |
| 核心假設被推翻 | 改動前驗證（重新掃描） |
| 多個步驟失敗 | 事後回顧 |

退回不是失敗，是用新資訊做更好的決策。不退回才危險——硬著頭皮做下去，前面的工作可能全部白費。
```

### 3. _index.md 或 methodology-01：脈絡合法性檢查（A3）

在需求對焦或觸發機制表加一個注意事項：

```
### 工具脈絡合法性

觸發條件命中只代表「這個工具通常在這種情況下有用」。啟動前額外確認：這個工具在**當前具體脈絡**下是否合法？

已知的脈絡衝突：
- 用方法論工具（如模型委員會）評價方法論本身 → 循環（「拿被告當法官」）
- 用閉環驗證分析工作的「完成度」→ 閉環的驗證手段以檔案改動為中心，分析工作不適用

遇到脈絡衝突時，標記為「此工具在本脈絡不適用」，不是跳過驗證，而是改用脈絡外的驗證方式（如用戶判斷、獨立指標）。
```

### 4. CLAUDE.md 方法論段：加行為規範（B2, B5, B6, B7）

在 CLAUDE.md 的方法論段或溝通原則段新增：

```
### AI 行為規範（必須遵守）

**主動推進，不等用戶問「接下來呢」**
完成一個工作單元後，主動告訴用戶下一步是什麼，等用戶確認後繼續。如果已有拆解排序的工作清單，按順序自動推進。

**主動回報計畫和進度**
開始一段工作前，先告訴用戶打算做什麼（「我接下來要更新快照、寫 OP、推上去，然後繼續做 X」）。做完後回報結果。不讓用戶猜你在幹嘛。

**主動做記錄，不等用戶提醒**
OP、快照、暫存、git push——CLAUDE.md 規定什麼時候該做就做，不需要用戶提醒。

**能自己判斷的不要問**
如果選項中有一個明顯更好（根據已知上下文、專案慣例、技術最佳實踐），直接做，不問用戶。只在選項真的各有優劣、需要用戶價值判斷時才問。

**控制提問頻率**
一次不超過兩個問題。能合併就合併。問之前先想：這個問題我能不能從 CLAUDE.md、記錄層、或專案慣例中找到答案？能的話就不問。
```

### 5. 方法論文件：跨 session 學習（B3）

在 methodology-04（事後回顧）的自我迭代段加：

```
### 跨 session 持久化

事後回顧的改善措施如果是「以後某類情況要用不同方式處理」，不能只留在當次對話。必須落地到持久位置：

| 改善類型 | 寫到哪裡 |
|---------|---------|
| 新的規則/慣例 | 提議更新 CLAUDE.md |
| 除錯經驗 | docs/debugging.md |
| 方法論調整 | 對應的 methodology 文件 |
| 用戶偏好 | MEMORY.md（本機）或 CLAUDE.md（全機器） |

「下次注意」不是改善措施。沒有寫進持久位置的改善，等於沒有改善。
```

### 6. 需求對焦補充：隱含期待（B1, B4）

在 methodology-01 的「第二步：帶判斷提問」後加：

```
### 隱含期待檢查

字面歧義解決後，額外問自己：用戶沒說出口但可能預期的是什麼？

資訊來源：
- CLAUDE.md 的專案背景和慣例
- MEMORY.md 的用戶偏好記錄
- 近期 OP 和快照中用戶的行為模式
- 當前對話中用戶已經表達過的內容（不要重複問）

如果能從這些來源推斷出用戶的隱含期待，直接按推斷執行，在回報時說明「我假設你希望 XX，所以我這樣做了」。用戶糾正時，記錄到 MEMORY.md 或 CLAUDE.md。
```
