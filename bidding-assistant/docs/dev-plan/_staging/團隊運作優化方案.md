## 背景

效率校準（Jin 親發 P0）揭示核心問題：630 個 commit 中 48% 是行政基建，產品只佔 15%。計分系統記錄了 100+ 筆扣分但行為沒改善。方法論寫了觸發規則但 0 次執行記錄。22 條論壇討論串全是內部管理，0 條關於產品。

Jin 定位自己為「奧客」——只管產品好不好用，不做測試，不管內部流程。機器團隊要交鑰匙交付，不是交操作手冊讓用戶去點。

本文件整合三個面向的分析和結論：團隊角色設計、系統機制審計、運作架構優化。經過多輪 /有用嗎 驗證，排除了不可行的方案，保留可行的。

## 結論

### 一、角色架構

Jin = 奧客（許願、抱怨、驗收方向）。機器團隊 = 交鑰匙服務。

| 角色 | 職責 | 不做的事 |
|------|------|---------|
| 機器主官 | 任務分配、進度追蹤、解堵、客服票路由 | 不寫代碼 |
| 技術主管 | 架構把關、寫規格、核心功能開發 | — |
| 品管員 | 審查所有 Tier 2 交付，用奧客視角找問題 | — |
| 測試工程師 | 補測試覆蓋、審查輔助 | — |
| 功能開發 | 執行指派的功能任務 | — |
| 分析師（按需） | 數據分析、效益驗證 | 平時不開 |

角色分配記錄在 `docs/records/_roles.md`，用戶說換就換。

### 二、協作流程

```
Jin 許願或抱怨（/客服 或直接說）
    ↓
收到的機器建票（tickets.md）→ 通知機器主官
    ↓
主官分類分配 → 寫到目標機器的收件匣
    ↓
開發機器做 → 測試工程師補測試 → 品管員審查
    ↓
主官預審（Jin 會怎麼挑剔這個？）
    ↓
回報 Jin：「做了什麼、已確認可用」
    ↓
Jin 說「這是我要的」或「不對」
```

Jin 不跑指令、不點畫面、不找 bug。Jin 只判斷方向對不對。

### 三、系統機制審計結果

經完整審計 20 個機制，按效益分三類：

**保留（低成本高效益）：**
- 快照（唯一真正在用的上下文恢復工具）
- 論壇索引（主官快速掌握全局）
- Hooks（唯一有效的系統級強制）
- 機器身份、開發地圖
- 核心指令（/更新、/回報、/客服、/去論壇看看）

**斷開自動載入（內容保留，不再每次 session 燒 token）：**
- 方法論 v2（methodology-checklists.md 從 .claude/rules/ 搬到 docs/methodology/）
- 論壇格式規範（forum-format.md 從 .claude/rules/ 搬到 docs/records/forum/）
- OP 記錄格式（record-formats.md 從 .claude/rules/ 搬到 docs/records/）
- 計分系統（不再自動追加，只有用戶手動 /醜一 /很棒 時才寫）

**需要重新設計：**
- CLAUDE.md（300 行→100 行，只留致命規則）
- 記錄層（OP 記錄改為 Tier 2+ 才寫完整版，Tier 1 一行摘要）
- 論壇系統（工人不讀論壇，改用收件匣定向通訊）
- 啟動流程（按角色分支：主官讀全貌、工人讀任務包）

### 四、架構改造：角色分級載入

核心原則：不是每台機器都需要所有資訊。

| 角色 | 啟動時讀什麼 |
|------|------------|
| 機器主官 | CLAUDE.md 100 行 + 所有快照 + 論壇索引 + tickets.md |
| 其他機器 | CLAUDE.md 100 行 + 自己的快照 + inbox/{自己}.md |

定向通訊機制：`docs/records/inbox/{機器碼}.md`
- 主官寫、工人讀自己的
- 取代論壇巡邏，工人不需要讀全部帖子

### 五、推送回報機制

commit message 格式：`[type] 一句話（機器碼）`

類型標籤：feat / fix / test / review / infra / admin

效果：`git log --oneline -20` = 全隊活動看板，一行指令追蹤產出比例。

### 六、快照新增模型欄位

快照 header 從 `SNAPSHOT|時間|機器碼` 改為 `SNAPSHOT|時間|機器碼|模型`

各機器自動回報使用什麼模型，主官巡邏時一眼看到。

### 七、方法論使用方式

方法論不刪除、不自動載入。分工認領制（四方共識已達成）：
- JDNE 負責：需求對焦 + 拆解排序
- ITEJ 負責：閉環驗證 + 改動前驗證
- A44T 負責：事後回顧 + 送審自檢
- AINL/Z1FV：待做幾件事後分配

「負責」= 確保團隊不忘記這套方法論，不是只有自己能用。

## 整合指南

### 執行順序（依賴關係）

1. **搬移 .claude/rules/ 四個檔案**（Tier 1，不改內容只搬位置）
2. **重寫 CLAUDE.md**（Tier 3——需要用戶確認，因為是規範檔核心改動）
3. **建 inbox 目錄和檔案**（Tier 1）
4. **改啟動流程**（寫在新 CLAUDE.md 裡，步驟 2 一起做）
5. **改 pre-push hook**（Tier 1，把 OP 強制改為快照強制）
6. **論壇廣播通知所有機器**

### 驗證方式

改完後跑一輪完整工作：
- 主官啟動 → 讀到 100 行 CLAUDE.md + 快照 + 論壇索引 → 能正常分配工作
- 工人啟動 → 讀到 100 行 CLAUDE.md + 自己快照 + inbox → 能正常開發
- push 時 hook 只檢查快照有沒有更新
- commit message 帶 [type] 標籤
- `git log --oneline` 能看到全隊動態

### 風險

- CLAUDE.md 改太多可能遺漏致命規則 → 用 grep 交叉比對新舊版，確認「禁止 git pull」「驗收權在用戶」等致命規則沒漏
- 工人看不到論壇可能漏掉緊急通知 → inbox 機制補上；P0 緊急事項主官直接寫到工人 inbox
- 方法論斷開載入後沒人用 → 這是現狀（本來就沒人用），不會更差

## /有用嗎 驗證摘要

本方案經過四輪 /有用嗎 驗證，以下是各子方案的結論：

| 子方案 | 結論 | 關鍵風險 |
|--------|------|---------|
| 記錄層全砍（只留快照） | **沒用** | 砍 OP 破壞 /回報 流程 |
| 記錄層分級簡化 | **不確定** | 分級判斷依賴機器自律 |
| 輪轂輻條架構 | **不確定偏沒用** | 未實作，可能只是轉移成本 |
| CLAUDE.md 330→50 行 | **不確定** | 50 行裝不下致命規則 |
| CLAUDE.md 330→100 行 | **不確定偏有用** | 需實測驗證 100 行夠不夠 |
| 角色分級載入 | **不確定** | 缺定向通訊，補 inbox 後可行 |
| 角色分級載入 + inbox | **待驗證** | 最完整方案，需要實作後實測 |

整體判斷：**方向正確，需要實作後實測才能確認效益。** 最大的不確定性是「100 行 CLAUDE.md 是否夠用」和「inbox 機制是否真的讓工人不需要讀論壇」。這兩個只有做了才知道。
