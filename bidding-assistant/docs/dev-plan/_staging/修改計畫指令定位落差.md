## 現況分析

### 大背景

核心目標是贏標案，這是一個複雜龐大的工程，有多條主線任務（Layer 0 知識庫、Layer 1 Bot、SaaS 架構等）和更多條支線任務。不可能一次通關，需要存檔/讀檔機制。

而 CLI 是線性結構（一次只能做一件事），面對多線有分支的現況，需要不同指令來切換主/支線任務的推進。

### 現有指令的角色

| 指令 | RPG 比喻 | 狀態 |
|------|---------|------|
| /暫存 | 存檔（保存支線碎片） | 正常運作 |
| /待辦 | 查進度（看所有任務線） | 正常運作 |
| /小結 | 冒險日誌（記錄做了什麼） | 正常運作 |
| /更新 | 同步存檔（多人連線同步） | 正常運作 |
| /修改計畫 | 讀檔繼續打（載入碎片推進主線） | **壞了** |

### 原始設計意圖

用戶的原始構想是主線/支線模式：

- **暫存（支線任務）**：討論是碎片化的，隨時把各種碎片保存起來
- **修改計畫（主線任務）**：定期檢視某個專題的碎片是否夠多，夠了就拼進去推進主線規劃——不一定是完成，可能只是推進一步

簡言之：主線任務所需要的碎片，要從支線任務中取得。

### 現在的實作

現在的 /修改計畫 是批次搬運工具：

1. 列出所有 plan type 暫存檔
2. 逐檔顯示 diff → 用戶確認 → 寫入目標文件
3. 寫完刪掉暫存檔

### 落差

| 應該做的事 | 現在有沒有做 |
|-----------|------------|
| 看主線走到哪了 | 沒有（不讀目標文件現況） |
| 按專題分組碎片 | 沒有（逐檔處理，不分組） |
| 判斷碎片夠不夠推進 | 沒有（列出就處理，不評估成熟度） |
| 拼接碎片成更完整的規劃 | 沒有（直接搬內容，不做合成） |

### 佐證：指令從未被使用

17 份「待推」暫存檔從建專案以來沒有人跑過 /修改計畫。可能原因：

- 暫存檔本身已經是可讀的結構化文件，不急著搬
- 目標文件可能根本不存在（如「新增 A02」「新增 M07」）
- 多份暫存檔指向同一目標，整合本身是大工程
- 有些暫存檔的前提已變（如 Discord Bot → SaaS 網頁）

## 可能方向

/修改計畫 應該重新設計為「讀檔繼續打」工具：

1. 選一條任務線（主線或支線）
2. 載入該任務線的現況（主線文件 + 相關暫存碎片）
3. 評估碎片是否足夠推進
4. 合成碎片，推進任務線（可能只是推一步，不一定完成）
5. 更新任務線的存檔

具體設計待下次討論。
