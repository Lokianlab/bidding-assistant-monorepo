## 背景

專案安裝了 54 個 Claude Code 插件（全部來自官方市集），加上 Claude.ai 端內建的 Notion、Canva、Gamma MCP。大部分插件跟我們的技術棧無關（其他語言、其他服務），需要整理出哪些真正有用，避免每次都從 54 個裡面猜。

## 結論

### 目前有用的（7 個）

| 插件 | 做什麼 | 怎麼用 |
|------|--------|--------|
| Notion（內建 MCP） | 讀寫 Notion 工作區 | 直接操作案件資料庫、知識庫，暫時不需要自建 Notion MCP |
| context7 | 查套件最新文件 | 開發時查 Next.js / React / Tailwind 等 API |
| superpowers | 開發工作流（brainstorming、TDD、debugging 等） | 已融入日常，自動觸發 |
| commit-commands | `/commit` 一鍵提交 | 三台機器頻繁推送，省去手動 git 操作 |
| feature-dev | 引導式功能開發（先看架構再寫碼） | 下一個功能（情報模組 M01）開發時用 |
| plugin-dev | 維護自訂 skill 和 hook | 專案有 8 個自訂指令 + 2 個 hook |
| explanatory-output-style | 回覆附帶教學說明 | 目前已啟用 |

### 以後看情況的（7 個）

| 插件 | 什麼時候需要 | 前提條件 |
|------|------------|---------|
| playwright | 自動操作瀏覽器，E2E 測試或抓政府網站資料 | Web App 功能夠多值得 E2E 時 |
| firecrawl | 爬網頁轉 markdown，抓招標公告 | 需要批量抓取政府公開資訊時 |
| frontend-design | 介面設計指導 | 做新頁面、改 UI 時 |
| pinecone | 向量搜尋（知識庫的第三個儲存選項） | 知識庫儲存方案定案時一併評估 |
| vercel | 一鍵部署 | SaaS 部署方案決定用 Vercel 時 |
| sentry | 錯誤監控 | Web App 上線後 |
| posthog | 使用者行為分析 | Web App 上線後 |

### 不需要的（40 個）

10 個其他語言 LSP + 30 個不相關服務（asana、slack、stripe、firebase 等）。跟我們的技術棧（TypeScript + Next.js + Notion）無關。
