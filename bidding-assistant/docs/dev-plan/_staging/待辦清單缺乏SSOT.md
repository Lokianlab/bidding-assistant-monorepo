# 待辦查詢不確定性

類別：issue
目標：（待定）
日期：2026-02-19
相關 OP：@op:20260219-ITEJ#1324

---

## 背景

討論 /暫存 改版時，用戶問「當我問待辦清單時，資料是從哪裡整理出來的？」。初步觀察是「待辦資訊散落 5 個來源，缺少 SSOT」。

經自問自答迭代後，問題定義修正：真正的問題不是資料分散，而是**查詢不確定性**——「待辦清單」沒有標準查詢程序，AI 每次即興決定查哪些來源，導致不同 session、不同機器掃出不同結果。

## 現況分析

### 活的待辦來源（3 個）

1. 各機器 `_snapshot-*.md` 的 `[ ]` `[?]` 項目
2. `_index.md` 中狀態為「進行中」的 topic
3. `_staging/_index.md` 中待推/待決的暫存檔

重啟流程（Layer 0-2）會讀這三個，但沒有「匯總成待辦清單」的步驟。壓縮流程更輕，只讀自己的快照。

### 實際漏項案例

A44T 機器被問「待辦清單」時掃出 4 項，但漏了「知識庫建置精靈」。該項目在 3 份暫存檔中都有提到（pain-points-analysis.md、saas-architecture-discussion.md、06-開發進程與優先序.md），卻因為不在任何機器的快照 `[ ]` 裡、也不在 `_index.md` 的「進行中」topic 中，所以被跳過。用戶主動提醒後 AI 才找到。

### 根因

- 「回答待辦清單」沒有標準程序（對比：寫 OP 有格式、更新快照有流程、/小結有步驟）
- AI 每次即興選擇查詢來源，覆蓋範圍不穩定
- 暫存檔中隱含的待辦事項（如精靈建置）不在 3 個主要來源的任何一個中

### 排除的錯誤方向

建 `_backlog.md` 做 SSOT：衍生檔案需要維護、會碰撞，而且不解決根因（即使有 SSOT，也可能漏更新）。

## 可能方向

1. **加 /待辦 指令** — 標準化查詢程序，明確列出要掃的來源和匯總邏輯
2. **在快照規範中加入「暫存檔掃描」** — 更新快照時也檢查 _staging/ 中與本機相關的項目
3. **維持現狀** — 接受不穩定，靠用戶補漏

尚未評估各方案。
