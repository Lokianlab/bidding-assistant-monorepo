# 待辦清單缺乏單一來源

類別：issue
目標：（待定）
日期：2026-02-19
相關 OP：@op:20260219-ITEJ#1254

---

## 背景

討論 /暫存 改版時，用戶問「當我問待辦清單時，資料是從哪裡整理出來的？」，發現待辦資訊散落在 5 個來源：

1. 各機器 `_snapshot-*.md`（`[ ]` 待做、`[?]` 未決）
2. `_index.md`（狀態=進行中的 topic）
3. 小結 C 章（待決策項目）
4. `_staging/_index.md`（待推/待決的暫存檔）
5. `plan-backlog` topic 的 OP（某次盤點的靜態數字）

每次有人問「待辦清單」，AI 要重新掃描這 5 個來源再拼裝。沒有一個檔案是待辦清單的 SSOT。

問題不在於掃描成本（3-4 次 Read），而在於：
- Compact 後 AI 不一定記得要查全部 5 個來源，可能漏掉
- 不同機器掃出來的結果可能不一致
- plan-backlog 的 OP 只是靜態快照，下次問又要重掃

## 現況分析

現有每個來源都有自己的本職，待辦資訊是附帶的：
- 快照的本職是「各機器工作備忘」
- _index.md 的本職是「主題現況」
- 小結 C 章的本職是「歷史決策記錄」

所以不是這些來源設計錯了，而是缺少一個匯總視角。

## 可能方向

1. **維持現狀** — 接受分散，每次手動掃描
2. **加 /待辦 指令** — 不建檔案，動態掃描彙整（像 /小結 是動態產出）
3. **建 `_backlog.md`** — 在 `docs/records/` 加全局待辦檔，每次 push 時自動刷新

尚未討論各方案的優缺。
