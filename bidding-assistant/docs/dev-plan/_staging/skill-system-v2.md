## 背景

Skill 系統在實際使用中暴露三個問題：

1. **思維型 skill（對焦/驗證/拆解/回顧）難用**：用戶要記指令名稱和使用時機，固定流程不分輕重，產出是方法論格式而非對話
2. **觸發規則不可靠**：9 條觸發條件依賴 AI 自律，實際遵守率不穩定，製造假安全感
3. **跨 session 無學習機制**：AI 犯的錯在下一個 session 會重犯，沒有行為回饋迴路

經 /回顧 追根因：skill 系統只優化品質（通過程序強制徹底性），犧牲了溝通效率和工作效率。根因是 LLM 注意力有限，詳細程序不保證更高遵守率。

進一步參考獎懲研究（RLHF 原理的無程式碼實作），發現計分板只是回饋機制的一部分。完整的獎懲系統需要：自動化評估迴路（不只靠用戶手動觸發）、結構化評分維度、門檻驅動迭代、以及 Constitutional AI 式的自我批評。

## 結論

### 一、Skill 分類與處置

| 類別 | 指令 | 處置 |
|------|------|------|
| 產出型 | /小結, /暫存, /待辦, /修改計畫, /kb, /地圖 | 保留 |
| 工具型 | /更新, /安裝 | 保留 |
| 溝通型 | /有用嗎（原 /評估）, /你說呢 | 新建 |
| 學習型 | /醜一（原 /記住）, /很棒, /你去吃屎 | 新建或改寫 |
| 思維型 | /對焦, /驗證, /拆解, /回顧 | **砍掉**，降為行為原則 + 內部清單 |

最終 13 個 skill。

### 二、行為原則（取代觸發規則表）

寫入 CLAUDE.md，取代「事件觸發規則」段落和「工具觸發指令」段落：

```markdown
## 工作行為原則

- 目標先行：動手前確認目標明確且可量化。含效益宣稱的要查證事實。說不清就問。
- 風險正比：改前檢查影響範圍。1 檔腦中過；≥3 檔 grep 並把影響點列給用戶看；規範檔全掃展示完整分析。
- 大事先拆：超過 3 步的任務先列步驟，風險最高或解鎖最多的先做。
- 失敗追因：同一件事失敗兩次，停下來找根因再繼續，不重試。
- 做完閉環：做對了→有效→做全了→沒矛盾。規模小腦中過，規模大展示給用戶。
```

### 三、內部清單（取代 skill 完整流程）

放在 `.claude/rules/methodology-checklists.md`（不加 paths:，永遠載入），提取各 skill 的核心檢查項：

```markdown
# 方法論內部清單

AI 內部參考，不對用戶展示步驟編號或表格。用對話方式呈現分析。

## 目標釐清（原 /對焦）
- 指令有模糊詞？展開可能含義，猜錯返工成本高的先問
- 效益宣稱有事實支撐嗎？能量化就量化
- 隱含期待：用戶沒說但可能預期的是什麼？

## 影響分析（原 /驗證）
- 四面向：程式碼引用、資料格式相容、文件描述、其他機器快照
- 間接依賴：影響點的消費者呢？
- 判斷：一定會壞 / 可能會壞 / 需要更新

## 任務分解（原 /拆解）
- 完成狀態用一句可驗證的話描述
- 列可獨立驗收的交付物
- 硬依賴 / 軟依賴 / 無依賴
- 高風險先做，解鎖最多的先做

## 根因追問（原 /回顧）
- 精確描述症狀（在什麼條件下，什麼行為與預期不同）
- 逐層追問，每層要有查證（不能只靠推理）
- 根因能改嗎？能改就是該修的地方，不能改就轉替代路徑
- 從根因往回走驗證因果鏈
```

### 四、計分板系統

#### 設計

| 元素 | 說明 |
|------|------|
| 存放位置 | `.claude/rules/scoring.md`（不加 paths:，永遠載入，在 git 中共享） |
| 格式 | `{行為描述}\|{累計分數}\|{機器碼:MMDD, ...}` |
| 寫入方式 | 合併式——同行為模式更新同一行，不追加新行 |
| 壓縮 | 超過 30 種不同模式時壓縮最舊的（只保留模式和總分） |
| 分數不清零 | 累積是特性。行為是否改善看日期（最後一筆離現在多久）。不同行為模式不相消。 |

#### 計分板觸發

| 指令 | 觸發 | 分數 |
|------|------|------|
| /醜一 | 用戶或 /你去吃屎 指出違規 | **-1** 每項 |
| /很棒 | 用戶認為 AI 表現出驚豔觀察/洞見/表現 | **+2** |
| /你說呢 | 用戶被迫要求 AI 表態（AI 本該主動） | **-1** |

#### 計分板範例

```markdown
# 計分板

總分: +3

## 扣分
不主動表態|-3|JDNE:0221, ITEJ:0221, JDNE:0219
效益宣稱未自驗|-1|JDNE:0221
主動表態但胡扯|-2|JDNE:0224, JDNE:0226

## 加分
主動承認提案無效|+2|JDNE:0221
提出用戶未想到的關鍵風險|+4|JDNE:0222, ITEJ:0225
```

### 五、獎懲回饋機制

#### 設計原則（從獎懲研究提煉）

| # | 原則 | 研究依據 | 對應到我們的系統 |
|---|------|---------|-----------------|
| 1 | 回饋迴路比生成能力重要 | Self-Refine (CMU 2024)：生成→批評→修正提升 20% 品質 | 每個工作環節都要有評估，不只是做完才檢查 |
| 2 | 結構化維度評分 | 分維度 1-5 分比整體打分更能定位問題 | 閉環四層各自追蹤，不是只看「過/沒過」 |
| 3 | 門檻驅動迭代，上限 2-3 次 | 超過 3 次邊際效益遞減 | 閉環修正最多 2 次，第 3 次停下來換路 |
| 4 | 人工修改距離 = 最真實的信號 | 用戶改了多少 = 品質指標，同類修改反覆 = 規則缺口 | 用戶同類糾正 ≥3 次 → 提示寫進規則 |
| 5 | 紅旗規則 | 單一維度嚴重失敗 → 不論總分攔截 | 閉環第一層沒過 → 後面跳過直接修 |
| 6 | 多評審交叉驗證 | PoLL：3 個 LLM 交叉評分優於單一評審，成本低 7-8 倍 | 模型委員會已有，可在獎懲中更輕量使用 |
| 7 | Constitutional AI 式自我批評 | 生成→依原則自我批評→修正 | 行為原則 = 憲法，工作單元完成後靜默自檢 |

#### 三種信號來源

| 來源 | 可靠度 | 觸發方式 | 範例 |
|------|--------|---------|------|
| 用戶觸發 | 最高 | 手動 | /醜一、/很棒、/你說呢 |
| 系統偵測 | 高（客觀事實） | 自動 | hook 攔截、build 失敗、同問題重複 |
| 自我批評 | 中（AI 判斷） | 自動，發現問題時才告訴用戶 | 行為原則自檢 |

#### 自動觸發規則

| 偵測條件 | 分數 | 說明 |
|---------|------|------|
| push 被 pre-push hook 攔截 | **-1**「推送缺記錄」 | 客觀事實，無誤判空間 |
| 閉環同一工作單元修正 ≥2 次 | **-1**「閉環反覆失敗」 | 第一次失敗不扣（正常），第二次扣（該找根因了） |
| 同 topic ≥3 筆修正 OP | **-1**「重複犯錯」 | 跨 session 也算，表示沒學到 |
| 用戶同類糾正 ≥3 次 | 不扣分，但自動提示 | 「這個模式出現 3 次了，要不要寫進規則？」 |

#### 行為原則的獎懲信號

每條行為原則都有對應的正向和負向信號。正向信號靠自我批評偵測（靜默記錄），負向信號靠系統偵測或用戶觸發。

| 原則 | 正向信號（自我批評偵測，+1） | 負向信號（系統/用戶偵測，-1） |
|------|---------------------------|---------------------------|
| 目標先行 | 動手前主動確認目標且用戶無異議 | 用戶被迫 /你說呢 或追問「你到底要做什麼」 |
| 風險正比 | 影響分析完整，用戶沒抓到遺漏 | 改了東西壞了別的地方（build/test 失敗） |
| 大事先拆 | 按步驟執行，無臨時大改 | 中途發現遺漏導致計畫重來 |
| 失敗追因 | 找到根因一次修好 | 同問題第 3 次出現（自動偵測） |
| 做完閉環 | 閉環四層一次通過 | 閉環被用戶打回或同一單元修 ≥2 次（自動偵測） |

正向信號的記錄條件較嚴：必須是用戶確認無異議的情況（不是 AI 自己覺得做得好）。不確定時不記正向。

#### 自我批評迴路（Constitutional AI 式）

```
觸發時機：每完成一個工作單元（準備 commit 前）
    ↓
用 5 條行為原則逐條過：這個工作單元有沒有違反？
    ↓
  ┌─ 沒問題 → 靜默，不打擾用戶。符合正向信號條件的記 +1
  └─ 有問題 → 主動告訴用戶「我發現 XX，要不要修？」→ 記計分板
    ↓
修正後重新自檢（最多 2 次，第 3 次升級告訴用戶「我修不好，需要你看」）
```

**不展示自檢過程，只展示結論。** 用戶不需要看到 AI 在逐條對照行為原則——那是 AI 內部的事。用戶只需要知道：(1) AI 有在自檢，(2) 自檢發現問題時 AI 會主動說。

#### 迭代上限與升級路徑

| 情境 | 上限 | 超過後 |
|------|------|--------|
| 閉環修正 | 2 次 | 停下來，告訴用戶問題在哪，換路或請用戶指示 |
| 自我批評修正 | 2 次 | 告訴用戶「我修不好這個問題」，讓用戶決定 |
| /你去吃屎 找到的違規修正 | 不限 | 每項獨立修，但同類問題 ≥3 項時觸發系統級改善 |

研究依據：Self-Refine 證實大部分改善在前 1-2 次迭代，超過 3 次邊際效益遞減。

#### 紅旗規則

以下情況不論整體表現如何，直接攔截：

- 閉環第一層（做對了）沒過 → 後面三層跳過，直接修
- build 或 test 失敗 → 不能繼續推進，先修
- 用戶說「不對」或「停」→ 立即停止當前方向
