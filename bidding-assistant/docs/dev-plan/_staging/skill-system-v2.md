## 背景

Skill 系統在實際使用中暴露三個問題：

1. **思維型 skill（對焦/驗證/拆解/回顧）難用**：用戶要記指令名稱和使用時機，固定流程不分輕重，產出是方法論格式而非對話
2. **觸發規則不可靠**：9 條觸發條件依賴 AI 自律，實際遵守率不穩定，製造假安全感
3. **跨 session 無學習機制**：AI 犯的錯在下一個 session 會重犯，沒有行為回饋迴路

經 /回顧 追根因：skill 系統只優化品質（通過程序強制徹底性），犧牲了溝通效率和工作效率。根因是 LLM 注意力有限，詳細程序不保證更高遵守率。

## 結論

### 一、Skill 分類與處置

| 類別 | 指令 | 處置 |
|------|------|------|
| 產出型 | /小結, /暫存, /待辦, /修改計畫, /kb, /地圖 | 保留 |
| 工具型 | /更新, /安裝 | 保留 |
| 溝通型 | /有用嗎（原 /評估）, /你說呢 | 新建 |
| 學習型 | /醜一（原 /記住）, /很棒, /你去吃屎 | 新建或改寫 |
| 思維型 | /對焦, /驗證, /拆解, /回顧 | **砍掉**，降為行為原則 + 內部清單 |

最終 13 個 skill。

### 二、行為原則（取代觸發規則表）

寫入 CLAUDE.md，取代「事件觸發規則」段落和「工具觸發指令」段落：

```markdown
## 工作行為原則

- 目標先行：動手前確認目標明確且可量化。含效益宣稱的要查證事實。說不清就問。
- 風險正比：改前檢查影響範圍。1 檔腦中過；≥3 檔 grep 並把影響點列給用戶看；規範檔全掃展示完整分析。
- 大事先拆：超過 3 步的任務先列步驟，風險最高或解鎖最多的先做。
- 失敗追因：同一件事失敗兩次，停下來找根因再繼續，不重試。
- 做完閉環：做對了→有效→做全了→沒矛盾。規模小腦中過，規模大展示給用戶。
```

### 三、內部清單（取代 skill 完整流程）

放在 `.claude/rules/methodology-checklists.md`（不加 paths:，永遠載入），提取各 skill 的核心檢查項：

```markdown
# 方法論內部清單

AI 內部參考，不對用戶展示步驟編號或表格。用對話方式呈現分析。

## 目標釐清（原 /對焦）
- 指令有模糊詞？展開可能含義，猜錯返工成本高的先問
- 效益宣稱有事實支撐嗎？能量化就量化
- 隱含期待：用戶沒說但可能預期的是什麼？

## 影響分析（原 /驗證）
- 四面向：程式碼引用、資料格式相容、文件描述、其他機器快照
- 間接依賴：影響點的消費者呢？
- 判斷：一定會壞 / 可能會壞 / 需要更新

## 任務分解（原 /拆解）
- 完成狀態用一句可驗證的話描述
- 列可獨立驗收的交付物
- 硬依賴 / 軟依賴 / 無依賴
- 高風險先做，解鎖最多的先做

## 根因追問（原 /回顧）
- 精確描述症狀（在什麼條件下，什麼行為與預期不同）
- 逐層追問，每層要有查證（不能只靠推理）
- 根因能改嗎？能改就是該修的地方，不能改就轉替代路徑
- 從根因往回走驗證因果鏈
```

### 四、計分板系統

#### 設計

| 元素 | 說明 |
|------|------|
| 存放位置 | `.claude/rules/scoring.md`（不加 paths:，永遠載入，在 git 中共享） |
| 格式 | `{行為描述}\|{累計分數}\|{機器碼:MMDD, ...}` |
| 寫入方式 | 合併式——同行為模式更新同一行，不追加新行 |
| 壓縮 | 超過 30 種不同模式時壓縮最舊的（只保留模式和總分） |
| 分數不清零 | 累積是特性。行為是否改善看日期（最後一筆離現在多久）。不同行為模式不相消。 |

#### 計分板觸發

| 指令 | 觸發 | 分數 |
|------|------|------|
| /醜一 | 用戶或 /你去吃屎 指出違規 | **-1** 每項 |
| /很棒 | 用戶認為 AI 表現出驚豔觀察/洞見/表現 | **+2** |
| /你說呢 | 用戶被迫要求 AI 表態（AI 本該主動） | **-1** |

#### 計分板範例

```markdown
# 計分板

總分: +3

## 扣分
不主動表態|-3|JDNE:0221, ITEJ:0221, JDNE:0219
效益宣稱未自驗|-1|JDNE:0221
主動表態但胡扯|-2|JDNE:0224, JDNE:0226

## 加分
主動承認提案無效|+2|JDNE:0221
提出用戶未想到的關鍵風險|+4|JDNE:0222, ITEJ:0225
```
