# 00E｜案後檢討筆記

> **用途**：每案結案（或確認未得標）後填寫，系統性記錄經驗並回饋至 00A-00D 知識庫
> **填寫時機**：得標案——驗收結案後 7 日內；未得標案——收到結果通知後 3 日內
> **填寫負責人**：該案 PM（或計畫主持人）
> **預計填寫時間**：20-30 分鐘（不需要寫文章，條列重點即可）

### 二、投標階段檢討

> 回想從看到案件到送出建議書的過程。

**▸ L1 戰略分析階段**
- 分析判斷是否準確？（例如：我們預判的競爭對手、案件風險、機關意圖，事後看來對不對？）
- 有沒有什麼資訊是投標前不知道、事後才發現很關鍵的？

**▸ L2-L3 規劃與企劃階段**
- 核心創意方向有打中機關嗎？評審的反應如何？
- 預算編列有沒有被質疑的地方？哪些項目編太高或太低？

**▸ L4 撰寫階段**
- 建議書哪些章節寫得好？（評審有特別提到的）
- 哪些章節被扣分或被質疑？原因是什麼？

**▹ 整體時程回顧**
- 從決定投標到送件，總共花了多少工作天？
- 哪個階段最趕？瓶頸在哪裡？

---

### 三、履約階段檢討（僅得標案填寫）

**▸ 時程執行 vs 規劃**
- 建議書中的時程規劃，實際執行時哪些階段比預期久？哪些比預期快？
- 具體數字：（例如「主視覺設計送審，建議書寫 3 週，實際花了 5 週」）

**▸ 預算執行 vs 規劃**
- 哪些費用項目嚴重超支或大幅結餘？原因？
- 有沒有當初沒編到、但實際需要花錢的項目？

**▸ 風險與應變**
- 有沒有發生預料之外的狀況？怎麼處理的？
- 建議書中的應變計畫有沒有實際派上用場？好不好用？

**▹ 機關合作經驗**
- 這個機關的審核速度如何？（快 / 正常 / 慢，大概幾天回覆）
- 承辦人的溝通風格？（例如：重視細節型 / 只看大方向型 / 會主動給意見型）
- 有沒有這個機關特有的眉角？（例如：「一定要紙本函送，不接受 email」）

### 四、知識庫更新建議

> 這是最重要的部分——把經驗轉化成系統知識。不確定的可以先寫下來，之後用精靈整理。

**▸ 00B 公司實績（得標案必填）**

新增一筆實績紀錄：
- 案名：
- 機關：
- 金額：
- 履約期間：
- 工作內容摘要（2-3 句）：
- 佐證文件位置（Google Drive 路徑）：

**▹ 00A 團隊成員（若有人員異動或新增經歷）**

- 例如：「王小明在這案中擔任計畫主持人，可新增此經歷」
- 例如：「新進同仁李小華參與了這案，需要建檔」

**▹ 00C 時程範本（若發現工期估計需修正）**

- 用到哪個範本？（T-EXH / T-EVT / T-LEC / T-MKT / T-OPS）
- 哪個階段的工期需要調整？調成多少？
- 有沒有需要新增的階段？
- 「⚠ 常見低估」有沒有新的要補充？

**▹ 00D 應變 SOP（若遇到新狀況或發現 SOP 需修正）**

- 有沒有遇到現有範本沒涵蓋的風險情境？描述一下發生了什麼、怎麼處理的。
- 現有 SOP 的步驟有沒有需要修正的？（例如順序不對、缺少某個步驟）

## ✏️ 使用說明

### 填完之後做什麼？

1. 存檔到 Google Drive 固定資料夾（建議路徑：`/標案智囊/案後檢討/[年度]/[案名]_檢討筆記.md`）
2. 開一個新的 Claude 對話，上傳這份筆記，用精靈指令處理知識庫更新（見精靈使用說明）
3. 精靈會自動讀取「四、知識庫更新建議」段落，引導你逐項確認並更新 00A-00D

### 這份筆記不是什麼？

- 不是結案報告（結案報告是給機關看的正式文件，這是給自己團隊的內部筆記）
- 不是績效考核（不是用來追究誰做不好，是用來改善系統流程）
- 不是非寫不可的公文（用條列、關鍵字就好，不需要寫完整句子）
