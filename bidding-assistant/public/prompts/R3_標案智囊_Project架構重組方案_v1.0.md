## 二、架構方案：單一 Project，分層載入

### 核心原則

> **Project = 你是誰 + 你的規矩**
> **對話 = 現在做什麼案子的什麼階段**

不拆成多個 Project，避免你在不同 Project 之間切換時失去脈絡。但 Project 裡只放「每一次對話都會用到的底層知識」，階段性內容改在對話中觸發。

### Project Knowledge 只放三份文件

| 文件 | 內容 | 估計大小 |
|---|---|---|
| `A_系統指令與公司背景.md` | 角色定義 + 全域約束規則 + 公司資料 + 實績摘要 | ~12KB |
| `B_撰寫鐵律與品質標準.md` | 超級驚豔建議書的「六大鐵律」精華版（刪除範例） | ~8KB |
| `C_階段指令索引.md` | 四個階段的觸發指令 + 產出規格 + 銜接規則 | ~6KB |

**合計約 26KB** — 相比現在的 123KB + 35MB PDF，負擔大幅降低。

### 從 Project 移除的東西

| 移除項目 | 去哪裡 | 原因 |
|---|---|---|
| `01~04` 各階段完整提示詞 | 改為對話中貼入 | 每次只用一個階段，不需要四個同時在場 |
| `標案智囊_提示詞工程整合參考手冊.md` | 留在你的 Notion/電腦 | 這是設計文件，給人看的，不是給 Claude 的操作指令 |
| `超級驚豔建議書` 完整版 | 精簡為鐵律版放 Project，完整版留你電腦 | 31KB 太大，且大部分範例只在撰寫階段才需要 |
| 所有 PDF（歷史完稿、當前需求書） | 歷史完稿做成文字摘要；需求書在對話時上傳 | PDF 太大，嚴重佔用空間 |

### 每個階段的詳細操作

#### 【對話 1】戰略分析

**你要做的**：
1. 開新對話
2. 上傳需求書 PDF（對話中上傳，不放 Project）
3. 貼入完整的 Layer 1 提示詞（從你電腦中的 `01_戰略分析報告_v2.md` 複製）
4. 輸入：`/分析`

**Claude 會做的**：
- 讀取 Project 中的公司背景 + 全域規則
- 讀取你上傳的 PDF
- 按 Layer 1 提示詞產出戰略分析報告

**對話結束後，你要做的**（斷點）：
- 開 30 分鐘決策會議：投標 or 放棄
- 若決定投標 → 致電機關做需求對焦
- 整理一份「分析摘要」（下一階段用）

**分析摘要模板**（你手動整理，約 300-500 字）：

```
## 案件摘要
- 案名：○○○
- 機關：○○○
- 預算：○○萬
- 決標方式：最有利標
- 截止日：○月○日

## 戰略結論
- 機關核心需求：（2-3 句）
- 我方主要優勢：（2-3 點）
- 主要風險：（2-3 點）
- 建議主戰略：（1 句）

## 需求對焦結果
- 致電確認事項 1：機關回覆○○○
- 致電確認事項 2：機關回覆○○○
```

---

#### 【對話 2】備標規劃

**你要做的**：
1. 開新對話
2. 貼入上一步整理的「分析摘要」
3. 貼入完整的 Layer 2 提示詞（從 `02_戰略備標規劃書.md` 複製）
4. 輸入：`/規劃`

**Claude 會做的**：
- 讀取 Project 中的公司背景 + 撰寫鐵律 + 階段索引
- 讀取你貼入的分析摘要
- 按 Layer 2 提示詞產出備標規劃書

**對話結束後，你要做的**（斷點）：
- 將規劃書分發給企劃、設計、報價人員
- 開團隊討論會，確認創意方向和經費框架
- 整理「規劃摘要 + 團隊決策」

---

#### 【對話 3】企劃稿與報價

**你要做的**：
1. 開新對話
2. 貼入「規劃摘要 + 團隊決策結果」
3. 如需要，再次上傳需求書 PDF 中的關鍵頁面（截圖或摘錄，不用整份）
4. 貼入 Layer 3 提示詞
5. 輸入：`/企劃`

**重要**：Layer 3 產出較長（企劃稿 + 報價），建議拆成多次對話：
- 對話 3a：企劃稿上半（計畫目的 + 執行規劃）
- 對話 3b：企劃稿下半（管理機制 + 效益評估）
- 對話 3c：經費概算

每次帶入相同的規劃摘要，指定不同的章節範圍即可。

---

#### 【對話 4】分章撰寫

**你要做的**：
1. 開新對話
2. 貼入「企劃稿定稿」（已經過團隊確認的版本）
3. 貼入 Layer 4 提示詞
4. 指定撰寫的章節範圍
5. 輸入：`/撰寫 第○章`

**重要**：一次只寫一章。服務建議書每章動輒 3000-5000 字，一次全寫一定會中斷。

## 五、為什麼這樣不會失去整體觀

| 你可能擔心的 | 實際情況 |
|---|---|
| Claude 不知道前面做了什麼 | 你貼入的「摘要」就是前面的結論，Claude 拿到的資訊比全部塞 Project 更精準 |
| 每次都要重新說明案件 | 摘要模板只需 300 字，複製貼上 10 秒鐘 |
| 撰寫時不知道分析結論 | 摘要裡已包含戰略結論，而且 Project 裡的公司背景和撰寫鐵律一直都在 |
| 不同對話風格不一致 | 全域約束規則在 Project 裡，每次對話都會載入，風格一致性有保障 |

**關鍵認知**：把所有東西塞進 Project 不等於 Claude「理解」了全局。當 context window 被塞滿時，Claude 的注意力反而會被稀釋，抓不到重點。精準的 300 字摘要，比 35MB 的 PDF 有效得多。

## 六、你現在要做的事

### Step 1：整理公司實績摘要表
把現有的三份 PDF 完稿，手動整理成一張表格（案名、機關、金額、年度、類型、關鍵成果），放入文件 A。

### Step 2：精簡「超級驚豔建議書」
從 31KB 的完整版中，只提取六大鐵律的規則文字（刪除範例對比表），壓縮到 ~8KB，成為文件 B。

### Step 3：建立階段指令索引
按本文件第三節「文件 C」的結構，建立一份簡潔的索引，成為文件 C。

### Step 4：清理 Project
移除所有 PDF 和多餘的 .md 檔案，只保留 A、B、C 三份文件。

### Step 5：把 01~04 階段提示詞存在你的電腦
這些不放 Project，改為每次對話時手動貼入對應階段的提示詞。建議存成四個 .txt 檔案，方便複製。

### Step 6：測試一輪
用一個實際案子跑一次 Layer 1 → 2 → 3 → 4，驗證流程順暢度和產出品質。

## 七、進階優化（未來可選）

### 7.1 建立 Notion 模板
在 Notion 建立「標案工作卡」模板，每個案子一張卡，欄位包含：
- 案件基本資訊
- 各階段摘要（分析摘要、規劃摘要、企劃摘要）
- 當前進度
- 團隊決策紀錄

這張卡就是你在不同對話之間「搬運脈絡」的中繼站。

### 7.2 階段提示詞版本管理
每次優化提示詞後，在檔名加版本號（如 `01_戰略分析_v3.md`），並記錄改了什麼。避免不同版本混用。

### 7.3 常用案型快捷包
如果某類案子（如展覽策展、活動執行）頻繁出現，可以針對該案型準備一份「快捷摘要模板」，預填常用的公司優勢和實績對應。
