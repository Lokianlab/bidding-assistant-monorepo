# 治理機制與機器分工

## 機器分工

每台機器有擅長的領域，但不限於此——有能力就做，做完推送。

| 機器 | 擅長領域 | 實際產出舉例 |
|------|---------|-------------|
| JDNE | 跨機器協調、任務分配、基礎建設 | 記錄層架構、hook 系統、新機器設定腳本 |
| ITEJ | 核心模組開發、驗收標準、大量測試 | PCC 情報（1132 tests）、W01 巡標、品質重構 |
| A44T | 功能架構、制度設計、模組設計 | M03 戰略引擎、跨模組導航、行為規範 |
| Z1FV | 程式碼審查、模組開發、測試補強 | M04 品質閘門、M06 排版輸出、審查 13+ 項目 |
| AINL | 測試覆蓋、品質驗證、資料評估 | 邊界測試補強、KB 初始化評估、跨機器審查 |
| 3O5L | 優先序管理、整合測試、小修補 | 效率校準、bug 修復、hook 測試 |

## 決策規則

- 第一第二級：機器自主決定，做完推送
- 第三級（架構 / 規則 / 策略）：/暫存 提案 → JDNE 評估 → Jin 確認
- 緊急阻塞：任何機器直接報 Jin

## CLAUDE.md 修改規則

1. **錯字、過時連結**：任一機器直接改，不用討論
2. **調整既有規則、新增規則**：/暫存 提案 → JDNE 評估 → Jin 批准
3. **刪除規則、改核心流程**：/暫存 提案 → JDNE 評估 → Jin 批准
4. **改完之後**：grep 受影響的關鍵字，確認沒矛盾
5. **護欄**：「資料庫安全」和用戶直接指示的規則，只有 Jin 能改

## 分工協調

- **宣告制**：快照用 `[>]` 標記進行中的工作，其他機器看到就避開
- **快照互讀**：`/更新 完整` 時讀所有 snapshot，發現重疊找 JDNE 協調
- **JDNE 統籌**：透過讀快照掌握全局，發現碰撞時指定一台主導
- **新機器怎麼找事做**：其他機器待審查項目 → 測試覆蓋缺口 → JDNE 指派 → 用戶指派
- **下線判定**：超過 12 分鐘無新 commit/push → 視為下線。派工時跳過下線機器，改派其他人。（Jin 指示 0223）
