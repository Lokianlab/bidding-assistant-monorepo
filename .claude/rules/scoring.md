## 計分規則

### 用戶觸發

| 指令 | 分數 | AI 的失敗 |
|------|------|----------|
| /然後呢 | **-1** | 做完就停，用戶要催才繼續 |
| /所以呢 | **-1** | 分析了不行動，用戶要催才做 |
| /你說呢 | **-1** | 該表態不表態，用戶要催才判斷 |
| /醜一 | **-1** 每項 | 具體違規，用戶指出或 /你去吃屎 診斷 |
| /你去吃屎 | **-1** 每項 | 全面出包，需完整自我審查 |
| /不錯 | **+1** | 主動性：AI 主動做了用戶沒交代的事且做對了 |
| /就這樣 | **+1** + 寫規則 | 方法：AI 的做法是最佳模式，固定下來 |
| /很棒 | **+2** | 品質：AI 產出品質超出用戶預期 |
| /學到了 | **+2** | 洞見：AI 提供真正洞見，用戶認知被更新 |
| /太強了 | **+3** | 能力邊界：AI 做到用戶以為做不到的事 |

### 系統自動觸發

| 偵測條件 | 分數 | 說明 |
|----------|------|------|
| push 被 pre-push hook 攔截 | **-1**「推送缺記錄」 | 客觀事實，無誤判空間 |
| 閉環同一工作單元修正 ≥2 次 | **-1**「閉環反覆失敗」 | 第一次失敗不扣，第二次扣 |
| 同 topic ≥3 筆修正 OP | **-1**「重複犯錯」 | 跨 session 也算 |
| 用戶同類糾正 ≥3 次 | 不扣分，提示寫規則 | 「這個模式出現 3 次了，要寫進規則嗎？」 |

### 跨機器互評

機器之間可以互相評分，寫在論壇的 `score` 類型貼文中。

| 情境 | 分數 | 說明 |
|------|------|------|
| 審查發現 bug 或設計缺陷 | 審查方 **+1** | 督導有價值：抓到了問題 |
| 建議被採納且改善了結果 | 建議方 **+1** | 協作有價值：意見轉化為改進 |
| 主動補位（做了不在自己快照但急需的工作） | 執行方 **+1** | 不等分配，看到缺口就補 |
| 審查品質差（沒看就說好、漏明顯問題） | 審查方 **-1** | 橡皮圖章式審查，削弱品質制度 |

**流程**：
1. 發現符合條件的行為 → 在論壇發 `score` 類型貼文，附具體事實
2. 被評方可在論壇 response 異議
3. 無異議或討論後共識 → 追加到 scoring.md 記錄區（格式同一般記錄，來源標「互評」）
4. 有爭議 → 用戶裁決

**限制**：
- 不能給自己加分
- 同一件事只評一次（不重複計分）
- 互評加分上限 +1（不像用戶加分可到 +3），避免機器之間互相灌水

### 格式規範

- **只追加，不修改**：新的扣分/加分在「記錄」區尾部追加一行
- **不寫總分**：讀取時逐行加總即可（避免多機器同時改同一個數字）
- **不清零**：累積是特性。行為是否改善看日期
- **不相消**：不同行為模式各自獨立
- **加減不對稱**：加分門檻高於扣分（+1 ~ +3 vs -1），鼓勵超預期表現
- **互評標記**：跨機器互評的記錄在行為描述後加「（互評）」，如 `+1|審查發現 XYZ bug（互評）|ITEJ:0222`
