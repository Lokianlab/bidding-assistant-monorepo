改動前驗證：動手前掃影響範圍（靜態）+ 用具體案例走過計畫（動態）。

## 參數

可選：要改的檔案名（如 `/驗證 settings-context.tsx`）。沒帶參數就對當前計畫做驗證。

## 入口判斷

- 改既有的東西 → 靜態掃描
- 多步驟計畫 → 動態模擬
- 兩者都是 → 先靜態再動態

## 檔位判斷

- **快速**：1-2 檔同模組
- **標準**：3-5 檔或跨模組
- **完整**：≥6 檔或規範文件

## 快速檢查

三個核心問題：
1. 這個改動直接影響幾個地方？（grep 確認）
2. 有沒有可能改了 A 壞了 B？
3. 用一筆具體資料走一遍，哪一步最可能出問題？

## 靜態掃描（改既有東西時）

每一步的分析和判斷都要展示給用戶，不能只在腦中過。

**1. 精確描述改動**
改什麼、從什麼改成什麼、為什麼改。不是「改一下設定」，要具體到可檢驗。

**2. 第一圈：四面向掃描**

| 面向 | 檢查什麼 |
|------|---------|
| 程式碼 | grep 所有 import / 引用這個檔案的地方 |
| 資料 | 改了欄位名？型別？格式？下游消費者能接住嗎？ |
| 文件 | CLAUDE.md、README、API 文件有沒有描述這個東西？ |
| 其他機器 | 快照中其他機器有沒有在改同一區域？ |

**3. 第二圈：間接依賴**
第一圈找到的影響點，各自再掃一圈：它們的消費者呢？

**4. 判斷每個影響點**
三種結論：一定會壞（必須處理）、可能會壞（評估風險）、需要更新（不會壞但要同步）。

**5. 行動計畫**
先處理一定會壞的 → 再處理需要更新的 → 可能會壞的排進測試重點。

## 動態模擬（多步驟計畫時）

**1. 把計畫攤成線性步驟**
每步寫：做什麼 → 做完後系統狀態是什麼。

**2. 拿一筆具體資料走進去**
不是抽象地「想像資料流」，是選一筆真實或合理的具體資料，逐步跑。

**3. 每步問四個問題**
- 輸入存在嗎？（上一步的輸出確實產生了嗎？）
- 操作會成功嗎？（權限、格式、型別都對嗎？）
- 輸出符合下一步預期嗎？
- 有副作用嗎？（改了不該改的？觸發了不該觸發的？）

**4. 刻意走歪路**
空值、特殊字元、不存在的 ID、重複提交。找最可能出錯的邊界。

**5. 記錄發現，做決策**
每個發現分類：阻塞（不處理不能繼續）、缺陷（能繼續但有風險）、待確認（需要更多資訊）。
每個做決策：照做 / 修後做 / 推翻計畫。

## 產出格式

影響點表：
```
| 影響點 | 類型 | 嚴重度 | 處理方式 |
```

## 完成標準

- 能列出所有已知影響點，每個有判斷和處理方式
- 正常路徑跑通，邊界已識別
- 阻塞問題全部有解法或已標記為待確認

## 下一步建議

驗證通過 → 開始執行。做完 → 閉環驗證（自動帶入，不需手動觸發）。
