彙整全專案的待辦事項，從固定來源掃描，確保結果穩定完整。

## 流程

1. **同步**：`git pull origin main`

2. **掃描來源**（全部必掃，順序固定）
   a. 所有 `docs/records/_snapshot-*.md` → 提取 `[ ]`（待做）和 `[?]`（未決）項目
   b. 所有 `docs/records/_snapshot-*.md` → 提取 `[v]`（待驗收）項目，按模組分組
   c. `docs/records/forum/_threads.md` → 提取狀態為「共識」且摘要含「待用戶」的 thread
   d. `docs/records/_index.md` → 提取狀態為「進行中」的 topic
   e. `bidding-assistant/docs/dev-plan/_staging/_index.md` → 列出所有暫存項目
   f. 暫存檔內容 → 逐檔讀取，提取明確命名的交付物或任務（如「建 X」「實作 Y」「T1 知識庫精靈」）
   g. `docs/summaries/_index.md` → 找近期回報 → 讀 C 章後續計畫

3. **去重合併**：同一概念出現在多個來源 → 合併為一項，標註所有來源

4. **交叉驗證**：逐項確認進度狀態和記錄一致
   - 對每一項，查 `_index.md` 狀態、各機器快照、相關 OP 記錄
   - 發現矛盾時（如快照寫待做，但其他來源顯示已結案）：
     1. 不預設任何來源為真——去看實際狀態（檔案存不存在、程式碼改了沒、git 歷史）
     2. 確認事實後，當場修改所有不一致的記錄（快照、主題索引等）
     3. 修正後的項目按真實狀態處理（已完成就剔除，進行中就保留）
   - 修正過的記錄在輸出表格後面列出，讓用戶知道改了什麼

5. **分類排序**
   - 待做：明確要建或實作的東西
   - 待決：需要討論或選擇的東西
   - 等待中：依賴外部條件（如等資料）
   - 排序：待做 → 待決 → 等待中

6. **輸出**：依序輸出三大區塊（任一區塊為空則省略）

## 輸出格式

### 區塊一：待核准決策（共識已達，等用戶拍板）

從 `_threads.md` 的「共識」thread 提取。每項一行，附「核准後會怎樣」。

```
## 待核准決策

| # | 決策 | 摘要 | 核准後動作 |
|---|------|------|-----------|
| 1 | 計分板架構重構 | 把計分板拆成規則檔+歷史檔+快照備註 | 改檔案結構 |
| 2 | ... | ... | ... |

回覆格式：「1/3 核准，2 暫緩，4 退回因為...」
```

### 區塊二：待驗收項目（做好了，等用戶確認）

從所有快照的 `[v]` 提取。按模組分組，每項附一行驗收方式。

```
## 待驗收

**基建類**
| # | 項目 | 機器 | 驗收方式 |
|---|------|------|---------|
| 1 | 方法論 v2 整合 | JDNE | 讀 methodology/_index.md 確認五個方法論完整 |
| 2 | ... | ... | ... |

**功能類**
| # | 項目 | 機器 | 驗收方式 |
|---|------|------|---------|
| 3 | 情報模組 | JDNE | npm run dev → /intelligence → 四個 Tab 各點一遍 |

回覆格式：「1/3/5 通過，2 退回，4 暫不驗」
```

### 區塊三：工作待辦

原有的待做/待決/等待中表格。

```
## 工作待辦

| # | 項目 | 狀態 | 來源 |
|---|------|------|------|
| 1 | ... | 待做 | ... |

共 N 項（X 待做、Y 待決、Z 等待中）
```

## 規則

- 只在交叉驗證發現矛盾時才修改記錄檔（快照、主題索引），不動其他檔案
- 暫存檔內容中，只提取明確命名的交付物或任務，不提取模糊的未來構想
- 用人話呈現，不出現 topic ID 或 OP 引用
