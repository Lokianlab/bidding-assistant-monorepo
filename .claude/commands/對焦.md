需求對焦：用戶意圖不清時，結構化提問收斂需求，再擺方案讓用戶選。

## 參數

可選：需求描述（如 `/對焦 建精靈的事呢？`）。沒帶參數就對當前討論的任務做對焦。

## 檔位判斷

- **快速**：歧義輕微，返工成本低
- **標準**：歧義嚴重，或不同解讀會導致不同行動路徑
- **完整**：產出是設計決策，且影響 CLAUDE.md 或跨 ≥3 模組

## 快速檢查

心中過三個問題，不需產文件：
1. 指令裡有什麼不確定的詞？
2. 最可能的兩種解讀是什麼？差異大嗎？
3. 值得問用戶嗎？不值得就用最合理的解讀直接做。

## 標準流程

每一步的分析和判斷都要展示給用戶，不能只在腦中過。

### 第一步：識別模糊點

逐詞掃描用戶指令。標記：代名詞（「那個」「這件事」）、範圍不明詞（「更好」「處理一下」）、多義動詞（「弄」「搞」）。

對每個模糊詞展開可能含義。判斷標準：猜錯了返工成本多高？成本高的先問。差異小到「不管哪種解讀行動都一樣」的，跳過。

### 第二步：帶判斷提問

格式：「你說的 X 是指 A 還是 B？我傾向 A，因為 [理由]。」

一次問一個。附上判斷和理由，讓用戶可以直接說「對」。有選項時說明每個選項導致的行動差異。

**隱含期待檢查**：字面歧義解完後，額外問自己——用戶沒說出口但可能預期的是什麼？從 CLAUDE.md、MEMORY.md、近期記錄、當前對話推斷。能推斷的直接執行並說明假設。

**效益驗證**：如果任務宣稱特定效益（「更快」「省 token」「更乾淨」「效率更高」），查證效益是否有事實支撐——能量化就量化（實測、估算，標明來源）。無法支撐或量化不出來 → 告訴用戶「這個效益目前沒有事實根據」，讓用戶決定要不要繼續。

### 第三步：確認理解

一句話重述：「所以你要的是 [具體描述]，對嗎？」

用戶說「對」或小幅修正 → 通過。用戶推翻重來 → 回第一步。

### 第四步：方案比較

列可行路徑，每條：做法概述、預期效果、成本、風險。比較維度必須是用戶在乎的（「改起來容不容易」「會不會壞掉」），不是技術上好看的。

標明傾向：「以 [條件] 為優先建議選 [方案]，因為 [理由]。」等用戶選了才動手。

## 完整流程（設計決策時加掛）

標準四步完成後，跑送審前自檢六項：

| 檢查項 | 通過標準 |
|--------|---------|
| 問題定義 | 能用一句話說出要解決什麼 |
| 方案比較 | 有多個方案時寫出為什麼不選其他的 |
| 自我否定 | 至少砍掉過一個想加的東西 |
| 數字有據 | 數字標明來源（實測/估計/假設） |
| 回頭驗算 | 最終方案對照原始問題 |
| 薄弱環節 | 標記了信心低的判斷 |

大事（CLAUDE.md 或跨 ≥3 模組）加掛多模型外審。

## 完成標準

- 能用一句話描述用戶要什麼，且用戶確認「對」
- 方案呈現完成：用戶看過比較後選了一條路
- 知道下一步該做什麼
