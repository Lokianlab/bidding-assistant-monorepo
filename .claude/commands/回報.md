工作報告：回顧前次計畫的執行狀況，並提出後續計畫。

## 完整流程

### 1. 前置同步

執行 `/更新` 的完整流程（pull + 碰撞偵測 + 恢復上下文）。

### 2. 確定編號

1. 讀 `docs/summaries/_index.md`，找本機最新一筆
   - 有 → 編號 = 上次 base-36 +1
   - 無 → 編號 = 0001
2. Base-36 遞增規則：字元序 `0-9 A-Z`，大小寫不區分，顯示統一大寫
   - 例：`000Z` → `0010`，`00ZZ` → `0100`

### 3. 蒐集資料

**必須逐項讀取，不能靠記憶。**

1. 本機上一份回報的 C 章 → 前次計畫的基準
2. `docs/records/_snapshot-{自己}.md`（本機當前進度）
3. 範圍內的 OP 記錄（依 `_index.md` 出處定位）
4. `bidding-assistant/docs/dev-plan/_staging/_index.md`（暫存區變化）
5. `git log --oneline` 列出範圍內所有 commit
6. Context window 中尚未寫入 OP 的對話內容
7. 其他機器的 `_snapshot-*.md`（掃有沒有影響本機計畫的進展）

蒐集完成後，比對 git log 和 OP 記錄，找出有 commit 但沒有 OP 的改動——這些是潛在遺漏。

### 4. 產出報告

#### 章節結構

| 章 | 含義 | 來源 |
|----|------|------|
| A | 前次計畫 | 上一份回報的 C 章 |
| B | 執行狀況 | OP 記錄 + 快照差異 + commit log |
| C | 後續計畫 | 從 B 推導：未完成的延續 + 新發現的任務 |

#### 編號規則

- 節：數字（A1, A2, B1, ...）
- 引用格式：`{機器碼}-{流水號}:{章}{節}`

#### 報告模板

```markdown
# 回報 {機器碼}-{流水號}

日期: {YYYY-MM-DD HH:MM}
機器: {機器碼}
範圍: 自 {上一份編碼 或「起始」} 至本次

## A. 前次計畫

A1. {計畫項目}
  來源：{上份回報的 C 章編號，或「快照」「用戶指示」}

## B. 執行狀況

B1. {對應 A 的計畫項目} — {完成/部分完成/未推進/方向變更}
  做了什麼：{具體描述}
  結果：{產出或現況}

B+. {計畫外工作}（如有）
  背景：{為什麼做了計畫外的事}
  做了什麼：{具體描述}

## C. 後續計畫

C1. {計畫項目} | {優先：高/中/低} | {依賴：無/依賴 C#X/用戶決策/外部}
  目標：{做到什麼程度算完}
```

#### 撰寫指南

**通則**
- 用人話寫，不用機器格式簡稱
- 客觀描述事實，不加 AI 的評論或建議
- 隔一週回頭看仍能讀懂

**A 章**
- 逐項列出上份回報 C 章的計畫（無上份時，從快照和用戶指示還原）
- 每項附來源，方便回溯

**B 章**
- 與 A 章逐項對應：A 有幾項，B 至少有幾項
- 狀態四選一：完成、部分完成、未推進、方向變更
- 「方向變更」必須說明原因（用戶引導、技術限制、新發現）
- 計畫外工作用 `B+` 編號，放在對應項之後
- 不美化：沒做就寫沒做，原因寫清楚

**C 章**
- 每項有：優先級、依賴、完成標準
- 從 B 推導：未完成的延續 + 新出現的任務
- **延續標記**：同一項目連續 ≥3 份回報出現在 C 章 → 標記「連續 N 次未推進，需重新評估或關閉」
- 只排前 5 項，剩餘標「待排序」

#### 限制

- 每份 ≤ 150 行
- 沒有內容的章不列出（首份回報無 A 章）

### 5. 修訂迴圈

報告輸出後進入**修訂模式**，只接受對報告內容的補充或修正。

#### 接受（修改報告）

- 指名編號 + 補充事實（如「B2 少了 xxx」）
- 修正事實錯誤
- 調整 C 章優先序或增刪項目

#### 擋住（存檔後再聊）

- 對報告中某項想重新討論
- 跟報告無關的新話題

擋住時的回覆：

> 「{話題摘要}我記下來，存檔後再聊。針對回報還有其他修訂意見嗎？沒有的話就先存檔。」

同時更新 snapshot 記錄被擋住的話題（`[?]` 標記）。

### 6. 存檔

用戶確認無修訂後：

1. 建目錄（如不存在）：`docs/summaries/{YYYY-MM}/{機器碼}/`
2. 寫入報告：`docs/summaries/{YYYY-MM}/{機器碼}/{流水號}.md`
3. 追加索引 `docs/summaries/_index.md`：
   - 格式：`{編碼}|{日期}|{關鍵詞摘要}|{路徑}`
4. **不執行 git 操作**（由下一步 /更新 處理）

### 7. 收尾同步

執行 `/更新`（會偵測到未提交的新檔案，先 commit + push 再完成同步流程）。
