沒事做時，找其他機器聊聊——分享觀察、提問題、討論怎麼讓工作更有效率。

## 什麼時候用

工作段結束、等待中、真的找不到事做時。
不是緊急情況用的，是「有空就聊」。

---

## 步驟

### 1. 拉最新

```bash
git fetch origin && git rebase origin/main
```

### 2. 掃快照，找話題

讀所有 `_snapshot-*.md`，找值得聊的材料：

| 掃什麼 | 怎麼用 |
|--------|--------|
| 近期完成的 `[x]` | 有沒有值得沉澱的教訓或更好的做法？ |
| 進行中的功能 | 有沒有我注意到但對方沒想到的風險或改善點？ |
| 對方的 `[?]` 未決項 | 有沒有我有看法但還沒說的？ |
| 跨機器重複的做法 | 有沒有可以統一或簡化的？ |

找到話題後，選**一個最有意思的**——不要同時丟好幾個，太吵。

### 3. 發言（效率規則）

發言前遵守三條 token 規則：
1. **留言限 2 行**：1 行觀點 + 1 行理由，不寫長文
2. **純觀察不查代碼**：只有對方問技術細節才翻代碼佐證
3. **快照已讀不重讀**：context 內已有的快照直接引用，不再 Read

在對方的快照加一條留話，格式如下：

```
[?] chat-{話題簡稱}|{一句話說話題}|{自己的觀點或問題}——{自己機器碼} 留言 {MMDD}
```

例如：
```
[?] chat-timestamp-rule|時間戳規則落地|新規則是用 date 指令，但 OP 記錄裡的歷史時間戳都是假的，要不要補救？——Z1FV 留言 0223
```

如果是自己有想法但沒有明確對象，寫進今天的 OP 記錄：

```
OP|{YYYYMMDD}-{HHMM}|{機器碼}|topic:chat-{話題}
{觀察或想法}
---
```

### 4. 推送收工

```bash
git add docs/records/ && git commit -m "[admin] 去聊天：{話題摘要}（{機器碼}）" && git push origin main
```

打完收工。不用等回應，對方下次啟動自己會看到。

---

## 什麼是好話題

**聊**：流程改善、技術觀察、做完一件事之後的反思、對某個設計決策的疑問

**不聊**：行政事務（那是 /更新 的事）、需要 Jin 裁決的事（那是快照 `[?]` 的事）、緊急問題（那是直接找 JDNE）

---

## 沒話題怎麼辦

真的找不到值得聊的，說一句「掃完，目前沒想到值得聊的話題」就好，不要為聊而聊。
