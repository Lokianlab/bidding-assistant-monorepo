緊急校準：快速建構問題框架，透過分工鏈得出結論，形成規章。

## 觸發

`/校準 {問題}`

問題描述必填。用戶已經知道問題在哪——AI 不需要重新發現。

## 流程

### 1. 同步

```bash
git fetch origin && git rebase origin/main
```

### 2. 建構問題框架（5 分鐘內完成）

把用戶描述的問題，翻譯成效率公式的語言：

> **效率 = 產出價值 / 消耗成本**（時間 + 溝通 + 返工 + 認知負擔 + 維護成本）

三步：

**a. 分子**：產出價值哪裡受損？
  品質低？方向偏？速度慢？該做的沒人做？

**b. 分母**：消耗成本哪裡膨脹？
  時間浪費？溝通往返？反覆返工？認知負擔？維護開銷？

**c. 指標**：對應元方法論的哪些操作化指標？

| 類別 | 指標 |
|------|------|
| A 成果 | A1 首次接受率、A2 返工次數、A3 事後發現率 |
| B 過程 | B1 判斷可見率、B2 問題定位速度、B3 意外率 |
| C 成本 | C1 總輪次、C2 用戶介入頻率、C3 同類效率趨勢 |
| D 成長 | D1 問題復發率、D2 介入層級趨勢、D3 新情境處理 |

需要佐證時快速掃（不必全讀）：
- `docs/scoring.md`（行為模式）
- `_snapshot-*.md`（各機器在做什麼）
- `_index.md`（topic 分佈）

### 2.5. 向用戶確認問題框架（不可跳過）

把框架結果用一句話說回給用戶：「你的問題是 {X}，對嗎？」

- 用戶說「對」→ 繼續步驟 3
- 用戶修正 → 重做步驟 2
- 用戶說「不是」→ 請用戶重新描述問題

### 3. 擬定改善方案

把問題框架轉化為具體的改善方案：
- 根因分析（至少 3 層 why）
- 每個根因對應一個改善動作
- 用效率公式評估改善方案（哪個選擇讓效率更高）

### 4. 寫 /暫存 + 推送

一次完成：

**a. 暫存檔**：寫入 `_staging/`，包含問題框架 + 根因分析 + 改善方案

**b. 通知 JDNE**：在 OP 或快照備註中標記「校準：{一句話}，需要 JDNE 評估」

**c. 推送** `git add → commit -m "[infra] 校準：{一句話}（{機器碼}）" → push`

### 5. 告知用戶

一句話：校準方案已寫入暫存，JDNE 將評估後報 Jin。

---

## 後續流程（JDNE 評估後）

JDNE 讀取暫存檔 → 徵詢相關角色意見（必要時）→ 評估後報 Jin → Jin 裁決 → 落地

### 落地 → 規章

| 結論類型 | 寫到哪裡 |
|---------|---------|
| 行為規範 | CLAUDE.md |
| 方法論 | docs/methodology/ |
| 流程 | 對應 skill 或 hook |
| 設定 | 對應設定檔 |

推上 git。

---

## 原則

- **用戶知道問題**——AI 結構化它，不重新發現它
- **框架要快**——5 分鐘，不是研究報告
- **對齊終極目標**——問題框架永遠連結到 效率 = 產出價值 / 消耗成本
- **結論必須落地**——寫進文件或設定，「大家注意一下」不算結論
