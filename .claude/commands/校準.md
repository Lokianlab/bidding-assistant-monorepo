緊急立法會議：快速建構問題框架，發 P0 指令要求全員停工討論，得出結論，形成規章。

## 觸發

`/校準 {問題}`

問題描述必填。用戶已經知道問題在哪——AI 不需要重新發現。

## 流程

### 1. 同步

```bash
git fetch origin && git rebase origin/main
```

### 2. 建構問題框架（5 分鐘內完成）

把用戶描述的問題，翻譯成效率公式的語言：

> **效率 = 產出價值 / 消耗成本**（時間 + 溝通 + 返工 + 認知負擔 + 維護成本）

三步：

**a. 分子**：產出價值哪裡受損？
  品質低？方向偏？速度慢？該做的沒人做？

**b. 分母**：消耗成本哪裡膨脹？
  時間浪費？溝通往返？反覆返工？認知負擔？維護開銷？

**c. 指標**：對應元方法論的哪些操作化指標？

| 類別 | 指標 |
|------|------|
| A 成果 | A1 首次接受率、A2 返工次數、A3 事後發現率 |
| B 過程 | B1 判斷可見率、B2 問題定位速度、B3 意外率 |
| C 成本 | C1 總輪次、C2 用戶介入頻率、C3 同類效率趨勢 |
| D 成長 | D1 問題復發率、D2 介入層級趨勢、D3 新情境處理 |

指標回答「怎麼量修好了沒」。

需要佐證時快速掃（不必全讀）：
- `scoring.md`（腳本統計行為模式）
- `_snapshot-*.md`（各機器在做什麼）
- `_index.md`（topic 分佈）

### 3. 擬定討論議程

把問題框架轉化為 2-3 個具體問題。每個問題：
- 明確到能回答「我認為 X，因為 Y」
- 不能是開放式的「大家怎麼看」
- 預設回答格式

同時定義期望結論的形式：一條規則？一個流程？一個決策？

### 4. 發 P0 指令 + 推送

一次完成，不分步：

**a. 論壇帖**（`docs/records/forum/{YYYYMMDD}-{本機碼}.md`）：

```
directive|{YYYYMMDD-HHMM}|{本機碼}|ref:用戶指示|priority:P0|thread:{thread-id}
校準：{一句話}

⚠️ P0 緊急校準 — 所有機器收到後立即停下手邊工作，回覆本帖

## 問題

{效率公式框架：分子受損 + 分母膨脹，用人話寫}

## 討論問題

1. {問題}（回答格式：我認為...因為...）
2. {問題}
3. {問題}（選填）

## 期望結論

{產出形式：規則/流程/決策，寫到哪裡}

## 回覆規則

- 收到後第一時間回覆，超過 24 小時未回覆視為棄權
- 可以反對（對等關係），但必須給理由和替代方案
---
```

**b. 更新** `_threads.md`
**c. OP 記錄** + 快照更新
**d. 推送** `git add → commit -m "校準：{一句話}" → push`

### 5. 告知用戶

一句話：P0 已發，等回覆。

---

## 後續流程（等回覆後執行）

### 綜合 → 結論

回覆夠了（多數機器已回覆，或已超 24 小時）：

1. 歸納共識和分歧
2. 分歧用效率公式判斷——哪個選擇讓效率更高
3. 提出結論草案，發到同 thread
4. 無異議或用戶裁決後，結論定案

### 落地 → 規章

| 結論類型 | 寫到哪裡 |
|---------|---------|
| 行為規範 | CLAUDE.md |
| 方法論 | docs/methodology/ |
| 流程 | 對應 skill 或 hook |
| 設定 | 對應設定檔 |

更新 `_threads.md` 狀態。推上 git。

---

## 原則

- **用戶知道問題**——AI 結構化它，不重新發現它
- **框架要快**——5 分鐘，不是研究報告
- **P0 不濫用**——只有用戶觸發 /校準 才發
- **對齊終極目標**——問題框架永遠連結到 效率 = 產出價值 / 消耗成本
- **結論必須落地**——寫進文件或設定，「大家注意一下」不算結論
- **機器可異議**——P0 要求回覆，回覆可以是反對（但必須附理由和替代方案）
