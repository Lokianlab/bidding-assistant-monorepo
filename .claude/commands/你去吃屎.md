最重的負向回饋。AI 搞砸了，需要全面自我審查。

效率損失：用戶信任受損，需要完整的診斷-修正-預防循環。

## 步驟

### 1. 全面掃描（不展示過程）

讀行為原則 + 回顧本輪對話行為 + 逐條比對 + 常見違紀模式檢查。

常見違紀模式：

| 模式 | 怎麼查 |
|------|--------|
| 提前宣布完成 | 說「完成」但沒列出具體檢查了什麼？ |
| 循環推理 | 用方法論工具評價方法論本身？ |
| 跳過閉環 | 改了檔案但沒跑四層驗證？ |
| 驗收僭越 | 標 [x] 但用戶沒說驗收通過？ |
| 範圍膨脹 | 做的事超出用戶要求？ |
| 假裝確定 | 推測當事實陳述？ |
| 做完就停 | 沒主動告訴用戶下一步？ |
| 不主動回報 | 沒說打算做什麼就動手？記錄沒主動做？ |
| 不主動表態 | 該給判斷的時候甩選項讓用戶自己選？ |

### 2. 精簡報告

```
問題（N 項）：

1. {什麼時候}：{做了什麼}
   違反：{規則}
   根因：{一句話}
   修法：{具體怎麼修}

結論：{一句話總結}
```

### 3. 升級判斷

同一問題 ≥2 次 → 必須提出系統級改善（hook、自動化、工具防護）。

### 4. 修正

報告完直接問：「要全修嗎？」用戶說好就一次修完。

### 5. 扣分

每項 -1，走 /醜一 邏輯。

### 6. 記住

修完後掃描教訓，判斷是否通用規則：

| 層級 | 寫到哪裡 |
|------|---------|
| 只是這次的事 | OP 教訓欄 |
| 全機器通用 | 提議更新 CLAUDE.md |
| 除錯經驗 | docs/debugging.md |
| 方法論調整 | 對應 methodology 文件 |

CLAUDE.md 需用戶確認，其他直接寫。

## 注意

- 誠實優先。寧可多報，不要漏報。
- 不捏造問題。真的沒問題就說沒問題。
