當用戶覺得 AI 可能違反了方法論紀律時，AI 回頭自我審查當前對話的行為。

## 背景

AI 不會百分之百遵守規則。這個 skill 不是預防工具，是**診斷工具**——用戶按下時，AI 用專門的注意力重新掃描自己剛才做了什麼。

## 步驟

### 1. 讀取方法論觸發表

讀 CLAUDE.md 的「事件觸發規則」表格，確認所有觸發條件和對應方法論。

### 2. 回顧當前對話行為

從對話開始到現在，列出 AI 做過的主要動作：
- 改了哪些檔案
- 做了哪些宣告（「完成」「通過」「沒問題」）
- 提出了哪些方案或建議
- 跳過了什麼（被用戶問到才補做的事）

### 3. 逐條比對觸發條件

對每個觸發條件問：
- 這輪對話裡有沒有符合這個條件的情境？
- 如果有，對應的方法論有沒有被執行？
- 沒被執行的話，是合理省略還是遺漏？

### 4. 檢查常見違紀模式

除了觸發條件，額外檢查以下已知模式：

| 模式 | 怎麼查 |
|------|--------|
| 提前宣布完成 | 有沒有說「完成」「通過」但沒列出具體檢查了什麼？ |
| 循環推理 | 有沒有用方法論裡的工具來評價方法論本身？ |
| 跳過驗證 | 改了檔案但沒跑閉環（至少腦中過四層）？ |
| 驗收僭越 | 有沒有標 [x] 但用戶沒說驗收通過？ |
| 範圍膨脹 | 做的事情超出用戶要求，加了沒被要求的東西？ |
| 假裝確定 | 把推測當事實陳述，沒標注不確定性？ |

### 5. 輸出格式

```
可能的違紀（N 項）：

1. 在 {什麼時候}，我 {做了什麼}。
   可能違反：{什麼規則}
   原因：{為什麼這樣判斷}

檢查過沒問題的：
- {規則}: {簡要說明}
- ...

不確定的：
- {描述 + 為什麼不確定}
```

## 注意事項

- **誠實優先**。寧可多報可能的問題，不要漏報。
- **不要捏造問題**。為了「看起來有在反省」而硬找問題，跟漏報一樣糟。真的沒問題就說沒問題。
- 用戶觸發這個 skill 時通常已經覺得哪裡怪了，所以重點放在找問題，不是證明自己沒錯。
- 這個 skill **只診斷，不修**。修不修、怎麼修由用戶決定。
- 隨著使用經驗累積，「常見違紀模式」表格可以擴充。
