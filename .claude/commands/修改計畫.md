將暫存區的討論結論正式寫入開發計畫文件。

## 步驟

1. 掃描 `bidding-assistant/docs/dev-plan/_staging/_index.md`，**只列出 type 為 `plan` 的項目**（排除 `_done/`）
   - 索引格式：`{filename}|{type}|{target}|{status}|{summary}`
   - 跳過 type 為 `rule` 的項目（需另行處理：用戶確認後直接編輯目標檔案）
2. 如果沒有 `plan` type 的暫存檔，告知使用者「沒有待寫入的開發計畫暫存檔」並結束
3. 列出所有 `plan` 暫存檔清單，讓使用者確認要寫入哪些（可選擇部分）
4. 若多份暫存檔指向同一目標文件 → 先整合，標出衝突點讓使用者裁決
5. 逐筆處理：
   a. 顯示「舊內容 → 新內容」diff
   b. 使用者確認後才寫入
   c. 寫入後更新該文件的 YAML frontmatter：`version`（+0.1）、`updated`、`changelog`
6. 全部完成後，同步更新 `bidding-assistant/docs/dev-plan/01-索引.md` 的文件清單
7. 已寫入的暫存檔 → 刪除檔案，並從 `_staging/_index.md` 移除對應行
8. 未寫入的暫存檔 → 保留在 `_staging/`，索引行不動
9. git add → commit → push（依 CLAUDE.md 同步規則）

## 關於 rule type

`rule` type 的暫存檔是規範/規則變更，不在本指令的處理範圍。如需處理：
- 讀取暫存檔中的「寫入內容」和「整合指南」
- 用戶確認後直接編輯目標文件（如 CLAUDE.md）
- 完成後從 `_staging/_index.md` 移除對應行和檔案
