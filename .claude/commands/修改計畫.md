載入暫存區碎片，推進開發計畫的任務線。

## 核心理念

這不是逐檔搬運工具。這是任務線推進器：讀懂碎片 + 目標現況 + 依賴脈絡，合成出新版本。

## 步驟

### 1. 同步

```bash
git pull origin main
```

### 2. 任務線地圖

掃描 `bidding-assistant/docs/dev-plan/_staging/_index.md`。

- 索引格式：`{filename}|{type}|{target}|{status}|{summary}`

按目標文件（target）分組，**所有 type（plan / rule / issue）一起分組**，顯示任務線地圖：

```
目前有碎片的任務線：
 1. 技術選型（定案 v1.0）— 3 個碎片：Agent 架構、AI 連接方式、知識庫儲存方案（待決）
 2. 開發進程（待寫）— 3 個碎片：四層路線、PCC 影響分析、知識庫精靈現況（待決）
 3. 方法論 — 2 個碎片：模型委員會規範變更 [rule 待決]、完備性盤點 [issue 待決]
 ...
 你想推進哪一條？
```

**分組規則**：
- target 相同的項目（不分 type）→ 同一組
- target 為「新增 XXX」→ 獨立一組，標注「（新文件）」
- target 為「無」→ 歸入「獨立碎片（不指向特定文件）」
- 每組顯示：目標文件的中文標題（不顯示檔名）、目標現有狀態/版號、碎片數量、碎片摘要
- **非 plan 碎片**：在摘要後標注類型，如 `[rule 待決]`、`[issue 已關閉]`
- **待決碎片**：在摘要後標注「（待決）」

**如果沒有任何暫存檔**：告知「暫存區沒有碎片」並結束。

**快速入口**：如果用戶在指令後附帶關鍵字（如 `$ARGUMENTS` 包含「技術選型」），直接跳到步驟 3，不顯示完整地圖。

### 3. 載入上下文

用戶選定一條任務線後：

a. **讀目標文件**：根據 target 欄位定位
   - dev-plan 文件 → `bidding-assistant/docs/dev-plan/{target}`
   - 其他位置（如 `docs/methodology/`、`CLAUDE.md`）→ 直接讀取
   - 待寫/不存在 → 標記為空
   - 已有內容 → 讀取完整內容

b. **讀所有碎片**：讀取該組所有暫存檔的完整內容
   - 路徑：`bidding-assistant/docs/dev-plan/_staging/{filename}`

c. **讀依賴文件**（僅 dev-plan 文件）：讀目標文件的 YAML `depends` 欄位，遞迴讀取依賴的文件內容
   - 目的：理解上下文（如「開發進程」依賴「技術選型」的決策）

### 4. 評估與提案

分析碎片覆蓋了目標文件的哪些部分，提出推進方案。

**對已有內容的目標**（定案/草案）：

```
「技術選型」目前有 7 個 ADR（v1.0）。
3 個碎片帶來：
 - Agent 架構 → 可新增 ADR-08（雙層策略）
 - AI 連接方式 → 可新增 ADR-09（$0 額外成本）
 - 知識庫儲存方案（待決）→ 結論未定，暫不處理
建議：新增兩個 ADR，版本升到 v1.1。
```

**對待寫的目標**（空檔/無檔）：

```
「開發進程」目前是空的。
2 個碎片提供了：四層路線 + PCC MCP 併入 Layer 0。
建議：建立初版（v0.1 草案），包含四層路線架構。
不足：缺少具體時程和里程碑，標記為草案。
```

**評估時注意**：
- **過時碎片**：碎片內容與目標現況或依賴文件矛盾時（如碎片提 Discord Bot 但架構已改為 SaaS），標出矛盾，由用戶決定取捨或丟棄
- **待決碎片**：提醒「這個碎片的結論還沒定」，建議暫不處理或僅納入草案
- **碎片重疊**：多個碎片覆蓋同一主題時，指出重疊，建議保留哪份或合併

用戶可以：確認方案、調整方案、或放棄（回到地圖重選）。

### 5. 執行

根據碎片成熟度，兩種結果：

#### 結果 A：推進正式文件（碎片夠成熟）

a. 合成新版本內容（不是逐段搬運，是讀懂所有碎片後重新組織）
b. 顯示完整 diff（已有內容 → 顯示修改差異）或新文件預覽（待寫 → 顯示完整初版）
c. **用戶確認後才寫入**
d. 更新 YAML frontmatter：
   - `version`：已有 → +0.1（如 1.0→1.1）；新建 → 0.1
   - `updated`：當天日期
   - `status`：根據完整度設定（定案/草案）
   - `changelog`：新增一行描述本次變更
e. **已完整消化的暫存檔** → 刪除檔案 + 從 `_staging/_index.md` 移除對應行
f. **未消化或部分相關的暫存檔** → 保留不動
g. **待決碎片** → 一律保留

#### 結果 B：整合碎片（碎片不夠成熟，但可以合併減少碎片數）

a. 將多個碎片合成為一份更完整、連貫的暫存檔
b. 新暫存檔命名：`{目標編號}-{整合主題}.md`
c. 新暫存檔加入 `_staging/_index.md`
d. 刪除被合併的舊碎片 + 從索引移除
e. **不動正式文件**
f. 用途：3 份零散碎片 → 1 份連貫的暫存檔，下次再推進

### 6. 收尾

a. 更新 `bidding-assistant/docs/dev-plan/01-索引.md`（文件清單的版號、狀態、修改日）
b. git add → commit → push（依 CLAUDE.md 同步規則）
c. commit message 格式：`修改計畫：推進「{目標名稱}」到 v{版號}` 或 `修改計畫：整合「{目標名稱}」碎片`

## 輸出風格

- **用人話**，不出現 topic ID、檔名路徑（除非用戶問）
- 任務線用目標文件的中文標題稱呼（如「技術選型」不是「03-技術選型.md」）
- 碎片用摘要稱呼（如「Agent 架構」不是「03-Agent架構決策.md」）

## 各類型碎片的處理方式

所有類型都在任務線地圖中按目標分組顯示，都可以被選中處理。

| type | 推進方式 | 消化條件 |
|------|---------|---------|
| plan | 合成進正式文件（步驟 4-5） | 內容已寫進目標文件 |
| rule | 讀「寫入內容」+「整合指南」，用戶確認後編輯目標文件 | 變更已寫進目標文件並驗證 |
| issue | 視狀態：待決→討論推進；已關閉→驗證後清理 | 問題已解決 |

### 統一刪除原則

暫存檔是中間產物，消化後即刪除。刪除前提：相關結論已有 OP 記錄或回報可追溯。
`_staging/_done/` 不再使用。
