知識庫維護精靈（CLI 版）。引導使用者透過對話建立/更新 00A-00E 知識庫，完成後直接寫入本地檔案。

## Step 1：前置同步

```bash
git pull origin main
```

然後讀取 T1 prompt，作為本次對話的行為規範：

```
Read bidding-assistant/public/prompts/T1_知識庫維護精靈_v2.0.md
```

**必須遵循 T1 prompt 的所有規則**：全域導航機制（6 條規則）、追問邏輯、品質檢查清單、輸出模板格式。

## Step 2：分流

如果使用者在指令後面帶了參數，直接進入對應 Phase：
- `/kb 成員` → Phase A
- `/kb 實績` → Phase B
- `/kb 萃取` → Phase C
- `/kb 檢討` → Phase D
- `/kb 範本` → Phase E

沒有參數 → 顯示 T1 的 Phase 0 分流選單，但把「上傳文件」改為「告訴我檔案路徑」。

## Step 3：讀取目標 KB 檔 + 掃描 ID

確定 Phase 後，讀取對應 KB 檔案：

| Phase | 讀取檔案 |
|-------|---------|
| A | `bidding-assistant/public/prompts/00A_團隊成員資料庫_v4.0.md` |
| B | `bidding-assistant/public/prompts/00B_公司實績資料庫_v4.0.md` |
| C/D | 讀所有可能寫入的 00A/00B/00C/00D |
| E | 讀 00C 或 00D（依使用者選擇） |

**掃描 ID**：
- 搜尋 `### M-`、`### E-`、`### P-` 開頭的行
- **佔位符排除**：條目 heading 含 `[姓名]`，或 body 含 `依此格式` 或 `繼續新增` → 不算已使用 ID
- 下一個 ID = 最大實際 ID + 1，三位數補零

## Step 4：執行對話

完全遵循 T1 prompt 的對話流程。以下是 CLI 版的差異：

### CLI 差異

- T1 說「上傳文件」→ 改為「告訴我檔案路徑，我直接讀」（用 Read 工具）
- T1 的 A-1「更新既有成員：請把資料貼過來」→ 改為「告訴我編號或姓名，我幫你找」（精靈自己 Grep 查找）

### 新增 vs 更新

在 Phase A-1（或 Phase B 開始時），如果使用者選擇「更新」：
1. 問：「要更新誰？告訴我編號（如 M-002）或姓名」
2. 在 KB 檔中搜尋，找到目標條目
3. 展示現有內容
4. 問：「要修改什麼？」
5. 只走需要修改的部分（不需重新跑全部 8 步）
6. 產出完整更新後的條目 → Step 7 覆寫

## Step 5：寫入前品質檢查

執行 T1 prompt 定義的品質檢查清單：
- 00A：5 項（姓名職稱狀態角色、學歷、經歷、動詞、隱私）
- 00B：5 項（案名機關主體角色、工作細項 ≥2、三要素、成果、協力廠商標註）
- 00C：3 項（工期合理、修正依據、順序正確）
- 00D：4 項（時序標記、負責人、預防+應變、地理資訊）

未通過 → 列出警告，問使用者是否仍要寫入。

## Step 6：寫入前再次 pull

```bash
git pull origin main
```

- 有衝突 → 解決
- 重新掃描 ID → 如果 ID 被其他機器佔了 → 自動遞增，告知使用者

## Step 7：寫入檔案

### 插入位置

| 檔案 | 寫入類型 | 錨點（在此行之前插入） |
|------|---------|---------------------|
| 00A | 新增成員 | `## 外部協力人員` |
| 00A | 新增外部人員 | `## 空白模板（複製此區塊` |
| 00B | 新增實績 | `## 空白模板（複製此區塊` |
| 00C/00D | 新增/修正範本 | `## ✏️ 維護須知` |

### 佔位符處理（首次寫入時）

| 佔位符 | 處理 |
|--------|------|
| 00A 的 `### M-003｜[姓名]` + `（依此格式逐一建立……）` | 替換為實際條目 |
| 00A 的 `### E-001｜[姓名]` + 表格 | 首次新增外部人員時替換 |
| 00B 的 `（繼續新增更早年度及前身社團時期的實績……）` | 刪除此行及前後 `---` |

**判定佔位符**：heading 含 `[姓名]`，或 body 含 `依此格式` 或 `繼續新增`。

### 更新既有條目

用 Edit 工具替換：從 `### {ID}｜` 到下一個 `---`（含 `---`），替換為完整更新後的條目。

### 條目格式

嚴格遵循 T1 prompt「輸出格式定義」段落的模板。重點：

- **每筆條目前面一個 `---` 分隔線**
- **不寫 HTML 註解**（`<!-- -->`）
- **不動空白模板和 AI 使用指引**
- **盡量不寫 `[待補]`**，精靈應追問到有答案
- 真的問不出來 → 標 `[待補]`，在摘要中提醒使用者

### 更新 `最後更新` 日期

寫入後，把檔案頂部的 `**最後更新**：[YYYY-MM-DD]` 改為今天日期。

## Step 8：收尾

顯示寫入摘要：
- 新增/更新了幾筆
- 寫入哪些檔案
- ID 是什麼
- 有哪些欄位標了 [待補]

```bash
git add <修改的檔案>
git commit -m "知識庫更新：{類別} {ID}｜{簡述}"
git push origin main
```
