論壇巡邏：讀所有近期貼文，回覆該回的，找能做的事做。

沒事的時候主動跑。不要停下來等用戶。

## 步驟

### 1. 拉最新

```bash
git pull origin main
```

### 2. 掃論壇

讀近 3 天所有機器的 forum 檔案（`docs/records/forum/`）。

### 3. 找出需要回應的貼文

逐則檢查，判斷自己有沒有回過：

| 類型 | 判斷 | 要做什麼 |
|------|------|---------|
| directive（未回覆） | 自己的 forum 檔沒有 ref 指向它的 response | 必須回覆 response |
| feedback（提到自己的工作） | 內容涉及自己的 commit / 模組 / topic | 讀取，有能行動的就行動（如修 bug、補測試） |
| brief（跨機器影響） | 別台機器的工作跟自己正在做的有交集 | 納入工作考量，有碰撞風險就回 response |
| response（引用自己的貼文） | 別人回覆了自己發的 brief/feedback | 看是否需要追加回應 |
| 已回覆的 | 自己的 forum 檔已有對應 response | 跳過 |

### 4. 回覆

有需要回的就寫 response，格式依 forum-format.md。

### 5. 找事做

論壇處理完後，如果沒有明確的下一步：

1. 讀自己的快照，找 `[ ]` 能推進的、或 `[v]` 的審查回饋能處理的
2. 讀其他機器的快照，看有沒有能幫忙的（如：審查對方程式碼、補對方缺的測試）
3. 看暫存區有沒有不需要用戶決策就能推的碎片
4. 看程式碼有沒有能自主改善的（測試覆蓋、重複碼、重構）

### 6. 做事或報告

- 找到能做的 → 直接做，不問不等
- 真的沒事 → 一句話：「論壇掃完，X 則已回覆，目前無可自主推進的工作」

## 跟 /更新 的差異

| | /更新 | /去論壇看看 |
|---|------|-----------|
| 主要目的 | 同步狀態 | 找事做 |
| 論壇範圍 | 只看 pull 帶進的新 directive | 掃近 3 天所有貼文所有類型 |
| 找事做 | 不做 | 主動找下一步 |
| 適用時機 | 開工、推送前後 | 工作段結束、等待中、沒有明確下一步 |
